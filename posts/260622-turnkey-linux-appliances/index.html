TurnKey Linux：预装好应用栈的 Debian 服务器镜像

2026-06-22 #linux#debian

想快速起一台 WordPress / Nextcloud / GitLab，从零装 Debian → 配栈 → 加固 → 设备份要几十步。TurnKey Linux 把这些提前做完，打包成 100+ 个开箱即用镜像（appliance）：下载 → 开机 → 向导设密码，几分钟能用。

不是黑箱：底座就是一台纯 Debian

除少数自研组件，底座是官方仓库里未改动的 Debian 二进制包。所以全部 Debian 文档、apt 命令、排错经验都通用，维护起来跟一台普通 Debian 无异。当前主线 v18.x = Debian 12 (Bookworm)。

每个镜像自带 TurnKey Core：

  • di-live 安装器 + Python 配置控制台（网络 / 重启等基础操作）
  • Webmin 网页管理：HTTPS :12321
  • 每日自动装安全补丁
  • TKLBAM：一键加密备份（基于 Duplicity），文件 + 数据库 + 包状态一起存，可整机恢复 / 迁移

代价是继承了 Debian stable 的反直觉默认：软件版本冻结，apt upgrade 只打安全补丁、不升应用大版本。想要新版应用得等官方出新的大版本镜像，别指望 apt 升上去。

部署：VM / 容器 / 云怎么选

同一份 rootfs 被 TKLDev 导出成多种格式，按平台挑：

目标平台用哪个格式怎么上
VirtualBox / VMwareOVA（v14.0 起官方推荐）双击或导入向导，转换几分钟
Proxmox / KVM / Hyper-V通用 hybrid ISO挂载启动，跑 di-live 装到磁盘
Proxmox（更省事）LXC 模板webUI 加 TurnKey channel 直接拉
OpenStackqcow2上传镜像
云预建 AMI 等市场直接开

非 headless 构建（ISO/OVA）在首次开机跑向导设 root / 应用密码 + 网络；headless 构建（LXC/云/Docker）在首次登录初始化，随机 root 密码要去日志里读。之后应用走 https://<ip>/，系统管理走 Webmin https://<ip>:12321。

KVM/QEMU 不需要特殊格式，官方明确「直接用 ISO 就行」。

Docker 也能跑，但那是「胖容器」且官方已掉队

Docker 镜像本质就是「文件系统 + 元数据」，从那份 rootfs 再导一个天经地义——但和 Docker 的哲学天生不合：

  • 标准容器跑一个进程（PID 1 = 应用）；TurnKey 是一整套系统（init + cron + ssh + Webmin + 应用 + 数据库），只能做成 full-OS system container，行为像 LXC。用 /usr/sbin/start.sh 一次拉起所有服务再 drop 到 shell，退出 shell 服务就停，日常得靠 SSH。
  • TurnKey 用 systemd 管服务，Docker 默认不支持 → 报 System has not been booted with systemd as init system (PID 1)。官网自己标注 Docker 文档 “does not work as intended”，社区反馈构建已坏。

结论：想把 TurnKey 容器化，真正顺的是 LXC（Proxmox）不是 Docker；需要 Docker 生态时直接找单应用官方镜像（nextcloud、gitlab/gitlab-ce）。

TKLDev：整个库可复现的根子

「全开源、可复现」不是口号——TKL 用一个叫 TKLDev 的 appliance（一台专门用来 build 的机器）产出库里全部镜像。每个 appliance 不是手工装出来的一坨，而是一个小 git 仓库里的声明式配方：

组成作用
Makefile入口；声明继承链（你的 app → common → core），别的不用重写
plan/要装的 Debian 包清单（支持 cpp 式条件），等于 apt install 的声明版
conf.d/构建期跑的编号 shell 脚本，做配置（建库、设服务、加固）
overlay/文件系统覆盖层，按目标路径摆好、构建时拷进根文件系统
removelist要从镜像里删掉的文件

因为有继承，做一个自定义 appliance 往往只写几十行。构建：

cd core            # 或你自己的 product 仓库
make               # 从源码构建，产出 build/product.iso
make clean         # 清理重来

底层靠 fab（把装包 + 配置 + 打包串成可复现流水线）和 deck（基于 overlay 的分层快照，反复 build 时秒回滚）。要定制自己的加固 / 内部证书版镜像、又想像官方一样可复现可分发，就用它——思路和 Dockerfile / Nix / Packer 同源：把「怎么造一台机器」写成代码。
