Git LFS

2026-06-03 #git

Git 为文本设计：每个版本的每个文件都永久留在历史里。代码这样很好——但二进制文件（视频、PSD、字体、数据集）会被同样对待。一个 50MB 的视频改 5 次，.git 就多 ~250MB，而且每次 clone 都要把全部历史下载下来。Git 还无法 diff 二进制，只能存整份拷贝，省不掉。

原理：把大文件换成 pointer

LFS 在仓库里只存一个 ~130 字节的 pointer 文本文件，真身放到独立的 LFS server：

version https://git-lfs.github.com/spec/v1
oid sha256:4d7a214614ab2935c943f9e0ff69d22eadbb8f32b1258daaa5e2ca24d17e2393
size 52428800

git 历史里流动的是这段文本（可 diff、可压缩），git checkout 时 LFS 认出 pointer 自动把真身拉回工作区。对你来说体感和普通 git 一样。

brew install git-lfs
git lfs install                 # 装 git hooks，每台机一次

git lfs track "*.mp4"           # 写规则到 .gitattributes
git add .gitattributes          # 这个文件必须提交，否则队友不会触发 LFS
git add intro.mp4               # 正常用 git
git commit -m "add intro video"

.gitattributes 里那条规则长这样：

*.mp4 filter=lfs diff=lfs merge=lfs -text

filter=lfs 背后:clean / smudge

.gitattributes 里那个 filter=lfs 指向 ~/.gitconfig 中 git lfs install 写的一段定义，真正干「真身 ↔ pointer」转换的就是它：

[filter "lfs"]
  clean = git-lfs clean -- %f       # add/commit:真身 → pointer
  smudge = git-lfs smudge -- %f     # checkout:pointer → 真身 (下载)
  process = git-lfs filter-process  # 上两步的常驻进程批量版
  required = true
  • clean 在 git add 触发 (工作区 → 仓库),smudge 在 checkout 触发 (仓库 → 工作区),%f 是文件名占位符。
  • process 是性能优化：默认每个文件 fork 一次 clean/smudge,process 改用一个常驻进程通过协议批量处理;git 优先用它，clean/smudge 作后备。
  • required = true 是关键安全项：过滤器失败就报错中止,而非把原始数据放过去。

没装 git-lfs 的队友 clone 出来是 pointer 文本而不是真身，根因就在这——他们 ~/.gitconfig 里没这段 filter,smudge 根本不会跑。

track 必须在 add 之前

LFS 的”拦截”发生在 git add 那一刻：此时它查 .gitattributes，匹配就把文件换成 pointer。这是一次性的——文件一旦以普通方式进了历史，后补规则也救不回来。

# ❌ 错误顺序：视频以 50MB 真身进了历史
git add video.mp4 && git commit -m "add video"
git lfs track "*.mp4"           # 规则管不到「已经」发生的提交，仓库照样臃肿

补救要靠 git lfs migrate，它改写历史把旧文件从 git 挖进 LFS（同 git filter-repo 一样会变 commit hash，需要 force push + 团队协调）：

git lfs migrate import --include="*.mp4"

类比 Laravel：git lfs track 像写了条新 migration 规则，但它不 backfill 旧数据，老记录得专门跑一次 migrate import 才能改造。

file locking：因为二进制无法 merge

文本冲突 git 能三方合并，二进制不能——两份 .psd 没有”合并”一说。于是 Alice、Bob 同时改 logo.psd，后推的人只能覆盖或丢掉自己的工。

LFS lock 把 SVN/Perforce 的 check-out / check-in 模型搬回 git：编辑前先认领，期间该文件在别人机器上变只读，server 拒绝他们 push。

git lfs lock   images/logo.psd  # 认领，别人变只读
git lfs unlock images/logo.psd  # 干完释放
git lfs locks                   # 看谁锁了什么

前提是在 .gitattributes 给文件加 lockable，这个 flag 让它们默认只读、持锁才可写：

*.psd filter=lfs diff=lfs merge=lfs -text lockable

锁是协调工具不是强保护——git lfs unlock --force 能撬开别人的锁，团队仍要靠沟通。GitHub / GitLab / Bitbucket 都支持 locking。

See git help lfs.
