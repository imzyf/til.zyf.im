MySQL VARCHAR `=` 忽略尾部空格（PAD SPACE）

2020-11-28 updated 2026-05-20 #mysql

WHERE uid = 'abc' 命中 'abc '（带尾部空格）的行，UNIQUE 索引把 'abc' 和 'abc ' 视为重复 —— 这是 collation 的 PAD SPACE 属性决定的：比较时两侧尾部空格当作不存在。前导空格保留，尾部空格忽略，是 SQL 标准行为。

CREATE TABLE t (
  uid VARCHAR(10) COLLATE utf8mb4_unicode_ci PRIMARY KEY
);
INSERT INTO t VALUES ('abc '), (' abc');

SELECT * FROM t WHERE uid = 'abc';     -- 命中 'abc '
SELECT * FROM t WHERE uid = ' abc';    -- 命中 ' abc'
INSERT INTO t VALUES ('abc');           -- ERROR 1062 Duplicate
INSERT INTO t VALUES ('abc   ');        -- ERROR 1062 Duplicate

8.0 默认 collation 已经改为 NO PAD

跨版本最容易翻车的点：

collationPAD 属性默认版本
utf8mb4_general_ciPAD SPACE5.7
utf8mb4_unicode_ciPAD SPACE(从未默认)
utf8mb4_0900_ai_ciNO PAD8.0
utf8mb4_binNO PAD—

8.0 默认建表的 = 比较尾部空格敏感，上面这段 SQL 用 8.0 默认 collation 不会产生 duplicate —— 和 5.7 完全反着来。查当前默认的 PAD 属性：

SHOW COLLATION WHERE Charset = 'utf8mb4' AND `Default` = 'Yes';
-- 看 Pad_attribute 列

三条解决路径

根治 —— 换成 NO PAD collation：

ALTER TABLE t CONVERT TO CHARACTER SET utf8mb4 COLLATE utf8mb4_0900_ai_ci;

迁移前先 SELECT * FROM t WHERE uid <> TRIM(uid) 扫一遍尾部带空格的脏数据。

临时绕 —— 单次查询加 COLLATE 子句：

SELECT * FROM t WHERE uid = 'abc' COLLATE utf8mb4_bin;

比老写法 WHERE uid = BINARY 'abc' 现代，保留字符集语义。

改用 LIKE —— LIKE 是逐字符模式匹配，不走 collation 的 PAD 规则：

SELECT * FROM t WHERE uid LIKE 'abc';    -- 0 行
SELECT * FROM t WHERE uid LIKE 'abc ';   -- 命中 'abc '

= 和 LIKE 在尾部空格上的语义差异根因在此。

经典攻击：SQL Constraint Truncation

非严格 mode + PAD SPACE 联手能绕过登录鉴权：注册超长用户名 'admin 1'（11 字符，超出 VARCHAR(10)）→ 数据库截断成 'admin ' → 登录时 PAD SPACE 让 'admin' = 'admin ' 等值，匹配到攻击者那行，应用层一看 username = admin 就当真 admin 处理 → 越权。

SET sql_mode = '';   -- 必须关掉严格模式才能复现

CREATE TABLE users (
  username VARCHAR(10) COLLATE utf8mb4_unicode_ci PRIMARY KEY,
  password VARCHAR(64)
);
INSERT INTO users VALUES ('admin', 'real_pwd');
INSERT INTO users VALUES ('admin     1', 'pwn');  -- 实际写入 'admin     '

SELECT * FROM users WHERE username = 'admin' AND password = 'pwn';
-- 命中攻击者那行：(admin     , pwn)

5.7+ 默认 sql_mode 含 STRICT_TRANS_TABLES（超长直接报错）、8.0 默认 NO PAD（等值不成立），两道默认都封死了这条路径 —— 主要是老库审计时查 sql_mode 和表 collation。

CHAR 是另一套机制

CHAR(N) 存储时空格补齐到 N 位，读取时剥掉尾部空格（除非 SQL mode 打开 PAD_CHAR_TO_FULL_LENGTH）。CHAR 上「找不到尾部空格」是存储层的事，VARCHAR 上是 collation 层的事 —— 同样的症状，两套机制，两套修法。
