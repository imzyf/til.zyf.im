#!/bin/bash vs #!/usr/bin/env bash

2026-06-13 #shell

区别就一句：#!/bin/bash 写死路径只认 /bin/bash，#!/usr/bin/env bash 让 env 去 $PATH 里找第一个 bash。

为什么要绕 env：shebang 那行只接受绝对路径，自己不查 PATH。但 bash 在各机器路径不一（/bin、/usr/bin、/opt/homebrew/bin…），写死就不通用。env 几乎永远在固定的 /usr/bin/env,而它能查 PATH——于是用「固定位置的 env」把「查 PATH」这个能力接进 shebang。python3 / node 同理。

macOS 上这不是风格问题而是会报错。Apple 不用 GPLv3，自带的 /bin/bash 冻结在 3.2（2007 年）;Homebrew 装的新 bash（5.x）在 /opt/homebrew/bin/bash,排在 PATH 前面：

echo "/bin/bash → $(/bin/bash --version | head -1)"
echo "env bash  → $(env bash --version | head -1)"
# /bin/bash → GNU bash, version 3.2.57(1)-release
# env bash  → GNU bash, version 5.3.9(1)-release

同一台机器差两个大版本。脚本一旦用到 declare -A（关联数组）、mapfile 这类 bash 4+ 特性，#!/bin/bash 在 mac 上直接 command not found,必须 #!/usr/bin/env bash 才跑到 5.x。

代价是 env bash 跟着 PATH 走——PATH 被改就换了 bash。所以反过来，要绝对确定环境的场景（root cron、系统安装脚本）才用写死的 #!/bin/bash,不受 PATH 干扰。

还有第三种 #!/bin/sh——它跑的根本不一定是 bash。sh 是接口名(POSIX shell),实现各发行版不同:Debian / Ubuntu 是 dash,macOS 是一份独立的 bash 3.2 拷贝。而且 bash 以 sh 名字被调用时自动进入 POSIX 模式,关掉 [[ ]]、declare -A、数组等 bashism。于是高频翻车：脚本写 #!/bin/sh 却用了数组，在 mac(sh 是宽松的 bash) 侥幸能跑，push 到 Debian CI(sh 是 dash) 立刻 [[: not found——「本地能跑」不等于「符合 POSIX」,两边的 sh 是不同程序。

口诀：要 bash 特性 → #!/usr/bin/env bash(默认);锁死环境 (root cron / 安装脚本)→ #!/bin/bash;只用 POSIX、求最大可移植 → #!/bin/sh(配 shellcheck 抓 bashism)。

顺带 env 还能临时改环境跑命令，调「环境变量导致的 bug」常用：

env FOO=bar ./script.sh   # 只给这次运行设 FOO，不污染当前 shell
env -i bash               # -i 空环境启动
env -u PATH command       # -u 临时删掉某变量
