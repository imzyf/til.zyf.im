CORS：带凭证跨域时 Allow-Origin 不能用 *

2020-05-09 #network

带 cookie 的跨域请求（fetch(..., { credentials: 'include' })、xhr.withCredentials = true）触发浏览器的「凭证模式」，硬性四条死规矩：

  1. Access-Control-Allow-Origin 必须是具体 origin，禁止 *
  2. 必须返回 Access-Control-Allow-Credentials: true
  3. 必须加 Vary: Origin 防缓存错乱
  4. 预检 OPTIONS 也要重复以上三条

Firefox 报：

Credential is not supported if the CORS header ‘Access-Control-Allow-Origin’ is ’*’

Chrome 报：

The value of the ‘Access-Control-Allow-Origin’ header in the response must not be the wildcard ’*’ when the request’s credentials mode is ‘include’

为什么禁用 * + 凭证

如果允许 * 同时带 cookie：攻击者站点 https://hack.com 直接请求 https://api.example.com，自动带上你已登录的 cookie，服务端无法区分调用方，敏感数据全开。规范因此硬性禁止——这是浏览器侧的强制 CSRF 防线。

Access-Control-Allow-Credentials: true 是服务端的显式授权开关，控制两件事：跨域请求能不能带凭证发出、跨域响应能不能把 Set-Cookie 写入当前域。前端 credentials:include + 服务端 Allow-Credentials: true 是双向确认，少一边浏览器都不携带凭证、也不写入 cookie。

服务端正确返回

读取请求头 Origin，比对白名单，动态回写当前 origin（不是写死一个）：

Access-Control-Allow-Origin: https://app.example.com
Access-Control-Allow-Credentials: true
Vary: Origin

Nginx 关键片段：

if ($http_origin ~* ^https://(app|admin)\.example\.com$) {
    add_header Access-Control-Allow-Origin $http_origin always;
    add_header Access-Control-Allow-Credentials true always;
    add_header Vary Origin always;
}

Vary: Origin 不可省

CDN / 反代按 URL 缓存响应。第一次 https://a.com 请求命中后，缓存里那条响应携带 Access-Control-Allow-Origin: https://a.com。下一次 https://b.com 请求同一 URL，缓存直接返回——浏览器拿到不匹配的 Allow-Origin，跨域失败。

Vary: Origin 告诉缓存层：把 Origin 也算进 cache key，不同 origin 各存各的。预检请求建议扩展为：

Vary: Origin, Access-Control-Request-Method, Access-Control-Request-Headers

常见踩坑

  • 配了具体 Origin，忘了 Allow-Credentials: true → 能过 CORS，但 cookie 发不上、Set-Cookie 也存不下
  • 多域名白名单写死了一个 origin → 其余域名全跨域失败，必须动态回传当前请求的 Origin
  • 同一响应出现多个 Access-Control-Allow-Origin（框架 + Nginx 重复添加）→ 浏览器报错
  • 本地开发 * 一切正常，上线开 credentials 立刻崩
  • 预检 OPTIONS 偷懒返回 * → 主请求过不了
