Git commit message 规范：Conventional Commits + 工具链分工

2019-06-04 #git

约定本身就是 Conventional Commits 的 Angular 变体：

<type>(<scope>): <subject>

<body>

<footer>

type 限定 7 个：feat / fix / docs / style / refactor / test / chore。scope 标受影响模块（auth、api、db），项目内自定。subject 三条硬规则：≤ 50 字符、动词原形开头（change 不是 changed）、首字母小写、不加句号。

约定的价值不在”整洁” —— type 是给机器读的：feat 触发 minor bump、fix 触发 patch、BREAKING CHANGE 触发 major，CHANGELOG 也按 type 分组。光人手写不行，工具链分两段 —— 本地”写 + 验”、远端”发布”。

本地两件套：

阶段工具职责
写commitizen + @commitlint/cz-commitlintgit cz 交互式生成 message，prompt 直接读 commitlint 规则 —— 不用像 cz-customizable 那样维护两份配置
验commitlint + huskygit hook 拦截不合规 message —— commitizen 是引导、commitlint 是底线

装一次：

npm i -D commitizen @commitlint/cli @commitlint/config-conventional @commitlint/cz-commitlint
echo "module.exports = { extends: ['@commitlint/config-conventional'] };" > commitlint.config.js

package.json 把 commitizen 指到 commitlint adapter，规则就只剩 commitlint.config.js 一份：

{
  "scripts": { "commit": "git-cz" },
  "config": {
    "commitizen": { "path": "@commitlint/cz-commitlint" }
  }
}

husky v9 起 husky add / husky install 都没了，改 init + 手写 hook 文件：

npm i -D husky
npx husky init
echo 'npx --no -- commitlint --edit "$1"' > .husky/commit-msg

发布工具按场景四选一：

场景工具
GitHub 项目，PR-based release flowrelease-please
CI 全自动 npm publish + GitHub Releasesemantic-release
自管 tag、本地跑 releasecommit-and-tag-version
Monorepo，多包独立版本号changesets

口诀：

  • release-please —— Google 维护，把”待发布”维持成一条长期 PR，merge 即发版。GitHub Actions 装一次就不再管，新项目首选。
  • semantic-release —— push 到 main 立即发版，最激进；适合 SDK / 工具库这种发版没人审的场景。
  • commit-and-tag-version —— standard-version 2022-05-15 弃用后的社区 drop-in fork，旧配置原样可用；适合不想给 CI 开发布权限、自己 npm publish 的项目。
  • changesets —— monorepo 默认选项（pnpm / turborepo 生态主流）。每个 PR 自带一份 markdown changeset 文件，发版时按包聚合 —— 比 conventional commits 派系更适合”一次 PR 改多包但版本号要各自 bump”。
