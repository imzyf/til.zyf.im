MySQL LIMIT offset 深分页慢：deferred join 与 keyset

2016-09-18 #mysql

LIMIT 100000, 10 不是”取 10 行”，是”取 100010 行扔掉前 100000 行”。执行器没法靠索引跳过 N 行，必须物化前 100000 行（含回表读所有 SELECT 列）数着扔。真正代价是回表读了 10 万行宽数据，跟最终返回多少无关。

方案 1：deferred join

逃不开 OFFSET 时（“跳到第 N 页”按钮），让 OFFSET 只在窄覆盖索引上扫：

-- 慢：100010 次回表
SELECT * FROM t1 WHERE ftype = 1 ORDER BY id DESC LIMIT 100000, 10;

-- 快：100010 次只走索引，10 次回表
SELECT t1.* FROM t1
INNER JOIN (
  SELECT id FROM t1 WHERE ftype = 1 ORDER BY id DESC LIMIT 100000, 10
) t2 USING (id);

ftype 上有二级索引即可——InnoDB 二级索引叶子隐式追加主键，(ftype) 等价 (ftype, id)，反向扫满足 ORDER BY id DESC，SELECT id 已覆盖，全程不回表。EXPLAIN 看子查询 Extra 必须是 Using index。

方案 2：keyset pagination

放弃页码，用上一页最后一行的 id 当游标：

SELECT * FROM t1 WHERE ftype = 1 AND id < ? ORDER BY id DESC LIMIT 10;

每页 O(页大小) 而非 O(offset)，第 1000 页和第 1 页一样快。排序列非唯一（如 created_at 有重复）需复合游标避免跳行：WHERE (created_at, id) < (?, ?) ORDER BY created_at DESC, id DESC。

怎么选

场景用哪个
无限滚动 / “下一页” / API 游标keyset
”跳到第 N 页”硬需求（后台列表）deferred join
