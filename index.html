Workflow / Agent / Function Calling：三层位置和分水岭（LangGraph demo）

2026-05-19 #ai#llm#langgraph#python#workflow#agent

讨论 LLM 应用编排，workflow、agent、function calling 是三个绕不过去的基础词；外加一个编排框架（LangGraph / CrewAI / Pydantic AI / OpenAI Agents SDK 任选其一）就构成完整讨论。它们不是平行关系 —— 先分清各自位置，再谈选型，问题会简单很多。

三层位置图

[ 应用形态 ]  Workflow（图固定） /  Agent（图运行时才长出来）
                              ↓ 用
[ 编排框架 ]  LangGraph：节点 + 边 + 共享 state
                              ↓ 调
[ LLM 协议 ]  Function Calling：强约束 JSON schema
  • Function Calling：LLM API 的协议能力，给 LLM 一份 JSON Schema，逼它按 schema 输出。最底层、最稳定，OpenAI / Anthropic / Gemini 都有。
  • LangGraph：本文用的编排框架（同族还有 CrewAI / Pydantic AI / OpenAI Agents SDK 等），把流程画成「节点 + 边 + 共享 state」。工具层，本身不发明能力，只是把节点间的传递标准化。下面的位置图和分水岭判断对其他框架同样适用。
  • Workflow / Agent：两种应用形态，区别只有一个 —— 控制流在你的代码手里，还是在 LLM 手里。抽象层的差异见 Workflow vs Agent 的本质差异。

关键认知：Workflow 和 Agent 不是 LangGraph 提供的「模式开关」，是你用同一套 LangGraph + Function Calling 拼出来的两种不同形态。LangGraph 两种都能做，分水岭在你怎么写边。

Function Calling 是两种形态共用的底层

同一个 function calling API，两种用法决定你做出的是 workflow 还是 agent：

维度Workflow 用法Agent 用法
给 LLM 几个 function1 个（= 输出 schema）N 个（每个工具一个）
tool_choicerequired，必须调auto，LLM 自己挑
function 的角色数据合同可调用工具
下一步谁定Python 看返回字段LLM 看 observation 决定
LangChain 写法llm.with_structured_output(Schema)llm.bind_tools([t1, t2, ...])

底层 HTTP 请求几乎一样，差别只在给几个 tool、tool_choice 设成什么。

一句话：function calling 是协议层；workflow / agent 是你怎么用这个协议。

demo：分水岭具体在代码哪一行

配套两个文件，同一段业务逻辑（理财顾问会议笔记 → 抽字段 → 合规检查 → 分支生成文档）、两种控制流形态，对照看一目了然：

  • financial_workflow.py —— workflow 版，每个关键点标 [WORKFLOW vs AGENT] 告诉你这一行换成 agent 会变什么样
  • financial_agent.py —— 同业务的 agent 版，每个关键点标 [WORKFLOW → AGENT] 给出实际改造结果

workflow 文件里三段注释对应三个设计决策，正好覆盖上面三层：

  1. Function Calling 层 —— with_structured_output(Pydantic) 锁死 LLM 返回 schema，下游节点拿到的是类型化对象，不是裸文本要 json.loads 加 try/except
  2. LangGraph 层 —— state 是节点间的「信封」，节点 return dict 由 LangGraph 自动 merge，加节点 / 改顺序 / 插 human-in-the-loop 都不动函数签名
  3. 应用形态层（分水岭） —— add_conditional_edges 的路由参数：是你的 Python 函数 → workflow；是 tools_condition（读 LLM last message 判定）→ agent

对照两个文件最直接的一组改造：workflow 里的 route_by_risk Python 函数 → agent 里替换成 tools_condition；workflow 的 TypedDict 业务字段 state → agent 的 MessagesState；workflow 的 (state)→dict 节点 → agent 的 @tool 函数。那一瞬间失去的代码控制力，就是 agent 多出来的不确定性的具体形态，也是你为什么需要 trace / eval / guardrail 那一整套基础设施的根因。

跑起来

python -m venv .venv && source .venv/bin/activate
pip install -r requirements.txt
echo "OPENAI_API_KEY=sk-xxx" > .env

python financial_workflow.py   # workflow 版：固定流程，编译期就能 draw_ascii
python financial_agent.py      # agent 版：LLM 决定每一步，stream 打印实际 loop

样本对白故意写满合规雷点（绝对能保证 8% 复利 / 从来没跌过 / 未审批私募），所以默认走 P1 分支。把这些词删掉重跑，能看到 safe 分支怎么走。两个文件用同一段样本，方便对照输出。

LLM CLI proxy：rtk 是命令字典而非算法

rtk 是夹在 AI 编码助手和 shell 之间的 Rust CLI proxy，宣称帮 LLM 省 60-90% 的 token。它做的是 context compression —— LLM 想跑 git status，rtk 把噪音过滤后再喂回去。看着像”魔法压缩算法”，拆开后发现根本不是。

核心：30+ 条命令各一份手写 parser

rtk 没有通用压缩引擎。代码里是 git_status.rs、cargo_test.rs、docker_logs.rs …… 每条命令一段 Rust 代码，里面写死”这条命令该怎么 fork、哪些字段是噪音、压缩后长什么样”。

新加一种工具支持 = 提一个 PR 写一个新文件。changelog 看得清清楚楚：v0.38.0 加 GitLab glab、v0.40.0 加 Gradle，一条一条加。

工作流（以 rtk git status 为例）

LLM 决定跑 git status
  ↓
Claude Code PreToolUse hook 拦截 Bash 调用
  ↓ 改写成 `rtk git status`
rtk 进程：
  1. fork 真命令但用机器格式：git status --porcelain=v2 --branch
  2. parser 切字段：branch / ahead-behind / modified / untracked
  3. 紧凑模板渲染成单行
  ↓
返给 LLM，LLM 完全不知道有中间层

步骤 1 是关键 —— 拿给机器看的格式做解析，比逆向给人看的输出稳得多。

四种策略是工具箱，不是原理

官网讲的 filtering / grouping / truncation / dedup 是共享字符串处理函数（dedup_lines()、truncate_middle() 之类），被 per-command parser 当乐高块挑用：

  • git_status 用 filtering（删 progress 行）
  • lint 用 grouping（按 rule 聚合）
  • read --aggressive 用 truncation（删函数体留 signature）
  • docker_logs 用 dedup（同行折叠 + 计数）

每条命令”该用哪几种、按什么字段切”是手写的领域知识，不是算法推导。

为什么不选其他路

候选方案为什么 rtk 没用
gzip 通用压缩LLM 不解压，喂进去还是压缩字节
让 LLM 自己 summarize慢、贵、丢信息，summarize 本身吃 token
盲截前 N 行没语义，可能切掉关键 error 行
per-command parser<10ms、确定性、保关键信号 —— 代价是每条命令要单独写

推论

  1. 护城河是命令字典而非算法。任何人能写 git_status.rs，但要覆盖 90% 日常命令、跟着每个工具升级（git 改输出 / pytest 升级）才是真功夫。
  2. 工具支持决定收益。rtk hugo、rtk astro 不存在就是裸过。
  3. 永远滞后于命令本身。git 改了 porcelain 格式得等 rtk 跟上。
  4. PreToolUse hook 才是 UX 关键。靠 prompt 教 LLM “记得用 rtk” 不可靠；hook 把改写做在协议层，LLM 不感知。

工程上这是一类典型的 protocol-aware tooling：不在 LLM 那侧改东西，而是在 LLM 和环境的接缝处做改写。比 prompt 工程稳，比 fine-tuning 轻。

state vs status

两个词中文都译”状态”，工程命名混用会被 review 反复打回。先记一句口诀：

state is what you ARE; status is where you ARE in a process.

判准两步：

  1. 取值是有限几个枚举标签吗？→ status
  2. 是描述对象当下完整情况的一坨数据吗？→ state

status 通常是 state 里的一个字段，不是同级概念。这一层嵌套关系是判断命名时最重要的视角。

最干净的演示：Kubernetes Pod

spec:      # desired state（期望）
  containers: [...]

status:    # 实际情况
  phase: Running           # ← 流程节点标签
  conditions: [...]
  containerStatuses: [...]

spec + status 一起构成 Pod 的完整 state。status.phase 是 state 内部的一个枚举字段。命名 / 查 API 文档时，先判断你在描述哪一层。

反例：GitHub PR 用 state 不用 status

GET /repos/:owner/:repo/pulls/:number
→ {
    "state": "open" | "closed",
    "merged": true | false,
    ...
  }

open / closed 明明是枚举标签，按规则该叫 status，但 GitHub API 字段就叫 state，merged 另起一个 boolean。

为什么？GitHub 把 PR 当成”一个对象当前是什么”在表达，不是把它当流程节点。语义优先于规则，不要死记。

常见场景速查

场景用 status用 state
HTTP200 OK、response.status_codeTCP 连接：ESTABLISHED / TIME_WAIT
数据库字段orders.status = 'paid'—
前端组件antd <Steps status="finish" />React useState({ count, loading })
CI/CDGitHub Actions job status: completedGitHub PR state: open（反例）
即时通讯用户可设 status: 🍵 Tea breakWebSocket readyState: OPEN
游戏—存档（整个世界快照）

易混 trap：OAuth 的 state 参数

/authorize?response_type=code&state=abc123xyz

这里的 state 不是本文讨论的”对象状态”，是 OAuth 2.0 防 CSRF 的随机 nonce（RFC 6749 §4.1.1），授权服务器回调时原样带回，客户端校验是否匹配。

遇到这个词别按”完整数据快照”理解，它跟 state machine 一点关系没有。

一句话决策

  • 数据库字段、订单 / 任务 / 用户的流程节点 → status
  • React 组件内部数据、连接 / 进程的运行时快照 → state
  • 卡在中间 → 看现有 codebase 怎么命名，跟它走，比”正确”重要

markdownlint × prettier：format 与 lint 分工

markdown 在工具链上有两条独立线：prettier 管”长什么样”（format），markdownlint 管”对不对”（lint）。共存方案不是把两者并列开就完事 —— 默认配是稳定的，但有一处 CJK 用户必踩的雷。

职责分工

维度prettiermarkdownlint
干什么format：表格对齐、空行、缩进、引号统一lint：规则告警（重复标题、缺 lang、emphasis 当标题、列表风格）
配置.prettierrc.json + .prettierignore.markdownlint-cli2.jsonc（自带 ignores / gitignore: true）
globs不能写在配置里，命令行 / Makefile 传配置里 globs 字段直接写
auto-fix全部规则（修一次到位）行内类（MD009/012/047 等）能修；表格类（MD055/056/058/060）整组不 fixable
互相感知零零

最后两行是关键 —— 决定了下面所有”为什么这么配”的判断。

默认值高度吻合，平时不打架

~/.config/markdownlint/base.jsonc 的关键规则（MD003 atx、MD004 dash、MD007 indent 2、MD035 ---、MD046 fenced、MD048 backtick、MD009/012/047 清理尾空格 / 多空行 / 末换行）和 prettier 默认逐项对齐。保存 → prettier format → markdownlint 跑完只剩 MD036 / MD024 / MD040 这类语义告警，没有 format 层冲突。

真踩雷：MD060 aligned + CJK 表格

prettier 按 east-asian-width 算 CJK 列宽（视觉对齐），MD060 按字符数算（字节对齐），两者在 CJK 表格上永远收敛不了 —— 把 ~/.config/markdownlint/base.jsonc 的 "MD060" 设为 { "style": "consistent" } 或 false 让给 prettier。

局部翻案：ignore 注释

  • prettier：<!-- prettier-ignore --> 跳下一行，<!-- prettier-ignore-start --> ... <!-- prettier-ignore-end --> 跳整段（保留 raw 表格 / 空格 / 折行）。
  • markdownlint：按规则号开关 —— <!-- markdownlint-disable MD013 MD056 --> ... <!-- markdownlint-enable --> 跳整段，配 disable-line / disable-next-line 跳单行；不带规则号的 disable = 全部关，慎用。

Makefile 把两者串成一个入口

prettier 不接受配置里的 globs，只能在命令行和 markdownlint 的 globs 字段手动对齐；顺序固定 prettier 先、markdownlint 后 —— prettier 修完 format，markdownlint 剩下的就纯是语义告警，信噪比最高。

# Keep MD_GLOBS in sync with .markdownlint-cli2.jsonc#globs.
MD_GLOBS := 'content-til/**/*.md' '*.md'

lint:
  prettier --write $(MD_GLOBS)
  markdownlint-cli2 --fix

lint-check:
  prettier --check $(MD_GLOBS)
  markdownlint-cli2

经验法则

VS Code 两个扩展共存：esbenp.prettier-vscode 读 .prettierrc.json，DavidAnson.vscode-markdownlint 读 .markdownlint-cli2.jsonc，各自互不感知，配置不冲突。规则索引见 https://github.com/DavidAnson/markdownlint/blob/main/doc/Rules.md。

Agent vs Workflow：控制流在谁手里

把 LLM 包进应用的写法分两种：Workflow（你的代码画好流程图，LLM 只填某些节点）和 Agent（LLM 自己决定下一步调哪个工具、什么时候停）。两者长得很像 —— 都接输入、都调 LLM、都返结果。本质差异只有一条：下一步该干什么，是你的代码决定，还是 LLM 自己决定。这条线划在哪，决定了你在写哪种东西。

Anthropic 把两者合称 agentic systems，但坚持把 workflow 和 agent 分清楚 —— 它们的成本、延迟、可控性差着一个数量级。

控制流在谁手里

Workflow：控制流写死在你的代码里。

用户输入
  ↓
你的代码：调 LLM 提取意图
  ↓
你的代码：switch case 分发
  ↓
你的代码：调对应函数
  ↓
返回

LLM 只是流程里的一个 text-in / text-out 节点。流程图你画，模型填格子。

Agent：控制流在 LLM 手里。

用户目标
  ↓
LLM 决定：调哪个工具、给什么参数
  ↓
工具返回结果
  ↓
LLM 决定：要不要继续，继续就调下一个
  ↓
（循环，直到 LLM 说"完了"）

你的代码只负责执行 LLM 给的指令。每一步「下一步干嘛」由模型自己判断。

判准：把工作流画出来 —— 如果存在「LLM 决定要不要进下一步 / 进哪一步」的节点 = Agent；只有「LLM 帮我填空 / 分类 / 改写」的节点 = Workflow。

五件套：Model + Prompt + Tools + Memory + Orchestrator

Agent 的最小完备架构。少一个就跑不起来：

组件干嘛工程上最容易翻车的点
Model决策核心；要支持 Function Calling指令遵循 / 工具选择准确率
Promptsystem prompt + tool 描述 + few-shot工具描述写糊 → Agent 调错或调不出
Tools真正改变外部世界的接口（读写文件、调 API、跑 SQL）工具粒度 / 错误信息可读性
Memory工作记忆（context）+ 短期（会话）+ 长期（跨会话）塞太多炸 context，塞太少失忆
Orchestrator主循环：把上面四个串起来跑最大轮数 / 超时 / 错误重试 / human-in-the-loop

OpenAI 的 Practical guide 只列三件套（Model / Tools / Instructions），把 Memory 和 Orchestrator 视作框架默认配置。从底层实现角度五件套缺一不可；从 API 使用者角度三件套是你能感知到的部分 —— 别被简化版误导以为不存在剩下两个。

反直觉的踩坑权重：大部分 Agent 做不好不是模型不够强，是 tool 描述写得稀烂 + memory 策略乱设计。换模型贵又简单，写描述便宜又关键。

ReAct 循环：Agent 的心跳

Thought → Action → Observation 三拍循环，是当前几乎所有 Agent 框架的内核：

┌─────────────────────────────────────────┐
│  Thought      —— 我现在该干嘛？          │
│     ↓                                    │
│  Action       —— 调 tool X，参数 {...}   │
│     ↓                                    │
│  Observation  —— 工具返回了 Y            │
│     ↓                                    │
│  （回到 Thought，直到决定 Finish）        │
└─────────────────────────────────────────┘

每一轮 Orchestrator 把 [目标 + 工具列表 + 历史 Thought/Action/Observation] 整包扔给 Model，Model 输出下一个 Thought + Action 或最终答案。模型本身无状态，所有「记忆」都靠 Orchestrator 把历史拼回 prompt 里 —— 这也是为什么 context window 总不够用，也是为什么 Agent 的 token 成本随轮数指数增长。

Claude Code 跑一句「把 User 的 email 字段改成 unique」内部大致就是：

Thought: 先定位 User 在哪
Action:  Grep "class User"
Observation: src/User.php:12

Thought: 看现在 email 怎么定义的
Action:  Read src/User.php
Observation: <文件内容>

Thought: 字段约束在 migration 不在 model，找 migration
Action:  Grep "users" 在 migration 目录
Observation: 2025_01_users_table.php

Thought: 写新 migration 加 unique
Action:  Write add_unique_to_users_email.php
Observation: ok

Thought: 跑测试确认
Action:  Bash 跑测试
Observation: 失败 —— 现有数据里有 3 行重复 email

Thought: 那要先去重再加约束，改方案
Action:  Edit migration，先 DELETE 重复行再 ALTER
...

每步「下一步该干嘛」全是模型根据上一步 Observation 自己决定的 —— 没人写死 if grep_success then read。报错也是模型自己看到 Observation 后调整策略，不是你代码 catch 的。Agent 与 Workflow 的本质差异，在终端里就长这样。

什么时候真的需要 Agent

Anthropic 和 OpenAI 在这点上罕见地一致：能用 Workflow 跑通的就不要上 Agent。控制流的不确定性是 Agent 所有成本和风险的来源 —— 慢、贵、不可预测。

场景选什么
用户提一个明确问题，答完就完直接调 LLM API
流程固定，只是某几步要 LLM 抽信息 / 总结 / 分类Workflow + LLM 节点（LangGraph 这类）
任务能拆但步骤不确定，路径依赖中间结果Agent
需要试错 / 自我纠错 / 调多个外部系统Agent
单步深推理（数学 / 代码）Reasoning model 直调，不需要 Agent

判准：「下一步该做什么」如果你能提前画清楚流程图，就不是 Agent 该解决的问题。

Reasoning model 别再写 CoT prompt

老的 CoT prompt 技巧（Let's think step by step、few-shot CoT 示例）在 reasoning model 上不仅多余，还会降低性能 —— OpenAI 在 o1 的 prompting guide 里明确说了这一点。原因不在”模型更聪明所以不需要”，而在 CoT 和 reasoning model 根本不是一回事。

CoT 是提示技巧，reasoning 是模型能力

维度CoT promptingReasoning model
来源inference 阶段的 in-context learningtraining 阶段的 RL（RLVR 等）
触发Let's think step by step / few-shot 示例默认就在思考，无需触发
思考过程普通输出 token，跟答案混在一起专门的 reasoning tokens，与最终答案分离
行为线性向前推，错了一路错下去backtrack、self-verify、试多条路径
能力天花板在已能做的任务上提准确率解锁了之前完全做不了的题

一句话：CoT 是”哄”出来的推理，reasoning model 是”烧”进权重里的推理。

老技巧为什么反而有害

模型已经有自己 RL 训出来的推理风格。prompt 层再叠一套等于污染：

  • Let's think step by step —— 纯噪音 tokens，模型本来就在 think
  • Few-shot CoT 示例 —— OpenAI 实测显示降低 o1 性能，因为示例会污染模型自有推理风格。注意区分：few-shot 本身 OK（给格式 / 领域示例没问题），few-shot CoT 才有害
  • “请仔细分析”、“深入思考”、“一步步来” —— 全是冗余 tokens，思考预算被消耗在”读你的废话”上
  • 角色扮演（“你是数学专家”）—— Kim 2024 实测在 12 个 reasoning 数据集里 7 个降低准确率

角色扮演翻车的机制是 persona 注入了和题目无关的 domain prior —— Kim 2024 的例子：一道数学题套了 civil engineering 故事壳，模型就把自己往土木工程师方向调，忽略了本质是道数学题。AQuA 上 13.78% 题因 persona 从对变错，Coin Flip 18%。匹配的 persona 也救不回来：Zheng 2024 测 in-domain persona（律师做法律题、医生做医学题）是 “minimal effect size”，期望收益约等于 0，且自动选 persona 的策略几乎打不过随机 —— 不存在”选对 persona”这件事。经验法则：匹配 persona 期望收益 ≈ 0，不匹配 < 0，默认别加。

新范式：减噪音胜过加技巧

OpenAI 官方建议是 “Keep prompts simple and direct” 和 “Avoid chain-of-thought prompts”。

老 prompt 风格（CoT 时代）：

You are an expert programmer. Let's think step by step.
First, analyze the problem. Then, consider edge cases.
Finally, write the code...

Reasoning model 时代：

Refactor this function to handle null inputs.

Constraints:
- Keep the existing public API
- Must pass tests in tests/UserTest.php
- PHP 8.2, no new dependencies

Return: the modified function only, no explanation.

短、直接、富约束、明确输出。把模型当聪明同事，不是当学生。

真正重要的维度

prompting 没消失，重心从”教它怎么想”变成”告诉它想什么、想多深、想完干嘛”：

  • 任务定义清晰度 —— 模型按字面意思深度推理，需求模糊 = 在错的方向上 deep think 几千 tokens
  • 约束 / 边界条件 —— 显式写”必须用 PHP 8.2”、“不能改 schema”，否则它会自由发挥
  • 输出格式 —— reasoning 完了的最终答案才是你要的，显式要 JSON / 表格 / 代码块
  • Thinking budget —— Claude / Gemini / GPT-5 都有显式参数控制思考长度，简单任务别让它想 30 秒
  • 上下文质量 —— garbage in, deeply-reasoned garbage out

品类正在消失

2025 下半年起，“是不是 reasoning model” 不再是模型品类，而是运行模式：

  • Claude Sonnet 4.6 / Opus 4.7：extended thinking 开关 + budget tokens
  • GPT-5：reasoning_effort 参数
  • Gemini 2.5：thinking_config
  • DeepSeek V3.x / R1 已合并为统一模型

未来不会再有”non-reasoning model”这个品类，只会有 thinking budget = 0 的运行模式。但小快省的 Haiku 类、端侧模型、embedding 这些纯非 reasoning 阵营，在长尾场景依然不可替代。

Agent 写代码时代：review 比 prompt 值钱

Reasoning model 让 prompt 技巧失效已是共识（CoT、persona 反而拖性能）。更大的变化在另一边 —— agent 一天写出比 2024 一整周还多的代码，看起来都很 idiomatic。瓶颈从「模型懂不懂我」换成了「我看不看得出它错」。

把力气从 prompt 转到 context engineering

省下 prompt 技巧的力气，花在塞进 context 的内容质量：

  • 任务定义 —— 模糊需求 = 在错的方向上 deep think 几千 tokens
  • 约束 / 边界 —— 必须 PHP 8.2、不能改 schema、通过 tests/UserTest.php
  • 输出格式 —— 显式 JSON / 表格 / 仅函数体
  • Thinking budget —— Claude budget_tokens / GPT-5 reasoning_effort / Gemini thinking_config。Token 成本暴跌，但思考预算成了新成本中心；简单任务别开 high
  • 上下文质量 —— garbage in, deeply-reasoned garbage out

Review 是新护城河

Agent 写 PR，你 merge。Agent 不签字，你签字。看不出问题就是你的问题，没有「AI 写错了」这个免责条款。

Agent 最爱踩的雷（PHP / Laravel 视角）：

类型典型样子
N+1测试都过，本地慢 200ms，prod 上炸
静默 scope creep让它修一个 method，顺手「改进」旁边三个
幻觉 API用了 Laravel 里不存在但听起来很像的方法
删测试测试碍它的事，agent 默默改 / 删，PR diff 漏看
过度抽象一个 if 包成三层 strategy pattern
危险 migration加 NOT NULL 列在 5000 万行表上，本地秒过 prod 锁半小时
摘 middleware改 controller 时把 auth / validation 顺手干掉
放宽类型int 改 int|null 让测试过，业务不该 null

这些都不是 prompt 写得不好导致，是 review 没看出来。再好的 prompt 也防不住。

2026 的「好 review」不是逐行读完，是几个具体动作：

  • 读 diff 时先问「我没让它改的它改了吗」—— 比「它写的对吗」更重要
  • 对 agent 盲区有 muscle memory —— 它爱过度抽象、爱兜底 try/catch、爱删测试、爱写「为了向后兼容」的 if
  • 守 invariants 而不是守语法 —— 类型对 / 测试过 / lint 过都没用，问「我的业务规则还成立吗」
  • 怀疑「太像样」的代码 —— agent 最危险的输出是 idiomatic 到让你想直接 approve 的

一句话

2024 年「会写代码」值钱，2025 年「会用 AI 写代码」值钱，2026 年「会判断 AI 写的代码对不对」值钱。

写代码的肌肉记忆贬值最快；taste、code smell 嗅觉、review 时的怀疑反射，越来越值钱。

GitHub URL 隐藏参数：`.diff` / `?w=1` / `compare@{时间}` 工作流

GitHub 网页 URL 上有一组 2012 年就存在、UI 从没暴露过的参数。LLM 工作流让它们价值翻倍 —— curl 一个 .diff URL 直接拿到纯文本 diff 喂给 prompt，比让 LLM 自己抓网页快也准。下面所有链接都点得开（用 laravel/framework 演示）。

.diff / .patch —— 任何 PR / commit / compare 页 URL 加扩展名，返回纯文本：

curl -L https://github.com/laravel/framework/pull/10000.diff
curl -L https://github.com/laravel/framework/pull/10000.patch

.diff 体积最小，适合喂 LLM；.patch 多带 commit metadata（作者、邮箱、时间），适合 git am / cherry-pick。commit / compare URL 同样支持。

?w=1 —— 任何 diff URL 加上，忽略纯 whitespace 改动：

https://github.com/laravel/framework/pull/10000/files?w=1

把 indent 调整 / 换行符变化彻底从视野抹掉，只剩真改动。refactor / formatter 大重写后救命。

?ts=4 —— diff / blob 页改 tab 渲染宽度：

https://github.com/laravel/framework/blob/master/src/Illuminate/Foundation/Application.php?ts=4

数值任意（?ts=2 / ?ts=8），只改你眼里的呈现，远端文件不变。在 tab 缩进的项目（Go / Linux kernel）里效果最直观。

#L16-L25 —— 行号锚点 + range：

https://github.com/laravel/framework/blob/master/src/Illuminate/Foundation/Application.php#L1-L20

提 PR 评论、写文档、给同事丢 issue 时定位用，比”Application.php 大概第 20 行”准多了。

compare/<base>@{<时间>}...<head> —— compare 页接受时间锚点，不只是 SHA / branch：

https://github.com/laravel/framework/compare/master@{1.week.ago}…master

https://github.com/laravel/framework/compare/master@{2026-05-01}…master

语法和 git log @{...} 一致。release notes 起草、“上周改了啥”周报、incident 后回溯都用得上。

Bun 加入 Anthropic：6 天 AI 重写 Zig→Rust

Bun 在 2025-12-02 被 Anthropic 收购（Anthropic 史上第一笔），五个月后 Sumner 用 Claude Code 6 天把 96 万行 Zig 翻译成 Rust 并合入主分支。这两件事是同一根逻辑链：Claude Code 自己跑在 Bun 上，Anthropic 有最直接的动力修 Bun 的内存泄漏；而 AI Agent 重写大型 codebase 这件事也只有现在的 Anthropic 做得了。

时间线

2025-12-02  Anthropic 收购 Bun，Bun 继续 MIT 开源 + 团队继续主导
2026-05-04  Sumner commit "Phase-A 移植指南" + LIFETIMES.tsv
2026-05-10  claude/phase-a-port 分支出现，6 天 96 万行 Rust
            Linux x64 通过 99.8% 现有测试
2026-05-13  Bun v1.3.14 发布（"如果 Rust 合入这是最后一个 Zig 版"）
2026-05-14  Rust 版本合入 main，二进制减小 3-8 MB

为什么收购：Claude Code 自己被 Bun 烧伤

Claude Code 以单文件 Bun executable 分发给百万级用户，长会话内存能飙到 14-23 GB —— 直接踩 Bun 的内存泄漏。Anthropic 当时 Claude Code ARR 已 $1B（2026/2 升到 $2.5B），Bun 一坏 Claude Code 就坏，直接收购比合作更快。

收购动机里 AI Agent 工作流也是关键：agent 写完代码要立刻装包、跑测试、产可执行 —— Bun 的”单二进制 + 冷启动 8-31ms + 装包比 npm 快 25×“对应每一步。

为什么重写：Zig 的 no-AI 政策 + 内存安全

四个理由叠加：

  1. Zig 项目禁止 AI 生成的 issue / PR / 评论 —— Bun 团队 fork 的 Zig 编译器（debug 编译快 4×）永远没法回上游。对一家 AI 公司是政治死结
  2. 内存泄漏与 unsafe —— Sumner 原话 “I am so tired of worrying about memory leaks and crashes”
  3. Zig 还是 0.16，破坏性变更频繁 —— 做生产基础设施长期不稳
  4. Rust 生态成熟 —— AI 训练数据、crate 库都更充足

注意：性能不是动机。Bun 比 Node 快不来自 Zig 比 V8 快，来自架构（JSC 替 V8、原生 HTTP 不走 libuv、内置打包器）。换 Rust 不动这些架构，性能不会变。

重写方法：600 行 Rosetta Stone + AI Agent

工作流：

docs/zig-to-rust.md      ← ~600 行类型 / 习惯用法映射文档
LIFETIMES.tsv            ← 每个指针字段的 ownership 分类
Phase A: 翻 Rust 代码（不要求编译）
Phase B: crate-by-crate 让它编译过

故意不用 Tokio / Rayon，保持低层执行模型，做”直译”非重设计。Sumner 在 X 上：

“this is already the status quo; we haven’t been typing code ourselves for many months now.”

“我预期 OSS 反过来走：禁止人类贡献代码。讨论和优先级仍由人定，但写代码、提 PR、回应 review，全 LLM 做。“

争议

uv (Astral, Rust)：350,000 行 / 73 个 unsafe
Bun Rust 移植：    681,000 行 / 13,000+ 个 unsafe

第二个 PR（删 60 万行 Zig）被 GitHub 自动 flag 为 “AI slop” 关闭。社区核心反对意见：unsafe 块里的 hallucination 风险无法人工 review；commit 大小让 review 不可能；“vibecoded disaster”。

对生产用户的实际影响

  • 性能 —— 几乎没差，架构没变
  • 稳定性 —— 内存泄漏修复（Claude Code 类长会话受益最大）
  • 二进制 —— 小 3-8 MB
  • 兼容性 —— v1.3.14 后跑同一份 npm 应用应当无感

真正值得追踪的事

Bun 是史上第一个把核心代码所有权交给 AI Agent 维护的生产级开源项目。这种重写姿势能成立，前提是 Sumner 作为 BDFL（Benevolent Dictator For Life —— 开源项目”仁慈终身独裁者”，源自 Guido van Rossum / Python，决策不走委员会投票）一个人就能拍板”6 天试一下，搞不定就扔”。基金会治理的项目（如 Node.js / Python 后 Steering Council 时代）光投票就要几个月。

两种结局：

  • 稳住 → “几人 reviewer + 一群 agent + 一个 BDFL” 模式被复制到很多基础设施项目
  • 崩了 → AI 生成代码不可信论的最大佐证

无论哪种，未来 1-2 年值得追踪。

git filter-repo 清除历史敏感文件

仓库不小心把 .env / database.yml 推到远程后，第一动作是轮换密钥，假定文件已被抓取 —— 改写历史只能挡住”还没看到的人”，已 clone / fork / 爬虫的副本拿不回来。这一步先做完，再谈 git filter-repo：它是第二步，不是第一步。

为什么不是 filter-branch

git filter-branch 官方已弃用。每个 commit fork 一个 shell 子进程，大仓库跑数小时；默认保留 refs/original/ 和 reflog，旧对象很容易”清理后又复活”。git filter-repo 是 Python 单进程流式管道，快 10–100 倍，且默认要求 fresh clone（拒绝在带遗留 ref 的工作目录上跑）。

brew install git-filter-repo

原理：重写而非删除

Git commit 的 SHA-1 由 tree + parent + author + message 哈希得出，不可变。“删一个文件”等价于生成一份新历史替代旧历史。

旧仓库 ── git fast-export ──▶ 事件流 ── 过滤器 ──▶ git fast-import ──▶ 新仓库

git fast-export 把仓库历史序列化成纯文本指令流（blob、commit、tag）；filter-repo 在内存里按规则改写事件流；git fast-import 把它灌成新对象库。

关键后果：一个 commit 的 tree 变了 → 它的 SHA 变 → 子 commit 的 parent 指向新 SHA → 子 SHA 也变 → 一路传染到 HEAD。所以 force push 是必须的 —— 远程的 commit 链被一条 SHA 完全不同的链替换。

--invert-paths：默认是白名单

非显然：filter-repo 默认把 --path 当保留列表，不是删除列表。

# 默认：只保留 src/，其他全删（拆分 monorepo 用法）
git filter-repo --path src/

# --invert-paths：删除 src/，其他保留（清理用法）
git filter-repo --path src/ --invert-paths

设计初衷之一是从 monorepo 抽子项目 —— “列出我要的那部分”比”列出我不要的”自然。清理敏感信息正好相反：99% 留 + 1% 删，必须加 --invert-paths。漏写就把整个仓库删干净只剩那个泄露文件，反应过来时已经覆盖远程了。

git filter-repo --path .env --invert-paths
git filter-repo --path-glob '*.pem' --invert-paths
git filter-repo --path-regex '^secrets/' --invert-paths

--mirror：不配它等于没删

普通 git clone + git push --force --all 看似清理，实际只覆盖 branches。远程上 branch 之外的 ref 没动，仍指向旧 commit：

  • Tags：旧 tag 仍含敏感 commit
  • Notes、其他自定义 ref

只要远程任何一个 ref 还能可达旧 commit，那个 blob 就不会被 GC 回收 —— 攻击者拿到 SHA 就能 git cat-file -p <sha> 读出来。删了等于没删。

git clone --mirror 拉一份裸仓库（.git 结构、无工作区）+ 所有 ref；git push --mirror 把本地 refs 一比一推到远程，远程多出来的 ref 会被删 —— “让远程完全等于本地”。这才是真清理：

git clone --mirror git@github.com:org/repo.git
cd repo.git
git filter-repo --path config/database.yml --invert-paths
git push --force --mirror

--mirror 只解决服务端的 ref 问题。任何一个老的本地 clone 仍能通过 reflog 和 remote-tracking refs (refs/remotes/origin/*) cat-file 出旧 blob —— 这正是 filter-repo 默认要求 fresh clone 的原因，也是清理后必须通知所有协作者删本地仓库重新 clone 的原因。

清理后必做

步骤说明
轮换密钥见开篇，第一优先
通知协作者所有人重新 clone，旧本地分支基于已消失的 commit 链
联系 GitHub SupportPR refs (refs/pull/<n>/head) 由 GitHub 服务端维护，push --mirror 覆盖不了；fork 拷贝也要走 Support 清
加 .gitignore防止再次提交
装 pre-commit 钩子gitleaks protect --staged 自动拦截
brew install gitleaks
gitleaks detect             # 扫整个历史，确认清理生效
gitleaks protect --staged   # pre-commit 钩子里跑

JS 生态分层：规范 / 引擎 / 运行时 / 类型

JavaScript 生态混乱的根源是没分清三件事：写规范的、实现引擎的、做宿主运行时的。再叠加 TypeScript 这个外挂类型层，就是常被混为一谈的”JS”。把四层拆开，整个服务端 JS 世界就不再奇怪。

规范层：ECMAScript（ECMA-262）

由 TC39 维护（Ecma International 第 39 号技术委员会），每年 6 月发新版。规范定义语法、内置对象、执行模型，但不定义 DOM、fetch、localStorage、fs、http —— 这些归别处管：

ECMAScript          ← TC39          (语法 / Promise / Map / async)
DOM / fetch         ← WHATWG        (浏览器 API)
HTTP / TLS          ← IETF          (RFC)
fs / http / process ← Node OpenJS   (Node 自加，不是 Web 标准)

写 JS 99% 在调宿主 API（fetch / document / fs），其中只有少数真正是 ECMAScript。fetch 不是 ECMAScript —— 它是 WHATWG 的，Node / Bun “借”过来内置。

JS 这语言官方名是 ECMAScript 不是 JavaScript，因为 JavaScript 是 Sun（现 Oracle）的商标。Brendan Eich 自嘲过 “ECMAScript 听起来像皮肤病”，但这名字沿用了 30 年。

引擎层：V8 / JavaScriptCore / SpiderMonkey

不是”标准统一就只该有一个实现” —— 优化目标真的不同：

V8（Chrome / Edge / Node）JSC（Safari / Bun）SpiderMonkey（Firefox）
出身场景标签页常驻几小时页面打开就关平衡
优化目标长跑峰值启动 + 内存平衡
JIT 策略激进多层克制（保持低层便宜）保守
许可证BSD 3-ClauseLGPL 2.1MPL 2.0
治理Google 主导WebKit 联盟（Apple + Igalia + Sony）Mozilla 基金会

iOS 上所有浏览器都是 JSC —— Chrome / Edge / Firefox iOS 版只是 Safari 套了 UI 壳，被 App Store 强制用 WebKit。这条 2024 年起 EU DMA 法案才开始松动，但实际进展缓慢。

Edge 自 2020 年起 = V8 —— Microsoft 当年用的 Chakra 死了，Edge 重写为 Chromium。所以”V8 阵营”实际控制桌面 80%+。

许可证差异决定生态形状：V8 的 BSD 才让 Node / Deno / Electron / Cloudflare Workers 这些闭源 / 商业派生敢用；JSC 的 LGPL 允许嵌入式调用但不允许闭源修改引擎本身。这是 Bun 选 JSC 的非技术理由之一。

运行时层：Node / Bun / Deno / 浏览器

V8 只是会跑 JS，不会读文件、不会发 HTTP、不会监听端口。Node 把 V8 嵌进 C++ 程序，再补上：

Node = V8 (引擎)
     + libuv (事件循环 + 跨平台异步 I/O：Linux epoll / macOS kqueue / Windows IOCP)
     + 标准库 (fs / http / crypto / stream)
     + N-API (C++ addon 接口)
     + 模块系统 (CommonJS / ESM)

同一个 V8 嵌进不同宿主就是不同运行时：

引擎事件循环默认权限
ChromeV8Blink 自家沙盒受限
NodeV8libuv完整 OS 访问
DenoV8Tokio (Rust)默认无权限，按需授权
BunJSC自家（io_uring / kqueue 直调）完整 OS 访问
Cloudflare WorkersV8V8 isolate + 自家 scheduler沙盒

这就是 Node 不能”换引擎追上 Bun”的根因：N-API、所有 native addon、调试器协议、profile 工具全围绕 V8 设计。换 V8 等于重写 Node。Bun 走的是从零写所有胶水的路，代价是 95-98% npm 兼容率（剩下的 2-5% 用了 V8/libuv 内部细节的边角库）。

类型层：TypeScript

TS = JS 严格超集 + 类型注解。类型在编译后完全消失 —— .ts → .js 后类型一个字符都没了，运行时拿不到任何”这是 User 不是 string”的信息。这是为什么 instanceof 对 interface 没用：

// 写
function greet(user: User): string { return `Hi, ${user.name}` }

// 编译后实际跑的
function greet(user) { return `Hi, ${user.name}` }

TS 现在多渠道编译：tsc（官方，慢）/ esbuild（Go）/ swc（Rust）/ bun（直接跑无中间步骤）/ Node 22+ --experimental-strip-types。Microsoft 2025 年用 Go 重写了 tsc，叫 TypeScript Native，快 10×。

TC39 有 Type Annotations 提案让 JS 引擎直接忽略类型注解（注释化），TS 代码不用编译就能跑。Stage 1 卡了 4 年还没动 —— 浏览器引擎不愿为”不执行的注解”做语法解析。短期内不会落地。

写代码时叫 TypeScript、调试时叫 JavaScript、查规范时叫 ECMAScript、谈性能时谈 V8 / JSC、谈部署时谈 Node / Bun / Workers —— 同一件事的五个面，分清面才不被术语骗。

PHP libvips 写文字：Pango 接管 layout，不用算 bbox

GD 那一套 baseline / bbox / 减偏移 在 libvips 里不存在。Image::text() 走 Pango 做 layout —— 你给一个 width 框 + align 方向，结果 image 直接拿到 $txt->width / $txt->height，没有 baseline 这回事。

Pango 是什么

GNOME 项目下的文字 layout 库。渲染一段文字本来要分两步，libvips 把第一步外包给了它：

阶段做什么谁来做
Layout选字体、断行、字距、对齐、Bidi（阿拉伯/希伯来从右到左）、CJK 换行规则、emoji shapingPango
Rasterize把矢量字形画成像素FreeType / Cairo

GD 的 imagettftext 只做第二步 —— 调用方必须自己算宽高、自己拆行、自己处理对齐，所以才有 imagettfbbox 8 元素数组、baseline 减偏移那堆脏活。libvips 直接传 width: 800, align: 'high'，Pango 帮你折行 + 对齐，结果 image 的 width / height 就是排版后的真实尺寸 —— baseline 这个概念在调用方完全不出现。

额外 bonus：开 rgba: true 之后可以传 Pango markup，单段里混排粗体 / 颜色 / 字号：

$txt = Image::text(
    '<span foreground="red" weight="bold">警告</span> 内容超过限制',
    ['font' => 'Sans 20', 'rgba' => true]
);

GD 想做这个得渲染 N 次再拼。

实际用法：右下角水印

同样的场景，libvips 写法：

use Jcupitt\Vips\Image;

$bg  = Image::newFromFile('photo.jpg');
$txt = Image::text('© 2026', [
    'font'  => 'Sans Bold 24',
    'width' => $bg->width - 40,   // 框内自动 word-wrap
    'align' => 'high',
    'rgba'  => true,              // 不开是灰度 mask，没法合到彩图
]);

$bg->composite($txt, 'over', [
    'x' => 20,
    'y' => $bg->height - $txt->height - 20,
])->writeToFile('out.png');

GD 同场景要：imagettfbbox 拿 8 元素 → $bbox[2] - $bbox[0] 算宽 → $bg_w - $width - 20 算 x → 还要担心 baseline 跟图像 y 轴方向相反。libvips 一步：align: high。

Image::text() 关键 options（官方文档）：

Option作用默认
fontFontconfig 名，如 'Sans Bold 24' / 'Noto Serif CJK SC 18'—
fontfile加载非系统字体的 .ttf 路径，需配合 font 用名字引用—
width换行宽度（像素），超过按 word boundary 折行不换行
align'low' / 'centre' / 'high' —— 多行的水平对齐—
justify两端对齐（拉伸词间空格），boolfalse
spacing行距（像素）字体自然行距
dpi渲染 DPI，影响实际像素大小72
rgba出彩色 / emoji / Pango markup，不开是 8-bit 灰度 maskfalse

部署成本是 libvips 在 PHP 圈用得不如 GD 普及的主因 —— ext-gd 几乎随 PHP 自动装好，libvips 要：

# Debian/Ubuntu
apt install libvips42 libvips-dev

# macOS
brew install vips

# PHP 端二选一
pecl install vips                                # 传统 C extension
# 或开 FFI（PHP 7.4+），用 jcupitt/php-vips 包，无需编译扩展
composer require jcupitt/vips

容器里加一行 apt install libvips42 就够，但这一行常被忘。装上之后水印 / 缩略图 / 多行排版量大时，libvips 的内存占用和吞吐都比 GD 强一个数量级 —— 它是流水线架构，不像 GD 把整张图按解压尺寸驻留内存。

cloc / tokei / scc：codebase LOC 统计的三代工具

数代码行数（LOC by language）这件事有三代工具，每代换语言、换取舍：

cloctokeiscc
实现 / 起年Perl / 2006Rust / 2016Go / 2018
Linux kernel 用时~64s~1.4s~0.8s
默认并行❌（要 --processes=N）✓✓
安装brew install clocbrew install tokeibrew install scc
独有能力--diff 对比两个 commit；读 .zip / .tarJSON / YAML 输出复杂度 / COCOMO / 重复检测

scc 那三个名词各自的意思：

  • 复杂度：数代码里 if / for / && 这类分支关键字，估算 cyclomatic complexity。
  • COCOMO：1981 年 Barry Boehm 提出的成本模型，给 KLOC 估”工业平均要多少人月、多少钱”。
  • 重复检测：按文件 hash 判定，相同内容只算一次（monorepo / vendor 目录里常见，cloc / tokei 会重复计入）。

对同一个目录（这个 blog 的 til/src/）跑默认命令，差异自己跳出来：

$ cloc til/src/
Language           files          blank        comment           code
Astro                 10            117             62           1340
TypeScript             7             14             49             89
SUM:                  17            131            111           1429

$ tokei til/src/
 Language         Files        Lines         Code     Comments       Blanks
 Astro               10         1519         1344           59          116
 TypeScript           7          152           97           44           11
 Total               17         1671         1441          103          127

$ scc til/src/
Language        Files       Lines    Blanks  Comments       Code Complexity
Astro              10       1,519        99        49      1,371         55
TypeScript          7         152        11        44         97          2
Total              17       1,671       110        93      1,468         57
Estimated Cost to Develop (organic) $40,425
Estimated Schedule Effort (organic) 4.06 months
Estimated People Required (organic) 0.88

cloc 没有 Lines 总数列（自己加 blank+comment+code）；tokei 加了 Lines；scc 多 Complexity 列 + 一段 COCOMO 估算块。同样的 17 个文件，三个工具给出三个 Code 总数：1429 / 1441 / 1468——blank 和 comment 怎么判，每家略不同。

cloc 的 --diff 是至今没被替代的核心——code review 想量化”这个 PR 改了多少 code 行”时，tokei / scc 都做不到：

$ cloc --git --diff HEAD~5 HEAD
Language        files          blank        comment           code
Markdown
 same              0              0              2            924
 modified          2              0              0              9
 added             0              0              0              4
 removed           0              0              0             19
Bourne Shell
 same              0              0              5             14
 modified          1              0              0              1
 added             0              0              0              1
 removed           0              0              0              0
SUM:
 same              0              0              7            938
 modified          3              0              0             10
 added             0              0              0              5
 removed           0              0              0             19

按语言分组，每组拆 same / modified / added / removed；想细到文件级加 --by-file。比 git diff --stat 信噪比高一截。

怎么选： 默认装 scc（最快 + 复杂度）；CI 输出要 JSON 接管线用 tokei；要 git diff 的 LOC 变化才掏 cloc。

PHP 安装工具的几条路径

PHP 安装按”装什么 + 在哪装”分四条路径，每条各自的工具：

纯 PHP 代码（复制即用）

  • PEAR — PHP Extension and Application Repository。最早的类库分发，用前要 include。已被 Composer 取代，基本退场。
  • Composer — PHP 依赖管理工具，灵感来自 npm / Bundler。composer.json 声明依赖，composer install 拉齐，事实标准。

C 扩展，裸机环境（编译 + 改 php.ini）

  • PECL — PHP Extension Community Library。PEAR 的子项目，分发 redis、apcu 这类底层扩展。已被官方标记 deprecated。
  • PIE — The PHP Installer for Extensions。PHP 基金会官方接管，取代 PECL，命令风格抄 Composer。

C 扩展，Docker 容器场景

  • docker-php-ext-install — 官方 PHP Docker 镜像内置脚本，容器里装 C 扩展的事实标准。
  • install-php-extensions — 上面的友好封装，自动处理依赖、编译选项、共享库。Dockerfile 里写 PHP 几乎人人用它。

PHAR 形式的 CLI 开发工具（PHPUnit、phpstan、psalm 这种）

  • PHIVE — PHAR Installation and Verification Environment。装 PHAR 工具并校验签名，避免污染 composer.json。

七个命令并排看：

# 纯 PHP 代码
pear install Mail                        # 老路子，已退场
composer require monolog/monolog         # 现代

# C 扩展（裸机）
pecl install apcu                        # 老路子（deprecated）
pie install apcu/apcu                    # 现代

# C 扩展（Docker），以装 gd 为例——需要系统库 + configure
apt-get install -y libfreetype-dev libjpeg-dev libpng-dev \
  && docker-php-ext-configure gd --with-freetype --with-jpeg \
  && docker-php-ext-install gd            # 官方：3 步联动
install-php-extensions gd                 # 友好封装：1 行搞定

# PHAR CLI 工具
phive install phpunit

记忆：

  • PHP 代码 → Composer
  • C 扩展，裸机 → PIE（PECL 已 deprecated）
  • C 扩展，Docker → install-php-extensions
  • PHAR CLI 工具 → PHIVE
