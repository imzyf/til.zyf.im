Helm 不是 controller：模板引擎 + release 状态机

2026-07-06 #devops#kubernetes

Helm 常被说成「Kubernetes 的 apt / npm」——对，但这个类比藏了最关键的一点：Helm 是一次性的（one-shot）。helm install 把模板渲染成 YAML、提交给 apiserver、把这次结果记下来，然后就退出了。之后没有任何进程持续盯着 drift。这是理解 Helm 一切行为的锚点：它不是 controller、不是 GitOps，跟 Operator / ArgoCD 的持续 reconcile 是两码事。记住它，下面每个「反直觉」都顺理成章。

release 状态存在集群里，不在你本地

helm install 之后，状态被写进 release namespace 的一个 Secret，名字是 sh.helm.release.v1.<name>.v<revision>。每 upgrade 一次多一个 revision Secret，helm history / helm rollback 就是读它们。内容是 json.Marshal → gzip → base64（Helm 3 默认 driver 从 Helm 2 的 ConfigMap 换成了 Secret）。想亲眼看：

kubectl get secret sh.helm.release.v1.my-pg.v3 -o jsonpath='{.data.release}' \
  | base64 -d | base64 -d | gunzip   # 两次 base64：先解 k8s Secret 层，再解 Helm 自己那层

两个非显然后果：① 这个 Secret 受 etcd ~1MB 上限约束，chart 塞太多东西（比如把大文件写进 ConfigMap 模板）会在 install 时炸；② 谁手滑删了它，Helm 就失忆——helm list 里 release 消失，但集群里的 Deployment 还在裸奔。换 driver 用 HELM_DRIVER=secret|configmap|sql（sql 是 beta，只支持 PostgreSQL）。

upgrade 是三方合并，不是覆盖

Helm 3 的 helm upgrade 用 3-way strategic merge patch：同时看旧 manifest、集群里的 live state、新 manifest。Helm 2 只做两方，kubectl edit 的带外改动直接被无视。差别很实：

  • 你手动把 replicas 从 3 调到 0。Helm 2 rollback 觉得 manifest 没变，保持 0；Helm 3 发现「记录 3 / 现状 0 / 目标 3」不一致，拉回 3。
  • 服务网格给 Pod 注入了 sidecar。Helm 2 upgrade 会把它抹掉，Helm 3 保留 sidecar，同时改你要改的 image。

Tiller 之死：一个被删掉的攻击面

Helm 2 有个集群内服务端 Tiller，默认以 cluster-admin 跑且无认证——任何能连上它的 Pod 都能操纵整个集群，是经典提权入口。Helm 3 直接砍掉，变成纯客户端，权限完全走你 kubeconfig 的 RBAC。所以网上任何出现 helm init / Tiller 的教程都是 Helm 2 时代的，别抄。

lookup 在 dry-run 会静默返回空

有个常见模式：用 lookup 查 Secret 是否已存在，没有才生成随机密码，好让密码在 upgrade 间保持稳定。坑在于——helm template 和 helm install --dry-run（客户端）根本不连 apiserver，lookup 永远返回空。于是 diff 工具（helm-diff、ArgoCD）每次 render 都看到「密码变了」，天天误报。要真查集群得用 --dry-run=server。

默认不等就绪；数组不合并只替换

两个静默 footgun：

  • helm install 返回 ≠ 应用 ready，它只保证 manifest 提交成功。CI 里不加 --wait 的绿是假绿；再配 --atomic 让失败自动 rollback。
  • values 合并对 map 是深合并，对数组是整体替换（不是 append）。想往默认 list 里加一项，结果整个 list 被你那一项顶掉。多个 -f 后者盖前者，--set 优先级最高。

chart 存哪：OCI 已取代 index.yaml

老式 repo 靠一个 index.yaml 索引（helm repo add）。3.8.0 起 OCI 支持转正 GA 且默认开启，底层是 ORAS——chart 像容器镜像一样 helm push / pull 到镜像仓库，没有 index.yaml。一个隐藏坑：OCI tag 不允许 +，Helm 会把 SemVer build metadata 里的 + 悄悄转成 _。

一句话记牢：Helm 只在你敲命令的那一刻工作——渲染、提交、记账，然后什么都不管了。想要持续 reconcile 抹平 drift，那是 ArgoCD / Flux 的活。

DN42 —— 改 git repo 申请 ASN 的 BGP 沙盒

DN42 是一张「长得跟真实互联网一模一样」的 overlay 网络——同一套 BGP / whois / DNS，但发号不用求 RIR：fork 一个 git 仓库、改几个文件、提 PR，脚本校验通过就合并，你就正式拥有了自己的 ASN 和地址段。核心价值是安全网：你在里面扮演一家 mini-ISP 练 BGP，配错了也不会有大运营商打电话来骂。

最反直觉的一点：它用 WireGuard 这类 VPN 隧道把成员互联，乍看像 Tor 那类 darknet，目标却相反——不做匿名，专做「去 NAT 的真实路由练习」。成员之间要的是能直连、能互相暴露服务，而不是藏起来。

注册是 git PR，不是找管理员

真实互联网上申请 ASN / IP 要走 APNIC 那套官僚流程；DN42 把它压成一次 PR——这是它整个设计的题眼。你要建的对象跟真实 whois 一一对应：

  • mntner（维护者，带签名 auth）、person（nic-hdl 以 -DN42 结尾）
  • aut-num（你的 ASN）、inetnum / inet6num（你的地址段）
  • route / route6——仿照真实互联网的 RPKI/ROA，没有它别人会因 ROA 校验失败而不收你的路由

坑在校验：一个自动脚本会在合并前检查语法，格式错一个字符就打回，拖慢管理员审核。所以是「严格按语法填」而不是「大概填填」。

号段为什么是私有的

资源DN42 用的段备注
ASN4242420000–4242429999（RFC 6996 私有段）新人建议在 4242420000–4242423999 里挑
IPv4172.20.0.0/14默认分 /27（32 个地址）够用
IPv6ULA fd00::/8默认分 /48

全用私有号段是刻意的：这样 DN42 的路由永远不会泄进公网 BGP 表、也不会跟公网 IP 撞车。代价是这些地址空间跟 ChaosVPN（172.31.0.0/16）、NeoNetwork 等兄弟网络重叠——DN42 只敢用 172.20.0.0/14 而非整个 172.16/12，就是给这些互联网络留位置，它们之间是打通的。

AS / ASN / BGP 选路的基础概念这里不重复，见同目录 CN2 那篇（那篇讲的是运营商的真实 ASN 就是线路名，这篇是你自己申请一个私有 ASN 来玩，正好两面）。

加入四步

  1. 注册：按上面建好对象，提 PR，合并即「持号」。
  2. 找 peer：没有中心服务器，必须连另一个成员。新手直接连 Kioubit——它连接数最多、中心性最高，且有全自动 peering 向导（填 ASN + WireGuard 公钥 + 隧道 IP，几分钟出结果），所以大半个网都直连它。也可用官方 Peer Finder 挑低延迟的。
  3. 建隧道：主流是 WireGuard（好配、够安全），也支持 OpenVPN / GRE / IPsec。
  4. 跑 BGP：装 BIRD 或 FRR，跟 peer 起 eBGP 会话，把自己的段 announce 出去。

门槛不高：一台 24/7 在线的 Linux VPS + 会配 WireGuard 和 BGP 就够。

dn42.us 不是官方入口

官方主站是 dn42.eu（wiki + registry 说明都在这）。dn42.us 是成员 xuu 自建的一个节点门户，挂着他自己的 wiki / paste / git 镜像等服务——它本身就是「在 DN42 里对外提供服务」的活样板，别把它当成注册入口。

TurnKey Linux：预装好应用栈的 Debian 服务器镜像

想快速起一台 WordPress / Nextcloud / GitLab，从零装 Debian → 配栈 → 加固 → 设备份要几十步。TurnKey Linux 把这些提前做完，打包成 100+ 个开箱即用镜像（appliance）：下载 → 开机 → 向导设密码，几分钟能用。

不是黑箱：底座就是一台纯 Debian

除少数自研组件，底座是官方仓库里未改动的 Debian 二进制包。所以全部 Debian 文档、apt 命令、排错经验都通用，维护起来跟一台普通 Debian 无异。当前主线 v18.x = Debian 12 (Bookworm)。

每个镜像自带 TurnKey Core：

  • di-live 安装器 + Python 配置控制台（网络 / 重启等基础操作）
  • Webmin 网页管理：HTTPS :12321
  • 每日自动装安全补丁
  • TKLBAM：一键加密备份（基于 Duplicity），文件 + 数据库 + 包状态一起存，可整机恢复 / 迁移

代价是继承了 Debian stable 的反直觉默认：软件版本冻结，apt upgrade 只打安全补丁、不升应用大版本。想要新版应用得等官方出新的大版本镜像，别指望 apt 升上去。

部署：VM / 容器 / 云怎么选

同一份 rootfs 被 TKLDev 导出成多种格式，按平台挑：

目标平台用哪个格式怎么上
VirtualBox / VMwareOVA（v14.0 起官方推荐）双击或导入向导，转换几分钟
Proxmox / KVM / Hyper-V通用 hybrid ISO挂载启动，跑 di-live 装到磁盘
Proxmox（更省事）LXC 模板webUI 加 TurnKey channel 直接拉
OpenStackqcow2上传镜像
云预建 AMI 等市场直接开

非 headless 构建（ISO/OVA）在首次开机跑向导设 root / 应用密码 + 网络；headless 构建（LXC/云/Docker）在首次登录初始化，随机 root 密码要去日志里读。之后应用走 https://<ip>/，系统管理走 Webmin https://<ip>:12321。

KVM/QEMU 不需要特殊格式，官方明确「直接用 ISO 就行」。

Docker 也能跑，但那是「胖容器」且官方已掉队

Docker 镜像本质就是「文件系统 + 元数据」，从那份 rootfs 再导一个天经地义——但和 Docker 的哲学天生不合：

  • 标准容器跑一个进程（PID 1 = 应用）；TurnKey 是一整套系统（init + cron + ssh + Webmin + 应用 + 数据库），只能做成 full-OS system container，行为像 LXC。用 /usr/sbin/start.sh 一次拉起所有服务再 drop 到 shell，退出 shell 服务就停，日常得靠 SSH。
  • TurnKey 用 systemd 管服务，Docker 默认不支持 → 报 System has not been booted with systemd as init system (PID 1)。官网自己标注 Docker 文档 “does not work as intended”，社区反馈构建已坏。

结论：想把 TurnKey 容器化，真正顺的是 LXC（Proxmox）不是 Docker；需要 Docker 生态时直接找单应用官方镜像（nextcloud、gitlab/gitlab-ce）。

TKLDev：整个库可复现的根子

「全开源、可复现」不是口号——TKL 用一个叫 TKLDev 的 appliance（一台专门用来 build 的机器）产出库里全部镜像。每个 appliance 不是手工装出来的一坨，而是一个小 git 仓库里的声明式配方：

组成作用
Makefile入口；声明继承链（你的 app → common → core），别的不用重写
plan/要装的 Debian 包清单（支持 cpp 式条件），等于 apt install 的声明版
conf.d/构建期跑的编号 shell 脚本，做配置（建库、设服务、加固）
overlay/文件系统覆盖层，按目标路径摆好、构建时拷进根文件系统
removelist要从镜像里删掉的文件

因为有继承，做一个自定义 appliance 往往只写几十行。构建：

cd core            # 或你自己的 product 仓库
make               # 从源码构建，产出 build/product.iso
make clean         # 清理重来

底层靠 fab（把装包 + 配置 + 打包串成可复现流水线）和 deck（基于 overlay 的分层快照，反复 build 时秒回滚）。要定制自己的加固 / 内部证书版镜像、又想像官方一样可复现可分发，就用它——思路和 Dockerfile / Nix / Packer 同源：把「怎么造一台机器」写成代码。

源与文档：https://www.turnkeylinux.org/ · TKLDev 文档

spec-kit vs. superpowers vs. gstack:三种 coding agent 工作流框架

三个都是套在 coding agent(主要是 Claude Code) 外面的工作流框架，但把它们放一起比不是因为像，而是因为它们赌的「正确性从哪来」是三个不同的答案:spec-kit 赌规格,superpowers 赌流程纪律,gstack 赌多视角评审。选型不是比谁命令多，是认领你当前最缺哪一种。

框架出身赌注载体
spec-kitGitHub 官方规格是可执行的唯一真理specify CLI + /speckit.* 命令
superpowersJesse Vincent (obra)流程纪律逼出质量自动触发的 skills 插件
gstackGarry Tan (YC)多角色评审补单视角盲区23 个纯 Markdown slash command

三者都是 MIT，都不止支持 Claude Code(各自能跨 7-30+ 个 agent),都不锁厂商——这些都不是差异点，差异全在下面。

spec-kit:规格是唯一真理来源

GitHub 官方的 Spec-Driven Development。核心主张一句话：规格是源码，代码是规格的编译产物。你维护的是 spec.md 不是 .py;改需求改 spec，再 implement。产物 spec.md → plan.md → tasks.md 被当成 agent 无关的规格层——换 agent 不换规格。

uv tool install specify-cli --from git+https://github.com/github/spec-kit.git@vX.Y.Z
specify init my-project --integration copilot

然后在 agent 里按顺序走命令，每步落一个文件：

/speckit.constitution   # 立项目宪法 (不可动的原则)
/speckit.specify        # 写「做什么 / 为什么」→ spec.md
/speckit.clarify        # 澄清模糊点 (可选，建议 plan 前跑)
/speckit.plan           # 定技术栈方案 → plan.md
/speckit.tasks          # 拆可执行任务 → tasks.md
/speckit.implement      # 执行全部任务

最适合：需求先行、跨会话、多文件的功能开发，团队要一层标准化的规格。

superpowers:用流程纪律逼出质量

Jesse Vincent(GitHub obra) 的 skills 框架。它不给你命令表，给一套自动触发的 skills——agent 一旦识别到该用某 skill 就必须用，把「偷懒跳步」这条路堵死。内核是真正的红绿 TDD + 每个任务派新鲜子代理，两段式审查 (先查规格符合、再查代码质量)。

Claude Code 装法一行：

/plugin install superpowers@claude-plugins-official

强制流水线 (skills 依次自动登场):

brainstorming               # 苏格拉底式追问，细化想法 → design doc
using-git-worktrees         # 设计批准后开隔离工作区
writing-plans               # 拆成 2-5 分钟的小任务，带确切路径 + 完整代码
subagent-driven-development # 每任务派新鲜子代理，两段式 review
test-driven-development     # RED → GREEN → REFACTOR，先写失败测试
requesting-code-review      # 对着 plan 审，critical 问题卡住不放行

最适合：在意测试纪律和执行严谨度，想防 agent「不写测试 / 静默改需求」。

gstack:把一个 agent 变成一支团队

Garry Tan(YC 总裁) 的设置：23 个专家 + 8 个 power tool,全是纯 Markdown slash command，无专有 runtime。它赌结构化 prompt 而非自建工具才是对的抽象层——每个角色只约束 agent 这一阶段盯一件事，靠视角切换补单一视角的盲区。

装法 (全局装 + ./setup):

git clone --single-branch --depth 1 https://github.com/garrytan/gstack.git ~/.claude/skills/gstack \
  && cd ~/.claude/skills/gstack && ./setup

一个 Sprint 里角色依次登场 (Think → Plan → Build → Review → Test → Ship → Reflect):

/office-hours       # YC 办公时间：这产品该不该做
/plan-ceo-review    # CEO:重想问题本身
/plan-eng-review    # 工程经理：锁架构 + 测试策略
/plan-design-review # 设计师：评设计维度，抓 AI slop
/review             # Staff 工程师：挖生产级 bug
/qa                 # QA:真开浏览器点流程，修完再验
/cso                # 安全官:OWASP + STRIDE 审计
/ship               # 发布工程师：测试 + PR + 覆盖率

外加 /codex(拉 OpenAI Codex 做跨模型二次 review)、/careful /freeze /guard(危险命令 / 目录锁护栏) 这类 power tool。最适合：一个人当全栈团队，要多视角决策 + 质量 / 安全 / 发布把关。

组合与选型

三者不互斥，常叠着用：gstack 多角色起草 → superpowers 头脑风暴 + TDD 执行 → spec-kit 固化成 spec/plan/tasks。但别为叠而叠，先认领你当前最缺的那一环。

口诀：缺规格标准化用 spec-kit，缺执行纪律用 superpowers，缺多视角把关用 gstack。

gitignore 里斜杠的位置决定匹配范围

gitignore 的匹配范围全看斜杠在哪，记两条：

  • 结尾 / = 只匹配目录;不带 = 文件、目录通吃。
  • 开头 / 中间有 / = 锚定到仓库根目录;pattern 里完全没有斜杠 = 任意层级都匹配。

结尾斜杠：对文件加 / 是静默 bug

.idea 是目录，.idea/ 和 .idea 效果一样——但 / 更精确，显式表达「我忽略的是目录」,哪天真有个叫 .idea 的文件不会被误伤。JetBrains / GitHub 模板都用 .idea/。

反过来对文件就是坑：.DS_Store 是 macOS 生成的文件,写成 .DS_Store/ 会匹配不到任何东西(没有叫这名的目录),规则静默失效、不报错。目标是文件时绝不能加 /。

.idea/         # 目录 → 带斜杠更严谨
.DS_Store      # 文件 → 加斜杠就失效

开头斜杠:monorepo 的陷阱

node_modules/(无开头斜杠) 匹配任意深度，等价于 **/node_modules/——所以 **/ 在这里是多余的。/node_modules/ 只锚定根目录那一个，子包里的会漏掉。

node_modules/                ← /node_modules/ 只命中这个
packages/app/node_modules/   ← 只有无斜杠写法 / **/ 才命中

monorepo(pnpm / turborepo 子包) 一定用能匹配嵌套的 node_modules/,别写 /node_modules/。

**/ 真正有用是「中间匹配多层」:docs/**/tmp(docs 下任意深度的 tmp)、src/**/*.log。

.DS_Store、.idea/ 这类跟项目无关的编辑器 / 系统噪音，放全局 ignore(git config --global core.excludesFile) 一次，所有仓库都干净。

nerd

装 Nerd Fonts 时顺手查的词。最反常识的一点先记住：

nerd 不是任何东西的首字母缩写，是一个被「造」出来的词。

很多人以为它是 acronym（比如流传的 “Northern Electric Research and Development”），全是 folk etymology（民间附会）。真实出处是 1950 年 Dr. Seuss（苏斯博士）童书 If I Ran the Zoo 里他自造的一个生物名，次年被学生俚语借去当「书呆子」用，固定至今。

词义随圈子翻转

  • 原意带贬义：书呆子、不擅社交、土气。
  • 科技圈现在基本是 褒义甚至自豪的自称：某领域的狂热钻研者。a computer nerd / a math nerd。说 “I’m a nerd” 是骄傲，不是骂人。

nerdfonts.com 的命名就是这层自嘲幽默——「只有我们这些爱折腾终端的 nerd 才在意的字体」。

近义词别混

词含义色彩
nerd技术 / 学术钻研者中性 → 褒义
geek极客，痴迷某领域偏褒义
dork笨拙、傻气偏贬义
dweeb呆子贬义

口诀：nerd / geek 在科技圈是夸，dork / dweeb 是损人。

CN2 GIA / 9929 / CMIN2：三网回国线路分层

回国线路好坏，本质是数据包进国内后走「精品网」还是「普通骨干网」——三网都能按 普通线 / 优质线 / 顶级线 三档理解（VPS、跨境专线圈的俗称）。看线的前提：回程比去程重要——下载占 90%+，回程决定体感。

AS 号就是线路名的真身

AS（Autonomous System，自治系统）= 由同一机构统一管理、对外采用统一路由策略的一组网络。互联网是「网络的网络」，每个大网络就是一个 AS。

  • ASN（AS 号）是网络层的身份证，全球唯一；BGP 选路靠它：IP 像门牌号定位主机，ASN 像区域号——BGP 先决定送到哪个 AS，再在 AS 内细分到具体地址。
  • ASN 原本 16 位（1–65535），号段不够用扩成了 32 位——移动的 AS58453 / AS58807 就是 32 位号，所以位数明显比电信 AS4809 长。

所谓 CN2 GIA、9929、CMIN2，本质就是 AS4809、AS9929、AS58807——运营商不同精品骨干网的「身份证号」。

精品网 vs 普通骨干：差在哪

普通骨干（电信 163、联通 169、移动 CMNET）是共享、超售的大网，国际带宽受限，晚高峰拥堵、抖动大。精品网是独立轻负载链路，绕开大量公共交换点（IX），一跳入境、一跳落地，所以延迟低、丢包少、高峰稳——代价是带宽小、价格贵。这就是 CN2 GIA 被当成回国「黄金标准」的根因。

优质线是半程精品：只在出国段走精品网，国内段仍回普通骨干（电信 CN2 GT 典型——国内段走 163，出境才汇入 CN2），比纯普通线好，但高峰国内段仍会堵；顶级线全程精品、不降级，才是天花板。

运营商普通线（普通骨干）优质线（半程精品）顶级线（全程精品）

商家宣传只说「CN2」是泛称，要追问是 GT 还是 GIA——两者差价常有 2–3 倍。

xxd —— 字节级显微镜，看出 BOM / CRLF / 不可见字符

文本工具看不出问题 (多了空格？BOM?tab 还是空格？CRLF 还是 LF?) 时，用 xxd 把真实字节 dump 出来。它随 Vim 分发，Linux / mac 基本自带。

xxd file.sh
# 00000000: 2321 2f62 696e 2f62 6173 680a       #!/bin/bash.
#           └偏移   └每字节 hex(两字节一组)     └ASCII(不可打印显示 .)

三列：偏移地址 / 每字节十六进制 / 对应 ASCII(不可打印字符用 . 占位)。行尾那个 0a 就是 LF;如果是 0d0a 就是 CRLF——这是肉眼分不出、xxd 一眼看穿的典型场景。查 UTF-8 BOM 同理：head -c3 file | xxd 出现 efbbbf 即有 BOM。

常用 flag:

xxd -l 16 file       # 只看前 16 字节(-l = length)
xxd -s 100 file      # 从第 100 字节开始(-s = seek)
echo -n hi | xxd     # 看一段文本的字节: 6869

-r(reverse) 能把 hex dump 还原成二进制，于是 xxd 成了简易二进制编辑器——dump、改 hex、再写回：

xxd file > dump.hex      # 导出
# 编辑 dump.hex 改某个字节
xxd -r dump.hex > file   # 还原写回

同类：od -c、hexdump -C。

#!/bin/bash vs #!/usr/bin/env bash

区别就一句：#!/bin/bash 写死路径只认 /bin/bash，#!/usr/bin/env bash 让 env 去 $PATH 里找第一个 bash。

为什么要绕 env：shebang 那行只接受绝对路径，自己不查 PATH。但 bash 在各机器路径不一（/bin、/usr/bin、/opt/homebrew/bin…），写死就不通用。env 几乎永远在固定的 /usr/bin/env,而它能查 PATH——于是用「固定位置的 env」把「查 PATH」这个能力接进 shebang。python3 / node 同理。

macOS 上这不是风格问题而是会报错。Apple 不用 GPLv3，自带的 /bin/bash 冻结在 3.2（2007 年）;Homebrew 装的新 bash（5.x）在 /opt/homebrew/bin/bash,排在 PATH 前面：

echo "/bin/bash → $(/bin/bash --version | head -1)"
echo "env bash  → $(env bash --version | head -1)"
# /bin/bash → GNU bash, version 3.2.57(1)-release
# env bash  → GNU bash, version 5.3.9(1)-release

同一台机器差两个大版本。脚本一旦用到 declare -A（关联数组）、mapfile 这类 bash 4+ 特性，#!/bin/bash 在 mac 上直接 command not found,必须 #!/usr/bin/env bash 才跑到 5.x。

代价是 env bash 跟着 PATH 走——PATH 被改就换了 bash。所以反过来，要绝对确定环境的场景（root cron、系统安装脚本）才用写死的 #!/bin/bash,不受 PATH 干扰。

还有第三种 #!/bin/sh——它跑的根本不一定是 bash。sh 是接口名(POSIX shell),实现各发行版不同:Debian / Ubuntu 是 dash,macOS 是一份独立的 bash 3.2 拷贝。而且 bash 以 sh 名字被调用时自动进入 POSIX 模式,关掉 [[ ]]、declare -A、数组等 bashism。于是高频翻车：脚本写 #!/bin/sh 却用了数组，在 mac(sh 是宽松的 bash) 侥幸能跑，push 到 Debian CI(sh 是 dash) 立刻 [[: not found——「本地能跑」不等于「符合 POSIX」,两边的 sh 是不同程序。

口诀：要 bash 特性 → #!/usr/bin/env bash(默认);锁死环境 (root cron / 安装脚本)→ #!/bin/bash;只用 POSIX、求最大可移植 → #!/bin/sh(配 shellcheck 抓 bashism)。

顺带 env 还能临时改环境跑命令，调「环境变量导致的 bug」常用：

env FOO=bar ./script.sh   # 只给这次运行设 FOO，不污染当前 shell
env -i bash               # -i 空环境启动
env -u PATH command       # -u 临时删掉某变量

2026-06-12 #shell

UTF-8 的 BOM 没用，还会静默破坏脚本和 JSON

BOM(Byte Order Mark) 是文件最开头一段肉眼不可见的字节，本职是标记编码的字节序:UTF-16 用 FE FF(大端)/ FF FE(小端) 告诉读取方高位在前还是低位在前。

关键非显然点：UTF-8 根本没有字节序问题——它的码元就是单字节，不存在「谁在前」。所以 UTF-8 的 BOM(EF BB BF) 纯属多余，标准也不推荐。但 Windows 系工具 (记事本、旧 Excel、部分 .NET) 会自动加，于是这三个看不见的字节就成了静默 bug 源：

  • shebang 失效:第一行变成 <BOM>#!/bin/bash,内核读头两字节看到的是 BOM 不是 #!,脚本不被当可执行 (参见 shebang 是 magic number)。
  • JSON 解析报错:json.loads / 严格 parser 把开头 BOM 当非法字符。
  • PHP “headers already sent”:<?php 前那三个 BOM 字节被当作输出先发出去，之后 header() / setcookie() 全废。

都因为字节在位置 0、又看不见，排查起来格外费时。检测和清除：

head -c3 file | xxd                  # efbbbf 就是有 UTF-8 BOM
sed -i '1s/^\xEF\xBB\xBF//' file      # 删掉开头 BOM
file file                            # 输出含 "with BOM" 也能认

一句话:Unix 工具链默认无 BOM,UTF-8 存盘一律选「无 BOM」那个选项。

shebang 是内核认的 magic number，不是 shell 语法

#! 不是 bash 的语法，而是文件开头的 magic number:字节 0x23 0x21。当你 ./script 执行一个文件，是 Linux 内核的 execve(2) 读头两个字节，看到 #! 就把后面那行当解释器路径，改成执行「解释器 + 脚本路径」。名字 = #(sharp) + !(bang)。

这带来几个非显然的硬约束：

  • 必须是文件最开头——#! 前面不能有空行、空格或 BOM，否则内核认不出，文件会被当普通数据。
  • 解释器后只允许一个参数。#!/usr/bin/env python -u 里 python -u 在多数系统被当作单个参数传给 env,于是去找名叫 "python -u" 的程序、失败。要传 flag 得在脚本内另想办法，别堆在 shebang 上。
  • 没有 shebang 的可执行文件，内核 execve 直接 ENOEXEC,是调用它的 shell 兜底用 /bin/sh 跑——所以「没写 shebang 也能跑」靠的是 shell 不是内核。
head -c2 script.sh | xxd   # 00000000: 2321  →  #!

选 #!/bin/bash 还是 #!/usr/bin/env bash,见 shebang vs env bash。

plutil vs PlistBuddy

macOS 自带两个改 .plist 的命令行工具，功能重叠却谁也没淘汰谁——因为出身不同：

  • /usr/bin/plutil：给开发者的文件级工具，在 PATH 里，Apple 主推、文档全。强项是校验和编码转换。
  • /usr/libexec/PlistBuddy：早期系统安装器留下的键值级编辑器。libexec = 内部 helper，故意不进 PATH，文档几乎没有，但改深层嵌套最顺手。

（plist = Apple 的配置文件格式，相当于苹果版 JSON/YAML；一个格式两种编码：XML 可读 / binary 紧凑。）

改一个嵌套值：两种风格

PlistBuddy 用类 DSL 命令，: 表层级，Set 不用声明类型：

/usr/libexec/PlistBuddy -c "Print :Server:Port" demo.plist     # 8080
/usr/libexec/PlistBuddy -c "Set :Server:Port 9090" demo.plist

plutil 用选项参数，. 表层级，且必须声明类型：

plutil -replace Server.Port -integer 7070 demo.plist
plutil -extract Server.Port raw demo.plist                     # 7070

两个都能读写任意路径的 plist，binary 编码也能直接改（不用先转 XML）。

只有 plutil 能做的：校验 + 转码

这是 PlistBuddy 没有的能力，CI / 构建脚本里最常用：

plutil -lint demo.plist               # demo.plist: OK   ← 语法体检
plutil -convert binary1 demo.plist    # XML → binary
plutil -convert xml1 demo.plist       # binary → XML（方便 diff）
plutil -p demo.plist                  # 人类可读打印，binary 也认

plutil -p 把 binary 也解出来给你看：

{
  "CFBundleVersion" => "1.0"
  "Server" => {
    "Port" => 9090
  }
}

怎么选

  • 体检 / 转码 / 在 CI 里跑 → plutil，-lint 和 -convert 是它独有
  • 改具体值、尤其深层嵌套 → PlistBuddy，Set/Add/Delete :a:b 比 -replace a.b -type 少打字、不用记类型

顺带：Apple 已预告未来 defaults 命令只认 preferences domain，按文件路径乱改 plist 的老用法要迁到这两个工具——所以它们短期不会消失。

See man plutil / /usr/libexec/PlistBuddy -h.

git interpret-trailers：读写 commit message trailer 的官方工具

trailer 是 commit message 末尾形如邮件头的 Key: Value 行——Signed-off-by / Reviewed-by / Co-authored-by。读写交给 git interpret-trailers,别 grep/awk:按字段位置取值会 silently drop(同 按作者统计 的定界符坑),只有 git 知道哪些行真是 trailer。

判定很严，同时满足才算 trailer、否则当正文:连续的含 : 行 + 前面隔一个空行 + 位于消息末尾(或 format-patch 的 --- 前)。所以 body 中段的 URL: ...、或紧贴正文没空行的，都不算——这是手写 parser 最易错处。

读用 --parse,只吐已有 trailer、忽略配置：

git log --format=%B -n1 HEAD | git interpret-trailers --parse

写用 --trailer,默认追加在 trailer 块末;与最后一条完全相同则不重复加：

git interpret-trailers --in-place \
  --trailer 'Reviewed-by: Bob <bob@example.com>' msg.txt

用 trailer.* 配置把常用 key 定义成简写。别名必须是完整 key 的前缀(大小写不敏感)——所以 ack 能映射 Acked-by,却没法拿 sign 映射 Reviewed-by。再配 .cmd 让别名自带取值逻辑，--trailer ack 不给值也能生成：

git config trailer.ack.key 'Acked-by'
git config trailer.ack.cmd 'echo "$(git config user.name) <$(git config user.email)>"'
git interpret-trailers --trailer ack   # 自动补 Acked-by: 你的名字 <邮箱>

.ifexists / .ifmissing 再单独控这个别名的重复与缺失行为。

最常见是 commit-msg 钩子里自动补 trailer($1 是 git 传入的消息文件，--trim-empty 丢掉空值行):

#!/bin/sh
git interpret-trailers --trim-empty \
  --trailer "git-version: $(git describe)" "$1" > "$1.new" && mv "$1.new" "$1"

git help interpret-trailers 看全。

reviewdog：把 linter 输出贴成 PR 行内评论

reviewdog 是一个通用适配器：吃任意 linter 的 stdout，按 errorformat 解析，然后把结果作为 PR 行内评论贴回 GitHub / GitLab。它自己不做静态分析，只解决”已有的 lint 结果怎么进 review”这一段。

价值全在它默认只报 diff 改动的行。整仓库跑 lint 有两个老问题：老代码一堆历史 warning 淹没真正要看的；报错埋在 CI 日志里 reviewer 根本不点开。reviewdog 用 -filter-mode 把结果过滤到本次改动，再走 review API 顶到代码行上，两个问题一起消。

# PHPStan checkstyle 输出 → 只评论本次 PR 改动行
vendor/bin/phpstan analyse --error-format=checkstyle | \
  reviewdog -f=checkstyle -reporter=github-pr-review

-f 选输入格式：内置一批（-f=golint / -f=checkstyle…，reviewdog -list 看全），没有就用 -efm='%f:%l:%c: %m' 自定义 scanf 风格的 pattern。

-filter-mode 决定报多少

值报什么
added（默认）只报新增 / 修改的行
diff_context改动行 ±N 行（默认 N=3）的上下文内
file只要文件被改了，整文件的问题都报
nofilter全报，不过滤

默认 added 是最克制的，新加的代码出新问题才说话——这正是”零历史噪音”的来源。本地想在 push 前自查，用 -diff 喂一份 diff：reviewdog -diff="git diff origin/main"。

两个 GitHub reporter，auth 不一样

-reporter 有十来个（gitlab-mr-discussion / bitbucket-code-report / gerrit-change-review…），GitHub 上常用的两个底层 API 不同，因此认证和能力都不同，选错会白配半天：

  • github-pr-review：走普通 review-comment API，只要一个 PAT（REVIEWDOG_GITHUB_API_TOKEN，private 仓库需 repo scope）。不依赖 reviewdog 的服务器，而且支持 code suggestion（评论里带可一键 apply 的改法）。
  • github-pr-check：走 GitHub Check API，能在 PR 顶部生成正经的 check 状态（可当 merge 门禁）。代价是 Check Runs 这套 API 普通 PAT 调不动——要么在 Actions 里用 secrets.GITHUB_TOKEN，要么装 reviewdog 的 GitHub App 配 REVIEWDOG_TOKEN。不支持 code suggestion。

口诀：想要能一键采纳的修改建议、配置最省 → github-pr-review；想要红绿状态做 merge gate → github-pr-check。

同类工具其实是两条线

reviewdog 属于 rule-based 那条——规则确定、几乎零误判，只把已有 linter 接进 review。同线还有 Super-Linter（一个容器塞几十种 linter，开箱即用）和 SonarQube/SonarCloud（带质量看板、覆盖率、quality gate，重）。

另一条是 AI-based：CodeRabbit、Qodo 读 diff 给自然语言评论，能看出逻辑层面的问题，但有概率出错。两条线不冲突：rule-based 兜底确定性问题，AI-based 补人类视角。

装：brew install reviewdog/tap/reviewdog。文档：https://github.com/reviewdog/reviewdog。

git commit --fixup 配 rebase autosquash 收拢历史

code review 改意见时，最容易留下 fix typo、address review 这种噪音 commit。--fixup + autosquash 让每个补丁自动并回它对应的原始 commit，历史只剩有意义的那几个。

发现 commit A 要补个改动时，不要新开一个独立 commit，而是：

git commit --fixup A      # message 自动写成 "fixup! <A 的标题>"

A 是占位符，换成目标 commit 的任意 commit-ish:hash(abc1234)、HEAD~2、分支名，或 :/登录 模糊匹配 message。下文 A / B 同理，只是示意图里给 commit 起的标签。

之后一把 rebase 收拢：

git rebase -i A^

打开 todo 时 git 已经自动把 fixup! 那条挪到 A 正下方、并标好 fixup 动作，直接存盘退出即可吃进 A:

pick   A   feat: 登录接口
fixup  A'  fixup! feat: 登录接口   ← 自动排序 + 自动标记
pick   B   feat: 注册接口

~/.gitconfig 里这行是关键——它让 rebase -i 默认就开自动排序，否则每次得手动加 --autosquash,或在编辑器里自己挪行：

[rebase]
  autosquash = true

两个易混点：

  • --fixup vs --squash:fixup! 丢弃补丁的 message，完全用原 commit 的;squash! 把两条 message 拼起来让你编辑。改 bug 用 fixup，要补充说明用 squash。
  • autosquash 只改 -i 的默认行为,不是全自动——仍要你手动触发一次 git rebase -i,它只负责把 fixup!/squash! 排到位。

--fixup 还能直接指向”动了哪个文件就修哪个 commit”:

git commit --fixup :/登录    # 模糊匹配最近 message 含"登录"的 commit

See man git-rebase → --autosquash,man git-commit → --fixup。

git 按目录自动切换提交身份 (includeIf)

多身份 (公司 / 个人) 最容易出的事是忘记切换、用错 email 提交。includeIf 让 git 按仓库所在目录自动加载对应身份，零手动操作。

~/.gitconfig:

[user]
 name = Your Name        # 默认 name;故意不写 email

[includeIf "gitdir:~/work/"]
 path = ~/.gitconfig_work
[includeIf "gitdir:~/personal/"]
 path = ~/.gitconfig_personal

~/.gitconfig_work / ~/.gitconfig_personal 各放一行 email:

[user]
 email = you@company.com

防呆关键：全局不设默认 email。任何没被规则覆盖的目录会直接报 unable to auto-detect email address 拒绝提交——把”忘记”变成”无法继续”,而不是悄悄用错身份提交后才发现。这一步才是护城河，includeIf 本身 google 5 秒可查。

三个坑：

  • 路径结尾的 / 必须有:gitdir:~/work/ 匹配该目录下所有 repo;漏了 / 只精确匹配 ~/work 这一个 repo。
  • 多条命中时后写的赢:git 按出现顺序逐个 include，后面覆盖前面;更具体的目录规则写在后面。
  • local 配置仍是最高优先级:某 repo 里 git config --local user.email 设过会盖掉 includeIf,这类历史 local 覆盖要清掉才轮到目录规则接管。

验证别直接 cd 到父目录 (非 repo 测不出),建个临时 repo 看实际结果：

mkdir -p ~/work/.idtest && git -C ~/work/.idtest init -q
git -C ~/work/.idtest config user.email   # → you@company.com
rm -rf ~/work/.idtest
git config --global --get-regexp includeIf  # 查所有规则

另一种匹配:hasconfig 按远程地址

目录分不开时 (公司 / 个人 repo 混在一处),改用远程 URL 匹配：

[includeIf "hasconfig:remote.*.url:git@github.com:acme/**"]
 path = ~/.gitconfig_work
  • 通配符除标准 glob 外，多了 **/ 和 /** 能跨多级路径匹配。git@github.com:acme/** = acme 这个 org 下所有 repo;git@github.com:*/** 则是 github 上所有 repo。
  • 非显然点 (scan-ahead):gitdir: 看路径就能判;但 remote URL 通常在 repo 的 .git/config 里，比全局 includeIf 晚解析。git 对 hasconfig 特殊处理——第一次遇到这个条件时，会预扫描后面所有配置文件找 remote URL，所以晚定义的 local remote 照样命中。
  • 代价:被它 include 进来的文件 (~/.gitconfig_work)禁止再声明任何 remote URL——否则「是否包含」会反过来取决于「被包含文件里的 remote」,形成死循环，git 直接禁掉这种写法。

按目录 (gitdir:) 仍是最简单可靠的默认，只有目录无法区分时才上 hasconfig。

See man git-config → “Conditional includes”。
