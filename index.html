PHP libvips 写文字：Pango 接管 layout，不用算 bbox

2026-05-12 #php

GD 那一套 baseline / bbox / 减偏移 在 libvips 里不存在。Image::text() 走 Pango 做 layout —— 你给一个 width 框 + align 方向，结果 image 直接拿到 $txt->width / $txt->height，没有 baseline 这回事。

Pango 是什么

GNOME 项目下的文字 layout 库。渲染一段文字本来要分两步，libvips 把第一步外包给了它：

阶段做什么谁来做
Layout选字体、断行、字距、对齐、Bidi（阿拉伯/希伯来从右到左）、CJK 换行规则、emoji shapingPango
Rasterize把矢量字形画成像素FreeType / Cairo

GD 的 imagettftext 只做第二步 —— 调用方必须自己算宽高、自己拆行、自己处理对齐，所以才有 imagettfbbox 8 元素数组、baseline 减偏移那堆脏活。libvips 直接传 width: 800, align: 'high'，Pango 帮你折行 + 对齐，结果 image 的 width / height 就是排版后的真实尺寸 —— baseline 这个概念在调用方完全不出现。

额外 bonus：开 rgba: true 之后可以传 Pango markup，单段里混排粗体 / 颜色 / 字号：

$txt = Image::text(
    '<span foreground="red" weight="bold">警告</span> 内容超过限制',
    ['font' => 'Sans 20', 'rgba' => true]
);

GD 想做这个得渲染 N 次再拼。

实际用法：右下角水印

同样的场景，libvips 写法：

use Jcupitt\Vips\Image;

$bg  = Image::newFromFile('photo.jpg');
$txt = Image::text('© 2026', [
    'font'  => 'Sans Bold 24',
    'width' => $bg->width - 40,   // 框内自动 word-wrap
    'align' => 'high',
    'rgba'  => true,              // 不开是灰度 mask，没法合到彩图
]);

$bg->composite($txt, 'over', [
    'x' => 20,
    'y' => $bg->height - $txt->height - 20,
])->writeToFile('out.png');

GD 同场景要：imagettfbbox 拿 8 元素 → $bbox[2] - $bbox[0] 算宽 → $bg_w - $width - 20 算 x → 还要担心 baseline 跟图像 y 轴方向相反。libvips 一步：align: high。

Image::text() 关键 options（官方文档）：

Option作用默认
fontFontconfig 名，如 'Sans Bold 24' / 'Noto Serif CJK SC 18'—
fontfile加载非系统字体的 .ttf 路径，需配合 font 用名字引用—
width换行宽度（像素），超过按 word boundary 折行不换行
align'low' / 'centre' / 'high' —— 多行的水平对齐—
justify两端对齐（拉伸词间空格），boolfalse
spacing行距（像素）字体自然行距
dpi渲染 DPI，影响实际像素大小72
rgba出彩色 / emoji / Pango markup，不开是 8-bit 灰度 maskfalse

部署成本是 libvips 在 PHP 圈用得不如 GD 普及的主因 —— ext-gd 几乎随 PHP 自动装好，libvips 要：

# Debian/Ubuntu
apt install libvips42 libvips-dev

# macOS
brew install vips

# PHP 端二选一
pecl install vips                                # 传统 C extension
# 或开 FFI（PHP 7.4+），用 jcupitt/php-vips 包，无需编译扩展
composer require jcupitt/vips

容器里加一行 apt install libvips42 就够，但这一行常被忘。装上之后水印 / 缩略图 / 多行排版量大时，libvips 的内存占用和吞吐都比 GD 强一个数量级 —— 它是流水线架构，不像 GD 把整张图按解压尺寸驻留内存。

cloc / tokei / scc：codebase LOC 统计的三代工具

数代码行数（LOC by language）这件事有三代工具，每代换语言、换取舍：

cloctokeiscc
实现 / 起年Perl / 2006Rust / 2016Go / 2018
Linux kernel 用时~64s~1.4s~0.8s
默认并行❌（要 --processes=N）✓✓
安装brew install clocbrew install tokeibrew install scc
独有能力--diff 对比两个 commit；读 .zip / .tarJSON / YAML 输出复杂度 / COCOMO / 重复检测

scc 那三个名词各自的意思：

  • 复杂度：数代码里 if / for / && 这类分支关键字，估算 cyclomatic complexity。
  • COCOMO：1981 年 Barry Boehm 提出的成本模型，给 KLOC 估”工业平均要多少人月、多少钱”。
  • 重复检测：按文件 hash 判定，相同内容只算一次（monorepo / vendor 目录里常见，cloc / tokei 会重复计入）。

对同一个目录（这个 blog 的 til/src/）跑默认命令，差异自己跳出来：

$ cloc til/src/
Language           files          blank        comment           code
Astro                 10            117             62           1340
TypeScript             7             14             49             89
SUM:                  17            131            111           1429

$ tokei til/src/
 Language         Files        Lines         Code     Comments       Blanks
 Astro               10         1519         1344           59          116
 TypeScript           7          152           97           44           11
 Total               17         1671         1441          103          127

$ scc til/src/
Language        Files       Lines    Blanks  Comments       Code Complexity
Astro              10       1,519        99        49      1,371         55
TypeScript          7         152        11        44         97          2
Total              17       1,671       110        93      1,468         57
Estimated Cost to Develop (organic) $40,425
Estimated Schedule Effort (organic) 4.06 months
Estimated People Required (organic) 0.88

cloc 没有 Lines 总数列（自己加 blank+comment+code）；tokei 加了 Lines；scc 多 Complexity 列 + 一段 COCOMO 估算块。同样的 17 个文件，三个工具给出三个 Code 总数：1429 / 1441 / 1468——blank 和 comment 怎么判，每家略不同。

cloc 的 --diff 是至今没被替代的核心——code review 想量化”这个 PR 改了多少 code 行”时，tokei / scc 都做不到：

$ cloc --git --diff HEAD~5 HEAD
Language        files          blank        comment           code
Markdown
 same              0              0              2            924
 modified          2              0              0              9
 added             0              0              0              4
 removed           0              0              0             19
Bourne Shell
 same              0              0              5             14
 modified          1              0              0              1
 added             0              0              0              1
 removed           0              0              0              0
SUM:
 same              0              0              7            938
 modified          3              0              0             10
 added             0              0              0              5
 removed           0              0              0             19

按语言分组，每组拆 same / modified / added / removed；想细到文件级加 --by-file。比 git diff --stat 信噪比高一截。

怎么选： 默认装 scc（最快 + 复杂度）；CI 输出要 JSON 接管线用 tokei；要 git diff 的 LOC 变化才掏 cloc。

git summary 是接手陌生仓库的瑞士军刀，一行看到全貌：

$ git effort --above 3 content/posts/

PHP 安装工具的几条路径

PHP 安装按”装什么 + 在哪装”分四条路径，每条各自的工具：

纯 PHP 代码（复制即用）

  • PEAR — PHP Extension and Application Repository。最早的类库分发，用前要 include。已被 Composer 取代，基本退场。
  • Composer — PHP 依赖管理工具，灵感来自 npm / Bundler。composer.json 声明依赖，composer install 拉齐，事实标准。

C 扩展，裸机环境（编译 + 改 php.ini）

  • PECL — PHP Extension Community Library。PEAR 的子项目，分发 redis、apcu 这类底层扩展。已被官方标记 deprecated。
  • PIE — The PHP Installer for Extensions。PHP 基金会官方接管，取代 PECL，命令风格抄 Composer。

C 扩展，Docker 容器场景

  • docker-php-ext-install — 官方 PHP Docker 镜像内置脚本，容器里装 C 扩展的事实标准。
  • install-php-extensions — 上面的友好封装，自动处理依赖、编译选项、共享库。Dockerfile 里写 PHP 几乎人人用它。

PHAR 形式的 CLI 开发工具（PHPUnit、phpstan、psalm 这种）

  • PHIVE — PHAR Installation and Verification Environment。装 PHAR 工具并校验签名，避免污染 composer.json。

七个命令并排看：

# 纯 PHP 代码
pear install Mail                        # 老路子，已退场
composer require monolog/monolog         # 现代

# C 扩展（裸机）
pecl install apcu                        # 老路子（deprecated）
pie install apcu/apcu                    # 现代

# C 扩展（Docker），以装 gd 为例——需要系统库 + configure
apt-get install -y libfreetype-dev libjpeg-dev libpng-dev \
  && docker-php-ext-configure gd --with-freetype --with-jpeg \
  && docker-php-ext-install gd            # 官方：3 步联动
install-php-extensions gd                 # 友好封装：1 行搞定

# PHAR CLI 工具
phive install phpunit

记忆：

  • PHP 代码 → Composer
  • C 扩展，裸机 → PIE（PECL 已 deprecated）
  • C 扩展，Docker → install-php-extensions
  • PHAR CLI 工具 → PHIVE

Dockerfile 中三组易混指令怎么选

Dockerfile 里有几组指令名字相近、用途易混。按”什么时候执行 + 写到哪里”两条轴归位即可：

  • 构建时执行（写进 image 层）：RUN、COPY、ADD、ARG
  • 运行时执行（启动 container 时跑）：CMD、ENTRYPOINT
  • 构建期声明、运行时仍可见：ENV
  • 只是元数据（不执行也不影响行为）：LABEL

COPY vs ADD

两者都把构建上下文里的文件复制进 image，但 ADD 多了两个隐式行为：

  • 源是 .tar / .tar.gz 时自动解压到目标目录
  • 源是远程 URL 时自动下载

这两个 “智能” 行为正是 ADD 比 COPY 危险的地方——upstream tarball 内容变了，image 跟着变，且解压结果难审计。

口诀：默认用 COPY，确实需要解压本地归档时才上 ADD。远程文件用 RUN curl -O 更显式、更可控。

RUN vs CMD vs ENTRYPOINT

RUN 在构建时执行并提交一层。CMD 和 ENTRYPOINT 都只描述启动命令，区别在能不能被 docker run 的尾参覆盖：

指令何时跑默认参数？被尾参覆盖？
RUN构建时——
ENTRYPOINT启动时否不会（除非 --entrypoint）
CMD启动时是会

两种写法行为不同：

ENTRYPOINT ["/scripts/entrypoint.sh"]    # exec form
CMD "/scripts/start-fpm.sh"              # shell form

exec form（JSON 数组）= 直接 execve，进程是 PID 1，能正常收到 SIGTERM。 shell form（裸字符串）= Docker 包成 /bin/sh -c "..."，再整串作为参数传给 ENTRYPOINT，最终启动等价于：

/scripts/entrypoint.sh /bin/sh -c "/scripts/start-fpm.sh"

只有 CMD ["nginx"] 没 ENTRYPOINT 时，CMD 自己就是启动命令；docker run img bash 会把 nginx 整串替换成 bash。

口诀：主程序用 ENTRYPOINT exec form；可被覆盖的默认参数走 CMD exec form。两个都用 shell form 会丢掉信号转发，容器收不到 docker stop 的 SIGTERM，10 秒后被 SIGKILL 强杀。

LABEL vs ARG vs ENV

三者都是 “声明值”，但生命周期完全不同：

指令构建时可见？运行时可见？写进 image？
ARG是否否（但值会出现在 docker history，别塞密钥）
ENV是（声明之后的层）是（容器环境变量）是
LABEL否否是（仅元数据，docker inspect 可见）

ARG 还有一条作用域陷阱：FROM 之前声明的 ARG 只能在 FROM 行里展开，要在后续指令里用同名 ARG，得在 FROM 之后再声明一次：

ARG PHP_VERSION=8.3
FROM php:${PHP_VERSION}-fpm    # OK，这里能用

ARG PHP_VERSION                # 必须重新声明，下面才能用
RUN echo "Building PHP ${PHP_VERSION}"

口诀：版本号 / 镜像 tag 用 ARG；运行时配置（端口、路径、非密钥配置）用 ENV；作者 / VCS 地址 / 镜像描述用 LABEL。

git clean 删未跟踪文件：先 -n 看，-X 单独清 ignored

git clean 删的是 working tree 里未被追踪的文件——.gitignore 命中的、和未 add 过的新文件。删完没有 git checkout 可以救，所以永远先用 -n 走一遍：

git clean -n           # 列出会删的文件，不动磁盘
git clean -nd          # 含未跟踪的目录
git clean -nXd         # 只列 .gitignore 命中的（如 build/ dist/）

确认后把 -n 换成 -f（force，必填——不写就拒绝执行）：

git clean -fd          # 删未跟踪的文件 + 目录
git clean -fXd         # 只删 ignored（清 build artifacts、.cache）
git clean -fd -- src   # 限定路径

-X 大小写反义：

  • -X（大写）：只删 .gitignore 命中的，未跟踪的新文件（如手动创建忘 add 的）保留
  • -x（小写）：把 ignored 也算成未跟踪一并删——破坏性更大，会清掉 .env、本地 IDE 配置等被 ignore 的工作文件，慎用

2022-10-01 #git

git 改提交时间：--date 只动 author，要改 committer 用环境变量

git 每个 commit 有两个时间字段：author date（谁写的代码、什么时候写的）和 committer date（谁应用的这个 commit、什么时候应用的）。git log 默认显示 author date，但 git log --format=%ci、GitHub 时间线排序、git log --since 都看 committer date——只改 --date 常出现”日志看着对了，GitHub 顺序没动”。

新 commit 同时改两个：

GIT_COMMITTER_DATE="2019-06-11T17:50:50+08:00" \
  git commit -m "fix..." --date="2019-06-11T17:50:50+08:00"

修上一个 commit（会改 hash，已 push 的不要碰）：

GIT_COMMITTER_DATE="$(date -R)" \
  git commit --amend --no-edit --date="$(date -R)"

验证两个字段：

git log -1 --format='author %ai%ncommit %ci'

2022-10-01 #git

git 忽略文件权限位变更：core.filemode false

跨 OS 协作时（macOS / Linux ↔ WSL ↔ NTFS / FAT 网络盘），git 会把权限位变化（100755 ↔ 100644）当成内容变更，git status 满屏 mode-only diff 但其实没东西可改。

仓库局部关掉：

git config core.filemode false

core.filemode 不在版本控制里——它是 .git/config 的本地设置，不会同步给协作者（git 设计如此：权限策略是工作环境属性，不是仓库属性）。所以改完不用 commit。

陷阱：一刀切关掉后，你新加的可执行脚本权限不会被同事看到——CI 上跑 ./deploy.sh 直接 permission denied。要保留”真正的可执行位”只压”杂讯”，用：

git update-index --chmod=+x path/to/script.sh

显式把这个文件标记成 executable 写进 index，比关 filemode 精准。

2022-10-01 #git

git 建独立无历史的分支用 --orphan

git checkout --orphan 创建一个不指向任何已有 commit 的新分支——HEAD 移过去后 working tree 保留，但 staging 和 commit history 都从零开始。常用场景：一个 repo 同时管代码和文档站，文档站需要独立 history（如 gh-pages），或者发布开源版本时不想暴露内部 commit 史。

git checkout --orphan gh-pages
git rm -rf .                         # 关键：清掉父分支带过来的 working tree
echo "<h1>hi</h1>" > index.html
git add . && git commit -m "init gh-pages"
git push origin gh-pages

陷阱：--orphan 切过去后 git status 会显示一堆 staged 文件——那是从父分支继承的 working tree，没 git rm -rf . 就 commit 等于把整套源码塞进了所谓的”独立”分支。

查 Linux 包：pkgs.org 找上游 + rpm/dpkg 看本地

排查”这台机器上 X 是哪个版本""我想装的版本哪个发行版有”——两边问题，两类工具。

pkgs.org 把主流发行版（CentOS/RHEL/Fedora/Debian/Ubuntu/openSUSE/Arch…）的官方仓库索引到一起，按包名搜能直接看到各发行版各版本提供的版本号、依赖、下载链接。比挨个翻 apt search / yum search 快，也方便确认”上游有没有这个版本”。

本地查”装了什么版本”：

# Red Hat / CentOS / Fedora (rpm 系)
rpm -qa | grep docker            # 列出所有匹配的已装包
rpm -q docker-ce                 # 查具体包
rpm -qi docker-ce                # 详细信息（描述、维护者、构建时间）

# Debian / Ubuntu (dpkg 系)
dpkg -l | grep docker            # 已装包，含版本和架构
apt list --installed | grep docker
dpkg -L docker-ce                # 列出这个包装了哪些文件

rpm -qa / dpkg -l 走本地包数据库，不联网；pkgs.org 是上游索引，不知道你本地装了啥。两者互补——前者答”装没装”，后者答”哪儿有装的”。

2022-07-28 #network

查 CNAME：dig / host / nslookup 三选一

CNAME（canonical name）是 DNS 把一个域名指向另一个域名的别名记录。常见场景：www.example.com CNAME 到 CDN 边缘域名，或 SaaS 给你一个 <your>.heroku.com 让你 CNAME 过去。

dig example.com CNAME              # 完整记录
dig +short example.com CNAME       # 只要目标
dig +trace example.com             # 完整解析路径（root → TLD → 权威）

host -t CNAME example.com          # 输出最人话："is an alias for ..."

nslookup -type=CNAME example.com   # Windows 时代遗留，少用

CNAME 两条硬规则：

  • 不能和其他记录共存：一个 name 上有 CNAME 就不能再有 A / MX / TXT。所以根域名 example.com 通常不能用 CNAME（MX 在那里），子域才行
  • 链式解析：CNAME 可以指向另一个 CNAME，resolver 会一路 chase，最终必须落到 A/AAAA。链长了拖慢首次解析

挂在云厂商前的根域名查不出 CNAME 多半是因为对方用了 ANAME / ALIAS / flattening——根域 CNAME 限制的绕过手段，权威服务器替你预解析到 A 记录返回，外部看到的就是一条 A 而不是 CNAME。

删了文件 df 不掉：lsof | grep deleted 找仍持有的进程

经典症状：rm 掉一个大日志，df 看磁盘空间没变。

根因：unix 文件系统下 rm 做的只是 unlink——把目录项里的引用删掉。inode 真正释放要等到所有持有它的 file descriptor 都关闭。如果某个进程还开着这个文件（典型场景：app 把日志 open 后一直 append），inode 就一直挂在 fd 上不释放，磁盘空间也回不来。

定位是哪个进程：

lsof | grep deleted
# nginx  1234  www-data  3w  REG  8,1  524288000  131  /var/log/nginx/access.log (deleted)

第 7 列就是占的字节数，第 1-2 列是进程名 + PID。两条出路：

# A. reload / 重启持有进程，让它关 fd
nginx -s reopen                  # nginx 专门有这个信号
systemctl restart <svc>          # 通用兜底

# B. 源头改成 truncate 而不是 rm
truncate -s 0 /var/log/access.log    # 文件保留，内容清空，进程继续写同一 inode
> /var/log/access.log                # 同效果，shell 内置

别用 echo "" > file——会写一个换行符进去，不是真清空。

预防：日志类场景上 logrotate 并开 copytruncate 选项，它就是为了解决 rotate 时进程仍持有老 fd 的问题。

shell job control：& nohup Ctrl+Z fg bg jobs

把命令丢后台、再拽回前台的全套手势。

& 直接后台跑，但关 terminal 时会被 SIGHUP 打死——shell 退出会向自己所有子进程发 HUP。nohup 的作用就是让进程忽略 HUP，于是关 terminal 也不死：

cmd &                            # 后台跑，关终端就没了
nohup cmd &                      # 关终端不死，stdout 默认写 nohup.out
nohup cmd > out.log 2>&1 &       # 自己接管输出

已经在前台跑的命令想丢后台——按 Ctrl+Z 暂停（SIGTSTP），shell 把它放进 jobs 列表，然后 bg 让它在后台继续跑：

$ long-running-cmd
^Z
[1]+ Stopped  long-running-cmd
$ bg                             # 继续在后台跑
$ jobs -l                        # 看 jobs 列表（带 PID）
[1]+ 12345 Running   long-running-cmd &
$ fg                             # 拽回前台

多个 job 时用 %N 选：

fg %1                            # 拽 job 1 回前台
fg %vim                          # 按命令名匹配
bg %2                            # 让 job 2 在后台继续
kill %1                          # 杀 job 1

nohup 只挡 HUP，不解决长跑任务的”我下次想接回 session 看实时输出”。生产场景认真用 tmux / screen / systemd unit，nohup 是临时手段。

shell 脚本必加 set -euo pipefail

bash 默认行为对脚本极不友好——命令失败默认继续往下跑、未定义变量默认空串、管道里前面命令失败被后面成功的命令掩盖。每个长一点的脚本开头都该加：

#!/usr/bin/env bash
set -euo pipefail

各位含义：

  • -e（errexit）：任一命令非零退出立即终止脚本。陷阱：在 || / && / if / while 条件里的命令不算
  • -u（nounset）：用未定义变量直接报错。把 rm -rf /$UNDEFINED/ 这种灾难挡在执行前。可能未定义的变量用 ${VAR:-default} 显式给默认值
  • -o pipefail：默认 cmd1 | cmd2 的退出码只看 cmd2——curl bad-url | jq . 即便 curl 失败也返 0。开 pipefail 后任一段失败整个 pipeline 就失败
  • -x（xtrace）：执行前打印每条命令。debug 用，生产脚本别开——会在日志里泄露 password / token

不开 errexit 时的手动错误处理：

cmd || { echo "cmd failed: $?" >&2; exit 1; }

if ! cmd; then
  echo "cmd failed" >&2
  exit 1
fi

errexit 不是银弹——在 pipeline、command substitution、subshell 等场景下都有边界，set -e 会”看起来开了实际没生效”。强约束脚本（部署 / 数据迁移）建议直接上 shellcheck 做静态检查，比 set -e 兜底更可靠。

PHP __invoke：实例 callable，相当于带状态闭包

__invoke 是 PHP 的 magic method —— 类里定义了它，实例就能当函数直接调用：

class Adder {
    public function __construct(private int $base) {}
    public function __invoke(int $x): int { return $this->base + $x; }
}

$add5 = new Adder(5);
$add5(3);                     // 8
is_callable($add5);           // true

价值不在”语法甜”，在承载状态的 closure。原生 closure 用 use ($base) 捕获变量；__invoke 用 __construct 注入依赖、$this 持有任意复杂状态，还能继承、被 mock、实现接口。

单动作 controller（Laravel）

路由绑类不绑方法，文件 = 端点。一个 controller 做一件事，不再有 8 个 action 挤一个 class：

class ShowProfile extends Controller
{
    public function __invoke(int $id)
    {
        return view('user.profile', ['user' => User::findOrFail($id)]);
    }
}

Route::get('user/{id}', ShowProfile::class);

文档：Single Action Controllers | Laravel。

策略 / provider 对象

需要”可配置的函数”时 —— 查表、解析、路由匹配 —— 比 closure 更适合，因为有完整 OOP toolkit 可用。AWS SDK 的 PatternEndpointProvider 就是这个套路：构造时吃配置 patterns，调用时按 service/region 查规则、返回 endpoint：

$provider = new PatternEndpointProvider($patterns);
$endpoint = $provider(['service' => 's3', 'region' => 'us-east-1']);

调用方不必知道里面是查表、远程 fetch 还是 ML 推理 —— 签名永远是”参数 in, 结果 out”。换实现不动 caller。

一个坑：callable ≠ Closure

is_callable($obj) 返回 true，但 $obj instanceof Closure 是 false。type-hint 写 Closure 会拒掉 __invoke-able 对象，写 callable 才接受：

function run(callable $fn) { return $fn(1); }     // ✅ 接受 Adder 实例
function run(Closure $fn)  { return $fn(1); }     // ❌ TypeError

要把 __invoke 对象转成真正的 Closure 实例（比如某 API 强制 Closure 入参），用：

$closure = Closure::fromCallable($add5);  // PHP 7.1+
$closure = $add5(...);                    // PHP 8.1+ first-class callable

PHP strict_types 看调用方，不看声明方

declare(strict_types=1) 关掉 PHP 的隐式类型转换 —— 函数 type-hint 写 int $x，不再吞 1.0 / "15" / true，传错直接 TypeError：

declare(strict_types=1);
function add(int $a, int $b): int { return $a + $b; }
add(1.0, 2.0);
// PHP Fatal error: Uncaught TypeError: add(): Argument #1 ($a) must be of type int, float given

不写 declare 行（默认 weak mode）时 add(1.0, 2.0) 返回 int(3)，安静吞掉。

反直觉点：strict 看 caller，不看 callee

文件级声明会让人下意识以为”哪边定义函数 / 类，哪边的 mode 控制类型检查”。反了：检查谁，看 call site 所在文件的 declare。

A.php 严格 / B.php 默认：

// A.php
declare(strict_types=1);
function add(int $a, int $b): int { return $a + $b; }

// B.php
require 'A.php';
add(1.0, 2.0);  // ✅ int(3) —— B 没声明 strict，照旧 coerce

A.php 默认 / B.php 严格：

// A.php
function add(int $a, int $b): int { return $a + $b; }

// B.php
declare(strict_types=1);
require 'A.php';
add(1.0, 2.0);  // ❌ TypeError —— B 声明了 strict，调用按 strict 检查

实际后果：第三方库无论自家标了多少 strict_types，决定你调它会不会抛 TypeError 的是你自己文件顶部那行。Composer 装回来的 vendor/ 大概率是 strict，但你在 weak 文件里传 1.0 给它的 int 参数，PHP 不会拦。

要”全项目 strict”，每个 .php 顶部都得写一遍。靠 php-cs-fixer 的 declare_strict_types rule 或 symplify/easy-coding-standard 自动补，别手写。

写法约束

必须是脚本里最早的语句（<?php 后第一行，前面不能有 use / 注释外的任何内容）：

PHP Fatal error: strict_types declaration must be the very first statement in the script

不能用 block 形式：

declare(strict_types=1) { ... }   // ❌ must not use block mode

declare 行只作用于当前文件，不会污染 require / include 进来的脚本 —— 这正是允许逐文件渐进迁移的设计。

SQL AND / OR 执行优先级

AND 优先级高于 OR，所有 SQL 方言通用：

SELECT id FROM t WHERE c1 OR c2 AND c3;

-- 实际等价于（AND 先绑定）：
SELECT id FROM t WHERE c1 OR (c2 AND c3);

-- 不是这样（很多人下意识这么想）：
SELECT id FROM t WHERE (c1 OR c2) AND c3;

口诀：AND 看成 *，OR 看成 +，乘号先算。

Bug 不会报错——query 跑成功，只是结果错——所以 WHERE 里只要 AND 和 OR 混用，永远写括号，别赌读者（或几个月后的自己）记得优先级。

MySQL DELETE 子查询自引 1093 错误

想删一张表里的重复数据，按 name 分组保留 MIN(id) 那条：

DELETE FROM student
WHERE id NOT IN (SELECT MIN(id) FROM student GROUP BY `name`);

直接跑会报：

ERROR 1093 (HY000): You can't specify target table 'student' for update in FROM clause

MySQL 不允许在 DELETE / UPDATE 的子查询里直接引用目标表——执行时一边扫一边改会让结果不确定，干脆禁掉。

绕开的办法：再套一层 derived table，强制 MySQL 把子查询物化成临时表后再用，与目标表解耦：

DELETE FROM student
WHERE id NOT IN (
  SELECT id FROM (
    SELECT MIN(id) AS id FROM student GROUP BY `name`
  ) tmp                                              -- 关键：再包一层
);

MySQL 8.0+ 用 CTE 写更清晰，CTE 同样会被物化，所以一样能绕开 1093：

WITH keep AS (SELECT MIN(id) AS id FROM student GROUP BY `name`)
DELETE FROM student WHERE id NOT IN (SELECT id FROM keep);

PostgreSQL / SQL Server 没这个限制，直接写第一种形式就行。

MySQL FOR UPDATE / FOR SHARE 真实加锁范围

SELECT ... FOR UPDATE 和 FOR SHARE（8.0.1 之前叫 LOCK IN SHARE MODE）加的锁经常比写 SQL 时以为的大一圈。规则不复杂，但每条单独看都反直觉。

S 锁 vs X 锁

已有 S已有 X
申请 S✅❌
申请 X❌❌
  • FOR SHARE = S 锁，多事务可同时拿
  • FOR UPDATE / INSERT / UPDATE / DELETE = X 锁，互斥

死锁陷阱：两个事务都拿了同一行 S 锁，任何一方想升级成 X 锁都会死锁——必须先等对方释放。所以 FOR SHARE 后立刻 UPDATE 是高危模式。

锁加在索引上，不是行上

SELECT * FROM t WHERE non_indexed_col = 'x' FOR UPDATE;

WHERE 列没索引时 InnoDB 会对扫描过的所有索引项加锁——走全表扫描就等于锁全表。EXPLAIN 看到 type=ALL 配上 FOR UPDATE 基本就是事故现场。

确保 WHERE 列有索引，且 EXPLAIN type 不是 ALL，加锁才能精准。

REPEATABLE READ 下 next-key lock 把 gap 也锁了

MySQL 默认隔离级别 REPEATABLE READ，InnoDB 此时用 next-key lock = record lock + gap lock：

-- 表里现有 id: 1, 5, 10
SELECT * FROM t WHERE id BETWEEN 3 AND 8 FOR UPDATE;
-- 锁 id=5（record lock）+ (1,5)、(5,10) 区间（gap lock）
-- 另一事务 INSERT id=4 / id=7 全部阻塞

更反直觉的是没匹配上的查询也加 gap 锁：

-- 表里现有 id: 1, 5, 10
SELECT * FROM t WHERE id = 7 FOR UPDATE;
-- 没行可锁，但锁住 (5,10) 区间，INSERT id=6/7/8/9 全部阻塞

唯一索引 + 等值匹配且匹配到——只加 record lock 不加 gap lock。这是少数能精准加锁的场景。

READ COMMITTED 关掉 gap 锁

会话或事务级改成 RC，gap lock 全部消失，只剩 record lock：

SET SESSION TRANSACTION ISOLATION LEVEL READ COMMITTED;

代价是放弃可重复读、接受 phantom row。在高并发 OLTP 场景里去 gap lock 能显著降低死锁率，很多 ORM 默认推荐 RC。

FOR SHARE 在 8.0+ 多的 NOWAIT / SKIP LOCKED

默认拿不到锁会等 innodb_lock_wait_timeout（50s），8.0+ 加了两个修饰符——LOCK IN SHARE MODE 不支持，FOR SHARE 才有：

SELECT * FROM t WHERE id = 5 FOR UPDATE NOWAIT;
-- 拿不到立刻报 ER_LOCK_NOWAIT，不等

SELECT * FROM t WHERE status = 'pending' ORDER BY id FOR UPDATE SKIP LOCKED LIMIT 1;
-- 跳过被锁的行

SKIP LOCKED 是写消息队列 / worker 抢任务的标准模式——多个 worker 同时跑这条 SQL，自动每人拿到不同任务，零竞争零冲突。比 FOR UPDATE + 应用层重试简单一个数量级。

MySQL 看优化器决策的三件套

想知道 MySQL 内部怎么处理你的 SQL，按”看的层次”挑工具：

工具看到什么适用场景
EXPLAIN物理执行计划（join 顺序、用了哪个 index、估算扫描行数）日常 SQL 优化第一站
SHOW WARNINGS after EXPLAINMySQL 重写 / 改写后的 SQL联表、视图、子查询场景，看实际跑的版本
OPTIMIZER_TRACE优化器完整决策过程（cost 估算、为什么没选某 index）选 index 反常识时排查

EXPLAIN + SHOW WARNINGS

EXPLAIN SELECT * FROM `order` o JOIN user u ON o.uid = u.id WHERE u.email = 'a@b.com';
SHOW WARNINGS;

SHOW WARNINGS 里 Note 1003 行就是 MySQL 改写后的 SQL——优化器可能调换 join 顺序、把外连接转内连接、把子查询折成 join。看了重写版才知道实际跑的是什么，原 SQL 已经不重要了。

optimizer_trace

SET optimizer_trace = 'enabled=on';
SELECT ... ;                                         -- 你要分析的 SQL
SELECT * FROM information_schema.OPTIMIZER_TRACE;
SET optimizer_trace = 'enabled=off';

返回的 TRACE 列是一大坨 JSON，覆盖三个阶段：

  • join_preparation — SQL 解析、视图展开、子查询改写
  • join_optimization — 候选 index、各 index 的估算 cost、最终选谁
  • join_execution — 执行细节

最常看 join_optimization，里面 range_analysis / chosen_range_access_summary 直接告诉你”为什么没选 idx_email”——通常是 rows_estimation 估算 cost 比全表扫还高，或者 join 顺序变了导致 index 不可用。

trace 是 session 级别，默认只保留最近一次，且开启后所有 query 都会被 trace（有性能开销），用完立刻关。

illuminate/filesystem 不是 illuminate/validation 的依赖，用 FileLoader 必须单独装。

准备语言文件

从 Laravel-Lang/lang 复制要用的语言文件到本地，例如 lang/zh-CN/validation.php。

装配

use Illuminate\Filesystem\Filesystem;
use Illuminate\Translation\FileLoader;
use Illuminate\Translation\Translator;
use Illuminate\Validation\Factory;

$loader = new FileLoader(new Filesystem(), __DIR__ . '/../lang');
$translator = new Translator($loader, 'zh-CN');
$validator = new Factory($translator);

用

$v = $validator->make(
    ['name' => ''],
    ['name' => 'required|string|min:2'],
);

if ($v->fails()) {
    print_r($v->errors()->all());
}

完整规则参考 Laravel 文档的 Manually Creating Validators。

Redis 按 pattern 批量删除 key

DEL 不接受 pattern，要按前缀清一批 key 得绕一下。最常见是 EVAL 里用 KEYS 拿全集再分块 DEL：

redis-cli EVAL "local keys = redis.call('keys', ARGV[1])
for i=1,#keys,5000 do
  redis.call('del', unpack(keys, i, math.min(i+4999, #keys)))
end
return #keys" 0 'prefix:*'

分块是因为 Lua 的 unpack 在几千个参数后会撞 stack 上限，单次 DEL unpack(keys) 在大集合下会直接报错。

但 KEYS 本身是 O(N) 全表扫，会阻塞 Redis 主线程直到扫完整个 keyspace——线上库千万别这么干。生产环境用客户端侧 SCAN，游标式增量取，单次 DEL 控量：

redis-cli --scan --pattern 'prefix:*' | xargs -L 1000 redis-cli DEL

注意 KEYS / SCAN 的 pattern 是 glob（* ? [abc]），不是 regex。

2020-08-14 #mysql

MySQL JSON 列建索引

MySQL 5.7+ 支持原生 JSON 数据类型。两个坑：

  • JSON 列不能有 non-NULL 默认值——要么 nullable，要么应用层填
  • 不能直接在 JSON 字段上建索引——要靠 Generated Column 包一层

用 Generated Column 索引 JSON 字段

把要查的 JSON 路径抽成一个虚拟列，再在虚拟列上建索引：

CREATE TABLE players (
  id               INT UNSIGNED NOT NULL,
  player_and_games JSON NOT NULL,
  name_virtual     VARCHAR(20) GENERATED ALWAYS AS (player_and_games ->> '$.name') NOT NULL,
  PRIMARY KEY (id),
  KEY idx_name (name_virtual)
);

->> 是 -> + JSON_UNQUOTE 的简写——直接拿字符串而不是 quoted JSON value。

Virtual 列只存元数据不落盘（默认），Stored 列落盘到磁盘。索引 JSON 字段几乎都用 Virtual，省空间且没性能差。

验证索引命中

EXPLAIN SELECT * FROM players WHERE name_virtual = 'Sally';
-- key: idx_name
