GitHub URL 隐藏参数：`.diff` / `?w=1` / `compare@{时间}` 工作流

2026-05-15 #github

GitHub 网页 URL 上有一组 2012 年就存在、UI 从没暴露过的参数。LLM 工作流让它们价值翻倍 —— curl 一个 .diff URL 直接拿到纯文本 diff 喂给 prompt，比让 LLM 自己抓网页快也准。下面所有链接都点得开（用 laravel/framework 演示）。

.diff / .patch —— 任何 PR / commit / compare 页 URL 加扩展名，返回纯文本：

curl -L https://github.com/laravel/framework/pull/10000.diff
curl -L https://github.com/laravel/framework/pull/10000.patch

.diff 体积最小，适合喂 LLM；.patch 多带 commit metadata（作者、邮箱、时间），适合 git am / cherry-pick。commit / compare URL 同样支持。

?w=1 —— 任何 diff URL 加上，忽略纯 whitespace 改动：

https://github.com/laravel/framework/pull/10000/files?w=1

把 indent 调整 / 换行符变化彻底从视野抹掉，只剩真改动。refactor / formatter 大重写后救命。

?ts=4 —— diff / blob 页改 tab 渲染宽度：

https://github.com/laravel/framework/blob/master/src/Illuminate/Foundation/Application.php?ts=4

数值任意（?ts=2 / ?ts=8），只改你眼里的呈现，远端文件不变。在 tab 缩进的项目（Go / Linux kernel）里效果最直观。

#L16-L25 —— 行号锚点 + range：

https://github.com/laravel/framework/blob/master/src/Illuminate/Foundation/Application.php#L1-L20

提 PR 评论、写文档、给同事丢 issue 时定位用，比”Application.php 大概第 20 行”准多了。

compare/<base>@{<时间>}...<head> —— compare 页接受时间锚点，不只是 SHA / branch：

https://github.com/laravel/framework/compare/master@{1.week.ago}…master

https://github.com/laravel/framework/compare/master@{2026-05-01}…master

语法和 git log @{...} 一致。release notes 起草、“上周改了啥”周报、incident 后回溯都用得上。

Bun 加入 Anthropic：6 天 AI 重写 Zig→Rust

Bun 在 2025-12-02 被 Anthropic 收购（Anthropic 史上第一笔），五个月后 Sumner 用 Claude Code 6 天把 96 万行 Zig 翻译成 Rust 并合入主分支。这两件事是同一根逻辑链：Claude Code 自己跑在 Bun 上，Anthropic 有最直接的动力修 Bun 的内存泄漏；而 AI Agent 重写大型 codebase 这件事也只有现在的 Anthropic 做得了。

时间线

2025-12-02  Anthropic 收购 Bun，Bun 继续 MIT 开源 + 团队继续主导
2026-05-04  Sumner commit "Phase-A 移植指南" + LIFETIMES.tsv
2026-05-10  claude/phase-a-port 分支出现，6 天 96 万行 Rust
            Linux x64 通过 99.8% 现有测试
2026-05-13  Bun v1.3.14 发布（"如果 Rust 合入这是最后一个 Zig 版"）
2026-05-14  Rust 版本合入 main，二进制减小 3-8 MB

为什么收购：Claude Code 自己被 Bun 烧伤

Claude Code 以单文件 Bun executable 分发给百万级用户，长会话内存能飙到 14-23 GB —— 直接踩 Bun 的内存泄漏。Anthropic 当时 Claude Code ARR 已 $1B（2026/2 升到 $2.5B），Bun 一坏 Claude Code 就坏，直接收购比合作更快。

收购动机里 AI Agent 工作流也是关键：agent 写完代码要立刻装包、跑测试、产可执行 —— Bun 的”单二进制 + 冷启动 8-31ms + 装包比 npm 快 25×“对应每一步。

为什么重写：Zig 的 no-AI 政策 + 内存安全

四个理由叠加：

  1. Zig 项目禁止 AI 生成的 issue / PR / 评论 —— Bun 团队 fork 的 Zig 编译器（debug 编译快 4×）永远没法回上游。对一家 AI 公司是政治死结
  2. 内存泄漏与 unsafe —— Sumner 原话 “I am so tired of worrying about memory leaks and crashes”
  3. Zig 还是 0.16，破坏性变更频繁 —— 做生产基础设施长期不稳
  4. Rust 生态成熟 —— AI 训练数据、crate 库都更充足

注意：性能不是动机。Bun 比 Node 快不来自 Zig 比 V8 快，来自架构（JSC 替 V8、原生 HTTP 不走 libuv、内置打包器）。换 Rust 不动这些架构，性能不会变。

重写方法：600 行 Rosetta Stone + AI Agent

工作流：

docs/zig-to-rust.md      ← ~600 行类型 / 习惯用法映射文档
LIFETIMES.tsv            ← 每个指针字段的 ownership 分类
Phase A: 翻 Rust 代码（不要求编译）
Phase B: crate-by-crate 让它编译过

故意不用 Tokio / Rayon，保持低层执行模型，做”直译”非重设计。Sumner 在 X 上：

“this is already the status quo; we haven’t been typing code ourselves for many months now.”

“我预期 OSS 反过来走：禁止人类贡献代码。讨论和优先级仍由人定，但写代码、提 PR、回应 review，全 LLM 做。“

争议

uv (Astral, Rust)：350,000 行 / 73 个 unsafe
Bun Rust 移植：    681,000 行 / 13,000+ 个 unsafe

第二个 PR（删 60 万行 Zig）被 GitHub 自动 flag 为 “AI slop” 关闭。社区核心反对意见：unsafe 块里的 hallucination 风险无法人工 review；commit 大小让 review 不可能；“vibecoded disaster”。

对生产用户的实际影响

  • 性能 —— 几乎没差，架构没变
  • 稳定性 —— 内存泄漏修复（Claude Code 类长会话受益最大）
  • 二进制 —— 小 3-8 MB
  • 兼容性 —— v1.3.14 后跑同一份 npm 应用应当无感

真正值得追踪的事

Bun 是史上第一个把核心代码所有权交给 AI Agent 维护的生产级开源项目。这种重写姿势能成立，前提是 Sumner 作为 BDFL（Benevolent Dictator For Life —— 开源项目”仁慈终身独裁者”，源自 Guido van Rossum / Python，决策不走委员会投票）一个人就能拍板”6 天试一下，搞不定就扔”。基金会治理的项目（如 Node.js / Python 后 Steering Council 时代）光投票就要几个月。

两种结局：

  • 稳住 → “几人 reviewer + 一群 agent + 一个 BDFL” 模式被复制到很多基础设施项目
  • 崩了 → AI 生成代码不可信论的最大佐证

无论哪种，未来 1-2 年值得追踪。

git filter-repo 清除历史敏感文件

仓库不小心把 .env / database.yml 推到远程后，第一动作是轮换密钥，假定文件已被抓取 —— 改写历史只能挡住”还没看到的人”，已 clone / fork / 爬虫的副本拿不回来。这一步先做完，再谈 git filter-repo：它是第二步，不是第一步。

为什么不是 filter-branch

git filter-branch 官方已弃用。每个 commit fork 一个 shell 子进程，大仓库跑数小时；默认保留 refs/original/ 和 reflog，旧对象很容易”清理后又复活”。git filter-repo 是 Python 单进程流式管道，快 10–100 倍，且默认要求 fresh clone（拒绝在带遗留 ref 的工作目录上跑）。

brew install git-filter-repo

原理：重写而非删除

Git commit 的 SHA-1 由 tree + parent + author + message 哈希得出，不可变。“删一个文件”等价于生成一份新历史替代旧历史。

旧仓库 ── git fast-export ──▶ 事件流 ── 过滤器 ──▶ git fast-import ──▶ 新仓库

git fast-export 把仓库历史序列化成纯文本指令流（blob、commit、tag）；filter-repo 在内存里按规则改写事件流；git fast-import 把它灌成新对象库。

关键后果：一个 commit 的 tree 变了 → 它的 SHA 变 → 子 commit 的 parent 指向新 SHA → 子 SHA 也变 → 一路传染到 HEAD。所以 force push 是必须的 —— 远程的 commit 链被一条 SHA 完全不同的链替换。

--invert-paths：默认是白名单

非显然：filter-repo 默认把 --path 当保留列表，不是删除列表。

# 默认：只保留 src/，其他全删（拆分 monorepo 用法）
git filter-repo --path src/

# --invert-paths：删除 src/，其他保留（清理用法）
git filter-repo --path src/ --invert-paths

设计初衷之一是从 monorepo 抽子项目 —— “列出我要的那部分”比”列出我不要的”自然。清理敏感信息正好相反：99% 留 + 1% 删，必须加 --invert-paths。漏写就把整个仓库删干净只剩那个泄露文件，反应过来时已经覆盖远程了。

git filter-repo --path .env --invert-paths
git filter-repo --path-glob '*.pem' --invert-paths
git filter-repo --path-regex '^secrets/' --invert-paths

--mirror：不配它等于没删

普通 git clone + git push --force --all 看似清理，实际只覆盖 branches。远程上 branch 之外的 ref 没动，仍指向旧 commit：

  • Tags：旧 tag 仍含敏感 commit
  • Notes、其他自定义 ref

只要远程任何一个 ref 还能可达旧 commit，那个 blob 就不会被 GC 回收 —— 攻击者拿到 SHA 就能 git cat-file -p <sha> 读出来。删了等于没删。

git clone --mirror 拉一份裸仓库（.git 结构、无工作区）+ 所有 ref；git push --mirror 把本地 refs 一比一推到远程，远程多出来的 ref 会被删 —— “让远程完全等于本地”。这才是真清理：

git clone --mirror git@github.com:org/repo.git
cd repo.git
git filter-repo --path config/database.yml --invert-paths
git push --force --mirror

--mirror 只解决服务端的 ref 问题。任何一个老的本地 clone 仍能通过 reflog 和 remote-tracking refs (refs/remotes/origin/*) cat-file 出旧 blob —— 这正是 filter-repo 默认要求 fresh clone 的原因，也是清理后必须通知所有协作者删本地仓库重新 clone 的原因。

清理后必做

步骤说明
轮换密钥见开篇，第一优先
通知协作者所有人重新 clone，旧本地分支基于已消失的 commit 链
联系 GitHub SupportPR refs (refs/pull/<n>/head) 由 GitHub 服务端维护，push --mirror 覆盖不了；fork 拷贝也要走 Support 清
加 .gitignore防止再次提交
装 pre-commit 钩子gitleaks protect --staged 自动拦截
brew install gitleaks
gitleaks detect             # 扫整个历史，确认清理生效
gitleaks protect --staged   # pre-commit 钩子里跑

JS 生态分层：规范 / 引擎 / 运行时 / 类型

JavaScript 生态混乱的根源是没分清三件事：写规范的、实现引擎的、做宿主运行时的。再叠加 TypeScript 这个外挂类型层，就是常被混为一谈的”JS”。把四层拆开，整个服务端 JS 世界就不再奇怪。

规范层：ECMAScript（ECMA-262）

由 TC39 维护（Ecma International 第 39 号技术委员会），每年 6 月发新版。规范定义语法、内置对象、执行模型，但不定义 DOM、fetch、localStorage、fs、http —— 这些归别处管：

ECMAScript          ← TC39          (语法 / Promise / Map / async)
DOM / fetch         ← WHATWG        (浏览器 API)
HTTP / TLS          ← IETF          (RFC)
fs / http / process ← Node OpenJS   (Node 自加，不是 Web 标准)

写 JS 99% 在调宿主 API（fetch / document / fs），其中只有少数真正是 ECMAScript。fetch 不是 ECMAScript —— 它是 WHATWG 的，Node / Bun “借”过来内置。

JS 这语言官方名是 ECMAScript 不是 JavaScript，因为 JavaScript 是 Sun（现 Oracle）的商标。Brendan Eich 自嘲过 “ECMAScript 听起来像皮肤病”，但这名字沿用了 30 年。

引擎层：V8 / JavaScriptCore / SpiderMonkey

不是”标准统一就只该有一个实现” —— 优化目标真的不同：

V8（Chrome / Edge / Node）JSC（Safari / Bun）SpiderMonkey（Firefox）
出身场景标签页常驻几小时页面打开就关平衡
优化目标长跑峰值启动 + 内存平衡
JIT 策略激进多层克制（保持低层便宜）保守
许可证BSD 3-ClauseLGPL 2.1MPL 2.0
治理Google 主导WebKit 联盟（Apple + Igalia + Sony）Mozilla 基金会

iOS 上所有浏览器都是 JSC —— Chrome / Edge / Firefox iOS 版只是 Safari 套了 UI 壳，被 App Store 强制用 WebKit。这条 2024 年起 EU DMA 法案才开始松动，但实际进展缓慢。

Edge 自 2020 年起 = V8 —— Microsoft 当年用的 Chakra 死了，Edge 重写为 Chromium。所以”V8 阵营”实际控制桌面 80%+。

许可证差异决定生态形状：V8 的 BSD 才让 Node / Deno / Electron / Cloudflare Workers 这些闭源 / 商业派生敢用；JSC 的 LGPL 允许嵌入式调用但不允许闭源修改引擎本身。这是 Bun 选 JSC 的非技术理由之一。

运行时层：Node / Bun / Deno / 浏览器

V8 只是会跑 JS，不会读文件、不会发 HTTP、不会监听端口。Node 把 V8 嵌进 C++ 程序，再补上：

Node = V8 (引擎)
     + libuv (事件循环 + 跨平台异步 I/O：Linux epoll / macOS kqueue / Windows IOCP)
     + 标准库 (fs / http / crypto / stream)
     + N-API (C++ addon 接口)
     + 模块系统 (CommonJS / ESM)

同一个 V8 嵌进不同宿主就是不同运行时：

引擎事件循环默认权限
ChromeV8Blink 自家沙盒受限
NodeV8libuv完整 OS 访问
DenoV8Tokio (Rust)默认无权限，按需授权
BunJSC自家（io_uring / kqueue 直调）完整 OS 访问
Cloudflare WorkersV8V8 isolate + 自家 scheduler沙盒

这就是 Node 不能”换引擎追上 Bun”的根因：N-API、所有 native addon、调试器协议、profile 工具全围绕 V8 设计。换 V8 等于重写 Node。Bun 走的是从零写所有胶水的路，代价是 95-98% npm 兼容率（剩下的 2-5% 用了 V8/libuv 内部细节的边角库）。

类型层：TypeScript

TS = JS 严格超集 + 类型注解。类型在编译后完全消失 —— .ts → .js 后类型一个字符都没了，运行时拿不到任何”这是 User 不是 string”的信息。这是为什么 instanceof 对 interface 没用：

// 写
function greet(user: User): string { return `Hi, ${user.name}` }

// 编译后实际跑的
function greet(user) { return `Hi, ${user.name}` }

TS 现在多渠道编译：tsc（官方，慢）/ esbuild（Go）/ swc（Rust）/ bun（直接跑无中间步骤）/ Node 22+ --experimental-strip-types。Microsoft 2025 年用 Go 重写了 tsc，叫 TypeScript Native，快 10×。

TC39 有 Type Annotations 提案让 JS 引擎直接忽略类型注解（注释化），TS 代码不用编译就能跑。Stage 1 卡了 4 年还没动 —— 浏览器引擎不愿为”不执行的注解”做语法解析。短期内不会落地。

写代码时叫 TypeScript、调试时叫 JavaScript、查规范时叫 ECMAScript、谈性能时谈 V8 / JSC、谈部署时谈 Node / Bun / Workers —— 同一件事的五个面，分清面才不被术语骗。

PHP libvips 写文字：Pango 接管 layout，不用算 bbox

GD 那一套 baseline / bbox / 减偏移 在 libvips 里不存在。Image::text() 走 Pango 做 layout —— 你给一个 width 框 + align 方向，结果 image 直接拿到 $txt->width / $txt->height，没有 baseline 这回事。

Pango 是什么

GNOME 项目下的文字 layout 库。渲染一段文字本来要分两步，libvips 把第一步外包给了它：

阶段做什么谁来做
Layout选字体、断行、字距、对齐、Bidi（阿拉伯/希伯来从右到左）、CJK 换行规则、emoji shapingPango
Rasterize把矢量字形画成像素FreeType / Cairo

GD 的 imagettftext 只做第二步 —— 调用方必须自己算宽高、自己拆行、自己处理对齐，所以才有 imagettfbbox 8 元素数组、baseline 减偏移那堆脏活。libvips 直接传 width: 800, align: 'high'，Pango 帮你折行 + 对齐，结果 image 的 width / height 就是排版后的真实尺寸 —— baseline 这个概念在调用方完全不出现。

额外 bonus：开 rgba: true 之后可以传 Pango markup，单段里混排粗体 / 颜色 / 字号：

$txt = Image::text(
    '<span foreground="red" weight="bold">警告</span> 内容超过限制',
    ['font' => 'Sans 20', 'rgba' => true]
);

GD 想做这个得渲染 N 次再拼。

实际用法：右下角水印

同样的场景，libvips 写法：

use Jcupitt\Vips\Image;

$bg  = Image::newFromFile('photo.jpg');
$txt = Image::text('© 2026', [
    'font'  => 'Sans Bold 24',
    'width' => $bg->width - 40,   // 框内自动 word-wrap
    'align' => 'high',
    'rgba'  => true,              // 不开是灰度 mask，没法合到彩图
]);

$bg->composite($txt, 'over', [
    'x' => 20,
    'y' => $bg->height - $txt->height - 20,
])->writeToFile('out.png');

GD 同场景要：imagettfbbox 拿 8 元素 → $bbox[2] - $bbox[0] 算宽 → $bg_w - $width - 20 算 x → 还要担心 baseline 跟图像 y 轴方向相反。libvips 一步：align: high。

Image::text() 关键 options（官方文档）：

Option作用默认
fontFontconfig 名，如 'Sans Bold 24' / 'Noto Serif CJK SC 18'—
fontfile加载非系统字体的 .ttf 路径，需配合 font 用名字引用—
width换行宽度（像素），超过按 word boundary 折行不换行
align'low' / 'centre' / 'high' —— 多行的水平对齐—
justify两端对齐（拉伸词间空格），boolfalse
spacing行距（像素）字体自然行距
dpi渲染 DPI，影响实际像素大小72
rgba出彩色 / emoji / Pango markup，不开是 8-bit 灰度 maskfalse

部署成本是 libvips 在 PHP 圈用得不如 GD 普及的主因 —— ext-gd 几乎随 PHP 自动装好，libvips 要：

# Debian/Ubuntu
apt install libvips42 libvips-dev

# macOS
brew install vips

# PHP 端二选一
pecl install vips                                # 传统 C extension
# 或开 FFI（PHP 7.4+），用 jcupitt/php-vips 包，无需编译扩展
composer require jcupitt/vips

容器里加一行 apt install libvips42 就够，但这一行常被忘。装上之后水印 / 缩略图 / 多行排版量大时，libvips 的内存占用和吞吐都比 GD 强一个数量级 —— 它是流水线架构，不像 GD 把整张图按解压尺寸驻留内存。

cloc / tokei / scc：codebase LOC 统计的三代工具

数代码行数（LOC by language）这件事有三代工具，每代换语言、换取舍：

cloctokeiscc
实现 / 起年Perl / 2006Rust / 2016Go / 2018
Linux kernel 用时~64s~1.4s~0.8s
默认并行❌（要 --processes=N）✓✓
安装brew install clocbrew install tokeibrew install scc
独有能力--diff 对比两个 commit；读 .zip / .tarJSON / YAML 输出复杂度 / COCOMO / 重复检测

scc 那三个名词各自的意思：

  • 复杂度：数代码里 if / for / && 这类分支关键字，估算 cyclomatic complexity。
  • COCOMO：1981 年 Barry Boehm 提出的成本模型，给 KLOC 估”工业平均要多少人月、多少钱”。
  • 重复检测：按文件 hash 判定，相同内容只算一次（monorepo / vendor 目录里常见，cloc / tokei 会重复计入）。

对同一个目录（这个 blog 的 til/src/）跑默认命令，差异自己跳出来：

SUM:                  17            131            111           1429

$ tokei til/src/
 Language         Files        Lines         Code     Comments       Blanks
 Astro               10         1519         1344           59          116
 TypeScript           7          152           97           44           11
 Total               17         1671         1441          103          127

$ scc til/src/

proto versions node               # 看本地装了哪些 + 远端可装版本

项目级版本：.prototools

一文件管所有 runtime，应该提交进 git（团队共享工具链版本，类比 composer.json 锁版本但管的是工具链本身）：

node = "20"

--to 三档路径：

flag写到用途
--to local（默认）./.prototools项目级，提交进 git
--to user~/.prototools用户级，覆盖 global
--to global~/.proto/.prototools系统级默认

Dockerfile 中三组易混指令怎么选

Dockerfile 里有几组指令名字相近、用途易混。按”什么时候执行 + 写到哪里”两条轴归位即可：

  • 构建时执行（写进 image 层）：RUN、COPY、ADD、ARG
  • 运行时执行（启动 container 时跑）：CMD、ENTRYPOINT
  • 构建期声明、运行时仍可见：ENV
  • 只是元数据（不执行也不影响行为）：LABEL

COPY vs ADD

两者都把构建上下文里的文件复制进 image，但 ADD 多了两个隐式行为：

  • 源是 .tar / .tar.gz 时自动解压到目标目录
  • 源是远程 URL 时自动下载

这两个 “智能” 行为正是 ADD 比 COPY 危险的地方——upstream tarball 内容变了，image 跟着变，且解压结果难审计。

口诀：默认用 COPY，确实需要解压本地归档时才上 ADD。远程文件用 RUN curl -O 更显式、更可控。

RUN vs CMD vs ENTRYPOINT

RUN 在构建时执行并提交一层。CMD 和 ENTRYPOINT 都只描述启动命令，区别在能不能被 docker run 的尾参覆盖：

指令何时跑默认参数？被尾参覆盖？
RUN构建时——
ENTRYPOINT启动时否不会（除非 --entrypoint）
CMD启动时是会

两种写法行为不同：

ENTRYPOINT ["/scripts/entrypoint.sh"]    # exec form
CMD "/scripts/start-fpm.sh"              # shell form

exec form（JSON 数组）= 直接 execve，进程是 PID 1，能正常收到 SIGTERM。 shell form（裸字符串）= Docker 包成 /bin/sh -c "..."，再整串作为参数传给 ENTRYPOINT，最终启动等价于：

/scripts/entrypoint.sh /bin/sh -c "/scripts/start-fpm.sh"

只有 CMD ["nginx"] 没 ENTRYPOINT 时，CMD 自己就是启动命令；docker run img bash 会把 nginx 整串替换成 bash。

口诀：主程序用 ENTRYPOINT exec form；可被覆盖的默认参数走 CMD exec form。两个都用 shell form 会丢掉信号转发，容器收不到 docker stop 的 SIGTERM，10 秒后被 SIGKILL 强杀。

LABEL vs ARG vs ENV

三者都是 “声明值”，但生命周期完全不同：

指令构建时可见？运行时可见？写进 image？
ARG是否否（但值会出现在 docker history，别塞密钥）
ENV是（声明之后的层）是（容器环境变量）是
LABEL否否是（仅元数据，docker inspect 可见）

ARG 还有一条作用域陷阱：FROM 之前声明的 ARG 只能在 FROM 行里展开，要在后续指令里用同名 ARG，得在 FROM 之后再声明一次：

ARG PHP_VERSION=8.3
FROM php:${PHP_VERSION}-fpm    # OK，这里能用

ARG PHP_VERSION                # 必须重新声明，下面才能用
RUN echo "Building PHP ${PHP_VERSION}"

口诀：版本号 / 镜像 tag 用 ARG；运行时配置（端口、路径、非密钥配置）用 ENV；作者 / VCS 地址 / 镜像描述用 LABEL。

Laradock 偷师：6 个值得抄走的 Docker 工程模式

Laradock 是 PHP/Laravel 的 docker-compose 全家桶模板。近年活跃度下降，新 Laravel 项目更多用 Sail。但它的 Dockerfile / compose 配置里有些经过多年打磨的模式，值得偷师——以下 6 条不限 PHP 场景。

1. PUID/PGID：容器用户对齐宿主，否则 bind mount 文件全是 root

ARG PUID=1000
ARG PGID=1000
RUN groupadd -g ${PGID} laradock && \
    useradd -l -u ${PUID} -g laradock -m laradock

容器默认 root 跑，写到 bind mount 的文件宿主端归 root 所有——开发者下次 vim 撞 permission denied，rm 要 sudo。把 host 的 id -u / id -g 通过 .env 注入 ARG，容器内建同 uid 用户，文件落地就是宿主用户。Linux dev 必踩；Mac/Win 因 Docker Desktop 自动做 uid 映射，不会感受到这个坑。

2. DEBIAN_FRONTEND=noninteractive 写 RUN 不写 ENV

# 不要：会污染容器运行时
ENV DEBIAN_FRONTEND=noninteractive

# 这样：只作用于该层 build
RUN DEBIAN_FRONTEND=noninteractive apt-get install -y tzdata

ENV 写进的是镜像层、运行时也生效——之后 docker exec 进容器手动装东西时，所有交互配置询问也被跳过（默认值生效），可能错过冲突选项或时区选择。RUN 内赋值只对那一条命令生效，构建期免提示、运行期保留交互。Docker 官方 Dockerfile best practices · ENV 同此建议。

3. extra_hosts: dockerhost 把”容器访问宿主”统一掉

extra_hosts:
  - "dockerhost:${DOCKER_HOST_IP}"
  # Linux 20.10+ 现在还可以这样：
  - "host.docker.internal:host-gateway"

Mac/Win Docker Desktop 自带 host.docker.internal，Linux 长期没有（20.10 才支持 host-gateway magic value）。Laradock 给容器塞统一别名 dockerhost——Xdebug 回连 IDE 时永远写 xdebug.client_host=dockerhost，跨平台共用一份 ini。

4. ARG INSTALL_XXX=false 的 opt-in 扩展矩阵

ARG INSTALL_XDEBUG=false
RUN if [ ${INSTALL_XDEBUG} = true ]; then \
    pecl install xdebug && docker-php-ext-enable xdebug; \
fi
WORKSPACE_INSTALL_XDEBUG=true
PHP_FPM_INSTALL_XDEBUG=true

一份 Dockerfile 适配 N 个项目：要 swoole 的项目 INSTALL_SWOOLE=true，不要的 false——不用为每个项目 fork 一份 Dockerfile。代价：build cache 粒度变粗（任一 ARG 变全层失效），高频切换的扩展（xdebug）值得拆独立 stage 减小重建范围。

5. docker-php-ext-disable 切扩展不重编译

# 生产关 xdebug，.so 文件仍保留
docker-php-ext-disable xdebug pcov

# 开发再 enable
docker-php-ext-enable xdebug

PHP 扩展本质是 .so + 一行 extension=xdebug.so ini 引用。docker-php-ext-disable 只移走 ini，.so 还在。开发/生产同一个镜像，不为”是否带 xdebug”重编两套——秒级切换。这条 official PHP image 自带，跟 Laradock 无关，但 Laradock 大量在用，是个被低估的工具。

6. db / cache 用命名 volume，不要 bind mount

services:
  mysql:
    volumes:
      - mysql:/var/lib/mysql            # 这样
      # - ./data/mysql:/var/lib/mysql   # 不要这样
volumes:
  mysql:

源码和数据卷分两种挂法 —— 源码要编辑器实时改，bind mount；数据要长期存活、和代码仓库脱钩，named volume。后者具体好在哪：

  • 误删难 —— rm -rf ./data 碰不到，得 docker volume rm 才删得了
  • 不污染 git —— bind mount 的 ./data/mysql/ 会进工作树，.gitignore 漏一次就把 ibdata 提进仓库
  • 备份 / 迁移有现成模式 —— named volume 跑一次 helper 容器就能打包；bind mount 还得另处理 owner
  • 权限不打架 —— 容器里 mysql uid 999 经常 chown 不动 host 目录；named volume 由容器自己初始化，没这层

规则：源码 bind mount，数据卷（mysql / postgres / redis / elasticsearch）走 named volume。

git 忽略文件权限位变更：core.filemode false

跨 OS 协作时（macOS / Linux ↔ WSL ↔ NTFS / FAT 网络盘），git 会把权限位变化（100755 ↔ 100644）当成内容变更，git status 满屏 mode-only diff 但其实没东西可改。

仓库局部关掉：

git config core.filemode false

core.filemode 不在版本控制里——它是 .git/config 的本地设置，不会同步给协作者（git 设计如此：权限策略是工作环境属性，不是仓库属性）。所以改完不用 commit。

陷阱：一刀切关掉后，你新加的可执行脚本权限不会被同事看到——CI 上跑 ./deploy.sh 直接 permission denied。要保留”真正的可执行位”只压”杂讯”，用：

git update-index --chmod=+x path/to/script.sh

显式把这个文件标记成 executable 写进 index，比关 filemode 精准。

git 建独立无历史的分支用 --orphan

git checkout --orphan 创建一个不指向任何已有 commit 的新分支——HEAD 移过去后 working tree 保留，但 staging 和 commit history 都从零开始。常用场景：一个 repo 同时管代码和文档站，文档站需要独立 history（如 gh-pages），或者发布开源版本时不想暴露内部 commit 史。

git checkout --orphan gh-pages
git rm -rf .                         # 关键：清掉父分支带过来的 working tree
echo "<h1>hi</h1>" > index.html
git add . && git commit -m "init gh-pages"
git push origin gh-pages

陷阱：--orphan 切过去后 git status 会显示一堆 staged 文件——那是从父分支继承的 working tree，没 git rm -rf . 就 commit 等于把整套源码塞进了所谓的”独立”分支。

查 CNAME：dig / host / nslookup 三选一

CNAME（canonical name）是 DNS 把一个域名指向另一个域名的别名记录。常见场景：www.example.com CNAME 到 CDN 边缘域名，或 SaaS 给你一个 <your>.heroku.com 让你 CNAME 过去。

dig example.com CNAME              # 完整记录
dig +short example.com CNAME       # 只要目标
dig +trace example.com             # 完整解析路径（root → TLD → 权威）

host -t CNAME example.com          # 输出最人话："is an alias for ..."

nslookup -type=CNAME example.com   # Windows 时代遗留，少用

CNAME 两条硬规则：

  • 不能和其他记录共存：一个 name 上有 CNAME 就不能再有 A / MX / TXT。所以根域名 example.com 通常不能用 CNAME（MX 在那里），子域才行
  • 链式解析：CNAME 可以指向另一个 CNAME，resolver 会一路 chase，最终必须落到 A/AAAA。链长了拖慢首次解析

挂在云厂商前的根域名查不出 CNAME 多半是因为对方用了 ANAME / ALIAS / flattening——根域 CNAME 限制的绕过手段，权威服务器替你预解析到 A 记录返回，外部看到的就是一条 A 而不是 CNAME。

CDPATH：让 cd 自动搜多个父目录

cd foo 默认只在当前目录找 foo/。CDPATH 给 cd 一份”备选父目录列表”，找不到就依次往里查：

export CDPATH=.:~/projects:~/work

之后无论在哪个目录：

cd blog          # 命中 ~/projects/blog/
cd client-x      # 命中 ~/work/client-x/

三个坑：

  • 第一项必须是 .——否则 cd subdir 会优先去 CDPATH 列表里搜，遇到同名目录就把你”传送”到别处
  • CDPATH 命中时 cd 会把目标路径打到 stdout，会污染脚本里 cd "$dir" && do-something 这类捕获 cd 输出的逻辑。脚本开头 unset CDPATH，并用绝对路径不依赖它
  • 不要塞太多目录——顺序匹配，列表越长越容易撞同名；3-5 个就到头

适合场景：固定的几个 workspace 根目录（~/code、~/work、~/oss）。如果项目散得很（dotfiles 在 home、客户代码在 NAS、个人项目在 iCloud），用 zoxide 这种基于 frecency 的跳转工具更顺手。

删了文件 df 不掉：lsof | grep deleted 找仍持有的进程

经典症状：rm 掉一个大日志，df 看磁盘空间没变。

根因：unix 文件系统下 rm 做的只是 unlink——把目录项里的引用删掉。inode 真正释放要等到所有持有它的 file descriptor 都关闭。如果某个进程还开着这个文件（典型场景：app 把日志 open 后一直 append），inode 就一直挂在 fd 上不释放，磁盘空间也回不来。

定位是哪个进程：

lsof | grep deleted
# nginx  1234  www-data  3w  REG  8,1  524288000  131  /var/log/nginx/access.log (deleted)

第 7 列就是占的字节数，第 1-2 列是进程名 + PID。两条出路：

# A. reload / 重启持有进程，让它关 fd
nginx -s reopen                  # nginx 专门有这个信号
systemctl restart <svc>          # 通用兜底

# B. 源头改成 truncate 而不是 rm
truncate -s 0 /var/log/access.log    # 文件保留，内容清空，进程继续写同一 inode
> /var/log/access.log                # 同效果，shell 内置

别用 echo "" > file——会写一个换行符进去，不是真清空。

预防：日志类场景上 logrotate 并开 copytruncate 选项，它就是为了解决 rotate 时进程仍持有老 fd 的问题。

shell job control：& nohup Ctrl+Z fg bg jobs

把命令丢后台、再拽回前台的全套手势。

& 直接后台跑，但关 terminal 时会被 SIGHUP 打死——shell 退出会向自己所有子进程发 HUP。nohup 的作用就是让进程忽略 HUP，于是关 terminal 也不死：

cmd &                            # 后台跑，关终端就没了
nohup cmd &                      # 关终端不死，stdout 默认写 nohup.out
nohup cmd > out.log 2>&1 &       # 自己接管输出

已经在前台跑的命令想丢后台——按 Ctrl+Z 暂停（SIGTSTP），shell 把它放进 jobs 列表，然后 bg 让它在后台继续跑：

$ long-running-cmd
^Z
[1]+ Stopped  long-running-cmd
$ bg                             # 继续在后台跑
$ jobs -l                        # 看 jobs 列表（带 PID）
[1]+ 12345 Running   long-running-cmd &
$ fg                             # 拽回前台

多个 job 时用 %N 选：

fg %1                            # 拽 job 1 回前台
fg %vim                          # 按命令名匹配
bg %2                            # 让 job 2 在后台继续
kill %1                          # 杀 job 1

nohup 只挡 HUP，不解决长跑任务的”我下次想接回 session 看实时输出”。生产场景认真用 tmux / screen / systemd unit，nohup 是临时手段。

2022-07-28 #shell

shell 脚本必加 set -euo pipefail

bash 默认行为对脚本极不友好——命令失败默认继续往下跑、未定义变量默认空串、管道里前面命令失败被后面成功的命令掩盖。每个长一点的脚本开头都该加：

#!/usr/bin/env bash
set -euo pipefail

各位含义：

  • -e（errexit）：任一命令非零退出立即终止脚本。陷阱：在 || / && / if / while 条件里的命令不算
  • -u（nounset）：用未定义变量直接报错。把 rm -rf /$UNDEFINED/ 这种灾难挡在执行前。可能未定义的变量用 ${VAR:-default} 显式给默认值
  • -o pipefail：默认 cmd1 | cmd2 的退出码只看 cmd2——curl bad-url | jq . 即便 curl 失败也返 0。开 pipefail 后任一段失败整个 pipeline 就失败
  • -x（xtrace）：执行前打印每条命令。debug 用，生产脚本别开——会在日志里泄露 password / token

不开 errexit 时的手动错误处理：

cmd || { echo "cmd failed: $?" >&2; exit 1; }

if ! cmd; then
  echo "cmd failed" >&2
  exit 1
fi

errexit 不是银弹——在 pipeline、command substitution、subshell 等场景下都有边界，set -e 会”看起来开了实际没生效”。强约束脚本（部署 / 数据迁移）建议直接上 shellcheck 做静态检查，比 set -e 兜底更可靠。

  fdate     Date,
  year      Int,
  month     Int,
  category1 String,
  category2 String,
  income    Double
);

  ('2020-02-01', 2020, 2, '3C', '电脑', 1012),

PHP __invoke：实例 callable，相当于带状态闭包

__invoke 是 PHP 的 magic method —— 类里定义了它，实例就能当函数直接调用：

class Adder {
    public function __construct(private int $base) {}
    public function __invoke(int $x): int { return $this->base + $x; }
}

$add5 = new Adder(5);
$add5(3);                     // 8
is_callable($add5);           // true

价值不在”语法甜”，在承载状态的 closure。原生 closure 用 use ($base) 捕获变量；__invoke 用 __construct 注入依赖、$this 持有任意复杂状态，还能继承、被 mock、实现接口。

单动作 controller（Laravel）

路由绑类不绑方法，文件 = 端点。一个 controller 做一件事，不再有 8 个 action 挤一个 class：

class ShowProfile extends Controller
{
    public function __invoke(int $id)
    {
        return view('user.profile', ['user' => User::findOrFail($id)]);
    }
}

Route::get('user/{id}', ShowProfile::class);

文档：Single Action Controllers | Laravel。

策略 / provider 对象

需要”可配置的函数”时 —— 查表、解析、路由匹配 —— 比 closure 更适合，因为有完整 OOP toolkit 可用。AWS SDK 的 PatternEndpointProvider 就是这个套路：构造时吃配置 patterns，调用时按 service/region 查规则、返回 endpoint：

$provider = new PatternEndpointProvider($patterns);
$endpoint = $provider(['service' => 's3', 'region' => 'us-east-1']);

调用方不必知道里面是查表、远程 fetch 还是 ML 推理 —— 签名永远是”参数 in, 结果 out”。换实现不动 caller。

一个坑：callable ≠ Closure

is_callable($obj) 返回 true，但 $obj instanceof Closure 是 false。type-hint 写 Closure 会拒掉 __invoke-able 对象，写 callable 才接受：

function run(callable $fn) { return $fn(1); }     // ✅ 接受 Adder 实例
function run(Closure $fn)  { return $fn(1); }     // ❌ TypeError

要把 __invoke 对象转成真正的 Closure 实例（比如某 API 强制 Closure 入参），用：

$closure = Closure::fromCallable($add5);  // PHP 7.1+
$closure = $add5(...);                    // PHP 8.1+ first-class callable

PHP strict_types 看调用方，不看声明方

declare(strict_types=1) 关掉 PHP 的隐式类型转换 —— 函数 type-hint 写 int $x，不再吞 1.0 / "15" / true，传错直接 TypeError：

declare(strict_types=1);
function add(int $a, int $b): int { return $a + $b; }
add(1.0, 2.0);
// PHP Fatal error: Uncaught TypeError: add(): Argument #1 ($a) must be of type int, float given

不写 declare 行（默认 weak mode）时 add(1.0, 2.0) 返回 int(3)，安静吞掉。

反直觉点：strict 看 caller，不看 callee

文件级声明会让人下意识以为”哪边定义函数 / 类，哪边的 mode 控制类型检查”。反了：检查谁，看 call site 所在文件的 declare。

A.php 严格 / B.php 默认：

// A.php
declare(strict_types=1);
function add(int $a, int $b): int { return $a + $b; }

// B.php
require 'A.php';
add(1.0, 2.0);  // ✅ int(3) —— B 没声明 strict，照旧 coerce

A.php 默认 / B.php 严格：

// A.php
function add(int $a, int $b): int { return $a + $b; }

// B.php
declare(strict_types=1);
require 'A.php';
add(1.0, 2.0);  // ❌ TypeError —— B 声明了 strict，调用按 strict 检查

实际后果：第三方库无论自家标了多少 strict_types，决定你调它会不会抛 TypeError 的是你自己文件顶部那行。Composer 装回来的 vendor/ 大概率是 strict，但你在 weak 文件里传 1.0 给它的 int 参数，PHP 不会拦。

要”全项目 strict”，每个 .php 顶部都得写一遍。靠 php-cs-fixer 的 declare_strict_types rule 或 symplify/easy-coding-standard 自动补，别手写。

写法约束

必须是脚本里最早的语句（<?php 后第一行，前面不能有 use / 注释外的任何内容）：

PHP Fatal error: strict_types declaration must be the very first statement in the script

不能用 block 形式：

declare(strict_types=1) { ... }   // ❌ must not use block mode

declare 行只作用于当前文件，不会污染 require / include 进来的脚本 —— 这正是允许逐文件渐进迁移的设计。

2021-01-29 #mysql

SQL AND / OR 执行优先级

AND 优先级高于 OR，所有 SQL 方言通用：

SELECT id FROM t WHERE c1 OR c2 AND c3;

-- 实际等价于（AND 先绑定）：
SELECT id FROM t WHERE c1 OR (c2 AND c3);

-- 不是这样（很多人下意识这么想）：
SELECT id FROM t WHERE (c1 OR c2) AND c3;

口诀：AND 看成 *，OR 看成 +，乘号先算。

Bug 不会报错——query 跑成功，只是结果错——所以 WHERE 里只要 AND 和 OR 混用，永远写括号，别赌读者（或几个月后的自己）记得优先级。
