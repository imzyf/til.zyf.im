caffeinate：macOS 临时防睡眠

2026-05-25 #shell#macos

macOS 自带 /usr/bin/caffeinate，临时阻止系统进入睡眠。最常见用法是给一条长任务套一层 caffeinate，命令跑完它自己退：

caffeinate -i ./long-task.sh        # long-task.sh 跑完，防睡眠也撤销

核心设计：caffeinate 的生命周期跟着另一个进程走。子命令结束它跟着退，不会忘关导致整天不睡。

flag 阻止不同层级的「睡」

flag阻止的层级
-isystem idle sleep（默认，最常用）
-ddisplay sleep（屏幕睡）
-mdisk idle sleep（磁盘空闲休眠）
-ssystem sleep —— 仅在插电源时生效
-uuser-active assertion（声明用户活跃；屏幕关着会唤醒；默认 5s）
-t <秒>给 assertion 加上限，到点自动撤销
-w <pid>跟某 PID 命运绑，那 PID 退出就退

caffeinate 不带任何 flag 等同于 -i —— 很多教程不点明，以为「随便加 caffeinate」就万事大吉，结果屏幕照睡。需要屏幕也亮着（演示 / 仪表盘看板 / 长亮场景）就显式加 -d：

caffeinate -di

四种生命周期

对应四种调用形态：

caffeinate -i -t 7200          # 接下来 2 小时不空闲睡，到点自动退
caffeinate -i ./long-task.sh   # 跟着子进程走，子进程结束就退
caffeinate -i -w 12345         # 跟某已有 PID 命运绑，PID 退出就退
caffeinate -i                  # 永远不退，要 Ctrl+C

-w 是事后补救手段：长任务已经在跑了才想起套 caffeinate，不用 kill 重启，caffeinate -i -w "$!" 抓刚 bg 的任务，或 caffeinate -i -w $(pgrep -f long-task) 抓按名找到的 PID。

第四种是初学者最常踩的坑 —— 开了一个 caffeinate -i 然后 Ctrl+Z + bg（或一开始就 & 后台化），关掉那个终端 tab，zsh 默认 nohup on 不向后台任务转发 SIGHUP，进程留在后台，Mac 几天不睡。避免方式：永远用前三种，绝不裸跑。即使「先开着以防万一」也强制带 -t <秒> 给个上限。

怀疑有残留时：

pgrep -lf caffeinate     # 列出当前所有 caffeinate 进程及参数
pkill caffeinate         # 全部杀掉（确认没有正在用的）

-s 为什么只在插电源时生效

苹果的设计假设：电池模式下用户「确实希望它睡」，否则损耗续航。要在电池模式下也不睡，用 -i —— idle sleep 在两种电源模式都能挡住。-s 是写给「插电时跑 CI / build 别中途睡」这类场景的。

debug：谁在阻止 sleep

Mac 莫名不睡时，看当前所有 PowerManagement assertion：

pmset -g assertions

会列出每个进程持有的 assertion（PreventUserIdleSystemSleep、PreventUserIdleDisplaySleep 等）。caffeinate 在底层调用同一套 IOPMAssertion* API；Amphetamine 这类 GUI 工具、视频会议、多媒体播放也通过同样机制声明 assertion。pmset -g assertions 是排查根因的金钥匙。

跟 pmset / GUI 工具区分

  • caffeinate —— 临时声明，进程级，跟生命周期绑
  • pmset —— 改全局电源配置，持久（sudo pmset -a sleep 0 是核选项）
  • Amphetamine / Lungo —— GUI 包装的 assertion，本质同 caffeinate

man caffeinate 看完整文档。

tmux：detach 让进程独立于窗口

iTerm2 + tmux 不是二选一，是叠两层。iTerm2 画像素（terminal emulator），tmux 在里面再开一组逻辑终端（terminal multiplexer）—— 核心收益是 detach：关窗、断 SSH、合盖，里面的进程都不死，下次 tmux a 接回来。再加 session / window / pane 三层结构，一个工作区拆成多视图共存。

iTerm2 跟 tmux 各管什么

维度iTerm2tmux
角色terminal emulatorterminal multiplexer
实现画 PTY master 端的像素一个 server 进程持有多个 PTY pair
关掉窗口里面 zsh 收 SIGHUP 死client 断开，server 不受影响，PTY 和子进程继续跑
多窗口 / 分屏原生 tab / splitsession / window / pane 三层
多客户端 attach不支持同一 session 多客户端同时连（结对编程刚需）

tmux server 给每个 pane 自己 openpty() 拿一对 PTY，里面跑的 shell 以为自己面对真终端。client 跟 server 通过 socket（/tmp/tmux-501/default）通信 —— client 死了 server 不死，这是 detach 的根因。

三层结构

server
└── session "mywork"             ← detach / attach 的单位
    ├── window 0 "edit"           ← 一个 window 一个 tab
    │   ├── pane 0  vim
    │   └── pane 1  rg            ← 一个 window 内的 split
    └── window 1 "test"
        └── pane 0  pytest --watch

记忆口诀：session 是工作区，window 是 tab，pane 是 split。

4 个 session 命令

下面 mywork 是 session 名（自取），不是命令：

tmux new -s mywork              # 创建并 attach
tmux ls                         # 列出所有 session
tmux a -t mywork                # attach 到名为 mywork 的 session
tmux kill-session -t mywork     # 结束某个 session

tmux new-session -A -s mywork   # 幂等：没有就建，有就 attach

最后一条的 -A 是写脚本和 .zshrc 别名时的核心 —— 同一别名反复跑永远落到同一 session，不会因为重跑而创建第二个。

必记的 4 个 prefix 键

-CC 模式下切 window / pane / 滚屏 / 复制全被 iTerm2 native 接管（⌘ 系列），不再需要记几十个 tmux 按键。剩下这 4 个还得手动：

键作用
prefix + ddetach（最高频）
prefix + ?列所有按键（忘了直接查）
prefix + :进命令模式（什么都能干）
prefix + z当前 pane 全屏 / 还原（看长输出用）

默认 prefix 是 Ctrl-b，先按 prefix 松开，再按命令键，不是同时按。

不用 -CC（远端 SSH / Ghostty / Alacritty）时按键多用一些 —— 那就靠 prefix + ? 现场查，不必硬记。

detach 能扛哪些场景

tmux a -t mywork            # 合盖再开，session 状态完整
场景没 tmux有 tmux
⌘Q 关 iTerm进程收 SIGHUP 死继续跑
SSH 断线进程死继续跑
Mac 空闲睡眠全机 SIGSTOP 暂停全机 SIGSTOP 暂停（tmux 不解决，需 caffeinate）
Mac 重启进程必死进程必死，session 元信息靠 tmux-resurrect

Mac 重启时 tmux-resurrect 能再画一遍 window / pane 布局，但默认只恢复布局不重起程序 —— 除非 @resurrect-processes 配白名单（'~vim->vim *' 'ssh->ssh' 这种 pattern）。

iTerm2 + tmux -CC

iTerm2 独有的 -CC 控制模式：

tmux -CC new-session -A -s mywork

tmux 的 window / pane 被 iTerm2 渲染成原生 tab / split，复制粘贴、滚动、字体、tab 切换全走 iTerm2，detach 持久能力同时保留。肌肉记忆零迁移，是 iTerm2 用户的最佳形态。

长跑 agent 实战

Claude Code / Codex 跑大重构、批量测试改写，单次任务 30 分钟到几小时起步。关掉终端 / Mac 睡了，烧掉的 token 全废。配进 ~/.zshrc：

cc() { tmux new-session -A -s "cc-$(basename "$PWD")" -c "$PWD" "caffeinate -i claude"; }

每个项目目录一个独立 session，tmux ls 一眼看全。caffeinate -i 套在 tmux 内层 —— caffeinate 命脉跟 claude 进程绑，detach 不影响。反过来包错位（caffeinate -i tmux ...）会出问题：caffeinate 跟外层 tmux 命令绑，detach 后 caffeinate 退出，Mac 该睡照睡，session 里的 claude 被 SIGSTOP 暂停。

tmux 和 caffeinate 是两条正交的防线：tmux 让 agent 跟窗口脱钩，caffeinate 让 Mac 不进睡眠，缺一边都漏。agent 的对话上下文不在 tmux 这一层 —— 它存在 Claude Code 自己的 session 文件里（路径按 cwd 编码，如 ~/.claude/projects/-Users-you-myproj/），靠 --resume 接回。

选何时

screen 也能 detach + multi-attach（screen -x），但脚本化接口糙、配置默认值停留在 90 年代，agent 时代的工具链（mosh / overmind / 各家 AI CLI 的 tmux integration）几乎都只面向 tmux。

failure 与 /-jər/ 词尾

failure /ˈfeɪljər/ 的坑：拼写 -ure 容易被中文母语者读成 “your” 的尾音，实际是 /-jər/ —— 一个 /j/（y 滑音）紧跟 /ər/。

把它拆成 fail + y + er，重音在 fail。

字母 U / I 在词尾担任 /j/ 滑音

英语词尾 -ure / -ular / -liar 几乎都是这个套路：U 或 I 不发元音，担任 /j/ 滑音。

词尾发音例
-ure/jər/failure, figure /ˈfɪɡjər/
-ular/jələr/regular /ˈreɡjələr/, popular, particular
-liar/ljər/familiar /fəˈmɪljər/, peculiar /pɪˈkjuːljər/

念法套同一模式：fail-y-er、fig-y-er、reg-y-ler、fa-mil-y-er。

变体：-sure 词尾的 /ʒər/

measure  /ˈmeʒər/
pleasure /ˈpleʒər/
treasure /ˈtreʒər/
leisure  /ˈliːʒər/

U 仍然担任 /j/ 滑音，只是和前面的 s 融合（palatalization），合成一个浊擦音 /ʒ/，表面看不见 U 的痕迹了。

/ʒ/ 汉语里没有，最近似 热 / 日 / 人 的声母 /ʐ/ —— 把翘舌放平就是 /ʒ/。别和拼音的 zh 混：zh 是塞擦音 /ʈʂ/（先停后擦），跟 /ʒ/ 不是一类音。

fzf 一行做 SSH host picker

~/.ssh/config 已经把所有 host 写好了，fzf 把它当数据源就有了 picker，preview 窗口顺便显示选中 host 的完整配置：

s() {
  local config="${HOME}/.ssh/config"

  [[ -f "$config" ]] || { echo "no $config"; return 1; }
  command -v fzf >/dev/null || { echo "fzf not installed"; return 1; }

  local hosts
  hosts=$(awk 'tolower($1)=="host" && $2 !~ /\*/ {print $2}' "$config")
  [[ -z "$hosts" ]] && { echo "no hosts in $config"; return 1; }

  local host
  host=$(echo "$hosts" | fzf \
    --preview "awk '/^Host / && \$2 == \"{}\" {p=1; next} /^Host / {p=0} p' $config" \
    --preview-window 'right:50%')

  [[ -n "$host" ]] && ssh "$host"
}

跑 s → fzf 弹列表 → 选 → SSH 上。光标停某个 host 时右半屏实时显示该 host 的 HostName / User / Port / IdentityFile，连之前一眼确认是不是要那台。

几个细节：

  • 三个 fail-fast 检查（config 文件存在 / fzf 命令在 / awk 出非空），任一项不满足立刻报具体原因，不静默退出
  • awk tolower($1)=="host" 大小写不敏感识别（ssh config 本身就不区分），过滤 Host *.dev 这种通配符规则，$2 取 host 别名
  • preview 那条 awk 是个状态机：匹配到 Host <选中> 起 p=1 开始打印，遇到下一个 Host 行 p=0 停止——精确抓单段配置，比 grep -A 10 更准（不会跨到下一段）
  • preview 字符串的转义陷阱：外层用 "..." 让 fzf 展开 {} 和 shell 展开 $config，内层 \$2 / \" 转义是为了让 shell 把 $2 和 " 原样递给 awk
  • [[ -n "$host" ]] 空选保护，避免 ssh "" 报错

维护 ~/.ssh/config

picker 的数据源是 ~/.ssh/config，没这文件 picker 空。最小骨架：

Host blog
    HostName 1.2.3.4
    User deploy
    Port 22
    IdentityFile ~/.ssh/blog_ed25519

Host *.dev
    IdentityAgent ~/.bitwarden-ssh-agent.sock
    AddKeysToAgent yes
    IgnoreUnknown UseKeychain
    UseKeychain yes

几个不踩坑必知：

  • 文件权限必须 chmod 600 ~/.ssh/config，权限太松 ssh 直接拒绝读取
  • Host 一行可以多别名空格分隔：Host blog blog2 mysite 三个名字都连同一台
  • 行内 4-空格缩进非语法必须，但强烈建议——stanza 边界清晰，picker 的 awk 解析也稳
  • 项目 / 客户多时用 Include ~/.ssh/config.d/*.conf 拆文件（工作的、个人的、临时的分开）

Argon2id PHC 串拆解：salt 来源与 Base64 编码

PHP password_hash($pwd, PASSWORD_ARGON2ID) 吐出来这种东西：

$argon2id$v=19$m=65536,t=4,p=1$UlhrVTlVbVI2S08wa2szSQ$gGGk/8NtsxcSbFOxcSkWOSf3fFE/a2KbMEqRiTrFmes

这是 PHC string format —— 2013-2015 年那场 Password Hashing Competition 顺带产出的字符串格式规范（竞赛本身是算法选秀，Argon2 胜出后由 RFC 9106 标准化；格式规范是副产品，让所有现代密码 hash 都能用自包含字符串表达，便于跨算法升级 / 跨语言迁移）。6 段用 $ 分隔。

$<id>[$v=<version>][$<param>=<value>(,<param>=<value>)*][$<salt>[$<hash>]]
段值含义
算法argon2id变体（不是 argon2i / argon2d）
版本v=19Argon2 v1.3（0x13）
参数m=65536, t=4, p=1内存 64 MiB、迭代 4 轮、并行 1
SaltUlhrVTlVbVI2S08wa2szSQ22 字符 → 解码后 16 字节
HashgGGk/8NtsxcSbFOxcSkWOSf3fFE/a2KbMEqRiTrFmes43 字符 → 解码后 32 字节

Salt 必须来自 CSPRNG

CSPRNG = Cryptographically Secure Pseudo-Random Number Generator（密码学安全的伪随机数生成器）。判别标准：next-bit 不可预测 + 状态泄露不能反推历史输出 + seed 自身来自熵源。rand() / mt_rand() 都不满足。

password_hash() 内部路径：

password_hash()
  → php_password_make_salt()
    → php_random_bytes_silent(16)
      → getrandom(2) (Linux) / arc4random (macOS)
        → OS 熵池（硬件中断、键盘 / 鼠标时序、磁盘 I/O 抖动）

等价于：

$salt = random_bytes(16);

永远不要用以下函数生成 salt / token / key：

不安全替代
rand()random_int($min, $max)
mt_rand()random_int($min, $max)
uniqid()bin2hex(random_bytes(16))
md5(microtime())random_bytes(N)

mt_rand() 是 Mersenne Twister，速度快但状态可被反推 —— 拿到约 624 个连续输出就能完全恢复内部状态、预测后续所有值。已有 php_mt_seed 在毫秒级破解。

salt 不需要保密（和 hash 一起明文存数据库），它的作用是：

  1. 让相同密码产生不同 hash，rainbow table 失效
  2. 强制攻击者逐个密码爆破，不能批量

字节 ↔ 字符：编码的换算

字节是 8 bit 的存储单位；字符是表示单位。两者的转换由编码定义。

Base64 把每 3 字节 打包成 4 字符，因为 LCM(8, 6) = 24 bit：

[byte 1] [byte 2] [byte 3]      ← 24 bit
 6 bit  6 bit  6 bit  6 bit
[char 1][char 2][char 3][char 4]

每个 Base64 字符 6 bit（2⁶ = 64 种可能）。选 6 是因为可打印 ASCII 里挑得出 64 个绝对安全的字符（A-Z a-z 0-9 + /），再多就找不齐了。

三种常见编码膨胀率：

bit/字符字符表膨胀率16 字节 → 多少字符
Hex40-9 a-f2×32
Base646A-Z a-z 0-9 + /1.33×22~24
Base325A-Z 2-71.6×26~32

所以 PHC 串里 16 字节 salt → 22 字符；32 字节 hash → 43 字符。反推也成立：看见 32 hex 是 MD5 / UUID（16 字节），64 hex 是 SHA256（32 字节）。

PHC 的 Base64 不带 padding

标准 Base64 末尾会用 = 补齐到 4 的倍数：

16 字节 → 21.33 字符 → 22 字符 + `==` = 24 字符

PHC 格式故意去掉 =：

标准 Base64PHC Base64
16 字节 saltUlhrVTlVbVI2S08wa2szSQ==UlhrVTlVbVI2S08wa2szSQ

理由：= 在 URL 里要转义、在 shell 里是赋值符。padding 是冗余信息（长度本来就能反推），删了不丢数据。

PHP base64_decode() 两种都兼容。但 Python 严格：

import base64
base64.b64decode('UlhrVTlVbVI2S08wa2szSQ')     # binascii.Error
base64.b64decode('UlhrVTlVbVI2S08wa2szSQ==')   # OK

s = 'UlhrVTlVbVI2S08wa2szSQ'
base64.b64decode(s + '=' * (-len(s) % 4))      # 手动补齐才能解

bcrypt 的 “Base64” 不是标准 Base64

bcrypt hash 里也有看似 Base64 的字段，但字母表不同：

标准 Base64:  A B C D E F G H I J K ... + /
bcrypt:       . / A B C D E F G H I ... 8 9
              ↑ 起头就不一样

bcrypt 沿用 OpenBSD crypt_blowfish 的历史字母表，索引 0 是 . 不是 A。

直接 base64_decode($bcrypt_part) 得到完全错误的字节 —— silent failure，没有报错。要解 bcrypt 内部的 salt / hash 必须用专门的字母表映射，或者干脆别解（业务代码 password_verify() 就够了）。

主流 hash 格式的 salt 编码对比

放一起看，差异只在算法和字母表 —— salt 原始长度（16 字节 / 128 bit）和编码方式（base64 无 padding）三家全一致，这是密码学共识的”足够防 rainbow table”门槛：

Hash 格式原始 salt编码字母表编码后字符数
argon2id（PHC）16 字节标准 A-Z a-z 0-9 + /22
bcrypt16 字节私有 ./ A-Z a-z 0-922
scrypt / PBKDF2（PHC）16 字节标准22

→ 这也解释了为什么把 bcrypt hash 喂给 base64_decode() silent failure：算法是 base64，但字母表对不上，输出字节全是垃圾。

RFC 9106 (Argon2) / PHC string format spec / Password Hashing Competition 主页。

pgrep / pkill 替代 ps | grep

按进程名 / 命令行查 PID，默认手势是 pgrep，不是 ps | grep。两个老问题 pgrep 一次性修了。

1. 自匹配陷阱

ps aux | grep node
# zhaoyifan  9012  ...  node server.js
# zhaoyifan 61234  ...  grep node                   ← grep 自己也被列出来

grep node 这条命令行本身就含 node，会匹配到自己。古早绕过手段是 grep '[n]ode'——用字符类骗 grep 不匹配自己（字符串 [n]ode 在正则里展开成 node，但字面上不再包含 node 子串，所以 grep 自己那行不被匹配）。

更隐蔽的问题是假阳性：grep node 不光匹配命令叫 node 的，还匹配整行任何字段含 node 的——USER 叫 node 的、路径含 node-exporter 的、TTY 含 node 的统统中招。

pgrep 设计上按字段查（默认匹配 process name，-f 匹配完整命令行），不是 grep 整行——既排除自己的 PID，也不会被 USER / 路径里的同字符串拐进去：

pgrep -lf node
# 9012 node server.js
pgrep -x node            # 精确等于（process name 完全匹配，不含参数）

2. 脚本里的退出码语义

# pgrep —— 退出码即答案
if pgrep -f caffeinate > /dev/null; then ...

# ps | grep —— ps 永远 0，得靠 grep -q + 排除自己
if ps aux | grep -v grep | grep -q caffeinate; then ...

ps 的工作是「打印出来」不是「找有没有」，所以没匹配也算成功——退出码语义全靠后面的 grep 承担。

配套 pkill —— 一步杀

pkill -f caffeinate                  # 等价 kill $(pgrep -f caffeinate)

pgrep 和 pkill 共用同一套 flag 模型，会一个自动会另一个。

常用 flag

flag作用
-l输出 PID + 进程名（不加只有 PID）
-f匹配完整命令行而不止进程名——caffeinate -i -t 300 这种带参数的必须用 -f
-x精确匹配 process name（默认是子串匹配，没 -x 时 pgrep ode 也会命中 node）
-u <user>限定某用户（多用户机才有意义；单用户 Mac 上等同不加）

想看进程的 CPU / 内存 / start 时间，让 pgrep 出 PID、ps 取详情——把第 1 节那两个 grep bug 一起绕过：

ps -p "$(pgrep -d, -f caffeinate)" -o pid,ppid,user,%cpu,%mem,stat,start,command

pgrep -d, 输出逗号分隔的 PID 列表，ps -p 直接吃。ps 本身的列不写死——ps -axo pid,user,%cpu,... 也能自由挑列；pgrep + ps -p 的核心优势是省掉 | grep 这一步，而不是控列。

shellcheck —— 抓 set -e 漏过的坑

set -euo pipefail 是 runtime 早退；shellcheck 是 parse 时静态分析——抓 set -e 漏过的写法层 bug。部署 / 迁移 / CI 脚本两个都要上。

brew install shellcheck     # 或：sudo apt install shellcheck
shellcheck deploy.sh

set -e 漏过的 5 条

  • SC2155 local x=$(cmd) —— local / declare / export 自身返回 0，里头 $(cmd) 失败被吞，set -e 也救不了。修：先 local x 再 x=$(cmd)
  • SC2086 unquoted $VAR —— rm $FILE 在 FILE="a b" 时删两个文件，word splitting / pathname expansion 会拆开。默认 "$VAR"
  • SC2046 unquoted $(cmd) —— 同上但是命令替换：cd $(pwd) 路径含空格时炸
  • SC2068 $@ 没加引号 —— 含空格的参数被再次拆开（foo "a b" c 传进去变 a b c 三个）。修："$@"
  • SC2059 printf "$msg" —— $msg 含 % 时被当 format directive，输出错乱或 format string 注入。修：printf "%s" "$msg"

关掉警告

# 单行
# shellcheck disable=SC2155
local x=$(some_legit_command)

# 整文件（紧跟 shebang）
#!/usr/bin/env bash
# shellcheck disable=SC2086,SC2046

# ~/.shellcheckrc 或项目根 .shellcheckrc
disable=SC1090-SC1100

disable=all（v0.8+）关全部——通常意味着该重写。

dogfooding

软件工程黑话之一。直译”吃自己的狗粮”，指：

公司内部员工把自己开发的产品当日常工具用，靠真实使用暴露 QA / metrics 抓不到的体验问题。

更体面的同义说法是 drinking your own champagne。

1988 年微软内部一封题为 “Eating our own Dog Food” 的邮件（经理 Paul Maritz 发出）让这个说法在科技圈固定下来，沿用至今。

为什么 QA / 单元测试替代不了

QA 跟着 checklist 点一遍验功能存在；dogfooding 验长期真实使用下的体感 —— 慢、绕路、别扭、不直觉。这些既不会触发断言，也写不进测试用例。

行为算 dogfooding 吗
团队每天用自己写的内部 CRM✅
自己做金融 SaaS 但从不开账户跑投顾流程❌
QA 跟着用例点一遍❌（缺「真实工作场景 + 长期使用」）

失败模式：internal build ≠ public build

dogfooding 失效的经典姿势是：自己用的版本跟客户拿到的不是同一个。

Anthropic 在 2026-04-23 关于 Claude Code 质量退化的 postmortem 里坦白：员工用的是带内部 flag 的 test build，客户用的是 public build。其中一个缓存清理 bug 在 internal build 里被另一个并行实验掩盖了，员工 dogfooding 完全没暴露问题，客户端持续退化超过两周才定位。

改进措施第一条：让内部员工用 exact public build。

MySQL utf8mb4_0900_ai_ci 拼读

后缀是正交标志，按 <charset>_<Unicode 版本>_<accent>_<case> 拆读：

  • utf8mb4：字符集
  • 0900：Unicode 9.0 UCA（Unicode Collation Algorithm）
  • ai：accent insensitive，重音不敏感
  • ci：case insensitive，大小写不敏感

换后两位就拼出整个网格：_as_cs（严格区分）、_as_ci（保重音忽略大小写）、_bin（纯二进制）。

非显然之处在它真正按 UCA 处理 Unicode 等价，老的 utf8mb4_general_ci 全部按字节比，结果不可预测：

SELECT 'café' = 'cafe';   -- 1（_ai_ci 忽略重音；general_ci 为 0）
SELECT 'ß' = 'ss';         -- 1（UCA 定义为等价；general_ci 为 0）
SELECT 'Ａ' = 'A';         -- 1（全角 = 半角；general_ci 为 0）
SELECT '😀' < '😁';        -- 按 Unicode 9.0 顺序可预测；general_ci 不可预测

8.0 把默认从 utf8mb4_general_ci 换到这里就是为了这条「正确性」。Unicode 出新版时会跟进新 collation（如未来 utf8mb4_1400_ai_ci），版本号 pin 在 collation 名里，老 collation 永久保留，不会改语义破坏老库。

参考 Unicode Character Sets · MySQL 8 docs。

LLM CLI proxy：rtk 是命令字典而非算法

rtk 是夹在 AI 编码助手和 shell 之间的 Rust CLI proxy，宣称帮 LLM 省 60-90% 的 token。它做的是 context compression —— LLM 想跑 git status，rtk 把噪音过滤后再喂回去。看着像「魔法压缩算法」，拆开后发现根本不是。

核心：一个 CLI 工具一份手写 parser

rtk 没有通用压缩引擎。src/cmds/ 下按工具切文件：

  • git/git.rs —— git status / git diff / git push 等子命令
  • python/pytest_cmd.rs、ruff_cmd.rs、mypy_cmd.rs
  • rust/cargo_cmd.rs
  • cloud/container.rs —— docker
  • js/npm_cmd.rs、tsc_cmd.rs、vitest_cmd.rs、playwright_cmd.rs

约 45 个 parser 文件覆盖 100+ 子命令。每个文件用 match 分发子命令，里面写死「哪些字段是噪音、压缩后长什么样」—— 手写领域知识，不是算法推导。

官网讲的 filtering / grouping / truncation / dedup 也不是引擎，只是 4 个共享字符串处理函数（dedup_lines() / truncate_middle() 之类）。parser 各自挑用哪几个 —— 同一函数在 git diff 里删 progress 行，在 docker logs 里折叠重复行。

新加一种工具支持 = 提一个 PR 加一个 <tool>_cmd.rs。changelog 看得清清楚楚：v0.38.0 加 GitLab glab、v0.40.0 加 Gradle，一条一条加。

关键技巧：拿机器格式做解析

rtk 的稳定性靠一个简单决定 —— 不解析给人看的输出。常见命令几乎都有 --porcelain / --json / --message-format 之类的脚本契约，rtk 就走这条。

机器契约是工具维护者给脚本用的稳定 API，UI 文案改了不影响。逆向人类输出（彩色、对齐、本地化）则一升级就崩。rtk 选前者 —— 这是它能跟着 100+ 工具活下去的根因，比任何「压缩算法」都重要。

工作流（以 rtk git status 为例）

LLM 决定跑 git status
  ↓
Claude Code PreToolUse hook 拦截 Bash 调用
  ↓ 改写成 `rtk git status`
rtk 进程：
  1. fork 真命令但走机器格式：git status --porcelain=v2 --branch
  2. parser 切字段：branch / ahead-behind / modified / untracked
  3. 紧凑模板渲染成单行
  ↓
返给 LLM，LLM 完全不知道有中间层

为什么不选其他路

候选方案为什么 rtk 没用
gzip 通用压缩LLM 不解压，喂进去还是压缩字节
让 LLM 自己 summarize慢、贵、丢信息，summarize 本身吃 token
盲截前 N 行没语义，可能切掉关键 error 行；切到的还是不稳的人类格式
per-command parser<10ms、确定性、保关键信号 —— 代价是每个工具都得单独写

推论

  1. 护城河是命令字典而非算法。任何人能写 pytest_cmd.rs，但要覆盖 90% 日常命令、跟着每个工具升级（git 改 porcelain、pytest 升级 reporter）才是真功夫。
  2. 工具支持决定收益。rtk hugo、rtk astro 不存在就是裸过。
  3. 永远滞后于命令本身。git 改了 porcelain 格式得等 rtk 跟上。
  4. PreToolUse hook 才是 UX 关键。靠 prompt 教 LLM「记得用 rtk」不可靠；hook 把改写做在协议层，LLM 不感知。

state vs status

两个词中文都译”状态”，工程命名混用会被 review 反复打回。先记一句口诀：

state is what you ARE; status is where you ARE in a process.

判准两步：

  1. 取值是有限几个枚举标签吗？→ status
  2. 是描述对象当下完整情况的一坨数据吗？→ state

status 通常是 state 里的一个字段，不是同级概念。这一层嵌套关系是判断命名时最重要的视角。

最干净的演示：Kubernetes Pod

spec:      # desired state（期望）
  containers: [...]

status:    # 实际情况
  phase: Running           # ← 流程节点标签
  conditions: [...]
  containerStatuses: [...]

spec + status 一起构成 Pod 的完整 state。status.phase 是 state 内部的一个枚举字段。命名 / 查 API 文档时，先判断你在描述哪一层。

反例：GitHub PR 用 state 不用 status

GET /repos/:owner/:repo/pulls/:number
→ {
    "state": "open" | "closed",
    "merged": true | false,
    ...
  }

open / closed 明明是枚举标签，按规则该叫 status，但 GitHub API 字段就叫 state，merged 另起一个 boolean。

为什么？GitHub 把 PR 当成”一个对象当前是什么”在表达，不是把它当流程节点。语义优先于规则，不要死记。

常见场景速查

场景用 status用 state
HTTP200 OK、response.status_codeTCP 连接：ESTABLISHED / TIME_WAIT
数据库字段orders.status = 'paid'—
前端组件antd <Steps status="finish" />React useState({ count, loading })
CI/CDGitHub Actions job status: completedGitHub PR state: open（反例）
即时通讯用户可设 status: 🍵 Tea breakWebSocket readyState: OPEN
游戏—存档（整个世界快照）

易混 trap：OAuth 的 state 参数

/authorize?response_type=code&state=abc123xyz

这里的 state 不是本文讨论的”对象状态”，是 OAuth 2.0 防 CSRF 的随机 nonce（RFC 6749 §4.1.1），授权服务器回调时原样带回，客户端校验是否匹配。

遇到这个词别按”完整数据快照”理解，它跟 state machine 一点关系没有。

一句话决策

  • 数据库字段、订单 / 任务 / 用户的流程节点 → status
  • React 组件内部数据、连接 / 进程的运行时快照 → state
  • 卡在中间 → 看现有 codebase 怎么命名，跟它走，比”正确”重要

git worktree —— 一仓多目录

正在 feature 分支写代码，文件改了一半 —— 线上 bug 要紧急修。常规做法是 stash / WIP commit / 切分支 / 回来再 pop，这套动作越熟越说明 git 缺机制。git worktree 就是这个机制：一个仓库同时挂多个工作目录，每个目录一个 HEAD，共享同一份 .git。比 stash 干净，比 git clone 轻量（不复制 objects、不重新 fetch）。

工作树 ≠ 分支

  • 分支：一个指针（.git/refs/heads/<name>），仓库级元数据，几个字节
  • 工作树：一个目录 + 一个 HEAD。HEAD 可以 attached 到分支，也可以 detached 直接指向 commit

多棵工作树共享所有分支、objects、hooks。每棵 linked worktree 的 HEAD 独立存在 .git/worktrees/<name>/HEAD，主工作树的 HEAD 仍在 .git/HEAD。同一分支同一时刻只能被一棵工作树 checkout —— 否则两边 HEAD 互相覆盖、reflog 串台。这是设计上的保护。

推论：

  • 创建 worktree ≠ 创建分支：--detach 完全不动分支；复用已有分支也不创建
  • 删除 worktree ≠ 删除分支：worktree remove 只清目录和元数据，分支还在
  • 删分支前先 worktree list：被占用的分支删不掉，会报 error: Cannot delete branch 'X' checked out at '<path>'

三种 add 姿势

git worktree add -b feat/login ../login main       # 新建分支，基于 main
git worktree add ../review feat/login              # 复用已有分支
git worktree add --detach ../inspect origin/main   # 不开分支，detach 到某个 ref/commit

末位 <commit-ish> 接 branch / tag / SHA / HEAD~3 都行。其他子命令（list / remove / move / lock / prune / repair）按需查 git help worktree，日常 95% 是这三条 add + remove。

典型工作流

feature 写一半，hotfix 插队：

git worktree add -b hotfix/login ../hotfix main
cd ../hotfix
# 改代码、commit、push、PR
cd -
git worktree remove ../hotfix
git branch -d hotfix/login        # remove 不删分支，按需手动

Review 别人 PR：

git fetch origin pull/123/head:pr-123
git worktree add --detach ../pr-123 pr-123
# composer install / 跑测试 / 手测
git worktree remove ../pr-123 && git branch -D pr-123

AI agent 让 worktree 从奢侈变刚需

git 2.5（2015 年）就有 worktree，十年小众。AI agent 让”同时跑 N 个任务”从奢侈变日常 —— branch 是 git 给「异步协作的人」的礼物，worktree 是 git 给「并行执行的 agent」的礼物。

让 agent 在你正在工作的目录里改代码，风险是文件 race（A 改 Home.php 第 100 行同时 B 也改）、watcher 链式反应、dev server 端口抢占、test DB 互 truncate。worktree 把每棵任务关进独立目录，git 层面互不踩脚。但 worktree 只隔离 git 状态，周边资源（端口 / 测试 DB / vendor/ / node_modules/）得自己按 worktree 分。

Claude Code 内置 EnterWorktree 和 Agent(isolation: "worktree")，工作树落在 .claude/worktrees/<name>/，基于 origin/<default-branch>。Conductor / Cursor / Aider 底层同套机制。三条纪律：

  1. 一棵树一个任务 —— 否则 review 时分不清谁改了什么
  2. 完事立刻 remove —— 不清理 .claude/worktrees/ 堆几十个，磁盘和心智都崩
  3. 依赖各装各的 —— vendor/ node_modules/ 是文件不是 ref，不会跟着 worktree 同步

高频陷阱

现象原因 / 解
git branch -d feat/x 报 Cannot delete branch 'feat/x' checked out at '<path>'分支被某棵 worktree 占着。git worktree list 定位 → git worktree remove ../x 释放
手动 rm -rf ../x 后 git worktree list 仍列 ../x元数据残留在 .git/worktrees/x/。git worktree prune 清死链
想在两棵树同时 checkout main设计禁止。第二棵用 --detach main，或 -b <new> main 基于 main 开新分支
pre-commit / husky / lefthook 在某棵树改了，另一棵也变.git/hooks/ 是共享的，所有 worktree 共用一份 hook 配置。装一次即可，别多树重复 husky install

git help worktree 看完整 manpage，重点段是 DESCRIPTION 里 .git/worktrees/ 布局描述。

markdownlint × prettier：format 与 lint 分工

markdown 在工具链上有两条独立线：prettier 管”长什么样”（format），markdownlint 管”对不对”（lint）。共存方案不是把两者并列开就完事 —— 默认配是稳定的，但有一处 CJK 用户必踩的雷。

职责分工

维度prettiermarkdownlint
干什么format：表格对齐、空行、缩进、引号统一lint：规则告警（重复标题、缺 lang、emphasis 当标题、列表风格）
配置.prettierrc.json + .prettierignore.markdownlint-cli2.jsonc（自带 ignores / gitignore: true）
globs不能写在配置里，命令行 / Makefile 传配置里 globs 字段直接写
auto-fix全部规则（修一次到位）行内类（MD009/012/047 等）能修；表格类（MD055/056/058/060）整组不 fixable
互相感知零零

最后两行是关键 —— 决定了下面所有”为什么这么配”的判断。

默认值高度吻合，平时不打架

~/.config/markdownlint/base.jsonc 的关键规则（MD003 atx、MD004 dash、MD007 indent 2、MD035 ---、MD046 fenced、MD048 backtick、MD009/012/047 清理尾空格 / 多空行 / 末换行）和 prettier 默认逐项对齐。保存 → prettier format → markdownlint 跑完只剩 MD036 / MD024 / MD040 这类语义告警，没有 format 层冲突。

真踩雷：MD060 aligned + CJK 表格

prettier 按 east-asian-width 算 CJK 列宽（视觉对齐），MD060 按字符数算（字节对齐），两者在 CJK 表格上永远收敛不了 —— 把 ~/.config/markdownlint/base.jsonc 的 "MD060" 设为 { "style": "consistent" } 或 false 让给 prettier。

局部翻案：ignore 注释

  • prettier：HTML 注释里写指令 <!-- ... -->。单行 prettier-ignore 跳下一行；整段成对 prettier-ignore-start / prettier-ignore-end（保留 raw 表格 / 空格 / 折行）。
  • markdownlint：同上，按规则号开关。整段成对：markdownlint-disable MD013 MD056 起、markdownlint-enable 收；单行：markdownlint-disable-line / disable-next-line。不带规则号的 disable = 全部关，慎用。

Makefile 把两者串成一个入口

prettier 不接受配置里的 globs，只能在命令行和 markdownlint 的 globs 字段手动对齐；顺序固定 prettier 先、markdownlint 后 —— prettier 修完 format，markdownlint 剩下的就纯是语义告警，信噪比最高。

# Keep MD_GLOBS in sync with .markdownlint-cli2.jsonc#globs.
MD_GLOBS := 'content-til/**/*.md' '*.md'

lint:
  prettier --write $(MD_GLOBS)
  markdownlint-cli2 --fix

lint-check:
  prettier --check $(MD_GLOBS)
  markdownlint-cli2

经验法则

VS Code 两个扩展共存：esbenp.prettier-vscode 读 .prettierrc.json，DavidAnson.vscode-markdownlint 读 .markdownlint-cli2.jsonc，各自互不感知，配置不冲突。规则索引见 https://github.com/DavidAnson/markdownlint/blob/main/doc/Rules.md。

Agent vs Workflow：控制流在谁手里

把 LLM 包进应用的写法分两种：Workflow（你的代码画好流程图，LLM 只填某些节点）和 Agent（LLM 自己决定下一步调哪个工具、什么时候停）。两者长得很像 —— 都接输入、都调 LLM、都返结果。本质差异只有一条：下一步该干什么，是你的代码决定，还是 LLM 自己决定。这条线划在哪，决定了你在写哪种东西。

Anthropic 把两者合称 agentic systems，但坚持把 workflow 和 agent 分清楚 —— 它们的成本、延迟、可控性差着一个数量级。

控制流在谁手里

Workflow：控制流写死在你的代码里。

用户输入
  ↓
你的代码：调 LLM 提取意图
  ↓
你的代码：switch case 分发
  ↓
你的代码：调对应函数
  ↓
返回

LLM 只是流程里的一个 text-in / text-out 节点。流程图你画，模型填格子。

Agent：控制流在 LLM 手里。

用户目标
  ↓
LLM 决定：调哪个工具、给什么参数
  ↓
工具返回结果
  ↓
LLM 决定：要不要继续，继续就调下一个
  ↓
（循环，直到 LLM 说"完了"）

你的代码只负责执行 LLM 给的指令。每一步「下一步干嘛」由模型自己判断。

判准：把工作流画出来 —— 如果存在「LLM 决定要不要进下一步 / 进哪一步」的节点 = Agent；只有「LLM 帮我填空 / 分类 / 改写」的节点 = Workflow。

五件套：Model + Prompt + Tools + Memory + Orchestrator

Agent 的最小完备架构。少一个就跑不起来：

组件干嘛工程上最容易翻车的点
Model决策核心；要支持 Function Calling指令遵循 / 工具选择准确率
Promptsystem prompt + tool 描述 + few-shot工具描述写糊 → Agent 调错或调不出
Tools真正改变外部世界的接口（读写文件、调 API、跑 SQL）工具粒度 / 错误信息可读性
Memory工作记忆（context）+ 短期（会话）+ 长期（跨会话）塞太多炸 context，塞太少失忆
Orchestrator主循环：把上面四个串起来跑最大轮数 / 超时 / 错误重试 / human-in-the-loop

OpenAI 的 Practical guide 只列三件套（Model / Tools / Instructions），把 Memory 和 Orchestrator 视作框架默认配置。从底层实现角度五件套缺一不可；从 API 使用者角度三件套是你能感知到的部分 —— 别被简化版误导以为不存在剩下两个。

反直觉的踩坑权重：大部分 Agent 做不好不是模型不够强，是 tool 描述写得稀烂 + memory 策略乱设计。换模型贵又简单，写描述便宜又关键。

ReAct 循环：Agent 的心跳

Thought → Action → Observation 三拍循环，是当前几乎所有 Agent 框架的内核：

┌─────────────────────────────────────────┐
│  Thought      —— 我现在该干嘛？          │
│     ↓                                    │
│  Action       —— 调 tool X，参数 {...}   │
│     ↓                                    │
│  Observation  —— 工具返回了 Y            │
│     ↓                                    │
│  （回到 Thought，直到决定 Finish）        │
└─────────────────────────────────────────┘

每一轮 Orchestrator 把 [目标 + 工具列表 + 历史 Thought/Action/Observation] 整包扔给 Model，Model 输出下一个 Thought + Action 或最终答案。模型本身无状态，所有「记忆」都靠 Orchestrator 把历史拼回 prompt 里 —— 这也是为什么 context window 总不够用，也是为什么 Agent 的 token 成本随轮数指数增长。

Claude Code 跑一句「把 User 的 email 字段改成 unique」内部大致就是：

Thought: 先定位 User 在哪
Action:  Grep "class User"
Observation: src/User.php:12

Thought: 看现在 email 怎么定义的
Action:  Read src/User.php
Observation: <文件内容>

Thought: 字段约束在 migration 不在 model，找 migration
Action:  Grep "users" 在 migration 目录
Observation: 2025_01_users_table.php

Thought: 写新 migration 加 unique
Action:  Write add_unique_to_users_email.php
Observation: ok

Thought: 跑测试确认
Action:  Bash 跑测试
Observation: 失败 —— 现有数据里有 3 行重复 email

Thought: 那要先去重再加约束，改方案
Action:  Edit migration，先 DELETE 重复行再 ALTER
...

每步「下一步该干嘛」全是模型根据上一步 Observation 自己决定的 —— 没人写死 if grep_success then read。报错也是模型自己看到 Observation 后调整策略，不是你代码 catch 的。Agent 与 Workflow 的本质差异，在终端里就长这样。

什么时候真的需要 Agent

Anthropic 和 OpenAI 在这点上罕见地一致：能用 Workflow 跑通的就不要上 Agent。控制流的不确定性是 Agent 所有成本和风险的来源 —— 慢、贵、不可预测。

场景选什么
用户提一个明确问题，答完就完直接调 LLM API
流程固定，只是某几步要 LLM 抽信息 / 总结 / 分类Workflow + LLM 节点（LangGraph 这类）
任务能拆但步骤不确定，路径依赖中间结果Agent
需要试错 / 自我纠错 / 调多个外部系统Agent
单步深推理（数学 / 代码）Reasoning model 直调，不需要 Agent

判准：「下一步该做什么」如果你能提前画清楚流程图，就不是 Agent 该解决的问题。

Reasoning model 别再写 CoT prompt

老的 CoT prompt 技巧（Let's think step by step、few-shot CoT 示例）在 reasoning model 上不仅多余，还会降低性能 —— OpenAI 在 o1 的 prompting guide 里明确说了这一点。原因不在”模型更聪明所以不需要”，而在 CoT 和 reasoning model 根本不是一回事。

CoT 是提示技巧，reasoning 是模型能力

维度CoT promptingReasoning model
来源inference 阶段的 in-context learningtraining 阶段的 RL（RLVR 等）
触发Let's think step by step / few-shot 示例默认就在思考，无需触发
思考过程普通输出 token，跟答案混在一起专门的 reasoning tokens，与最终答案分离
行为线性向前推，错了一路错下去backtrack、self-verify、试多条路径
能力天花板在已能做的任务上提准确率解锁了之前完全做不了的题

一句话：CoT 是”哄”出来的推理，reasoning model 是”烧”进权重里的推理。

老技巧为什么反而有害

模型已经有自己 RL 训出来的推理风格。prompt 层再叠一套等于污染：

  • Let's think step by step —— 纯噪音 tokens，模型本来就在 think
  • Few-shot CoT 示例 —— OpenAI 实测显示降低 o1 性能，因为示例会污染模型自有推理风格。注意区分：few-shot 本身 OK（给格式 / 领域示例没问题），few-shot CoT 才有害
  • “请仔细分析”、“深入思考”、“一步步来” —— 全是冗余 tokens，思考预算被消耗在”读你的废话”上
  • 角色扮演（“你是数学专家”）—— Kim 2024 实测在 12 个 reasoning 数据集里 7 个降低准确率

角色扮演翻车的机制是 persona 注入了和题目无关的 domain prior —— Kim 2024 的例子：一道数学题套了 civil engineering 故事壳，模型就把自己往土木工程师方向调，忽略了本质是道数学题。AQuA 上 13.78% 题因 persona 从对变错，Coin Flip 18%。匹配的 persona 也救不回来：Zheng 2024 测 in-domain persona（律师做法律题、医生做医学题）是 “minimal effect size”，期望收益约等于 0，且自动选 persona 的策略几乎打不过随机 —— 不存在”选对 persona”这件事。经验法则：匹配 persona 期望收益 ≈ 0，不匹配 < 0，默认别加。

新范式：减噪音胜过加技巧

OpenAI 官方建议是 “Keep prompts simple and direct” 和 “Avoid chain-of-thought prompts”。

老 prompt 风格（CoT 时代）：

You are an expert programmer. Let's think step by step.
First, analyze the problem. Then, consider edge cases.
Finally, write the code...

Reasoning model 时代：

Refactor this function to handle null inputs.

Constraints:
- Keep the existing public API
- Must pass tests in tests/UserTest.php
- PHP 8.2, no new dependencies

Return: the modified function only, no explanation.

短、直接、富约束、明确输出。把模型当聪明同事，不是当学生。

真正重要的维度

prompting 没消失，重心从”教它怎么想”变成”告诉它想什么、想多深、想完干嘛”：

  • 任务定义清晰度 —— 模型按字面意思深度推理，需求模糊 = 在错的方向上 deep think 几千 tokens
  • 约束 / 边界条件 —— 显式写”必须用 PHP 8.2”、“不能改 schema”，否则它会自由发挥
  • 输出格式 —— reasoning 完了的最终答案才是你要的，显式要 JSON / 表格 / 代码块
  • Thinking budget —— Claude / Gemini / GPT-5 都有显式参数控制思考长度，简单任务别让它想 30 秒
  • 上下文质量 —— garbage in, deeply-reasoned garbage out

品类正在消失

2025 下半年起，“是不是 reasoning model” 不再是模型品类，而是运行模式：

  • Claude Sonnet 4.6 / Opus 4.7：extended thinking 开关 + budget tokens
  • GPT-5：reasoning_effort 参数
  • Gemini 2.5：thinking_config
  • DeepSeek V3.x / R1 已合并为统一模型

未来不会再有”non-reasoning model”这个品类，只会有 thinking budget = 0 的运行模式。但小快省的 Haiku 类、端侧模型、embedding 这些纯非 reasoning 阵营，在长尾场景依然不可替代。
