MSA / SOW / SLA / DPA / NDA 区别

2026-06-01 #english#jargon

B2B SaaS 谈企业客户时反复出现的五个合同缩写。最常见的错是把它们当并列选项挑一个签——实际是分层的：一份主协议套着其余几份，各管一层。

口诀：MSA 定规则，SOW 定范围，SLA 定质量，DPA 定数据，NDA 定保密。

缩写全称管什么签几次
MSAMaster Service Agreement长期合作的法律框架一次（关系建立时）
SOWStatement of Work单个项目的范围 / 工期 / 报价 / 交付物每个项目一份
SLAService Level Agreement服务质量承诺 + 不达标的赔偿一次（嵌进 MSA 或 SOW）
DPAData Processing Agreement数据怎么存 / 删 / 出境 / 合规一次（常作 MSA 附件）
NDANon-Disclosure Agreement保密义务一次（常在 MSA 之前）

分层关系

MSA  主协议（法律框架，长期不变）
 ├── DPA  数据处理协议（附件 Exhibit）
 ├── SLA  服务等级（可在 MSA，也可下放到每份 SOW）
 ├── SOW #1  上线核心模块
 └── SOW #2  做 SSO / 数据对接定制

MSA 是顶层框架，DPA / SLA 通常作为它的附件（Exhibit）；SOW 挂在 MSA 下面，一个项目一份。所以听到 “we have an MSA with them” 基本等于”我们是签了长期合同的正式企业客户关系”。

MSA 和 SOW 为什么拆开

拆开的价值是谈判频率不同：付款、责任上限、保密、终止这类通用条款变化慢，写进 MSA 谈一次管几年；范围、工期、报价这类项目细节每次都变，每个项目单独走一份 SOW，不重谈主合同。后续加功能只追加新 SOW——这就是框架协议省时间的地方。

冲突时谁优先？合同里一般写明 MSA 优先，除非某份 SOW 明确声明覆盖某条。判别归属也靠这条：看到”通用、反复用”的归 MSA，看到”这个项目、这个价”的归 SOW。

DPA：谁是 controller，谁是 processor

金融 / 医疗 / 涉及 PII 的 SaaS 几乎必签 DPA，核心是把双方角色定死：

  • Data Controller（控制方）——决定数据用来干什么的一方，通常是客户。
  • Data Processor（处理方）——替控制方处理数据的一方，通常是 SaaS 厂商。

这个区分不是文字游戏：GDPR / CCPA 下两种角色的法律责任不同。DPA 还会写明子处理商（sub-processor，如厂商用的云）清单、数据保留期、合同结束后的删除义务、以及发生 breach 后多少小时内必须通知。

NDA 已被 MSA 的保密条款覆盖，为什么还单独签

时间点不同。NDA 常在还没决定合作时就签——评估、技术对接、看 demo 阶段就要交换敏感信息，而这时 MSA 远没谈成。MSA 里的保密条款只在合作期内、且范围限于这次合作；NDA 是更早、更独立的一道闸。

另一个区别是方向：NDA 可以是 mutual（双向） 或 one-way（单向），谁向谁披露、谁负保密义务可以不对称；MSA 的保密条款默认是双向且绑死在主合作上。

English Vowel Reduction: Why Spelling ≠ Pronunciation

英语的元音弱化（vowel reduction）规律：重音所在的音节，元音发得清楚；非重读音节的元音，几乎都退化成 /ə/（schwa，中央元音）或 /ɪ/。跟字母拼写无关，只跟重音位置有关。

同词根，重音移动，读音跟着变

派生词加后缀时，重音常常跳到别的音节，原来被重读的元音就”塌陷”成 schwa。

基础词音标派生词音标弱化的元音
au·thor/ˈɔːθər/au·thor·i·ty/əˈθɒrɪti/au /ɔː/ → /ə/
pho·to/ˈfoʊtoʊ/pho·tog·ra·phy/fəˈtɒɡrəfi/o /oʊ/ → /ə/
a·tom/ˈeɪtəm/a·tom·ic/əˈtɒmɪk/a /eɪ/ → /ə/
e·qual/ˈiːkwəl/e·qual·i·ty/ɪˈkwɒlɪti/e /iː/ → /ɪ/
te·le·graph/ˈtelɪɡrɑːf/te·leg·ra·phy/tɪˈlɛɡrəfi/e /ɛ/ → /ɪ/
Di·vine/dɪˈvaɪn/di·vin·i·ty/dɪˈvɪnɪti/i /aɪ/ → /ɪ/

介词、冠词、助动词——弱读形式是默认形式

这类词在句子中几乎永远是非重读的，英语母语者日常会自动用弱读形式，重读版反而显得刻意或强调。

词强读（单独说 / 强调时）弱读（句中默认）
and/ænd//ənd/ 或 /ən/
to/tuː//tə/
for/fɔːr//fər/
of/ɒv//əv/ 或 /ə/
can/kæn//kən/
have/hæv//həv/ 或 /əv/

“I’ll go to the store and get some for you.” 实际说出来是：
/aɪl ɡoʊ tə ðə stɔːr ən ɡɛt səm fər juː/
其中 to / and / for 全部弱化，这才是自然语速。

-tion / -ment / -ness 后缀——固定弱化模式

这类后缀本身就是非重读音节，o / e 恒弱化为 schwa，不因词根变化。

词音标弱化点
na·tion/ˈneɪʃən/-tion 中 o → /ə/
sta·tion/ˈsteɪʃən/同上
gov·ern·ment/ˈɡʌvərnmənt/-ment 中 e → /ə/
hap·pi·ness/ˈhæpinəs/-ness 中 e → /ə/

为什么会弱化

英语是 stress-timed 语言——重读音节之间的时间间隔大致相等，为了维持节奏，非重读音节必须”压缩”。Schwa /ə/ 是嘴巴最放松时发出的音，发音成本最低，所以成了弱化的终点。

这也是为什么英语听力难：母语者不是”说清楚每个字母”，而是在保持节奏的前提下把非重读部分尽量压缩。

实践口诀

重音在哪，元音在哪；离开重音，退化成 schwa。

背新词时，先标重音位置，再确认非重读音节哪些会弱化——比死记每个字母的读法省力得多。

Task：YAML 写的 Make 替代，自带 sh 解释器

Task 是 Go 写的 task runner，单二进制 + YAML 配置，定位”现代 Make 替代”。同代工具里 Just 名气更大，但两者其实不在一条线上 —— Task 真正的差异点是自带 POSIX sh 解释器和原生增量 / 并行依赖，这两点决定了它能当轻量构建系统用，Just 不能。

Make 真痛点不在 Make 本身

GNU Make 三大平台都有，跨平台坑出在 Makefile 里每条命令依赖的宿主 shell 和 coreutils：

  • Shell 方言：/bin/sh 在 Ubuntu 是 dash、在老 macOS 是 bash、Alpine 是 ash，[[ ]] / 数组 / echo -e 行为各不一样。本机 macOS 跑通的 Makefile 一进 CI 就 silently 行为漂移
  • Coreutils 分裂：macOS 自带 BSD 版 sed / find / xargs，GNU 习惯的 sed -i 's/x/y/g' f 在 BSD 上必须写成 sed -i '' 's/x/y/g' f（-i 必须吃一个空字符串参数）；GNU 的 find -printf / xargs -r / date -d 在 BSD 上整个不存在
  • Tab vs 空格：Makefile 命令行必须 Tab 开头，VS Code 默认设置把 Tab 转空格 → missing separator 红一片

要让一个 Makefile 跨 macOS / Linux 都跑通，迟早塞进 ifeq ($(shell uname),Darwin) 分支，维护成本爆炸。

Task 解法：bundled sh + 原生 incremental

Task 内嵌 mvdan/sh，一个纯 Go 实现的 POSIX shell 解释器。for f in src/*.go; do echo $f; done 不再走 /bin/sh，macOS / Linux 行为一致，dash vs bash 的”在我机器上能跑”问题消失。

边界要写清楚：Task 只 bundle 了 shell 语法，没 bundle coreutils。rm / sed / find 仍然调系统的那份 —— BSD sed -i '' vs GNU sed -i 的坑 Task 不替你解决，要么显式 gsed，要么改写成只用 POSIX 公共子集。这一点官方文档也提：“any executables called must be available as a built-in or in the system’s PATH.”

增量构建走 sources + generates：

version: '3'

tasks:
  build:
    cmds:
      - go build -o bin/app ./cmd/app
    sources:
      - ./**/*.go
    generates:
      - bin/app

  test:
    deps: [build]          # deps 列表默认并行
    cmds:
      - go test ./...
  • sources 里所有文件算 checksum（可切 method: timestamp），下次跑发现指纹没变就跳过该 task —— 大项目 task build 没改代码时几毫秒返回
  • deps: 默认并行执行（前提是 deps 之间互不依赖），串行用 cmds: 里 - task: foo 调用
  • 配合 task --watch 监听文件变化自动重跑

跟 Just 的分野

Just 和 Task 经常被并列推荐，但 Just README 自己写得很清楚：“just is a command runner, not a build system, so it avoids much of make’s complexity and idiosyncrasies.” —— 故意不做依赖图也不做增量，任务每次从头跑。

维度JustTask
定位command runner（不是 build system）task runner + 轻量 build tool
Shell走宿主 shell自带 mvdan/sh，跨平台一致
依赖并行串行默认并行
增量跳过无sources + generates checksum
--watch无内置
语法自定义 DSL（精简 Makefile 风）YAML
参数传递位置参数一等公民：just deploy prod走变量：task deploy ENV=prod

判准：

  • 项目里命令独立、每次都该完整跑（lint、deploy、clear-cache） → Just，手感更轻
  • 任务有产物链、改一个文件只想重建受影响下游 → Task
  • 命令需要传位置参数（just deploy prod） → Just 直观
  • 团队里 Makefile 已经在用 sources / .PHONY / -j 这些 Make 特性，想替换 → Task 一对一替

装：brew install go-task。文档：https://taskfile.dev/。

httpbingo.org —— 按需返回任意 HTTP 响应

写 HTTP 客户端时最难造的是失败：500、慢响应、3xx 链、basic auth 401。靠真实业务 API 不可控也污染数据。httpbingo.org 就是干这个——按 URL 告诉它要什么响应，它就回什么。

curl https://httpbingo.org/status/503         # 强制 503
curl https://httpbingo.org/delay/5            # 5 秒后响应（测超时）
curl https://httpbingo.org/redirect/3         # 3 次 302 链（测 follow 上限）
curl -u user:pass https://httpbingo.org/basic-auth/user/pass
curl https://httpbingo.org/anything           # 把请求 headers / body / query 原样 echo 回来

/anything 是 debug 神器——客户端实际发了什么 header、是不是按预期带了 cookie、proxy 有没有偷改 body，一行 curl 看到。

域名容易和老前辈 httpbin.org 混。httpbingo.org 是 mccutchen/go-httpbin 跑的公网实例，Kenneth Reitz 那个老 httpbin 的 Go 重写。老版自 v0.6.1（2017-08）后没再发版，README 还挂着 travis-ci.org 的 badge（travis 早已对 OSS 停服）。go-httpbin 零依赖、stdlib only、持续发版（v2.22 / 2026），多了 /sse /jsonl /websocket/echo，少了 /brotli（Go stdlib 不带）。

CI 里别直接打公网

免费服务限流 / 下线一次就 red 一片。CI 里起个 sidecar：

docker run -p 8080:8080 ghcr.io/mccutchen/go-httpbin

写 Go 的话还能直接当 library 嵌进单测起 httptest.Server，连 sidecar 都省了。

CONCAT 遇到 NULL：MySQL vs PostgreSQL

CONCAT(a, NULL, b) 在 MySQL 是 NULL，在 PostgreSQL 是 'ab'。默认行为相反，跨库写 SQL 极易踩。

-- MySQL 8.0
SELECT CONCAT('hello', NULL, 'world');  -- NULL

-- PostgreSQL 16
SELECT CONCAT('hello', NULL, 'world');  -- 'helloworld'

为什么相反

SQL 标准守三值逻辑：NULL 表示「未知」，未知和任何值拼接结果仍是未知，所以传染整个表达式。MySQL 的 CONCAT 按这条走；PostgreSQL 引入 CONCAT() 时刻意违反标准 —— 文档直接写 NULL arguments are ignored —— 理由是实战拼姓名 / 地址 / URL 时 NULL 传染几乎从来不是想要的行为。

|| 是 SQL 标准的字符串拼接操作符，两个 DB 都让 NULL 传染：

-- PostgreSQL
SELECT 'hello' || NULL || 'world';  -- NULL

MySQL 默认 || 是逻辑 OR，要 SET sql_mode = 'PIPES_AS_CONCAT' 才当拼接 —— 所以 MySQL 项目里基本看不到 || 拼字符串。

记口诀：CONCAT 跨库不一致，|| 跨库一致传染。

写法对照

写法MySQLPostgreSQL
CONCAT(a, NULL, b)NULL'ab'
CONCAT_WS(',', a, NULL, b)'a,b''a,b'
CONCAT_WS(NULL, 'a', 'b')NULLNULL
CONCAT_WS(',', 'a', '', 'b')'a,,b''a,,b'
a || NULL || b逻辑 ORNULL

CONCAT_WS 跨库一致：其余 NULL 自动跳过，但分隔符自己是 NULL 整表达式仍是 NULL、空串保留不当 NULL —— 想把空串也跳过，外层先 NULLIF(col, '')。

最常踩 CONCAT(first_name, ' ', middle_name, ' ', last_name) —— middle_name 为 NULL 时整个名字消失，前端只显示空白，业务侧不报错。改 CONCAT_WS(' ', first_name, middle_name, last_name) 立刻修好。

二进制安全 是什么意思

PHP 文档里 crypt() 顶上挂着一行：

Warning: This function is not (yet) binary safe!

binary safe 指函数能完整处理任意字节序列，含 NUL 字节（\0、0x00）。失安全的两种姿势：遇 \0 截断、结果随 locale 变化。

crypt() 怎么”不 binary safe”

底层是 C crypt(3)，C 字符串以 \0 终止：

$salt = '$2y$10$' . str_repeat('a', 22);
crypt("secret\0extra1", $salt) === crypt("secret\0extra2", $salt);  // true

C 端只读到 "secret"，\0 后的字节全部丢弃。把 random_bytes() 出来的 32 字节 key 当密码直接喂 crypt() / password_hash() 时这就是定时炸弹 —— 32 字节里含 \0 的概率是 1 - (255/256)^32 ≈ 12%，碰上就把高熵 key 蒸发掉一截。

修法是先编码再哈希：

$key = random_bytes(32);
password_hash(base64_encode($key), PASSWORD_BCRYPT);

更对路的做法：哈希二进制 key 别走 crypt 家族，用 hash_hmac('sha256', $msg, $key)，它本身 binary safe。

“(yet)” 是什么意思

不是 typo —— 是文档作者留的”想修但修不动”的尾巴。crypt() 走 libc，要 binary safe 得绕开 libc 自己实现一份，跨平台一致性破坏。从 PHP 4 挂到 PHP 8，括号里这两字大概会一直挂下去。

不 binary safe 分两类

类型典型函数触发
\0 截断crypt()、所有 fs 路径参数（fopen / include / file_get_contents）入参含 \0
locale 依赖strcoll字节落在 0x80-0xFF

路径那条历史上是 CVE-2006-7243 —— include $_GET['page'] . '.php' + 攻击者传 evil.so\0 能绕扩展名白名单包含任意文件。PHP 5.3.4 起 fs 函数拒收含 \0 的路径（PHP 8.0 起直接抛 TypeError），但拼接前先过用户输入仍是写代码的人的责任。

locale 那条更隐蔽 —— strcoll 按 LC_COLLATE 排序，同样两字节给相反答案：

setlocale(LC_COLLATE, 'C');
strcoll("\xE4", "z");                    // > 0（按字节 0xE4 > 0x7A）

setlocale(LC_COLLATE, 'de_DE.UTF-8');
strcoll("\xE4", "z");                    // < 0（德语排序 ä 紧挨 a，在 z 前）

同一段代码在 CI（默认 C）、生产容器（可能装了 de_DE.UTF-8）、本地 mac 跑出不同结果，debug 极难找。要可移植的国际化排序走 Collator（intl 扩展）显式给 locale；要大小写转换走 mb_strtolower($s, 'UTF-8') 明确指定编码 —— strtolower 自 PHP 8.2 起只处理 ASCII，对非 ASCII 字节原样返回。

哪些是 binary safe 的

strlen strpos strrpos substr str_replace strcmp strncmp explode implode hash hash_hmac fread fwrite md5 sha1 —— 核心字符串和 hash 族基本全是。判别口诀：函数文档说”以字节为单位”且不依赖 locale。

strcmp() 是 binary safe，但不是 constant-time —— 校验 token / HMAC / CSRF 用 hash_equals()，否则吃时序攻击。两个正交属性，别混。

mb_* 系列是多字节安全（认得 UTF-8 多字节序列），不等于”任意二进制安全” —— 喂非法 UTF-8 字节会报警或返回 false。

caffeinate：macOS 临时防睡眠

macOS 自带 /usr/bin/caffeinate，临时阻止系统进入睡眠。最常见用法是给一条长任务套一层 caffeinate，命令跑完它自己退：

caffeinate -i ./long-task.sh        # long-task.sh 跑完，防睡眠也撤销

核心设计：caffeinate 的生命周期跟着另一个进程走。子命令结束它跟着退，不会忘关导致整天不睡。

flag 阻止不同层级的「睡」

flag阻止的层级
-isystem idle sleep（默认，最常用）
-ddisplay sleep（屏幕睡）
-mdisk idle sleep（磁盘空闲休眠）
-ssystem sleep —— 仅在插电源时生效
-uuser-active assertion（声明用户活跃；屏幕关着会唤醒；默认 5s）
-t <秒>给 assertion 加上限，到点自动撤销
-w <pid>跟某 PID 命运绑，那 PID 退出就退

caffeinate 不带任何 flag 等同于 -i —— 很多教程不点明，以为「随便加 caffeinate」就万事大吉，结果屏幕照睡。需要屏幕也亮着（演示 / 仪表盘看板 / 长亮场景）就显式加 -d：

caffeinate -di

四种生命周期

对应四种调用形态：

caffeinate -i -t 7200          # 接下来 2 小时不空闲睡，到点自动退
caffeinate -i ./long-task.sh   # 跟着子进程走，子进程结束就退
caffeinate -i -w 12345         # 跟某已有 PID 命运绑，PID 退出就退
caffeinate -i                  # 永远不退，要 Ctrl+C

-w 是事后补救手段：长任务已经在跑了才想起套 caffeinate，不用 kill 重启，caffeinate -i -w "$!" 抓刚 bg 的任务，或 caffeinate -i -w $(pgrep -f long-task) 抓按名找到的 PID。

第四种是初学者最常踩的坑 —— 开了一个 caffeinate -i 然后 Ctrl+Z + bg（或一开始就 & 后台化），关掉那个终端 tab，zsh 默认 nohup on 不向后台任务转发 SIGHUP，进程留在后台，Mac 几天不睡。避免方式：永远用前三种，绝不裸跑。即使「先开着以防万一」也强制带 -t <秒> 给个上限。

怀疑有残留时：

pgrep -lf caffeinate     # 列出当前所有 caffeinate 进程及参数
pkill caffeinate         # 全部杀掉（确认没有正在用的）

-s 为什么只在插电源时生效

苹果的设计假设：电池模式下用户「确实希望它睡」，否则损耗续航。要在电池模式下也不睡，用 -i —— idle sleep 在两种电源模式都能挡住。-s 是写给「插电时跑 CI / build 别中途睡」这类场景的。

debug：谁在阻止 sleep

Mac 莫名不睡时，看当前所有 PowerManagement assertion：

pmset -g assertions

fzf 一行做 SSH host picker

~/.ssh/config 已经把所有 host 写好了，fzf 把它当数据源就有了 picker，preview 窗口顺便显示选中 host 的完整配置：

s() {
  local config="${HOME}/.ssh/config"

  [[ -f "$config" ]] || { echo "no $config"; return 1; }
  command -v fzf >/dev/null || { echo "fzf not installed"; return 1; }

  local hosts
  hosts=$(awk 'tolower($1)=="host" && $2 !~ /\*/ {print $2}' "$config")
  [[ -z "$hosts" ]] && { echo "no hosts in $config"; return 1; }

  local host
  host=$(echo "$hosts" | fzf \
    --preview "awk '/^Host / && \$2 == \"{}\" {p=1; next} /^Host / {p=0} p' $config" \
    --preview-window 'right:50%')

  [[ -n "$host" ]] && ssh "$host"
}

跑 s → fzf 弹列表 → 选 → SSH 上。光标停某个 host 时右半屏实时显示该 host 的 HostName / User / Port / IdentityFile，连之前一眼确认是不是要那台。

几个细节：

  • 三个 fail-fast 检查（config 文件存在 / fzf 命令在 / awk 出非空），任一项不满足立刻报具体原因，不静默退出
  • awk tolower($1)=="host" 大小写不敏感识别（ssh config 本身就不区分），过滤 Host *.dev 这种通配符规则，$2 取 host 别名
  • preview 那条 awk 是个状态机：匹配到 Host <选中> 起 p=1 开始打印，遇到下一个 Host 行 p=0 停止——精确抓单段配置，比 grep -A 10 更准（不会跨到下一段）
  • preview 字符串的转义陷阱：外层用 "..." 让 fzf 展开 {} 和 shell 展开 $config，内层 \$2 / \" 转义是为了让 shell 把 $2 和 " 原样递给 awk
  • [[ -n "$host" ]] 空选保护，避免 ssh "" 报错

维护 ~/.ssh/config

picker 的数据源是 ~/.ssh/config，没这文件 picker 空。最小骨架：

Host blog
    HostName 1.2.3.4
    User deploy
    Port 22
    IdentityFile ~/.ssh/blog_ed25519

Host *.dev
    IdentityAgent ~/.bitwarden-ssh-agent.sock
    AddKeysToAgent yes
    IgnoreUnknown UseKeychain
    UseKeychain yes

几个不踩坑必知：

  • 文件权限必须 chmod 600 ~/.ssh/config，权限太松 ssh 直接拒绝读取
  • Host 一行可以多别名空格分隔：Host blog blog2 mysite 三个名字都连同一台
  • 行内 4-空格缩进非语法必须，但强烈建议——stanza 边界清晰，picker 的 awk 解析也稳
  • 项目 / 客户多时用 Include ~/.ssh/config.d/*.conf 拆文件（工作的、个人的、临时的分开）

Argon2id PHC 串拆解：salt 来源与 Base64 编码

PHP password_hash($pwd, PASSWORD_ARGON2ID) 吐出来这种东西：

$argon2id$v=19$m=65536,t=4,p=1$UlhrVTlVbVI2S08wa2szSQ$gGGk/8NtsxcSbFOxcSkWOSf3fFE/a2KbMEqRiTrFmes

这是 PHC string format —— 2013-2015 年那场 Password Hashing Competition 顺带产出的字符串格式规范（竞赛本身是算法选秀，Argon2 胜出后由 RFC 9106 标准化；格式规范是副产品，让所有现代密码 hash 都能用自包含字符串表达，便于跨算法升级 / 跨语言迁移）。6 段用 $ 分隔。

$<id>[$v=<version>][$<param>=<value>(,<param>=<value>)*][$<salt>[$<hash>]]
段值含义
算法argon2id变体（不是 argon2i / argon2d）
版本v=19Argon2 v1.3（0x13）
参数m=65536, t=4, p=1内存 64 MiB、迭代 4 轮、并行 1
SaltUlhrVTlVbVI2S08wa2szSQ22 字符 → 解码后 16 字节
HashgGGk/8NtsxcSbFOxcSkWOSf3fFE/a2KbMEqRiTrFmes43 字符 → 解码后 32 字节

Salt 必须来自 CSPRNG

CSPRNG = Cryptographically Secure Pseudo-Random Number Generator（密码学安全的伪随机数生成器）。判别标准：next-bit 不可预测 + 状态泄露不能反推历史输出 + seed 自身来自熵源。rand() / mt_rand() 都不满足。

password_hash() 内部路径：

password_hash()
  → php_password_make_salt()
    → php_random_bytes_silent(16)
      → getrandom(2) (Linux) / arc4random (macOS)
        → OS 熵池（硬件中断、键盘 / 鼠标时序、磁盘 I/O 抖动）

等价于：

$salt = random_bytes(16);

永远不要用以下函数生成 salt / token / key：

不安全替代
rand()random_int($min, $max)
mt_rand()random_int($min, $max)
uniqid()bin2hex(random_bytes(16))
md5(microtime())random_bytes(N)

mt_rand() 是 Mersenne Twister，速度快但状态可被反推 —— 拿到约 624 个连续输出就能完全恢复内部状态、预测后续所有值。已有 php_mt_seed 在毫秒级破解。

salt 不需要保密（和 hash 一起明文存数据库），它的作用是：

  1. 让相同密码产生不同 hash，rainbow table 失效
  2. 强制攻击者逐个密码爆破，不能批量

字节 ↔ 字符：编码的换算

字节是 8 bit 的存储单位；字符是表示单位。两者的转换由编码定义。

为什么不能 1 字节 → 1 字符？ 一个字节有 256 种值（0255），但可打印 ASCII 只有 95 个（0x200x7E，即 128 个 7-bit ASCII 编码点扣掉 33 个控制符 —— 0x00~0x1F + 0x7F。控制符是 NUL / Tab / 换行 / Esc / DEL 等，不能透传文本，所以 128 根本不是候选档位）。1:1 至少缺 161 个对应。退一步：能不能扩到 256 字符？

候选方案为什么不行
全 256 ASCII0x00~0x1F 是 NUL / Tab / 换行 / Esc，终端、字符串、协议都会炸
Latin-1 全 256 字符UTF-8 时代字节 0x80~0xFF 不是合法 UTF-8 序列，存数据库 / 走 JSON / 进 URL 全报错
强行透传二进制URL、JSON、shell、HTTP header、email、.env、grep 输出全部假设 ASCII-safe

所以真正约束是：字符集必须落在「绝对安全的可打印 ASCII 子集」里。理论上界 95，实操还要扣掉 " \ < > & = + ? # 等在 URL / JSON / shell 里有特殊含义的字符，安全集合一般 60~80 个。各档编码的取舍：

字符集大小编码bit/字符膨胀率16 字节后
16Hex42×32 字符
32Base3251.6×26~32
64Base6461.33×22~24
85Base85 / Ascii856.41.25×20~21
95全可打印 ASCII 上界~6.571.22×20

Base64 字符表是 A-Z a-z 0-9 + /（62 个字母数字 + 2 个填空），「安全字符够多」+「6 bit 整齐对齐 24 bit 包」的双重最优解：再上一档要 128 字符，超出可打印 ASCII 上界；再下一档 32 字符膨胀率明显更高。

Base64 把每 3 字节 打包成 4 字符，因为 LCM(8, 6) = 24 bit —— 字节 8 bit 和 Base64 字符 6 bit 不能整除，得找两边都数得齐的最小公共单位。24 bit 既是 3 个字节，也是 4 个 Base64 字符：

[byte 1] [byte 2] [byte 3]      ← 24 bit
 6 bit  6 bit  6 bit  6 bit
[char 1][char 2][char 3][char 4]

试别的字节数都凑不齐：

输入字节bit÷ 6 = 字符问题
181.33余 2 bit，半字符存不下
2162.67余 4 bit，半字符存不下
3244整除
4325.33回到 8 bit 周期

所以 3 字节是「最小成块单位」，每 3 字节切一刀、吐 4 字符、循环。输入不是 3 的倍数时，结尾凑不齐 24 bit，要补 0 bit 到下一个 6 bit 边界，再用 = 标记「这是补的、不是真数据」 —— padding 的来历，下一节正好展开 PHC 为什么把 = 删了。

所以 PHC 串里 16 字节 salt → 22 字符；32 字节 hash → 43 字符。反推也成立：看见 32 hex 是 MD5 / UUID（16 字节），64 hex 是 SHA256（32 字节）。

PHC 的 Base64 不带 padding

标准 Base64 末尾会用 = 补齐到 4 的倍数：

16 字节 → 21.33 字符 → 22 字符 + `==` = 24 字符

PHC 格式故意去掉 =：

标准 Base64PHC Base64
16 字节 saltUlhrVTlVbVI2S08wa2szSQ==UlhrVTlVbVI2S08wa2szSQ

理由：= 在 URL 里要转义、在 shell 里是赋值符。padding 是冗余信息（长度本来就能反推），删了不丢数据。

PHP base64_decode() 两种都兼容。但 Python 严格：

import base64
base64.b64decode('UlhrVTlVbVI2S08wa2szSQ')     # binascii.Error
base64.b64decode('UlhrVTlVbVI2S08wa2szSQ==')   # OK

s = 'UlhrVTlVbVI2S08wa2szSQ'
base64.b64decode(s + '=' * (-len(s) % 4))      # 手动补齐才能解

bcrypt 的 “Base64” 不是标准 Base64

bcrypt hash 里也有看似 Base64 的字段，但字母表不同：

标准 Base64:  A B C D E F G H I J K ... + /
bcrypt:       . / A B C D E F G H I ... 8 9
              ↑ 起头就不一样

bcrypt 沿用 OpenBSD crypt_blowfish 的历史字母表，索引 0 是 . 不是 A。

直接 base64_decode($bcrypt_part) 得到完全错误的字节 —— silent failure，没有报错。要解 bcrypt 内部的 salt / hash 必须用专门的字母表映射，或者干脆别解（业务代码 password_verify() 就够了）。

放一起看，差异只在算法和字母表 —— salt 原始长度（16 字节 / 128 bit）和编码方式（base64 无 padding）三家全一致，这是密码学共识的”足够防 rainbow table”门槛：

Hash 格式原始 salt编码字母表编码后字符数
argon2id（PHC）16 字节标准 A-Z a-z 0-9 + /22
bcrypt16 字节私有 ./ A-Z a-z 0-922

pgrep / pkill 替代 ps | grep

按进程名 / 命令行查 PID，默认手势是 pgrep，不是 ps | grep。两个老问题 pgrep 一次性修了。

1. 自匹配陷阱

ps aux | grep node
# zhaoyifan  9012  ...  node server.js
# zhaoyifan 61234  ...  grep node                   ← grep 自己也被列出来

grep node 这条命令行本身就含 node，会匹配到自己。古早绕过手段是 grep '[n]ode'——用字符类骗 grep 不匹配自己（字符串 [n]ode 在正则里展开成 node，但字面上不再包含 node 子串，所以 grep 自己那行不被匹配）。

更隐蔽的问题是假阳性：grep node 不光匹配命令叫 node 的，还匹配整行任何字段含 node 的——USER 叫 node 的、路径含 node-exporter 的、TTY 含 node 的统统中招。

pgrep 设计上按字段查（默认匹配 process name，-f 匹配完整命令行），不是 grep 整行——既排除自己的 PID，也不会被 USER / 路径里的同字符串拐进去：

pgrep -lf node
# 9012 node server.js
pgrep -x node            # 精确等于（process name 完全匹配，不含参数）

2. 脚本里的退出码语义

# pgrep —— 退出码即答案
if pgrep -f caffeinate > /dev/null; then ...

# ps | grep —— ps 永远 0，得靠 grep -q + 排除自己
if ps aux | grep -v grep | grep -q caffeinate; then ...

ps 的工作是「打印出来」不是「找有没有」，所以没匹配也算成功——退出码语义全靠后面的 grep 承担。

配套 pkill —— 一步杀

pkill -f caffeinate                  # 等价 kill $(pgrep -f caffeinate)
pkill -9 -f long-running-task        # SIGKILL 强杀
pkill -HUP nginx                     # 发 SIGHUP 让 nginx reload 配置（不止于杀）

pgrep 和 pkill 共用同一套 flag 模型，会一个自动会另一个。

常用 flag

flag作用
-l输出 PID + 进程名（不加只有 PID）
-f匹配完整命令行而不止进程名——caffeinate -i -t 300 这种带参数的必须用 -f
-x精确匹配 process name（默认是子串匹配，没 -x 时 pgrep ode 也会命中 node）
-u <user>限定某用户（多用户机才有意义；单用户 Mac 上等同不加）
-P <ppid>找某 PPID 的所有子进程（debug 进程树用）
-n / -onewest / oldest 各取一个
-d,用逗号分隔多个 PID（喂给别的工具，见下）
-i忽略大小写

跟 ps 组合而不是 ps | grep

想看进程的 CPU / 内存 / start 时间，让 pgrep 出 PID、ps 取详情——把第 1 节那两个 grep bug 一起绕过：

ps -p "$(pgrep -d, -f caffeinate)" -o pid,ppid,user,%cpu,%mem,stat,start,command

pgrep -d, 输出逗号分隔的 PID 列表，ps -p 直接吃。ps 本身的列不写死——ps -axo pid,user,%cpu,... 也能自由挑列；pgrep + ps -p 的核心优势是省掉 | grep 这一步，而不是控列。

shellcheck —— 抓 set -e 漏过的坑

set -euo pipefail 是 runtime 早退；shellcheck 是 parse 时静态分析——抓 set -e 漏过的写法层 bug。部署 / 迁移 / CI 脚本两个都要上。

brew install shellcheck     # 或：sudo apt install shellcheck
shellcheck deploy.sh

set -e 漏过的 5 条

  • SC2155 local x=$(cmd) —— local / declare / export 自身返回 0，里头 $(cmd) 失败被吞，set -e 也救不了。修：先 local x 再 x=$(cmd)
  • SC2086 unquoted $VAR —— rm $FILE 在 FILE="a b" 时删两个文件，word splitting / pathname expansion 会拆开。默认 "$VAR"
  • SC2046 unquoted $(cmd) —— 同上但是命令替换：cd $(pwd) 路径含空格时炸
  • SC2068 $@ 没加引号 —— 含空格的参数被再次拆开（foo "a b" c 传进去变 a b c 三个）。修："$@"
  • SC2059 printf "$msg" —— $msg 含 % 时被当 format directive，输出错乱或 format string 注入。修：printf "%s" "$msg"

关掉警告

# 单行
# shellcheck disable=SC2155
local x=$(some_legit_command)

# 整文件（紧跟 shebang）
#!/usr/bin/env bash
# shellcheck disable=SC2086,SC2046

# ~/.shellcheckrc 或项目根 .shellcheckrc
disable=SC1090-SC1100

disable=all（v0.8+）关全部——通常意味着该重写。

dogfooding

软件工程黑话之一。直译”吃自己的狗粮”，指：

公司内部员工把自己开发的产品当日常工具用，靠真实使用暴露 QA / metrics 抓不到的体验问题。

更体面的同义说法是 drinking your own champagne。

1988 年微软内部一封题为 “Eating our own Dog Food” 的邮件（经理 Paul Maritz 发出）让这个说法在科技圈固定下来，沿用至今。

为什么 QA / 单元测试替代不了

QA 跟着 checklist 点一遍验功能存在；dogfooding 验长期真实使用下的体感 —— 慢、绕路、别扭、不直觉。这些既不会触发断言，也写不进测试用例。

行为算 dogfooding 吗
团队每天用自己写的内部 CRM✅
自己做金融 SaaS 但从不开账户跑投顾流程❌
QA 跟着用例点一遍❌（缺「真实工作场景 + 长期使用」）

失败模式：internal build ≠ public build

dogfooding 失效的经典姿势是：自己用的版本跟客户拿到的不是同一个。

Anthropic 在 2026-04-23 关于 Claude Code 质量退化的 postmortem 里坦白：员工用的是带内部 flag 的 test build，客户用的是 public build。其中一个缓存清理 bug 在 internal build 里被另一个并行实验掩盖了，员工 dogfooding 完全没暴露问题，客户端持续退化超过两周才定位。

改进措施第一条：让内部员工用 exact public build。

MySQL utf8mb4_0900_ai_ci 拼读

后缀是正交标志，按 <charset>_<Unicode 版本>_<accent>_<case> 拆读：

  • utf8mb4：字符集
  • 0900：Unicode 9.0 UCA（Unicode Collation Algorithm）
  • ai：accent insensitive，重音不敏感
  • ci：case insensitive，大小写不敏感

换后两位就拼出整个网格：_as_cs（严格区分）、_as_ci（保重音忽略大小写）、_bin（纯二进制）。

非显然之处在它真正按 UCA 处理 Unicode 等价，老的 utf8mb4_general_ci 全部按字节比，结果不可预测：

SELECT 'café' = 'cafe';   -- 1（_ai_ci 忽略重音；general_ci 为 0）
SELECT 'ß' = 'ss';         -- 1（UCA 定义为等价；general_ci 为 0）
SELECT 'Ａ' = 'A';         -- 1（全角 = 半角；general_ci 为 0）
SELECT '😀' < '😁';        -- 按 Unicode 9.0 顺序可预测；general_ci 不可预测

8.0 把默认从 utf8mb4_general_ci 换到这里就是为了这条「正确性」。Unicode 出新版时会跟进新 collation（如未来 utf8mb4_1400_ai_ci），版本号 pin 在 collation 名里，老 collation 永久保留，不会改语义破坏老库。

参考 Unicode Character Sets · MySQL 8 docs。

LLM 应用编排三层：Workflow / Agent / Function Calling

讨论 LLM 应用编排，workflow、agent、function calling 是三个绕不过去的基础词；外加一个编排框架（LangGraph / CrewAI / Pydantic AI / OpenAI Agents SDK 任选其一）就构成完整讨论。它们不是平行关系 —— 先分清各自位置，再谈选型，问题会简单很多。

三层位置图

[ 应用形态 ]  Workflow（图固定） /  Agent（图运行时才长出来）
                              ↓ 用
[ 编排框架 ]  LangGraph：节点 + 边 + 共享 state
                              ↓ 调
[ LLM 协议 ]  Function Calling：强约束 JSON schema
  • Function Calling：LLM API 的协议能力，给 LLM 一份 JSON Schema，逼它按 schema 输出。最底层、最稳定，OpenAI / Anthropic / Gemini 都有。
  • LangGraph：本文用的编排框架（同族还有 CrewAI / Pydantic AI / OpenAI Agents SDK 等），把流程画成「节点 + 边 + 共享 state」。工具层，本身不发明能力，只是把节点间的传递标准化。下面的位置图和分水岭判断对其他框架同样适用。
  • Workflow / Agent：两种应用形态，区别只有一个 —— 控制流在你的代码手里，还是在 LLM 手里。抽象层的差异见 Workflow vs Agent 的本质差异。

关键认知：Workflow 和 Agent 不是 LangGraph 提供的「模式开关」，是你用同一套 LangGraph + Function Calling 拼出来的两种不同形态。LangGraph 两种都能做，分水岭在你怎么写边。

Function Calling 是两种形态共用的底层

同一个 function calling API，两种用法决定你做出的是 workflow 还是 agent：

维度Workflow 用法Agent 用法
给 LLM 几个 function1 个（= 输出 schema）N 个（每个工具一个）
tool_choicerequired，必须调auto，LLM 自己挑
function 的角色数据合同可调用工具
下一步谁定Python 看返回字段LLM 看 observation 决定
LangChain 写法llm.with_structured_output(Schema)llm.bind_tools([t1, t2, ...])

底层 HTTP 请求几乎一样，差别只在给几个 tool、tool_choice 设成什么。

一句话：function calling 是协议层；workflow / agent 是你怎么用这个协议。

demo：分水岭具体在代码哪一行

配套两个文件，同一段业务逻辑（理财顾问会议笔记 → 抽字段 → 合规检查 → 分支生成文档）、两种控制流形态，对照看一目了然：

  • financial_workflow.py —— workflow 版，每个关键点标 [WORKFLOW vs AGENT] 告诉你这一行换成 agent 会变什么样
  • financial_agent.py —— 同业务的 agent 版，每个关键点标 [WORKFLOW → AGENT] 给出实际改造结果

workflow 文件里三段注释对应三个设计决策，正好覆盖上面三层：

  1. Function Calling 层 —— with_structured_output(Pydantic) 锁死 LLM 返回 schema，下游节点拿到的是类型化对象，不是裸文本要 json.loads 加 try/except
  2. LangGraph 层 —— state 是节点间的「信封」，节点 return dict 由 LangGraph 自动 merge，加节点 / 改顺序 / 插 human-in-the-loop 都不动函数签名
  3. 应用形态层（分水岭） —— add_conditional_edges 的路由参数：是你的 Python 函数 → workflow；是 tools_condition（读 LLM last message 判定）→ agent
