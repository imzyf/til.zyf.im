PhpStorm + Xdebug 容器联调，3 个名字必须对齐

PhpStorm 容器化 Xdebug 跑不通，最常见的根因不是端口、不是网络，而是 3 个魔法字符串没对齐 —— 错位时 IDE 静默忽略，没有任何报错。挨个对完，剩下的就是常规网络问题。

前置：mode + 触发方式

Xdebug 3 默认完全 disabled，要先开 mode；然后选 trigger 还是 always：

xdebug.mode=debug
xdebug.start_with_request=trigger    ; 推荐 dev 默认
  • trigger：只有请求带 XDEBUG_SESSION cookie / GET / POST / env 时才挂 debug；其他请求零开销。HTTP 场景永远选这个
  • yes：所有请求都进 debug session。每个请求都要给 IDE 发 connect，开发体感慢；只在 CLI 一次性脚本场景用（见最后一节）

1. serverName ↔ Servers > Name

容器内 PHP 通过环境变量告诉 Xdebug “我是哪个项目”：

PHP_IDE_CONFIG=serverName=laradock

Preferences > PHP > Servers 里那条 Server 的 Name 字段必须完全等于 serverName= 的值（这里是 laradock）。匹配上之后 PhpStorm 才会用这条 Server 上配的 path mapping 把容器里的 /var/www/... 翻译回宿主项目根，断点才落在正确文件上。

失败模式：debug session 建立成功（右下角图标变绿、Variables 面板出现），但断点永远不停 —— path mapping 找不到对应 Server，PhpStorm 不知道容器路径对应宿主哪个文件。

2. xdebug.idekey ↔ XDEBUG_SESSION value ↔ Run Configuration IDE key

三处的 IDE key 必须一致，默认值约定俗成是 PHPSTORM：

xdebug.idekey=PHPSTORM
  • 浏览器：Xdebug helper 扩展给请求加 Cookie: XDEBUG_SESSION=PHPSTORM
  • CLI：XDEBUG_SESSION=PHPSTORM php some-script.php
  • PhpStorm：Run > Edit Configurations > PHP Remote Debug 里 IDE key 填 PHPSTORM

cookie / env 的 value 同时承担两个角色 —— 触发 start_with_request=trigger 模式启动 session，且把 value 当作本次 session 的 IDE key 上报给 IDE。

失败模式：连接建立、断点也命中，但 PhpStorm 把 session 路由给错的 Run Configuration —— 看不到该 config 里配的环境变量、看不到自定义 evaluator。多人共享一台 Xdebug proxy 时尤其容易撞。

3. xdebug.client_host ↔ 容器回连宿主的地址

Xdebug 是 reverse connection：容器里的 PHP 主动回连 IDE，不是 IDE 拨进容器。所以 xdebug.client_host 必须能从容器解析到宿主 IP：

xdebug.client_host=host.docker.internal   ; Mac / Win / Linux 20.10+
xdebug.client_port=9003

跨平台统一写法（dockerhost 别名、host-gateway magic）。

失败模式：容器里 tail -f Xdebug 日志能看到 Could not connect to client. Tried: host.docker.internal:9003 —— 这条是少数不静默的失败，因为 Xdebug 自己在抱怨连不上。

CLI / Artisan 用 -d 覆盖

PHP CLI 走的不是 fpm，加载的是 /etc/php/X.Y/cli/conf.d/*.ini 而不是 fpm 的 ini。fpm 那边配好 Xdebug、CLI 仍然没动静是常见盲区。两种解法：

  • 把同样的 xdebug.* 配到 cli/conf.d/20-xdebug.ini，CLI 全局开（开发机 OK，生产别这么干）
  • 用 php -d 临时覆盖，只在需要时调试某条命令：
alias artisandebug='php \
  -dxdebug.mode=debug \
  -dxdebug.client_host=host.docker.internal \
  -dxdebug.client_port=9003 \
  -dxdebug.start_with_request=yes \
  artisan'

CLI 场景用 start_with_request=yes 是对的 —— 一次性进程，没机会注入 trigger，进来就要挂 debug。然后 artisandebug migrate:status 跑起来就能停断点。

排查顺序

跑不通时按这个顺序查，每一步都是上一步的前提：

  1. 容器里 php -i | grep xdebug 确认 mode=debug、idekey、client_host 都加载（CLI 调试要 php-cli 而不是 fpm 的输出）
  2. PhpStorm 右下角点过 Start Listening for PHP Debug Connections（电话图标变绿）
  3. 三对名字（serverName / idekey / client_host）逐一对齐
  4. Server 里 path mapping 设了容器 ↔ 宿主路径

排到 4 还不行，看 Help > Show Log in Finder 的 idea.log，搜 xdebug —— 99% 能直接定位到具体 mismatch。

PhpStorm "Multiple definitions exist for class" 警告

PhpStorm 报：

Multiple definitions exist for class Foo\Bar

自动跳转 / 补全在两个文件之间反复横跳 —— 同一份代码被索引了两遍。

最常见场景：本地 dev 一个 package（packages/foo/），它又被 Composer path repository 或 symlink 挂进 vendor/foo/，PhpStorm 把两份都索引了。

Cmd + O 输入类名看候选列表，确认是哪两个文件，然后保留 source-of-truth 那份，另一份处理：

  • 项目内的副本：右键 dir → Mark Directory as > Excluded
  • 系统级 Include Path 里的：Settings > Languages & Frameworks > PHP > Include Path 删条目

PhpStorm 会增量重建索引，不必走 File > Invalidate Caches。

PhpStorm 新建文件突然没有语法高亮

新建 foo.php 突然没语法高亮，删了重建还是没有，扩展名也没错 —— 大概率是某个 pattern 被误加进了 Text 文件类型的白名单。最常见的源头是某次手滑点了 Override File Type → Text，PhpStorm 就把这个 pattern 永久记下了。

修复：

Settings > Editor > File Types > Recognized File Types > Text > Registered Patterns

删掉多余的 *.php / foo.php 即可。改动立刻生效，文件不用重开。

PhpStorm 选中文本后输引号自动包裹

很多编辑器默认开启的 “选中 + 输引号 = 自动包裹” 在 PhpStorm 里默认关。开启路径：

Settings > Editor > General > Smart Keys > Surround selection on typing quote or brace

勾上后，选中一个 word 再敲 ' " ( [ { 都会用对应符号把它包起来 —— 加日志、加翻译函数、试不同表达式优先级时省一大堆光标移动。
