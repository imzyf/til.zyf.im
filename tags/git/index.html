gitignore 里斜杠的位置决定匹配范围

gitignore 的匹配范围全看斜杠在哪，记两条：

  • 结尾 / = 只匹配目录;不带 = 文件、目录通吃。
  • 开头 / 中间有 / = 锚定到仓库根目录;pattern 里完全没有斜杠 = 任意层级都匹配。

结尾斜杠：对文件加 / 是静默 bug

.idea 是目录，.idea/ 和 .idea 效果一样——但 / 更精确，显式表达「我忽略的是目录」,哪天真有个叫 .idea 的文件不会被误伤。JetBrains / GitHub 模板都用 .idea/。

反过来对文件就是坑：.DS_Store 是 macOS 生成的文件,写成 .DS_Store/ 会匹配不到任何东西(没有叫这名的目录),规则静默失效、不报错。目标是文件时绝不能加 /。

.idea/         # 目录 → 带斜杠更严谨
.DS_Store      # 文件 → 加斜杠就失效

开头斜杠:monorepo 的陷阱

node_modules/(无开头斜杠) 匹配任意深度，等价于 **/node_modules/——所以 **/ 在这里是多余的。/node_modules/ 只锚定根目录那一个，子包里的会漏掉。

node_modules/                ← /node_modules/ 只命中这个
packages/app/node_modules/   ← 只有无斜杠写法 / **/ 才命中

monorepo(pnpm / turborepo 子包) 一定用能匹配嵌套的 node_modules/,别写 /node_modules/。

**/ 真正有用是「中间匹配多层」:docs/**/tmp(docs 下任意深度的 tmp)、src/**/*.log。

.DS_Store、.idea/ 这类跟项目无关的编辑器 / 系统噪音，放全局 ignore(git config --global core.excludesFile) 一次，所有仓库都干净。

git interpret-trailers：读写 commit message trailer 的官方工具

trailer 是 commit message 末尾形如邮件头的 Key: Value 行——Signed-off-by / Reviewed-by / Co-authored-by。读写交给 git interpret-trailers,别 grep/awk:按字段位置取值会 silently drop(同 按作者统计 的定界符坑),只有 git 知道哪些行真是 trailer。

判定很严，同时满足才算 trailer、否则当正文:连续的含 : 行 + 前面隔一个空行 + 位于消息末尾(或 format-patch 的 --- 前)。所以 body 中段的 URL: ...、或紧贴正文没空行的，都不算——这是手写 parser 最易错处。

读用 --parse,只吐已有 trailer、忽略配置：

git log --format=%B -n1 HEAD | git interpret-trailers --parse

写用 --trailer,默认追加在 trailer 块末;与最后一条完全相同则不重复加：

git interpret-trailers --in-place \
  --trailer 'Reviewed-by: Bob <bob@example.com>' msg.txt

用 trailer.* 配置把常用 key 定义成简写。别名必须是完整 key 的前缀(大小写不敏感)——所以 ack 能映射 Acked-by,却没法拿 sign 映射 Reviewed-by。再配 .cmd 让别名自带取值逻辑，--trailer ack 不给值也能生成：

git config trailer.ack.key 'Acked-by'
git config trailer.ack.cmd 'echo "$(git config user.name) <$(git config user.email)>"'
git interpret-trailers --trailer ack   # 自动补 Acked-by: 你的名字 <邮箱>

.ifexists / .ifmissing 再单独控这个别名的重复与缺失行为。

最常见是 commit-msg 钩子里自动补 trailer($1 是 git 传入的消息文件，--trim-empty 丢掉空值行):

#!/bin/sh
git interpret-trailers --trim-empty \
  --trailer "git-version: $(git describe)" "$1" > "$1.new" && mv "$1.new" "$1"

git help interpret-trailers 看全。

git commit --fixup 配 rebase autosquash 收拢历史

code review 改意见时，最容易留下 fix typo、address review 这种噪音 commit。--fixup + autosquash 让每个补丁自动并回它对应的原始 commit，历史只剩有意义的那几个。

发现 commit A 要补个改动时，不要新开一个独立 commit，而是：

git commit --fixup A      # message 自动写成 "fixup! <A 的标题>"

A 是占位符，换成目标 commit 的任意 commit-ish:hash(abc1234)、HEAD~2、分支名，或 :/登录 模糊匹配 message。下文 A / B 同理，只是示意图里给 commit 起的标签。

之后一把 rebase 收拢：

git rebase -i A^

打开 todo 时 git 已经自动把 fixup! 那条挪到 A 正下方、并标好 fixup 动作，直接存盘退出即可吃进 A:

pick   A   feat: 登录接口
fixup  A'  fixup! feat: 登录接口   ← 自动排序 + 自动标记
pick   B   feat: 注册接口

~/.gitconfig 里这行是关键——它让 rebase -i 默认就开自动排序，否则每次得手动加 --autosquash,或在编辑器里自己挪行：

[rebase]
  autosquash = true

两个易混点：

  • --fixup vs --squash:fixup! 丢弃补丁的 message，完全用原 commit 的;squash! 把两条 message 拼起来让你编辑。改 bug 用 fixup，要补充说明用 squash。
  • autosquash 只改 -i 的默认行为,不是全自动——仍要你手动触发一次 git rebase -i,它只负责把 fixup!/squash! 排到位。

--fixup 还能直接指向”动了哪个文件就修哪个 commit”:

git commit --fixup :/登录    # 模糊匹配最近 message 含"登录"的 commit

See man git-rebase → --autosquash,man git-commit → --fixup。

git 按目录自动切换提交身份 (includeIf)

多身份 (公司 / 个人) 最容易出的事是忘记切换、用错 email 提交。includeIf 让 git 按仓库所在目录自动加载对应身份，零手动操作。

~/.gitconfig:

[user]
 name = Your Name        # 默认 name;故意不写 email

[includeIf "gitdir:~/work/"]
 path = ~/.gitconfig_work
[includeIf "gitdir:~/personal/"]
 path = ~/.gitconfig_personal

~/.gitconfig_work / ~/.gitconfig_personal 各放一行 email:

[user]
 email = you@company.com

防呆关键：全局不设默认 email。任何没被规则覆盖的目录会直接报 unable to auto-detect email address 拒绝提交——把”忘记”变成”无法继续”,而不是悄悄用错身份提交后才发现。这一步才是护城河，includeIf 本身 google 5 秒可查。

三个坑：

  • 路径结尾的 / 必须有:gitdir:~/work/ 匹配该目录下所有 repo;漏了 / 只精确匹配 ~/work 这一个 repo。
  • 多条命中时后写的赢:git 按出现顺序逐个 include，后面覆盖前面;更具体的目录规则写在后面。
  • local 配置仍是最高优先级:某 repo 里 git config --local user.email 设过会盖掉 includeIf,这类历史 local 覆盖要清掉才轮到目录规则接管。

验证别直接 cd 到父目录 (非 repo 测不出),建个临时 repo 看实际结果：

mkdir -p ~/work/.idtest && git -C ~/work/.idtest init -q
git -C ~/work/.idtest config user.email   # → you@company.com
rm -rf ~/work/.idtest
git config --global --get-regexp includeIf  # 查所有规则

另一种匹配:hasconfig 按远程地址

目录分不开时 (公司 / 个人 repo 混在一处),改用远程 URL 匹配：

[includeIf "hasconfig:remote.*.url:git@github.com:acme/**"]
 path = ~/.gitconfig_work
  • 通配符除标准 glob 外，多了 **/ 和 /** 能跨多级路径匹配。git@github.com:acme/** = acme 这个 org 下所有 repo;git@github.com:*/** 则是 github 上所有 repo。
  • 非显然点 (scan-ahead):gitdir: 看路径就能判;但 remote URL 通常在 repo 的 .git/config 里，比全局 includeIf 晚解析。git 对 hasconfig 特殊处理——第一次遇到这个条件时，会预扫描后面所有配置文件找 remote URL，所以晚定义的 local remote 照样命中。
  • 代价:被它 include 进来的文件 (~/.gitconfig_work)禁止再声明任何 remote URL——否则「是否包含」会反过来取决于「被包含文件里的 remote」,形成死循环，git 直接禁掉这种写法。

按目录 (gitdir:) 仍是最简单可靠的默认，只有目录无法区分时才上 hasconfig。

See man git-config → “Conditional includes”。

Git LFS

Git 为文本设计：每个版本的每个文件都永久留在历史里。代码这样很好——但二进制文件（视频、PSD、字体、数据集）会被同样对待。一个 50MB 的视频改 5 次，.git 就多 ~250MB，而且每次 clone 都要把全部历史下载下来。Git 还无法 diff 二进制，只能存整份拷贝，省不掉。

原理：把大文件换成 pointer

LFS 在仓库里只存一个 ~130 字节的 pointer 文本文件，真身放到独立的 LFS server：

version https://git-lfs.github.com/spec/v1
oid sha256:4d7a214614ab2935c943f9e0ff69d22eadbb8f32b1258daaa5e2ca24d17e2393
size 52428800

git 历史里流动的是这段文本（可 diff、可压缩），git checkout 时 LFS 认出 pointer 自动把真身拉回工作区。对你来说体感和普通 git 一样。

brew install git-lfs
git lfs install                 # 装 git hooks，每台机一次

git lfs track "*.mp4"           # 写规则到 .gitattributes
git add .gitattributes          # 这个文件必须提交，否则队友不会触发 LFS
git add intro.mp4               # 正常用 git
git commit -m "add intro video"

.gitattributes 里那条规则长这样：

*.mp4 filter=lfs diff=lfs merge=lfs -text

filter=lfs 背后:clean / smudge

.gitattributes 里那个 filter=lfs 指向 ~/.gitconfig 中 git lfs install 写的一段定义，真正干「真身 ↔ pointer」转换的就是它：

[filter "lfs"]
  clean = git-lfs clean -- %f       # add/commit:真身 → pointer
  smudge = git-lfs smudge -- %f     # checkout:pointer → 真身 (下载)
  process = git-lfs filter-process  # 上两步的常驻进程批量版
  required = true
  • clean 在 git add 触发 (工作区 → 仓库),smudge 在 checkout 触发 (仓库 → 工作区),%f 是文件名占位符。
  • process 是性能优化：默认每个文件 fork 一次 clean/smudge,process 改用一个常驻进程通过协议批量处理;git 优先用它，clean/smudge 作后备。
  • required = true 是关键安全项：过滤器失败就报错中止,而非把原始数据放过去。

没装 git-lfs 的队友 clone 出来是 pointer 文本而不是真身，根因就在这——他们 ~/.gitconfig 里没这段 filter,smudge 根本不会跑。

track 必须在 add 之前

LFS 的”拦截”发生在 git add 那一刻：此时它查 .gitattributes，匹配就把文件换成 pointer。这是一次性的——文件一旦以普通方式进了历史，后补规则也救不回来。

# ❌ 错误顺序：视频以 50MB 真身进了历史
git add video.mp4 && git commit -m "add video"
git lfs track "*.mp4"           # 规则管不到「已经」发生的提交，仓库照样臃肿

补救要靠 git lfs migrate，它改写历史把旧文件从 git 挖进 LFS（同 git filter-repo 一样会变 commit hash，需要 force push + 团队协调）：

git lfs migrate import --include="*.mp4"

类比 Laravel：git lfs track 像写了条新 migration 规则，但它不 backfill 旧数据，老记录得专门跑一次 migrate import 才能改造。

file locking：因为二进制无法 merge

文本冲突 git 能三方合并，二进制不能——两份 .psd 没有”合并”一说。于是 Alice、Bob 同时改 logo.psd，后推的人只能覆盖或丢掉自己的工。

LFS lock 把 SVN/Perforce 的 check-out / check-in 模型搬回 git：编辑前先认领，期间该文件在别人机器上变只读，server 拒绝他们 push。

git lfs lock   images/logo.psd  # 认领，别人变只读
git lfs unlock images/logo.psd  # 干完释放
git lfs locks                   # 看谁锁了什么

前提是在 .gitattributes 给文件加 lockable，这个 flag 让它们默认只读、持锁才可写：

*.psd filter=lfs diff=lfs merge=lfs -text lockable

锁是协调工具不是强保护——git lfs unlock --force 能撬开别人的锁，团队仍要靠沟通。GitHub / GitLab / Bitbucket 都支持 locking。

See git help lfs.

git show 读对象数据库

git show 读的是 git 对象数据库里存的东西，不是磁盘工作区。这个区别决定了它能干 cat 干不了的事。

<rev>:<path>：看任意 commit 时刻的文件

git show HEAD:config.php          # 当前 commit 里这个文件的样子
git show HEAD~3:config.php        # 往前数 3 个 commit 时的样子
git show main:config.php          # main 分支上的样子
git show abc1234:src/app.js       # 指定 commit hash

不用 checkout、不动工作区，直接把历史里某一版的文件内容打到 stdout。想对比”线上分支的这个文件和我本地差在哪”时特别顺手。

为什么 cat 看不到 pointer

LFS / CRLF 转换 / .gitattributes 里的 filter，都是在文件进出工作区时做的变换。所以同一个文件两个来源不同：

cat video.mp4                     # 工作区：被 LFS 还原后的视频真身
git show HEAD:video.mp4           # 对象库：git 实际存的 pointer 文本

要看 git 仓库里”真正存了什么”，只能走 git show（或 git cat-file -p）。

不带 :path 时

git show                          # 最新 commit 的 message + 完整 diff
git show abc1234                  # 指定 commit 的 message + diff
git show v1.2.0                   # tag 指向的对象

判准：git show <X> 看的是”X 这个对象的内容”——commit 给你 diff，文件 blob 给你文件内容，tag 给你 tag 信息。

See git help show.

git worktree —— 一仓多目录

正在 feature 分支写代码，文件改了一半 —— 线上 bug 要紧急修。常规做法是 stash / WIP commit / 切分支 / 回来再 pop，这套动作越熟越说明 git 缺机制。git worktree 就是这个机制：一个仓库同时挂多个工作目录，每个目录一个 HEAD，共享同一份 .git。比 stash 干净，比 git clone 轻量（不复制 objects、不重新 fetch）。

工作树 ≠ 分支

  • 分支：一个指针（.git/refs/heads/<name>），仓库级元数据，几个字节
  • 工作树：一个目录 + 一个 HEAD。HEAD 可以 attached 到分支，也可以 detached 直接指向 commit

多棵工作树共享所有分支、objects、hooks。每棵 linked worktree 的 HEAD 独立存在 .git/worktrees/<name>/HEAD，主工作树的 HEAD 仍在 .git/HEAD。同一分支同一时刻只能被一棵工作树 checkout —— 否则两边 HEAD 互相覆盖、reflog 串台。这是设计上的保护。

推论：

  • 创建 worktree ≠ 创建分支：--detach 完全不动分支；复用已有分支也不创建
  • 删除 worktree ≠ 删除分支：worktree remove 只清目录和元数据，分支还在
  • 删分支前先 worktree list：被占用的分支删不掉，会报 error: Cannot delete branch 'X' checked out at '<path>'

三种 add 姿势

git worktree add -b feat/login ../login main       # 新建分支，基于 main
git worktree add ../review feat/login              # 复用已有分支
git worktree add --detach ../inspect origin/main   # 不开分支，detach 到某个 ref/commit

末位 <commit-ish> 接 branch / tag / SHA / HEAD~3 都行。其他子命令（list / remove / move / lock / prune / repair）按需查 git help worktree，日常 95% 是这三条 add + remove。

典型工作流

feature 写一半，hotfix 插队：

git worktree add -b hotfix/login ../hotfix main
cd ../hotfix
# 改代码、commit、push、PR
cd -
git worktree remove ../hotfix
git branch -d hotfix/login        # remove 不删分支，按需手动

Review 别人 PR：

git fetch origin pull/123/head:pr-123
git worktree add --detach ../pr-123 pr-123
# composer install / 跑测试 / 手测
git worktree remove ../pr-123 && git branch -D pr-123

AI agent 让 worktree 从奢侈变刚需

git 2.5（2015 年）就有 worktree，十年小众。AI agent 让”同时跑 N 个任务”从奢侈变日常 —— branch 是 git 给「异步协作的人」的礼物，worktree 是 git 给「并行执行的 agent」的礼物。

让 agent 在你正在工作的目录里改代码，风险是文件 race（A 改 Home.php 第 100 行同时 B 也改）、watcher 链式反应、dev server 端口抢占、test DB 互 truncate。worktree 把每棵任务关进独立目录，git 层面互不踩脚。但 worktree 只隔离 git 状态，周边资源（端口 / 测试 DB / vendor/ / node_modules/）得自己按 worktree 分。

Claude Code 内置 EnterWorktree 和 Agent(isolation: "worktree")，工作树落在 .claude/worktrees/<name>/，基于 origin/<default-branch>。Conductor / Cursor / Aider 底层同套机制。三条纪律：

  1. 一棵树一个任务 —— 否则 review 时分不清谁改了什么
  2. 完事立刻 remove —— 不清理 .claude/worktrees/ 堆几十个，磁盘和心智都崩
  3. 依赖各装各的 —— vendor/ node_modules/ 是文件不是 ref，不会跟着 worktree 同步

高频陷阱

现象原因 / 解
git branch -d feat/x 报 Cannot delete branch 'feat/x' checked out at '<path>'分支被某棵 worktree 占着。git worktree list 定位 → git worktree remove ../x 释放
手动 rm -rf ../x 后 git worktree list 仍列 ../x元数据残留在 .git/worktrees/x/。git worktree prune 清死链
想在两棵树同时 checkout main设计禁止。第二棵用 --detach main，或 -b <new> main 基于 main 开新分支
pre-commit / husky / lefthook 在某棵树改了，另一棵也变.git/hooks/ 是共享的，所有 worktree 共用一份 hook 配置。装一次即可，别多树重复 husky install

git help worktree 看完整 manpage，重点段是 DESCRIPTION 里 .git/worktrees/ 布局描述。

git filter-repo 清除历史敏感文件

仓库不小心把 .env / database.yml 推到远程后，第一动作是轮换密钥，假定文件已被抓取 —— 改写历史只能挡住”还没看到的人”，已 clone / fork / 爬虫的副本拿不回来。这一步先做完，再谈 git filter-repo：它是第二步，不是第一步。

为什么不是 filter-branch

git filter-branch 官方已弃用。每个 commit fork 一个 shell 子进程，大仓库跑数小时；默认保留 refs/original/ 和 reflog，旧对象很容易”清理后又复活”。git filter-repo 是 Python 单进程流式管道，快 10–100 倍，且默认要求 fresh clone（拒绝在带遗留 ref 的工作目录上跑）。

brew install git-filter-repo

原理：重写而非删除

Git commit 的 SHA-1 由 tree + parent + author + message 哈希得出，不可变。“删一个文件”等价于生成一份新历史替代旧历史。

旧仓库 ── git fast-export ──▶ 事件流 ── 过滤器 ──▶ git fast-import ──▶ 新仓库

git fast-export 把仓库历史序列化成纯文本指令流（blob、commit、tag）；filter-repo 在内存里按规则改写事件流；git fast-import 把它灌成新对象库。

关键后果：一个 commit 的 tree 变了 → 它的 SHA 变 → 子 commit 的 parent 指向新 SHA → 子 SHA 也变 → 一路传染到 HEAD。所以 force push 是必须的 —— 远程的 commit 链被一条 SHA 完全不同的链替换。

--invert-paths：默认是白名单

非显然：filter-repo 默认把 --path 当保留列表，不是删除列表。

# 默认：只保留 src/，其他全删（拆分 monorepo 用法）
git filter-repo --path src/

# --invert-paths：删除 src/，其他保留（清理用法）
git filter-repo --path src/ --invert-paths

设计初衷之一是从 monorepo 抽子项目 —— “列出我要的那部分”比”列出我不要的”自然。清理敏感信息正好相反：99% 留 + 1% 删，必须加 --invert-paths。漏写就把整个仓库删干净只剩那个泄露文件，反应过来时已经覆盖远程了。

git filter-repo --path .env --invert-paths
git filter-repo --path-glob '*.pem' --invert-paths
git filter-repo --path-regex '^secrets/' --invert-paths

--mirror：不配它等于没删

普通 git clone + git push --force --all 看似清理，实际只覆盖 branches。远程上 branch 之外的 ref 没动，仍指向旧 commit：

  • Tags：旧 tag 仍含敏感 commit
  • Notes、其他自定义 ref

只要远程任何一个 ref 还能可达旧 commit，那个 blob 就不会被 GC 回收 —— 攻击者拿到 SHA 就能 git cat-file -p <sha> 读出来。删了等于没删。

git clone --mirror 拉一份裸仓库（.git 结构、无工作区）+ 所有 ref；git push --mirror 把本地 refs 一比一推到远程，远程多出来的 ref 会被删 —— “让远程完全等于本地”。这才是真清理：

git clone --mirror git@github.com:org/repo.git
cd repo.git
git filter-repo --path config/database.yml --invert-paths
git push --force --mirror

--mirror 只解决服务端的 ref 问题。任何一个老的本地 clone 仍能通过 reflog 和 remote-tracking refs (refs/remotes/origin/*) cat-file 出旧 blob —— 这正是 filter-repo 默认要求 fresh clone 的原因，也是清理后必须通知所有协作者删本地仓库重新 clone 的原因。

清理后必做

步骤说明
轮换密钥见开篇，第一优先
通知协作者所有人重新 clone，旧本地分支基于已消失的 commit 链
联系 GitHub SupportPR refs (refs/pull/<n>/head) 由 GitHub 服务端维护，push --mirror 覆盖不了；fork 拷贝也要走 Support 清

装 pre-commit 钩子gitleaks protect --staged 自动拦截
brew install gitleaks

cloc / tokei / scc：codebase LOC 统计的三代工具

数代码行数（LOC by language）这件事有三代工具，每代换语言、换取舍：

实现 / 起年Perl / 2006Rust / 2016Go / 2018

scc 那三个名词各自的意思：

  • 复杂度：数代码里 if / for / && 这类分支关键字，估算 cyclomatic complexity。
  • COCOMO：1981 年 Barry Boehm 提出的成本模型，给 KLOC 估”工业平均要多少人月、多少钱”。
  • 重复检测：按文件 hash 判定，相同内容只算一次（monorepo / vendor 目录里常见，cloc / tokei 会重复计入）。

对同一个目录（这个 blog 的 til/src/）跑默认命令，差异自己跳出来：

$ cloc til/src/

Astro                 10            117             62           1340
TypeScript             7             14             49             89
SUM:                  17            131            111           1429

$ tokei til/src/
 Language         Files        Lines         Code     Comments       Blanks
 Astro               10         1519         1344           59          116
 TypeScript           7          152           97           44           11
 Total               17         1671         1441          103          127

$ scc til/src/
Language        Files       Lines    Blanks  Comments       Code Complexity
Astro              10       1,519        99        49      1,371         55
TypeScript          7         152        11        44         97          2
Total              17       1,671       110        93      1,468         57
Estimated Cost to Develop (organic) $40,425
Estimated Schedule Effort (organic) 4.06 months
Estimated People Required (organic) 0.88

cloc 没有 Lines 总数列（自己加 blank+comment+code）；tokei 加了 Lines；scc 多 Complexity 列 + 一段 COCOMO 估算块。同样的 17 个文件，三个工具给出三个 Code 总数：1429 / 1441 / 1468——blank 和 comment 怎么判，每家略不同。

cloc 的 --diff 是至今没被替代的核心——code review 想量化”这个 PR 改了多少 code 行”时，tokei / scc 都做不到：

Markdown

 modified          2              0              0              9

 removed           0              0              0             19
Bourne Shell
 same              0              0              5             14
 modified          1              0              0              1
 added             0              0              0              1
 removed           0              0              0              0
SUM:
 same              0              0              7            938
 modified          3              0              0             10
 added             0              0              0              5
 removed           0              0              0             19

按语言分组，每组拆 same / modified / added / removed；想细到文件级加 --by-file。比 git diff --stat 信噪比高一截。

怎么选： 默认装 scc（最快 + 复杂度）；CI 输出要 JSON 接管线用 tokei；要 git diff 的 LOC 变化才掏 cloc。

git clean 删未跟踪文件：先 -n 看，-X 单独清 ignored

git clean 删的是 working tree 里未被追踪的文件——.gitignore 命中的、和未 add 过的新文件。删完没有 git checkout 可以救，所以永远先用 -n 走一遍：

git clean -n           # 列出会删的文件，不动磁盘
git clean -nd          # 含未跟踪的目录
git clean -nXd         # 只列 .gitignore 命中的（如 build/ dist/）

确认后把 -n 换成 -f（force，必填——不写就拒绝执行）：

git clean -fd          # 删未跟踪的文件 + 目录
git clean -fXd         # 只删 ignored（清 build artifacts、.cache）
git clean -fd -- src   # 限定路径

-X 大小写反义：

  • -X（大写）：只删 .gitignore 命中的，未跟踪的新文件（如手动创建忘 add 的）保留
  • -x（小写）：把 ignored 也算成未跟踪一并删——破坏性更大，会清掉 .env、本地 IDE 配置等被 ignore 的工作文件，慎用

git 改提交时间：--date 只动 author，要改 committer 用环境变量

git 每个 commit 有两个时间字段：author date（谁写的代码、什么时候写的）和 committer date（谁应用的这个 commit、什么时候应用的）。git log 默认显示 author date，但 git log --format=%ci、GitHub 时间线排序、git log --since 都看 committer date——只改 --date 常出现”日志看着对了，GitHub 顺序没动”。

新 commit 同时改两个：

GIT_COMMITTER_DATE="2019-06-11T17:50:50+08:00" \
  git commit -m "fix..." --date="2019-06-11T17:50:50+08:00"

修上一个 commit（会改 hash，已 push 的不要碰）：

GIT_COMMITTER_DATE="$(date -R)" \
  git commit --amend --no-edit --date="$(date -R)"

验证两个字段：

git log -1 --format='author %ai%ncommit %ci'

2022-10-01 #git

git 忽略文件权限位变更：core.filemode false

跨 OS 协作时（macOS / Linux ↔ WSL ↔ NTFS / FAT 网络盘），git 会把权限位变化（100755 ↔ 100644）当成内容变更，git status 满屏 mode-only diff 但其实没东西可改。

仓库局部关掉：

git config core.filemode false

core.filemode 不在版本控制里——它是 .git/config 的本地设置，不会同步给协作者（git 设计如此：权限策略是工作环境属性，不是仓库属性）。所以改完不用 commit。

陷阱：一刀切关掉后，你新加的可执行脚本权限不会被同事看到——CI 上跑 ./deploy.sh 直接 permission denied。要保留”真正的可执行位”只压”杂讯”，用：

git update-index --chmod=+x path/to/script.sh

显式把这个文件标记成 executable 写进 index，比关 filemode 精准。

git 建独立无历史的分支用 --orphan

git checkout --orphan 创建一个不指向任何已有 commit 的新分支——HEAD 移过去后 working tree 保留，但 staging 和 commit history 都从零开始。常用场景：一个 repo 同时管代码和文档站，文档站需要独立 history（如 gh-pages），或者发布开源版本时不想暴露内部 commit 史。

git checkout --orphan gh-pages
git rm -rf .                         # 关键：清掉父分支带过来的 working tree
echo "<h1>hi</h1>" > index.html
git add . && git commit -m "init gh-pages"
git push origin gh-pages

陷阱：--orphan 切过去后 git status 会显示一堆 staged 文件——那是从父分支继承的 working tree，没 git rm -rf . 就 commit 等于把整套源码塞进了所谓的”独立”分支。

Git `--allow-empty`：init repo 先打个空 commit 占位

git commit -m "init repo" --allow-empty

第一个 commit 是 git history 的根。git rebase -i HEAD~N 默认从根之后开始，根 commit 不在 todo list 里 —— 除非显式 git rebase --root -i（操作复杂度陡增）。先放一个空 init commit 占位，后面所有内容性 commit 都可以正常 rebase / squash / reorder。

--allow-empty 的其他真实场景：

  • CI 只在 push 时触发但 working tree 没改 —— 强推一个空 commit 触发流水线
  • 给一批工作起点打锚点（feat: kick off auth refactor）方便后续 git log --grep

按作者统计 git 贡献量

只看 commit 次数（git 内置，0 依赖）：

git shortlog -sn --no-merges

要带行数的版本，用 --numstat 给出每文件 +N -N filename，再用 --pretty=format 把作者名作定界符让 awk 聚合：

git log --numstat --pretty='COMMIT %aN' \
  | awk '/^COMMIT / { a = substr($0, 8); next }
         /^[0-9]/   { ins[a] += $1; del[a] += $2 }
         END        { for (k in ins) printf "%s\t+%d -%d\n", k, ins[k], del[k] }'

%aN 会走 .mailmap（同一人不同邮箱自动合并）。^[0-9] 自动跳过 binary 文件（numstat 对 binary 输出 - - file）。加 --author="..." 限单人；加 --since="2024-01-01" 限时段。

行数易被 reformatting / rename / 生成代码放大，跟 commit 数对照看比单看任一项准。

CRLF will be replaced by LF

$ git add .
warning: CRLF will be replaced by LF in X.

文件含 CRLF 行尾，git 按当前 core.autocrlf / .gitattributes 配置正在把它改成 LF。只是 warning 不是 error，提交不会被拦。

跨平台团队（Mac/Linux + Windows）想把行尾问题从 PR review 里彻底抹掉，仓库根放一份 .gitattributes：

* text=auto
*.sh text eol=lf
*.bat text eol=crlf

老仓库加这份配置后，跑 git add --renormalize . && git commit 让已入库的文件按新规则重写一遍——否则规则只对未来的修改生效。

* text=auto 一行覆盖 90% 场景。*.sh text eol=lf 是经典救场——Windows 同事 checkout 后行尾被换成 CRLF，bash deploy.sh 直接报 bad interpreter；*.bat 是对称情况。完整规则见 GitHub Docs: Configuring Git to handle line endings。

Git commit message 规范：Conventional Commits + 工具链分工

约定本身就是 Conventional Commits 的 Angular 变体：

<type>(<scope>): <subject>

<body>

<footer>

type 限定 7 个：feat / fix / docs / style / refactor / test / chore。scope 标受影响模块（auth、api、db），项目内自定。subject 三条硬规则：≤ 50 字符、动词原形开头（change 不是 changed）、首字母小写、不加句号。

约定的价值不在”整洁” —— type 是给机器读的：feat 触发 minor bump、fix 触发 patch、BREAKING CHANGE 触发 major，CHANGELOG 也按 type 分组。光人手写不行，工具链分两段 —— 本地”写 + 验”、远端”发布”。

阶段工具职责
写commitizen + @commitlint/cz-commitlintgit cz 交互式生成 message，prompt 直接读 commitlint 规则 —— 不用像 cz-customizable 那样维护两份配置
验commitlint + huskygit hook 拦截不合规 message —— commitizen 是引导、commitlint 是底线

echo "module.exports = { extends: ['@commitlint/config-conventional'] };" > commitlint.config.js

package.json 把 commitizen 指到 commitlint adapter，规则就只剩 commitlint.config.js 一份：

{
  "scripts": { "commit": "git-cz" },

npm i -D husky
npx husky init
echo 'npx --no -- commitlint --edit "$1"' > .husky/commit-msg

场景工具

git checkout --ours / --theirs

冲突里某个文件不想逐行 merge，整文件保留一边即可。批量加 .：

git checkout --ours .    # 保留"当前视角"那边，丢另一边
git checkout --theirs .  # 反之
git add .

坑：--ours 在 merge 和 rebase 下指向相反的分支。

操作--ours--theirs
git merge feature当前分支 (HEAD)被合入的 feature
git rebase upstreamupstream（rebase 落点）你正在 rebase 的分支

rebase 把你的提交逐个”重放”到 upstream 上，每一步 HEAD 都是 upstream，所以 --ours 指 upstream，反直觉但自洽。
