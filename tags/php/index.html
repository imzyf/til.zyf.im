二进制安全 是什么意思

PHP 文档里 crypt() 顶上挂着一行：

Warning: This function is not (yet) binary safe!

binary safe 指函数能完整处理任意字节序列，含 NUL 字节（\0、0x00）。失安全的两种姿势：遇 \0 截断、结果随 locale 变化。

crypt() 怎么”不 binary safe”

底层是 C crypt(3)，C 字符串以 \0 终止：

$salt = '$2y$10$' . str_repeat('a', 22);
crypt("secret\0extra1", $salt) === crypt("secret\0extra2", $salt);  // true

C 端只读到 "secret"，\0 后的字节全部丢弃。把 random_bytes() 出来的 32 字节 key 当密码直接喂 crypt() / password_hash() 时这就是定时炸弹 —— 32 字节里含 \0 的概率是 1 - (255/256)^32 ≈ 12%，碰上就把高熵 key 蒸发掉一截。

修法是先编码再哈希：

$key = random_bytes(32);
password_hash(base64_encode($key), PASSWORD_BCRYPT);

更对路的做法：哈希二进制 key 别走 crypt 家族，用 hash_hmac('sha256', $msg, $key)，它本身 binary safe。

“(yet)” 是什么意思

不是 typo —— 是文档作者留的”想修但修不动”的尾巴。crypt() 走 libc，要 binary safe 得绕开 libc 自己实现一份，跨平台一致性破坏。从 PHP 4 挂到 PHP 8，括号里这两字大概会一直挂下去。

不 binary safe 分两类

类型典型函数触发
\0 截断crypt()、所有 fs 路径参数（fopen / include / file_get_contents）入参含 \0
locale 依赖strcoll字节落在 0x80-0xFF

路径那条历史上是 CVE-2006-7243 —— include $_GET['page'] . '.php' + 攻击者传 evil.so\0 能绕扩展名白名单包含任意文件。PHP 5.3.4 起 fs 函数拒收含 \0 的路径（PHP 8.0 起直接抛 TypeError），但拼接前先过用户输入仍是写代码的人的责任。

locale 那条更隐蔽 —— strcoll 按 LC_COLLATE 排序，同样两字节给相反答案：

setlocale(LC_COLLATE, 'C');
strcoll("\xE4", "z");                    // > 0（按字节 0xE4 > 0x7A）

setlocale(LC_COLLATE, 'de_DE.UTF-8');
strcoll("\xE4", "z");                    // < 0（德语排序 ä 紧挨 a，在 z 前）

同一段代码在 CI（默认 C）、生产容器（可能装了 de_DE.UTF-8）、本地 mac 跑出不同结果，debug 极难找。要可移植的国际化排序走 Collator（intl 扩展）显式给 locale；要大小写转换走 mb_strtolower($s, 'UTF-8') 明确指定编码 —— strtolower 自 PHP 8.2 起只处理 ASCII，对非 ASCII 字节原样返回。

哪些是 binary safe 的

strlen strpos strrpos substr str_replace strcmp strncmp explode implode hash hash_hmac fread fwrite md5 sha1 —— 核心字符串和 hash 族基本全是。判别口诀：函数文档说”以字节为单位”且不依赖 locale。

strcmp() 是 binary safe，但不是 constant-time —— 校验 token / HMAC / CSRF 用 hash_equals()，否则吃时序攻击。两个正交属性，别混。

mb_* 系列是多字节安全（认得 UTF-8 多字节序列），不等于”任意二进制安全” —— 喂非法 UTF-8 字节会报警或返回 false。

Argon2id PHC 串拆解：salt 来源与 Base64 编码

PHP password_hash($pwd, PASSWORD_ARGON2ID) 吐出来这种东西：

$argon2id$v=19$m=65536,t=4,p=1$UlhrVTlVbVI2S08wa2szSQ$gGGk/8NtsxcSbFOxcSkWOSf3fFE/a2KbMEqRiTrFmes

这是 PHC string format —— 2013-2015 年那场 Password Hashing Competition 顺带产出的字符串格式规范（竞赛本身是算法选秀，Argon2 胜出后由 RFC 9106 标准化；格式规范是副产品，让所有现代密码 hash 都能用自包含字符串表达，便于跨算法升级 / 跨语言迁移）。6 段用 $ 分隔。

$<id>[$v=<version>][$<param>=<value>(,<param>=<value>)*][$<salt>[$<hash>]]
段值含义
算法argon2id变体（不是 argon2i / argon2d）
版本v=19Argon2 v1.3（0x13）
参数m=65536, t=4, p=1内存 64 MiB、迭代 4 轮、并行 1
SaltUlhrVTlVbVI2S08wa2szSQ22 字符 → 解码后 16 字节
HashgGGk/8NtsxcSbFOxcSkWOSf3fFE/a2KbMEqRiTrFmes43 字符 → 解码后 32 字节

Salt 必须来自 CSPRNG

CSPRNG = Cryptographically Secure Pseudo-Random Number Generator（密码学安全的伪随机数生成器）。判别标准：next-bit 不可预测 + 状态泄露不能反推历史输出 + seed 自身来自熵源。rand() / mt_rand() 都不满足。

password_hash() 内部路径：

password_hash()
  → php_password_make_salt()
    → php_random_bytes_silent(16)
      → getrandom(2) (Linux) / arc4random (macOS)
        → OS 熵池（硬件中断、键盘 / 鼠标时序、磁盘 I/O 抖动）

等价于：

$salt = random_bytes(16);

永远不要用以下函数生成 salt / token / key：

不安全替代
rand()random_int($min, $max)
mt_rand()random_int($min, $max)
uniqid()bin2hex(random_bytes(16))
md5(microtime())random_bytes(N)

mt_rand() 是 Mersenne Twister，速度快但状态可被反推 —— 拿到约 624 个连续输出就能完全恢复内部状态、预测后续所有值。已有 php_mt_seed 在毫秒级破解。

salt 不需要保密（和 hash 一起明文存数据库），它的作用是：

  1. 让相同密码产生不同 hash，rainbow table 失效
  2. 强制攻击者逐个密码爆破，不能批量

字节 ↔ 字符：编码的换算

字节是 8 bit 的存储单位；字符是表示单位。两者的转换由编码定义。

为什么不能 1 字节 → 1 字符？ 一个字节有 256 种值（0255），但可打印 ASCII 只有 95 个（0x200x7E，即 128 个 7-bit ASCII 编码点扣掉 33 个控制符 —— 0x00~0x1F + 0x7F。控制符是 NUL / Tab / 换行 / Esc / DEL 等，不能透传文本，所以 128 根本不是候选档位）。1:1 至少缺 161 个对应。退一步：能不能扩到 256 字符？

候选方案为什么不行
全 256 ASCII0x00~0x1F 是 NUL / Tab / 换行 / Esc，终端、字符串、协议都会炸
Latin-1 全 256 字符UTF-8 时代字节 0x80~0xFF 不是合法 UTF-8 序列，存数据库 / 走 JSON / 进 URL 全报错
强行透传二进制URL、JSON、shell、HTTP header、email、.env、grep 输出全部假设 ASCII-safe

所以真正约束是：字符集必须落在「绝对安全的可打印 ASCII 子集」里。理论上界 95，实操还要扣掉 " \ < > & = + ? # 等在 URL / JSON / shell 里有特殊含义的字符，安全集合一般 60~80 个。各档编码的取舍：

字符集大小编码bit/字符膨胀率16 字节后
16Hex42×32 字符
32Base3251.6×26~32
64Base6461.33×22~24
85Base85 / Ascii856.41.25×20~21
95全可打印 ASCII 上界~6.571.22×20

Base64 字符表是 A-Z a-z 0-9 + /（62 个字母数字 + 2 个填空），「安全字符够多」+「6 bit 整齐对齐 24 bit 包」的双重最优解：再上一档要 128 字符，超出可打印 ASCII 上界；再下一档 32 字符膨胀率明显更高。

Base64 把每 3 字节 打包成 4 字符，因为 LCM(8, 6) = 24 bit —— 字节 8 bit 和 Base64 字符 6 bit 不能整除，得找两边都数得齐的最小公共单位。24 bit 既是 3 个字节，也是 4 个 Base64 字符：

[byte 1] [byte 2] [byte 3]      ← 24 bit
 6 bit  6 bit  6 bit  6 bit
[char 1][char 2][char 3][char 4]

试别的字节数都凑不齐：

输入字节bit÷ 6 = 字符问题
181.33余 2 bit，半字符存不下
2162.67余 4 bit，半字符存不下
3244整除
4325.33回到 8 bit 周期

所以 3 字节是「最小成块单位」，每 3 字节切一刀、吐 4 字符、循环。输入不是 3 的倍数时，结尾凑不齐 24 bit，要补 0 bit 到下一个 6 bit 边界，再用 = 标记「这是补的、不是真数据」 —— padding 的来历，下一节正好展开 PHC 为什么把 = 删了。

所以 PHC 串里 16 字节 salt → 22 字符；32 字节 hash → 43 字符。反推也成立：看见 32 hex 是 MD5 / UUID（16 字节），64 hex 是 SHA256（32 字节）。

PHC 的 Base64 不带 padding

标准 Base64 末尾会用 = 补齐到 4 的倍数：

16 字节 → 21.33 字符 → 22 字符 + `==` = 24 字符

PHC 格式故意去掉 =：

标准 Base64PHC Base64
16 字节 saltUlhrVTlVbVI2S08wa2szSQ==UlhrVTlVbVI2S08wa2szSQ

理由：= 在 URL 里要转义、在 shell 里是赋值符。padding 是冗余信息（长度本来就能反推），删了不丢数据。

PHP base64_decode() 两种都兼容。但 Python 严格：

import base64
base64.b64decode('UlhrVTlVbVI2S08wa2szSQ')     # binascii.Error
base64.b64decode('UlhrVTlVbVI2S08wa2szSQ==')   # OK

s = 'UlhrVTlVbVI2S08wa2szSQ'
base64.b64decode(s + '=' * (-len(s) % 4))      # 手动补齐才能解

bcrypt 的 “Base64” 不是标准 Base64

bcrypt hash 里也有看似 Base64 的字段，但字母表不同：

标准 Base64:  A B C D E F G H I J K ... + /
bcrypt:       . / A B C D E F G H I ... 8 9
              ↑ 起头就不一样

bcrypt 沿用 OpenBSD crypt_blowfish 的历史字母表，索引 0 是 . 不是 A。

直接 base64_decode($bcrypt_part) 得到完全错误的字节 —— silent failure，没有报错。要解 bcrypt 内部的 salt / hash 必须用专门的字母表映射，或者干脆别解（业务代码 password_verify() 就够了）。

主流 hash 格式的 salt 编码对比

放一起看，差异只在算法和字母表 —— salt 原始长度（16 字节 / 128 bit）和编码方式（base64 无 padding）三家全一致，这是密码学共识的”足够防 rainbow table”门槛：

Hash 格式原始 salt编码字母表编码后字符数
argon2id（PHC）16 字节标准 A-Z a-z 0-9 + /22
bcrypt16 字节私有 ./ A-Z a-z 0-922
scrypt / PBKDF2（PHC）16 字节标准22

→ 这也解释了为什么把 bcrypt hash 喂给 base64_decode() silent failure：算法是 base64，但字母表对不上，输出字节全是垃圾。

RFC 9106 (Argon2) / PHC string format spec / Password Hashing Competition 主页。

PHP libvips 写文字：Pango 接管 layout，不用算 bbox

GD 那一套 baseline / bbox / 减偏移 在 libvips 里不存在。Image::text() 走 Pango 做 layout —— 你给一个 width 框 + align 方向，结果 image 直接拿到 $txt->width / $txt->height，没有 baseline 这回事。

Pango 是什么

GNOME 项目下的文字 layout 库。渲染一段文字本来要分两步，libvips 把第一步外包给了它：

阶段做什么谁来做
Layout选字体、断行、字距、对齐、Bidi（阿拉伯/希伯来从右到左）、CJK 换行规则、emoji shapingPango
Rasterize把矢量字形画成像素FreeType / Cairo

GD 的 imagettftext 只做第二步 —— 调用方必须自己算宽高、自己拆行、自己处理对齐，所以才有 imagettfbbox 8 元素数组、baseline 减偏移那堆脏活。libvips 直接传 width: 800, align: 'high'，Pango 帮你折行 + 对齐，结果 image 的 width / height 就是排版后的真实尺寸 —— baseline 这个概念在调用方完全不出现。

额外 bonus：开 rgba: true 之后可以传 Pango markup，单段里混排粗体 / 颜色 / 字号：

$txt = Image::text(
    '<span foreground="red" weight="bold">警告</span> 内容超过限制',
    ['font' => 'Sans 20', 'rgba' => true]
);

GD 想做这个得渲染 N 次再拼。

实际用法：右下角水印

同样的场景，libvips 写法：

use Jcupitt\Vips\Image;

$bg  = Image::newFromFile('photo.jpg');
$txt = Image::text('© 2026', [
    'font'  => 'Sans Bold 24',
    'width' => $bg->width - 40,   // 框内自动 word-wrap
    'align' => 'high',
    'rgba'  => true,              // 不开是灰度 mask，没法合到彩图
]);

$bg->composite($txt, 'over', [
    'x' => 20,
    'y' => $bg->height - $txt->height - 20,
])->writeToFile('out.png');

GD 同场景要：imagettfbbox 拿 8 元素 → $bbox[2] - $bbox[0] 算宽 → $bg_w - $width - 20 算 x → 还要担心 baseline 跟图像 y 轴方向相反。libvips 一步：align: high。

Image::text() 关键 options（官方文档）：

Option作用默认
fontFontconfig 名，如 'Sans Bold 24' / 'Noto Serif CJK SC 18'—
fontfile加载非系统字体的 .ttf 路径，需配合 font 用名字引用—
width换行宽度（像素），超过按 word boundary 折行不换行
align'low' / 'centre' / 'high' —— 多行的水平对齐—
justify两端对齐（拉伸词间空格），boolfalse
spacing行距（像素）字体自然行距
dpi渲染 DPI，影响实际像素大小72
rgba出彩色 / emoji / Pango markup，不开是 8-bit 灰度 maskfalse

部署成本是 libvips 在 PHP 圈用得不如 GD 普及的主因 —— ext-gd 几乎随 PHP 自动装好，libvips 要：

# Debian/Ubuntu
apt install libvips42 libvips-dev

# macOS
brew install vips

# PHP 端二选一
pecl install vips                                # 传统 C extension
# 或开 FFI（PHP 7.4+），用 jcupitt/php-vips 包，无需编译扩展
composer require jcupitt/vips

容器里加一行 apt install libvips42 就够，但这一行常被忘。装上之后水印 / 缩略图 / 多行排版量大时，libvips 的内存占用和吞吐都比 GD 强一个数量级 —— 它是流水线架构，不像 GD 把整张图按解压尺寸驻留内存。

PHP 安装工具的几条路径

PHP 安装按”装什么 + 在哪装”分四条路径，每条各自的工具：

纯 PHP 代码（复制即用）

  • PEAR — PHP Extension and Application Repository。最早的类库分发，用前要 include。已被 Composer 取代，基本退场。
  • Composer — PHP 依赖管理工具，灵感来自 npm / Bundler。composer.json 声明依赖，composer install 拉齐，事实标准。

C 扩展，裸机环境（编译 + 改 php.ini）

  • PECL — PHP Extension Community Library。PEAR 的子项目，分发 redis、apcu 这类底层扩展。已被官方标记 deprecated。
  • PIE — The PHP Installer for Extensions。PHP 基金会官方接管，取代 PECL，命令风格抄 Composer。

C 扩展，Docker 容器场景

  • docker-php-ext-install — 官方 PHP Docker 镜像内置脚本，容器里装 C 扩展的事实标准。
  • install-php-extensions — 上面的友好封装，自动处理依赖、编译选项、共享库。Dockerfile 里写 PHP 几乎人人用它。

PHAR 形式的 CLI 开发工具（PHPUnit、phpstan、psalm 这种）

  • PHIVE — PHAR Installation and Verification Environment。装 PHAR 工具并校验签名，避免污染 composer.json。

七个命令并排看：

# 纯 PHP 代码
pear install Mail                        # 老路子，已退场
composer require monolog/monolog         # 现代

# C 扩展（裸机）
pecl install apcu                        # 老路子（deprecated）
pie install apcu/apcu                    # 现代

# C 扩展（Docker），以装 gd 为例——需要系统库 + configure
apt-get install -y libfreetype-dev libjpeg-dev libpng-dev \
  && docker-php-ext-configure gd --with-freetype --with-jpeg \
  && docker-php-ext-install gd            # 官方：3 步联动
install-php-extensions gd                 # 友好封装：1 行搞定

# PHAR CLI 工具
phive install phpunit

记忆：

  • PHP 代码 → Composer
  • C 扩展，裸机 → PIE（PECL 已 deprecated）
  • C 扩展，Docker → install-php-extensions
  • PHAR CLI 工具 → PHIVE

PHP 图像处理：GD / Imagick / libvips 选型与层级

PHP 圈做图像处理有三个 native 选择 —— GD、Imagick、libvips。它们看起来功能重叠（都能 resize / 转格式 / 合成），关键差异不在「支持哪些格式」（最终都调 libpng / libjpeg / libwebp 同一层 codec），而在 C 引擎怎么调度内存，以及它们在调用链里所处的层级。

三个 lib 对比

Lib安装内存模型适用
GDapt install php-gd（PHP 自带 ext）整张图按解压尺寸驻留90% 项目默认
Imagickapt install imagemagick php-imagick整图驻留，内存最大要 HEIC / PSD / PDF / 色彩配置文件
libvipsapt install libvips42 + composer require jcupitt/vips流水线，按 chunk 处理高吞吐缩略图、大图（> 4K）

GD 的「整张图驻留」是常被忽略的成本：一张 4000×3000 的 JPEG 压缩后 2 MB，解压到内存是 4000 × 3000 × 4 byte ≈ 46 MB raw。PHP-FPM 默认 memory_limit = 128M，并发处理两三张就 OOM kill。libvips 是 demand-driven streaming，同一张图峰值内存可压到几 MB（详见 libvips 写文字那篇）。

Imagick 在 2026 仍是「功能最全但坑最多」 —— CVE 历史长，上游 policy.xml 默认禁用一堆 coder（PDF / PS / HTTPS / MSL），运维要反向解锁才能用全。容器镜像也最大。除非真要 HEIC / PSD，否则 GD 或 libvips 更划算。

调用层级

┌──────────────────────────────────────────────────────────────┐
│ Layer 1  应用 wrapper（PHP userland）                         │
│   Intervention v3 / Glide / 自写代码                          │
│   ↑ driver 可切换                                             │
├──────────────────────────────────────────────────────────────┤
│ Layer 2  PHP 扩展（C 绑定 / FFI）                             │
│   ext-gd        ext-imagick       ext-vips / FFI              │
├──────────────────────────────────────────────────────────────┤
│ Layer 3  Native C 引擎                                        │
│   libgd         ImageMagick       libvips                     │
│   （内存模型差异在这一层）                                     │
├──────────────────────────────────────────────────────────────┤
│ Layer 4  格式 codec（三个引擎共享）                            │
│   libpng  libjpeg(IJG)  libwebp  libavif  FreeType            │
├──────────────────────────────────────────────────────────────┤
│ Layer 5  系统库                                                │
│   zlib（libpng 依赖）fontconfig（FreeType 字体查找）         │
└──────────────────────────────────────────────────────────────┘

两个非显然结论：

Layer 4 是共享的 —— 三个 C 引擎最终都调 libpng / libjpeg / libwebp 读写格式字节。所以问「GD 能不能输出 WebP」本质是问「GD 编译时有没有 link libwebp」，而不是 GD 算法层面的能力。apt install php-gd 把 libpng / libjpeg 作为依赖一起拉进来，要 WebP 还得加 --with-webp 重编。Imagick 多出来的 HEIC / PSD 也只是它在 Layer 3 自己挂了更多 codec wrapper，不是 Layer 4 神奇变多。

Layer 1 跟 Layer 2/3 解耦 —— Intervention v3 是 driver-swappable，业务代码写在 Layer 1 抽象层，未来从 GD 升 libvips 只换 driver 注册不动业务：

// config/image.php
'driver' => env('IMAGE_DRIVER', 'gd'),   // 切 'imagick' / 'vips' 不动业务代码

写代码时该有意识地利用这一层 —— 直接 imagecreatefromjpeg() 把自己钉死在 Layer 2 上，迁移成本就再也下不来了。

PHP __invoke：实例 callable，相当于带状态闭包

__invoke 是 PHP 的 magic method —— 类里定义了它，实例就能当函数直接调用：

class Adder {
    public function __construct(private int $base) {}
    public function __invoke(int $x): int { return $this->base + $x; }
}

$add5 = new Adder(5);
$add5(3);                     // 8
is_callable($add5);           // true

价值不在”语法甜”，在承载状态的 closure。原生 closure 用 use ($base) 捕获变量；__invoke 用 __construct 注入依赖、$this 持有任意复杂状态，还能继承、被 mock、实现接口。

单动作 controller（Laravel）

路由绑类不绑方法，文件 = 端点。一个 controller 做一件事，不再有 8 个 action 挤一个 class：

class ShowProfile extends Controller
{
    public function __invoke(int $id)
    {
        return view('user.profile', ['user' => User::findOrFail($id)]);
    }
}

Route::get('user/{id}', ShowProfile::class);

文档：Single Action Controllers | Laravel。

策略 / provider 对象

需要”可配置的函数”时 —— 查表、解析、路由匹配 —— 比 closure 更适合，因为有完整 OOP toolkit 可用。AWS SDK 的 PatternEndpointProvider 就是这个套路：构造时吃配置 patterns，调用时按 service/region 查规则、返回 endpoint：

$provider = new PatternEndpointProvider($patterns);
$endpoint = $provider(['service' => 's3', 'region' => 'us-east-1']);

调用方不必知道里面是查表、远程 fetch 还是 ML 推理 —— 签名永远是”参数 in, 结果 out”。换实现不动 caller。

一个坑：callable ≠ Closure

is_callable($obj) 返回 true，但 $obj instanceof Closure 是 false。type-hint 写 Closure 会拒掉 __invoke-able 对象，写 callable 才接受：

function run(callable $fn) { return $fn(1); }     // ✅ 接受 Adder 实例
function run(Closure $fn)  { return $fn(1); }     // ❌ TypeError

要把 __invoke 对象转成真正的 Closure 实例（比如某 API 强制 Closure 入参），用：

$closure = Closure::fromCallable($add5);  // PHP 7.1+
$closure = $add5(...);                    // PHP 8.1+ first-class callable

PHP strict_types 看调用方，不看声明方

declare(strict_types=1) 关掉 PHP 的隐式类型转换 —— 函数 type-hint 写 int $x，不再吞 1.0 / "15" / true，传错直接 TypeError：

declare(strict_types=1);
function add(int $a, int $b): int { return $a + $b; }
add(1.0, 2.0);
// PHP Fatal error: Uncaught TypeError: add(): Argument #1 ($a) must be of type int, float given

不写 declare 行（默认 weak mode）时 add(1.0, 2.0) 返回 int(3)，安静吞掉。

反直觉点：strict 看 caller，不看 callee

文件级声明会让人下意识以为”哪边定义函数 / 类，哪边的 mode 控制类型检查”。反了：检查谁，看 call site 所在文件的 declare。

A.php 严格 / B.php 默认：

// A.php
declare(strict_types=1);
function add(int $a, int $b): int { return $a + $b; }

// B.php
require 'A.php';
add(1.0, 2.0);  // ✅ int(3) —— B 没声明 strict，照旧 coerce

A.php 默认 / B.php 严格：

// A.php
function add(int $a, int $b): int { return $a + $b; }

// B.php
declare(strict_types=1);
require 'A.php';
add(1.0, 2.0);  // ❌ TypeError —— B 声明了 strict，调用按 strict 检查

实际后果：第三方库无论自家标了多少 strict_types，决定你调它会不会抛 TypeError 的是你自己文件顶部那行。Composer 装回来的 vendor/ 大概率是 strict，但你在 weak 文件里传 1.0 给它的 int 参数，PHP 不会拦。

要”全项目 strict”，每个 .php 顶部都得写一遍。靠 php-cs-fixer 的 declare_strict_types rule 或 symplify/easy-coding-standard 自动补，别手写。

写法约束

必须是脚本里最早的语句（<?php 后第一行，前面不能有 use / 注释外的任何内容）：

PHP Fatal error: strict_types declaration must be the very first statement in the script

不能用 block 形式：

declare(strict_types=1) { ... }   // ❌ must not use block mode

declare 行只作用于当前文件，不会污染 require / include 进来的脚本 —— 这正是允许逐文件渐进迁移的设计。

Composer vendor 提交进 git 的 submodule 陷阱

Composer 官方 FAQ 答案是别 commit vendor/：仓库膨胀、依赖历史在你 repo 里重演一遍、vendor/<pkg>/.git 会被 git 当 submodule。最佳实践是 CI / 部署脚本里跑 composer install。

但有些场景必须把 vendor/ 进 git —— 离线 / 无 composer 的生产机、内网部署、国内网络拉国外包不稳 —— 这时 submodule 这条会静默出事：git 不报错，部署后才发现包是空的。

静默踩坑

composer install 在两种情况下会留下 vendor/<pkg>/.git：

  • 包没发 dist tarball，只能从 git 拉 source
  • 显式 --prefer-source，或 composer.json 配了 "preferred-install": "source"

git add vendor/ 看到子目录里有 .git，不会把里面的代码加进来，而是记一条 gitlink。git index 里每个条目都带一个类 Unix 八进制 mode —— 100644 普通文件、100755 可执行、120000 符号链接、160000 gitlink（指向另一个 git 仓库的某个 commit，即 submodule 引用）。git add 撞到带 .git 的子目录就自动选 160000，记一条 <sha> vendor/foo，里面文件一行不进。

git commit 通过，git push 通过，部署机 git pull 拉出来 —— 那个包目录是空的，运行时 class not found。报错栈帧跟 composer 无关，定位常要绕半天。

两个诊断信号：

git ls-files -s | grep 160000
# 160000 d4e5f6... 0 vendor/foo
#  ^ mode               ^ 被当 submodule 的路径

git diff --cached vendor/foo
# new file mode 160000
# Subproject commit d4e5f6...
#                   ^ 只 commit 引用，没有具体文件 diff

正确姿势

要 commit vendor/ 就显式走 dist：

composer install --prefer-dist

或写进 composer.json，所有人共享：

{
  "config": {
    "preferred-install": "dist"
  }
}

再加一条兜底 .gitignore，万一某个包没 dist 又被 fallback 到 source 也不会变 submodule：

/vendor/**/.git

已经踩了

vendor 已经以 gitlink 形式 commit 进去了，清掉重 add：

git rm -r --cached vendor
echo '/vendor/**/.git' >> .gitignore
git add vendor .gitignore
git commit -m "track vendor as files, not submodules"

--cached 只动 index 不删工作区文件，关键参数 —— 丢了会把磁盘上的 vendor 一起删掉。

See Composer FAQ: Should I commit the dependencies in my vendor directory?.

PhpStorm + Xdebug 容器联调，3 个名字必须对齐

PhpStorm 容器化 Xdebug 跑不通，最常见的根因不是端口、不是网络，而是 3 个魔法字符串没对齐 —— 错位时 IDE 静默忽略，没有任何报错。挨个对完，剩下的就是常规网络问题。

前置：mode + 触发方式

Xdebug 3 默认完全 disabled，要先开 mode；然后选 trigger 还是 always：

xdebug.mode=debug
xdebug.start_with_request=trigger    ; 推荐 dev 默认
  • trigger：只有请求带 XDEBUG_SESSION cookie / GET / POST / env 时才挂 debug；其他请求零开销。HTTP 场景永远选这个
  • yes：所有请求都进 debug session。每个请求都要给 IDE 发 connect，开发体感慢；只在 CLI 一次性脚本场景用（见最后一节）

1. serverName ↔ Servers > Name

容器内 PHP 通过环境变量告诉 Xdebug “我是哪个项目”：

PHP_IDE_CONFIG=serverName=laradock

Preferences > PHP > Servers 里那条 Server 的 Name 字段必须完全等于 serverName= 的值（这里是 laradock）。匹配上之后 PhpStorm 才会用这条 Server 上配的 path mapping 把容器里的 /var/www/... 翻译回宿主项目根，断点才落在正确文件上。

失败模式：debug session 建立成功（右下角图标变绿、Variables 面板出现），但断点永远不停 —— path mapping 找不到对应 Server，PhpStorm 不知道容器路径对应宿主哪个文件。

2. xdebug.idekey ↔ XDEBUG_SESSION value ↔ Run Configuration IDE key

三处的 IDE key 必须一致，默认值约定俗成是 PHPSTORM：

xdebug.idekey=PHPSTORM
  • 浏览器：Xdebug helper 扩展给请求加 Cookie: XDEBUG_SESSION=PHPSTORM
  • CLI：XDEBUG_SESSION=PHPSTORM php some-script.php
  • PhpStorm：Run > Edit Configurations > PHP Remote Debug 里 IDE key 填 PHPSTORM

cookie / env 的 value 同时承担两个角色 —— 触发 start_with_request=trigger 模式启动 session，且把 value 当作本次 session 的 IDE key 上报给 IDE。

失败模式：连接建立、断点也命中，但 PhpStorm 把 session 路由给错的 Run Configuration —— 看不到该 config 里配的环境变量、看不到自定义 evaluator。多人共享一台 Xdebug proxy 时尤其容易撞。

3. xdebug.client_host ↔ 容器回连宿主的地址

Xdebug 是 reverse connection：容器里的 PHP 主动回连 IDE，不是 IDE 拨进容器。所以 xdebug.client_host 必须能从容器解析到宿主 IP：

xdebug.client_host=host.docker.internal   ; Mac / Win / Linux 20.10+
xdebug.client_port=9003

跨平台统一写法（dockerhost 别名、host-gateway magic）。

失败模式：容器里 tail -f Xdebug 日志能看到 Could not connect to client. Tried: host.docker.internal:9003 —— 这条是少数不静默的失败，因为 Xdebug 自己在抱怨连不上。

CLI / Artisan 用 -d 覆盖

PHP CLI 走的不是 fpm，加载的是 /etc/php/X.Y/cli/conf.d/*.ini 而不是 fpm 的 ini。fpm 那边配好 Xdebug、CLI 仍然没动静是常见盲区。两种解法：

  • 把同样的 xdebug.* 配到 cli/conf.d/20-xdebug.ini，CLI 全局开（开发机 OK，生产别这么干）
  • 用 php -d 临时覆盖，只在需要时调试某条命令：
alias artisandebug='php \
  -dxdebug.mode=debug \
  -dxdebug.client_host=host.docker.internal \
  -dxdebug.client_port=9003 \
  -dxdebug.start_with_request=yes \
  artisan'

CLI 场景用 start_with_request=yes 是对的 —— 一次性进程，没机会注入 trigger，进来就要挂 debug。然后 artisandebug migrate:status 跑起来就能停断点。

排查顺序

跑不通时按这个顺序查，每一步都是上一步的前提：

  1. 容器里 php -i | grep xdebug 确认 mode=debug、idekey、client_host 都加载（CLI 调试要 php-cli 而不是 fpm 的输出）
  2. PhpStorm 右下角点过 Start Listening for PHP Debug Connections（电话图标变绿）
  3. 三对名字（serverName / idekey / client_host）逐一对齐
  4. Server 里 path mapping 设了容器 ↔ 宿主路径

排到 4 还不行，看 Help > Show Log in Finder 的 idea.log，搜 xdebug —— 99% 能直接定位到具体 mismatch。

PHP GD 文字对齐：imagettfbbox 8 元素 + 坐标系是相对 baseline

GD 写文字最容易踩的坑：imagettftext($img, $size, $angle, $x, $y, ...) 里的 $y 不是文字顶部、也不是中线，而是 baseline——拉丁字母 Hxg 中 H 的脚底。所以 imagettftext($img, 12, 0, 0, 0, ...) 写出来的字会完全跑到画布上方消失。

imagettfbbox 返回的也是 baseline 坐标系下的 8 元素：

索引   含义        典型值（angle=0）
[0]   左下 x        0
[1]   左下 y       -1     ← baseline 之下 1 px
[2]   右下 x      198
[3]   右下 y       -1
[4]   右上 x      198
[5]   右上 y      -20    ← baseline 之上 20 px（负值在上）
[6]   左上 x        0
[7]   左上 y      -20

y 轴在 baseline 下为正、之上为负——和 GD 全局图像坐标（左上角原点、向下增长）相反。这是混淆的根源。

实用算法

文字宽度：$bbox[2] - $bbox[0]；高度：$bbox[1] - $bbox[7]。

右对齐到 $rightX：

$bbox = imagettfbbox($size, 0, $font, $text);
$x = $rightX - ($bbox[2] - $bbox[0]);
imagettftext($img, $size, 0, $x, $y, $color, $font, $text);

水平居中到 $centerX：

$x = $centerX - intval(($bbox[2] - $bbox[0]) / 2) - $bbox[0];

减 $bbox[0] 是因为有些字体首字符左缘会突出到 x=0 之前（$bbox[0] 为负），不补回来会左偏。

垂直贴到顶边 $topY（让文字最高点对齐 $topY）：

$y = $topY - $bbox[7];   // bbox[7] 是负的，等价 topY + |bbox[7]|

隐藏坑

  • descender 不算 bounding box：g/p/q/y 的下伸尾巴在 bbox 外，多行排版按 bbox 算行高会让下行字头部蹭到上行尾巴。靠 $size * 1.2 这种经验系数加行距更稳
  • $angle 旋转的是文字，不是 bbox 坐标系含义——bbox 永远按”如果这段文字是水平正向的”来给点，旋转后想知道屏幕坐标得自己用 $angle 套旋转矩阵
  • PHP 8.0 起 imagettfbbox / imagettftext 是 imageftbbox / imagefttext 的别名——行为一致，新代码两套都行

字体文件路径必须是真实可读的 .ttf / .otf，相对路径默认相对 CWD 不是 script 目录——线上跑用 __DIR__ . '/font.ttf' 兜底。

Composer `--ignore-platform-reqs` 是安静的 bug 触发器

经典踩坑：本地 PHP 7.1，要装的包声明需要 PHP 7.4，composer require 直接拒：

Your requirements could not be resolved to an installable set of packages.
- hellogerard/jobby requires php >=7.4

Google 第一条答案是加 --ignore-platform-reqs：

composer require hellogerard/jobby --ignore-platform-reqs

报错没了，包装上了，看起来一切正常 —— 直到运行时撞上 PHP Fatal error: Cannot use null coalescing assignment operator (??=) 之类，因为这是 PHP 7.4 才有的语法。

--ignore-platform-reqs 字面就是”不检查平台依赖”，它把 composer.json 里 "require": { "php": ">=7.4" } 这条约束跳过去 —— composer 不知道这个包用了 7.4 才有的 syntax / API，硬装下去，运行才炸。安装时无错、运行时无关栈帧、composer.lock 还带着错版本 —— 是教科书级的静默 bug。

正解：用目标 PHP 版本的 binary 跑 composer。Composer 本身是个 PHP 脚本，跟着哪个 PHP 走就按哪个 PHP 检查约束：

which composer
# /usr/local/bin/composer

# 用 7.4 解释器跑 composer
/usr/local/opt/php@7.4/bin/php /usr/local/bin/composer require hellogerard/jobby

# 内存不够时临时关掉 memory_limit
/usr/local/opt/php@7.4/bin/php -d memory_limit=-1 /usr/local/bin/composer update -vvv

如果之前误用 --ignore-platform-reqs 装过，composer.lock 已经被污染，要 composer update --lock 重生成约束树。

PHP CLI 排查速查：`--ri` / `--re` / `php -a` 常被忽略

需要 PHP build 时带 readline，Homebrew 装的 php 默认有。

php-fpm -tt —— FPM 配置语法检查 + dump 解析后的最终值（双 t，单 t 只验语法不 dump）：

关键字指向LSB静态方法可用
self::写它的那个类（词法）否✓
parent::self 的父类否✓
static::实际调用时的类（late static binding，PHP 5.3+）✓✓
$this->当前实例所属的类✓✗（静态方法里无 $this）

self:: 和 static:: 的差别只在被子类继承后才显现：

class A
{
    public static function newSelf()   { return new self();   }
    public static function newStatic() { return new static(); }
}

class B extends A {}

var_dump(B::newSelf());   // object(A) — self 锁死在写它的 A 里
var_dump(B::newStatic()); // object(B) — static 跟着调用方

口诀：工厂方法要返回子类型 → static::；要锁定当前类不被覆盖 → self::。

CGI vs FastCGI vs PHP-CGI vs PHP-FPM

PHP 处理 web 请求要解决两件事——和 web server 通信的协议、PHP 自己的进程管理。这四个名词分两组踩在这两件事上：协议层是 CGI / FastCGI，PHP 实现是 PHP-CGI / PHP-FPM。

协议层

CGIFastCGI
进程模型每个请求 fork-and-exec 一次master + 长驻 worker pool
启动开销每次都加载解释器解释器常驻内存
数据流环境变量 + stdin/stdout二进制协议复用 socket 连接
现状历史遗留现代 web 标准

CGI 慢在反复 fork 和解释器反复初始化。FastCGI 把解释器留在内存，请求过来直接派给空闲 worker。

PHP 实现

PHP-CGIPHP-FPM
性质PHP 的 CGI binaryPHP 官方 FastCGI 进程管理器
进程管理无master + 动态 worker pool
改 php.ini必须杀进程重启，PHP 会停服kill -USR2 平滑 reload，请求不中断
现状淘汰主流（Laravel / WordPress / 大多数 PHP 栈）

PHP-FPM = PHP 解释器 + FastCGI 协议实现 + 进程管理。三件事的合体，不是单独某个组件。

实战：nginx + php-fpm

nginx 把 .php 请求按 FastCGI 协议丢给 php-fpm 的 socket：

location ~ \.php$ {
    fastcgi_pass   unix:/run/php/php8.2-fpm.sock;
    fastcgi_index  index.php;
    fastcgi_param  SCRIPT_FILENAME $document_root$fastcgi_script_name;
    include        fastcgi_params;
}

php-fpm 那头按 pool 配置 worker 数（pm.max_children 等），nginx 不感知 PHP 进程。

怎么选

现代栈一律 nginx + php-fpm，CGI / PHP-CGI 了解概念即可，生产不用。

PHP empty('0.0') 不是空

PHP 的 empty() 把 '0'（字符串零）当空，但 '0.0'（字符串小数零）当非空。DB 里把 decimal 字段存成 string 时会咬人——查到 '0.00' 误以为有值。

empty('0');           // true
empty('0.0');         // false ← 反直觉
empty((float) '0.0'); // true

判断数值前先 cast：empty((float) $value)。
