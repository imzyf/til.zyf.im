PHP 安装工具的几条路径

PHP 安装按”装什么 + 在哪装”分四条路径，每条各自的工具：

纯 PHP 代码（复制即用）

  • PEAR — PHP Extension and Application Repository。最早的类库分发，用前要 include。已被 Composer 取代，基本退场。
  • Composer — PHP 依赖管理工具，灵感来自 npm / Bundler。composer.json 声明依赖，composer install 拉齐，事实标准。

C 扩展，裸机环境（编译 + 改 php.ini）

  • PECL — PHP Extension Community Library。PEAR 的子项目，分发 redis、apcu 这类底层扩展。已被官方标记 deprecated。
  • PIE — The PHP Installer for Extensions。PHP 基金会官方接管，取代 PECL，命令风格抄 Composer。

C 扩展，Docker 容器场景

  • docker-php-ext-install — 官方 PHP Docker 镜像内置脚本，容器里装 C 扩展的事实标准。
  • install-php-extensions — 上面的友好封装，自动处理依赖、编译选项、共享库。Dockerfile 里写 PHP 几乎人人用它。

PHAR 形式的 CLI 开发工具（PHPUnit、phpstan、psalm 这种）

  • PHIVE — PHAR Installation and Verification Environment。装 PHAR 工具并校验签名，避免污染 composer.json。

七个命令并排看：

# 纯 PHP 代码
pear install Mail                        # 老路子，已退场
composer require monolog/monolog         # 现代

# C 扩展（裸机）
pecl install apcu                        # 老路子（deprecated）
pie install apcu/apcu                    # 现代

# C 扩展（Docker），以装 gd 为例——需要系统库 + configure
apt-get install -y libfreetype-dev libjpeg-dev libpng-dev \
  && docker-php-ext-configure gd --with-freetype --with-jpeg \
  && docker-php-ext-install gd            # 官方：3 步联动
install-php-extensions gd                 # 友好封装：1 行搞定

# PHAR CLI 工具
phive install phpunit

记忆：

  • PHP 代码 → Composer
  • C 扩展，裸机 → PIE（PECL 已 deprecated）
  • C 扩展，Docker → install-php-extensions
  • PHAR CLI 工具 → PHIVE

PHP strpos 找到时返回 0

strpos($haystack, $needle): int|false 找不到时返回 false，找到时返回位置 int——位置 0 在 if 里跟 false 等价，会被吞掉。

$s = 'abc';

if (strpos($s, 'a')) {            // 不输出，0 被当 false
    echo 'found';
}

if (strpos($s, 'a') !== false) {  // 正解
    echo 'found';
}

PHP 8.0+ 直接用 str_contains($haystack, $needle): bool，没有这个坑。

https://www.php.net/manual/zh/function.strpos.php

this / self / parent / static 在 PHP 类内部的区别

类内部访问成员有 4 个关键字，差别在「指向哪个类」与「是否跟着子类走」（多态 / late static binding）：

关键字指向LSB静态方法可用
self::写它的那个类（词法）否✓
parent::self 的父类否✓
static::实际调用时的类（late static binding，PHP 5.3+）✓✓
$this->当前实例所属的类✓✗（静态方法里无 $this）

self:: 和 static:: 的差别只在被子类继承后才显现：

class A
{
    public static function newSelf()   { return new self();   }
    public static function newStatic() { return new static(); }
}

class B extends A {}

var_dump(B::newSelf());   // object(A) — self 锁死在写它的 A 里
var_dump(B::newStatic()); // object(B) — static 跟着调用方

口诀：工厂方法要返回子类型 → static::；要锁定当前类不被覆盖 → self::。

CGI vs FastCGI vs PHP-CGI vs PHP-FPM

PHP 处理 web 请求要解决两件事——和 web server 通信的协议、PHP 自己的进程管理。这四个名词分两组踩在这两件事上：协议层是 CGI / FastCGI，PHP 实现是 PHP-CGI / PHP-FPM。

协议层

CGIFastCGI
进程模型每个请求 fork-and-exec 一次master + 长驻 worker pool
启动开销每次都加载解释器解释器常驻内存
数据流环境变量 + stdin/stdout二进制协议复用 socket 连接
现状历史遗留现代 web 标准

CGI 慢在反复 fork 和解释器反复初始化。FastCGI 把解释器留在内存，请求过来直接派给空闲 worker。

PHP 实现

PHP-CGIPHP-FPM
性质PHP 的 CGI binaryPHP 官方 FastCGI 进程管理器
进程管理无master + 动态 worker pool
改 php.ini必须杀进程重启，PHP 会停服kill -USR2 平滑 reload，请求不中断
现状淘汰主流（Laravel / WordPress / 大多数 PHP 栈）

PHP-FPM = PHP 解释器 + FastCGI 协议实现 + 进程管理。三件事的合体，不是单独某个组件。

实战：nginx + php-fpm

nginx 把 .php 请求按 FastCGI 协议丢给 php-fpm 的 socket：

location ~ \.php$ {
    fastcgi_pass   unix:/run/php/php8.2-fpm.sock;
    fastcgi_index  index.php;
    fastcgi_param  SCRIPT_FILENAME $document_root$fastcgi_script_name;
    include        fastcgi_params;
}

php-fpm 那头按 pool 配置 worker 数（pm.max_children 等），nginx 不感知 PHP 进程。

怎么选

现代栈一律 nginx + php-fpm，CGI / PHP-CGI 了解概念即可，生产不用。

PHP empty('0.0') 不是空

PHP 的 empty() 把 '0'（字符串零）当空，但 '0.0'（字符串小数零）当非空。DB 里把 decimal 字段存成 string 时会咬人——查到 '0.00' 误以为有值。

empty('0');           // true
empty('0.0');         // false ← 反直觉
empty((float) '0.0'); // true

判断数值前先 cast：empty((float) $value)。

https://www.php.net/manual/zh/function.empty.php
