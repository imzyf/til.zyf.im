LLM CLI proxy：rtk 是命令字典而非算法

rtk 是夹在 AI 编码助手和 shell 之间的 Rust CLI proxy，宣称帮 LLM 省 60-90% 的 token。它做的是 context compression —— LLM 想跑 git status，rtk 把噪音过滤后再喂回去。看着像「魔法压缩算法」，拆开后发现根本不是。

核心：一个 CLI 工具一份手写 parser

rtk 没有通用压缩引擎。src/cmds/ 下按工具切文件：

  • git/git.rs —— git status / git diff / git push 等子命令
  • python/pytest_cmd.rs、ruff_cmd.rs、mypy_cmd.rs
  • rust/cargo_cmd.rs
  • cloud/container.rs —— docker
  • js/npm_cmd.rs、tsc_cmd.rs、vitest_cmd.rs、playwright_cmd.rs

约 45 个 parser 文件覆盖 100+ 子命令。每个文件用 match 分发子命令，里面写死「哪些字段是噪音、压缩后长什么样」—— 手写领域知识，不是算法推导。

官网讲的 filtering / grouping / truncation / dedup 也不是引擎，只是 4 个共享字符串处理函数（dedup_lines() / truncate_middle() 之类）。parser 各自挑用哪几个 —— 同一函数在 git diff 里删 progress 行，在 docker logs 里折叠重复行。

新加一种工具支持 = 提一个 PR 加一个 <tool>_cmd.rs。changelog 看得清清楚楚：v0.38.0 加 GitLab glab、v0.40.0 加 Gradle，一条一条加。

关键技巧：拿机器格式做解析

rtk 的稳定性靠一个简单决定 —— 不解析给人看的输出。常见命令几乎都有 --porcelain / --json / --message-format 之类的脚本契约，rtk 就走这条。

机器契约是工具维护者给脚本用的稳定 API，UI 文案改了不影响。逆向人类输出（彩色、对齐、本地化）则一升级就崩。rtk 选前者 —— 这是它能跟着 100+ 工具活下去的根因，比任何「压缩算法」都重要。

工作流（以 rtk git status 为例）

LLM 决定跑 git status
  ↓
Claude Code PreToolUse hook 拦截 Bash 调用
  ↓ 改写成 `rtk git status`
rtk 进程：
  1. fork 真命令但走机器格式：git status --porcelain=v2 --branch
  2. parser 切字段：branch / ahead-behind / modified / untracked
  3. 紧凑模板渲染成单行
  ↓
返给 LLM，LLM 完全不知道有中间层

为什么不选其他路

候选方案为什么 rtk 没用
gzip 通用压缩LLM 不解压，喂进去还是压缩字节
让 LLM 自己 summarize慢、贵、丢信息，summarize 本身吃 token
盲截前 N 行没语义，可能切掉关键 error 行；切到的还是不稳的人类格式
per-command parser<10ms、确定性、保关键信号 —— 代价是每个工具都得单独写

推论

  1. 护城河是命令字典而非算法。任何人能写 pytest_cmd.rs，但要覆盖 90% 日常命令、跟着每个工具升级（git 改 porcelain、pytest 升级 reporter）才是真功夫。
  2. 工具支持决定收益。rtk hugo、rtk astro 不存在就是裸过。
  3. 永远滞后于命令本身。git 改了 porcelain 格式得等 rtk 跟上。
  4. PreToolUse hook 才是 UX 关键。靠 prompt 教 LLM「记得用 rtk」不可靠；hook 把改写做在协议层，LLM 不感知。
