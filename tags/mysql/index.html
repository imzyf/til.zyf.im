GROUP_CONCAT 截断

生产踩了一个：GROUP_CONCAT 聚合的结果被悄悄截断，默认上限只有 1024 bytes。

SELECT @@GROUP_CONCAT_MAX_LEN;
-- 1024

会话临时调大：

SET SESSION group_concat_max_len = 1000000;

永久改 my.cnf：

[mysqld]
group_concat_max_len = 1000000

SQL AND / OR 执行优先级

AND 优先级高于 OR，所有 SQL 方言通用：

SELECT id FROM t WHERE c1 OR c2 AND c3;

-- 实际等价于（AND 先绑定）：
SELECT id FROM t WHERE c1 OR (c2 AND c3);

-- 不是这样（很多人下意识这么想）：
SELECT id FROM t WHERE (c1 OR c2) AND c3;

口诀：AND 看成 *，OR 看成 +，乘号先算。

Bug 不会报错——query 跑成功，只是结果错——所以 WHERE 里只要 AND 和 OR 混用，永远写括号，别赌读者（或几个月后的自己）记得优先级。

MySQL DELETE 子查询自引 1093 错误

想删一张表里的重复数据，按 name 分组保留 MIN(id) 那条：

DELETE FROM student
WHERE id NOT IN (SELECT MIN(id) FROM student GROUP BY `name`);

直接跑会报：

ERROR 1093 (HY000): You can't specify target table 'student' for update in FROM clause

MySQL 不允许在 DELETE / UPDATE 的子查询里直接引用目标表——执行时一边扫一边改会让结果不确定，干脆禁掉。

绕开的办法：再套一层 derived table，强制 MySQL 把子查询物化成临时表后再用，与目标表解耦：

DELETE FROM student
WHERE id NOT IN (
  SELECT id FROM (
    SELECT MIN(id) AS id FROM student GROUP BY `name`
  ) tmp                                              -- 关键：再包一层
);

MySQL 8.0+ 用 CTE 写更清晰，CTE 同样会被物化，所以一样能绕开 1093：

WITH keep AS (SELECT MIN(id) AS id FROM student GROUP BY `name`)
DELETE FROM student WHERE id NOT IN (SELECT id FROM keep);

PostgreSQL / SQL Server 没这个限制，直接写第一种形式就行。

MySQL FOR UPDATE / FOR SHARE 真实加锁范围

SELECT ... FOR UPDATE 和 FOR SHARE（8.0.1 之前叫 LOCK IN SHARE MODE）加的锁经常比写 SQL 时以为的大一圈。规则不复杂，但每条单独看都反直觉。

S 锁 vs X 锁

已有 S已有 X
申请 S✅❌
申请 X❌❌
  • FOR SHARE = S 锁，多事务可同时拿
  • FOR UPDATE / INSERT / UPDATE / DELETE = X 锁，互斥

死锁陷阱：两个事务都拿了同一行 S 锁，任何一方想升级成 X 锁都会死锁——必须先等对方释放。所以 FOR SHARE 后立刻 UPDATE 是高危模式。

锁加在索引上，不是行上

SELECT * FROM t WHERE non_indexed_col = 'x' FOR UPDATE;

WHERE 列没索引时 InnoDB 会对扫描过的所有索引项加锁——走全表扫描就等于锁全表。EXPLAIN 看到 type=ALL 配上 FOR UPDATE 基本就是事故现场。

确保 WHERE 列有索引，且 EXPLAIN type 不是 ALL，加锁才能精准。

REPEATABLE READ 下 next-key lock 把 gap 也锁了

MySQL 默认隔离级别 REPEATABLE READ，InnoDB 此时用 next-key lock = record lock + gap lock：

-- 表里现有 id: 1, 5, 10
SELECT * FROM t WHERE id BETWEEN 3 AND 8 FOR UPDATE;
-- 锁 id=5（record lock）+ (1,5)、(5,10) 区间（gap lock）
-- 另一事务 INSERT id=4 / id=7 全部阻塞

更反直觉的是没匹配上的查询也加 gap 锁：

-- 表里现有 id: 1, 5, 10
SELECT * FROM t WHERE id = 7 FOR UPDATE;
-- 没行可锁，但锁住 (5,10) 区间，INSERT id=6/7/8/9 全部阻塞

唯一索引 + 等值匹配且匹配到——只加 record lock 不加 gap lock。这是少数能精准加锁的场景。

READ COMMITTED 关掉 gap 锁

会话或事务级改成 RC，gap lock 全部消失，只剩 record lock：

SET SESSION TRANSACTION ISOLATION LEVEL READ COMMITTED;

代价是放弃可重复读、接受 phantom row。在高并发 OLTP 场景里去 gap lock 能显著降低死锁率，很多 ORM 默认推荐 RC。

FOR SHARE 在 8.0+ 多的 NOWAIT / SKIP LOCKED

默认拿不到锁会等 innodb_lock_wait_timeout（50s），8.0+ 加了两个修饰符——LOCK IN SHARE MODE 不支持，FOR SHARE 才有：

SELECT * FROM t WHERE id = 5 FOR UPDATE NOWAIT;
-- 拿不到立刻报 ER_LOCK_NOWAIT，不等

SELECT * FROM t WHERE status = 'pending' ORDER BY id FOR UPDATE SKIP LOCKED LIMIT 1;
-- 跳过被锁的行

SKIP LOCKED 是写消息队列 / worker 抢任务的标准模式——多个 worker 同时跑这条 SQL，自动每人拿到不同任务，零竞争零冲突。比 FOR UPDATE + 应用层重试简单一个数量级。

MySQL 看优化器决策的三件套

想知道 MySQL 内部怎么处理你的 SQL，按”看的层次”挑工具：

工具看到什么适用场景
EXPLAIN物理执行计划（join 顺序、用了哪个 index、估算扫描行数）日常 SQL 优化第一站
SHOW WARNINGS after EXPLAINMySQL 重写 / 改写后的 SQL联表、视图、子查询场景，看实际跑的版本
OPTIMIZER_TRACE优化器完整决策过程（cost 估算、为什么没选某 index）选 index 反常识时排查

EXPLAIN + SHOW WARNINGS

EXPLAIN SELECT * FROM `order` o JOIN user u ON o.uid = u.id WHERE u.email = 'a@b.com';
SHOW WARNINGS;

SHOW WARNINGS 里 Note 1003 行就是 MySQL 改写后的 SQL——优化器可能调换 join 顺序、把外连接转内连接、把子查询折成 join。看了重写版才知道实际跑的是什么，原 SQL 已经不重要了。

optimizer_trace

SET optimizer_trace = 'enabled=on';
SELECT ... ;                                         -- 你要分析的 SQL
SELECT * FROM information_schema.OPTIMIZER_TRACE;
SET optimizer_trace = 'enabled=off';

返回的 TRACE 列是一大坨 JSON，覆盖三个阶段：

  • join_preparation — SQL 解析、视图展开、子查询改写
  • join_optimization — 候选 index、各 index 的估算 cost、最终选谁
  • join_execution — 执行细节

最常看 join_optimization，里面 range_analysis / chosen_range_access_summary 直接告诉你”为什么没选 idx_email”——通常是 rows_estimation 估算 cost 比全表扫还高，或者 join 顺序变了导致 index 不可用。

trace 是 session 级别，默认只保留最近一次，且开启后所有 query 都会被 trace（有性能开销），用完立刻关。

MySQL JSON 列建索引

MySQL 5.7+ 支持原生 JSON 数据类型。两个坑：

  • JSON 列不能有 non-NULL 默认值——要么 nullable，要么应用层填
  • 不能直接在 JSON 字段上建索引——要靠 Generated Column 包一层

用 Generated Column 索引 JSON 字段

把要查的 JSON 路径抽成一个虚拟列，再在虚拟列上建索引：

CREATE TABLE players (
  id               INT UNSIGNED NOT NULL,
  player_and_games JSON NOT NULL,
  name_virtual     VARCHAR(20) GENERATED ALWAYS AS (player_and_games ->> '$.name') NOT NULL,
  PRIMARY KEY (id),
  KEY idx_name (name_virtual)
);

->> 是 -> + JSON_UNQUOTE 的简写——直接拿字符串而不是 quoted JSON value。

Virtual 列只存元数据不落盘（默认），Stored 列落盘到磁盘。索引 JSON 字段几乎都用 Virtual，省空间且没性能差。

验证索引命中

EXPLAIN SELECT * FROM players WHERE name_virtual = 'Sally';
-- key: idx_name

参考 MySQL 5.7 新特性 JSON · 如何索引 JSON 字段 | mysql.taobao.org · JSON 函数参考

MySQL Illegal mix of collations

Illegal mix of collations (utf8mb4_general_ci,IMPLICIT) and (utf8mb4_unicode_ci,IMPLICIT) for operation

utf8mb4_general_ci 和 utf8mb4_unicode_ci 两列直接做 JOIN / 比较时报错——MySQL 不会自动转换 collation。

解决

根治：统一字段 collation。MySQL 8.0+ 默认 utf8mb4_0900_ai_ci（基于 Unicode 9.0），5.7 默认 utf8mb4_general_ci，老库迁 8.0 时 collation 不会自动改，要小心遗留表。

临时绕：SQL 里 COLLATE 显式转：

SELECT * FROM a JOIN b ON a.name = b.name COLLATE utf8mb4_unicode_ci;

三种 collation

排序依据性能用途
utf8mb4_general_ci简化 ASCII 规则最快5.7 老库默认，新库不推荐
utf8mb4_unicode_ciUnicode 4.0略慢5.7 推荐
utf8mb4_0900_ai_ciUnicode 9.0≈ unicode_ci8.0+ 默认

性能差距在绝大多数业务里可忽略，同库统一比选哪个更重要。

MySQL 取每组最大行：子查询 ORDER BY 不生效

经典需求：每个用户按某字段取最大那一行。MySQL 5.6 时代流传一个”民间偏方”——子查询先 ORDER BY 倒序、外层 GROUP BY 拿”第一条”：

-- 5.6 上看似生效，5.7 起失效
SELECT * FROM (
  SELECT id, no, name FROM testorder ORDER BY no DESC
) a GROUP BY a.name;

5.6 上巧合返回每组 no 最大的那一行；5.7 默认 sql_mode=ONLY_FULL_GROUP_BY 直接拒；即便关掉 ONLY_FULL_GROUP_BY，返回的也是按物理存储顺序的”任意一行”，ORDER BY 被忽略。

根因不是 MySQL 的 bug，是 SQL 标准本来就这么定义：

A “table” (and a subquery in the FROM clause too) is — according to the SQL standard — an unordered set of rows.

FROM 后的子查询返回的是集合不是序列。子查询里写 ORDER BY，优化器允许直接丢弃。GROUP BY 选哪一行也是自由的，更不受外层 ORDER BY 影响——5.6 那种”先排序再分组取第一”是优化器实现细节，不是契约。

正确写法是 self-join：先聚合算出每组最大值，再回表取整行。

SELECT a.id, a.no, a.name
FROM testorder a
INNER JOIN (
  SELECT name, MAX(no) AS no FROM testorder GROUP BY name
) b ON a.name = b.name AND a.no = b.no;

MySQL 8.0+ 改用 window function 更直观：

SELECT id, no, name FROM (
  SELECT id, no, name,
         ROW_NUMBER() OVER (PARTITION BY name ORDER BY no DESC) AS rn
  FROM testorder
) t WHERE rn = 1;

ANY_VALUE(col) 只是绕开 ONLY_FULL_GROUP_BY 的报错，不解决”取哪一行”的不确定性——MySQL 文档原话：服务器自由选择，结果 indeterminate。别拿它当”5.7 兼容补丁”用。

MySQL 5.7 默认 sql_mode 多了 6 项：升级踩坑清单

5.6 → 5.7 升级最容易踩的坑：默认 sql_mode 从只有 NO_ENGINE_SUBSTITUTION 一项变成 7 项。多出来的每一条都可能让”5.6 跑得好好的 SQL”在 5.7 直接报错。

SELECT @@SQL_MODE;
-- 5.6: NO_ENGINE_SUBSTITUTION
-- 5.7: ONLY_FULL_GROUP_BY,STRICT_TRANS_TABLES,NO_ZERO_IN_DATE,
--      NO_ZERO_DATE,ERROR_FOR_DIVISION_BY_ZERO,NO_AUTO_CREATE_USER,
--      NO_ENGINE_SUBSTITUTION

新增 6 项各自封禁的”老 MySQL 容忍行为”：

  • ONLY_FULL_GROUP_BY：SELECT 引用了 GROUP BY 外的非聚合列直接拒（5.7.5 起还能识别 functional dependency：主键 + 全字段唯一确定的列免检）。绕开用 ANY_VALUE(col)，但要清楚它不解决”取哪一行不确定”的问题
  • STRICT_TRANS_TABLES：事务表上 INSERT/UPDATE 拿到非法值（字符串超长被截断、无符号字段写负数、NOT NULL 字段无默认值且未提供）直接报错而不是静默修正
  • NO_ZERO_IN_DATE + NO_ZERO_DATE：禁 2017-00-15、0000-00-00 这种”日期占位符”。老业务把 0000-00-00 当 NULL 用的全部踩雷
  • ERROR_FOR_DIVISION_BY_ZERO：INSERT/UPDATE 中除零报错（之前默认返 NULL）
  • NO_AUTO_CREATE_USER：GRANT ... TO 'user' 不再隐式创建用户。8.0 起这条被移除——CREATE USER 是唯一合法路径

升级前的 checklist：

-- 1. 测试库上跑生产读流量，盯 ONLY_FULL_GROUP_BY 报错
-- 2. 写流量看 STRICT_TRANS_TABLES、NO_ZERO_DATE 报错
-- 3. 应急可临时降级，但只是续命，不是解法
SET GLOBAL sql_mode = 'NO_ENGINE_SUBSTITUTION';

主备复制场景额外注意：先升级备库再升主库——主库以高 sql_mode 写入的 binlog，老 sql_mode 的备库未必能 replay。

MySQL 查重去重：HAVING + JOIN 子查询

按某列查重——GROUP BY 之后用 HAVING 筛 COUNT(*) > 1：

SELECT col_a, COUNT(*)
FROM t1
GROUP BY col_a
HAVING COUNT(*) > 1;

记法：WHERE 在聚合之前过滤行、不能引用 COUNT(*)；HAVING 在聚合之后过滤组、可以。

去重保留每组 id 最大的——先把 keep set 算出来当 derived table，再 JOIN 回原表删剩下的：

DELETE a
FROM t1 a
JOIN (
  SELECT col_a, MAX(id) AS keep_id
  FROM t1
  GROUP BY col_a
  HAVING COUNT(*) > 1
) b ON a.col_a = b.col_a
WHERE a.id < b.keep_id;

不能直接写 DELETE FROM t1 WHERE id NOT IN (SELECT MAX(id) FROM t1 GROUP BY col_a)——MySQL 禁止 DELETE / UPDATE 子查询里直接引目标表（错误 1093）。包一层 derived table 触发物化、与目标表解耦就能绕开，机制详见 210129-delete-self-reference-1093.md。

要保留最早一条把 MAX → MIN、< → >。
