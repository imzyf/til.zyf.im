spec-kit vs. superpowers vs. gstack:三种 coding agent 工作流框架

三个都是套在 coding agent(主要是 Claude Code) 外面的工作流框架，但把它们放一起比不是因为像，而是因为它们赌的「正确性从哪来」是三个不同的答案:spec-kit 赌规格,superpowers 赌流程纪律,gstack 赌多视角评审。选型不是比谁命令多，是认领你当前最缺哪一种。

框架出身赌注载体
spec-kitGitHub 官方规格是可执行的唯一真理specify CLI + /speckit.* 命令
superpowersJesse Vincent (obra)流程纪律逼出质量自动触发的 skills 插件
gstackGarry Tan (YC)多角色评审补单视角盲区23 个纯 Markdown slash command

三者都是 MIT，都不止支持 Claude Code(各自能跨 7-30+ 个 agent),都不锁厂商——这些都不是差异点，差异全在下面。

spec-kit:规格是唯一真理来源

GitHub 官方的 Spec-Driven Development。核心主张一句话：规格是源码，代码是规格的编译产物。你维护的是 spec.md 不是 .py;改需求改 spec，再 implement。产物 spec.md → plan.md → tasks.md 被当成 agent 无关的规格层——换 agent 不换规格。

uv tool install specify-cli --from git+https://github.com/github/spec-kit.git@vX.Y.Z
specify init my-project --integration copilot

然后在 agent 里按顺序走命令，每步落一个文件：

/speckit.constitution   # 立项目宪法 (不可动的原则)
/speckit.specify        # 写「做什么 / 为什么」→ spec.md
/speckit.clarify        # 澄清模糊点 (可选，建议 plan 前跑)
/speckit.plan           # 定技术栈方案 → plan.md
/speckit.tasks          # 拆可执行任务 → tasks.md
/speckit.implement      # 执行全部任务

最适合：需求先行、跨会话、多文件的功能开发，团队要一层标准化的规格。

superpowers:用流程纪律逼出质量

Jesse Vincent(GitHub obra) 的 skills 框架。它不给你命令表，给一套自动触发的 skills——agent 一旦识别到该用某 skill 就必须用，把「偷懒跳步」这条路堵死。内核是真正的红绿 TDD + 每个任务派新鲜子代理，两段式审查 (先查规格符合、再查代码质量)。

Claude Code 装法一行：

/plugin install superpowers@claude-plugins-official

强制流水线 (skills 依次自动登场):

brainstorming               # 苏格拉底式追问，细化想法 → design doc
using-git-worktrees         # 设计批准后开隔离工作区
writing-plans               # 拆成 2-5 分钟的小任务，带确切路径 + 完整代码
subagent-driven-development # 每任务派新鲜子代理，两段式 review
test-driven-development     # RED → GREEN → REFACTOR，先写失败测试
requesting-code-review      # 对着 plan 审，critical 问题卡住不放行

最适合：在意测试纪律和执行严谨度，想防 agent「不写测试 / 静默改需求」。

gstack:把一个 agent 变成一支团队

Garry Tan(YC 总裁) 的设置：23 个专家 + 8 个 power tool,全是纯 Markdown slash command，无专有 runtime。它赌结构化 prompt 而非自建工具才是对的抽象层——每个角色只约束 agent 这一阶段盯一件事，靠视角切换补单一视角的盲区。

装法 (全局装 + ./setup):

git clone --single-branch --depth 1 https://github.com/garrytan/gstack.git ~/.claude/skills/gstack \
  && cd ~/.claude/skills/gstack && ./setup

一个 Sprint 里角色依次登场 (Think → Plan → Build → Review → Test → Ship → Reflect):

/office-hours       # YC 办公时间：这产品该不该做
/plan-ceo-review    # CEO:重想问题本身
/plan-eng-review    # 工程经理：锁架构 + 测试策略
/plan-design-review # 设计师：评设计维度，抓 AI slop
/review             # Staff 工程师：挖生产级 bug
/qa                 # QA:真开浏览器点流程，修完再验
/cso                # 安全官:OWASP + STRIDE 审计
/ship               # 发布工程师：测试 + PR + 覆盖率

外加 /codex(拉 OpenAI Codex 做跨模型二次 review)、/careful /freeze /guard(危险命令 / 目录锁护栏) 这类 power tool。最适合：一个人当全栈团队，要多视角决策 + 质量 / 安全 / 发布把关。

组合与选型

三者不互斥，常叠着用：gstack 多角色起草 → superpowers 头脑风暴 + TDD 执行 → spec-kit 固化成 spec/plan/tasks。但别为叠而叠，先认领你当前最缺的那一环。

口诀：缺规格标准化用 spec-kit，缺执行纪律用 superpowers，缺多视角把关用 gstack。
