nerd

装 Nerd Fonts 时顺手查的词。最反常识的一点先记住：

nerd 不是任何东西的首字母缩写，是一个被「造」出来的词。

很多人以为它是 acronym（比如流传的 “Northern Electric Research and Development”），全是 folk etymology（民间附会）。真实出处是 1950 年 Dr. Seuss（苏斯博士）童书 If I Ran the Zoo 里他自造的一个生物名，次年被学生俚语借去当「书呆子」用，固定至今。

词义随圈子翻转

  • 原意带贬义：书呆子、不擅社交、土气。
  • 科技圈现在基本是 褒义甚至自豪的自称：某领域的狂热钻研者。a computer nerd / a math nerd。说 “I’m a nerd” 是骄傲，不是骂人。

nerdfonts.com 的命名就是这层自嘲幽默——「只有我们这些爱折腾终端的 nerd 才在意的字体」。

近义词别混

词含义色彩
nerd技术 / 学术钻研者中性 → 褒义
geek极客，痴迷某领域偏褒义
dork笨拙、傻气偏贬义
dweeb呆子贬义

口诀：nerd / geek 在科技圈是夸，dork / dweeb 是损人。

MSA / SOW / SLA / DPA / NDA 区别

B2B SaaS 谈企业客户时反复出现的五个合同缩写。最常见的错是把它们当并列选项挑一个签——实际是分层的：一份主协议套着其余几份，各管一层。

口诀：MSA 定规则，SOW 定范围，SLA 定质量，DPA 定数据，NDA 定保密。

缩写全称管什么签几次
MSAMaster Service Agreement长期合作的法律框架一次（关系建立时）
SOWStatement of Work单个项目的范围 / 工期 / 报价 / 交付物每个项目一份
SLAService Level Agreement服务质量承诺 + 不达标的赔偿一次（嵌进 MSA 或 SOW）
DPAData Processing Agreement数据怎么存 / 删 / 出境 / 合规一次（常作 MSA 附件）
NDANon-Disclosure Agreement保密义务一次（常在 MSA 之前）

分层关系

MSA  主协议（法律框架，长期不变）
 ├── DPA  数据处理协议（附件 Exhibit）
 ├── SLA  服务等级（可在 MSA，也可下放到每份 SOW）
 ├── SOW #1  上线核心模块
 └── SOW #2  做 SSO / 数据对接定制

MSA 是顶层框架，DPA / SLA 通常作为它的附件（Exhibit）；SOW 挂在 MSA 下面，一个项目一份。所以听到 “we have an MSA with them” 基本等于”我们是签了长期合同的正式企业客户关系”。

MSA 和 SOW 为什么拆开

拆开的价值是谈判频率不同：付款、责任上限、保密、终止这类通用条款变化慢，写进 MSA 谈一次管几年；范围、工期、报价这类项目细节每次都变，每个项目单独走一份 SOW，不重谈主合同。后续加功能只追加新 SOW——这就是框架协议省时间的地方。

冲突时谁优先？合同里一般写明 MSA 优先，除非某份 SOW 明确声明覆盖某条。判别归属也靠这条：看到”通用、反复用”的归 MSA，看到”这个项目、这个价”的归 SOW。

DPA：谁是 controller，谁是 processor

金融 / 医疗 / 涉及 PII 的 SaaS 几乎必签 DPA，核心是把双方角色定死：

  • Data Controller（控制方）——决定数据用来干什么的一方，通常是客户。
  • Data Processor（处理方）——替控制方处理数据的一方，通常是 SaaS 厂商。

这个区分不是文字游戏：GDPR / CCPA 下两种角色的法律责任不同。DPA 还会写明子处理商（sub-processor，如厂商用的云）清单、数据保留期、合同结束后的删除义务、以及发生 breach 后多少小时内必须通知。

NDA 已被 MSA 的保密条款覆盖，为什么还单独签

时间点不同。NDA 常在还没决定合作时就签——评估、技术对接、看 demo 阶段就要交换敏感信息，而这时 MSA 远没谈成。MSA 里的保密条款只在合作期内、且范围限于这次合作；NDA 是更早、更独立的一道闸。

另一个区别是方向：NDA 可以是 mutual（双向） 或 one-way（单向），谁向谁披露、谁负保密义务可以不对称；MSA 的保密条款默认是双向且绑死在主合作上。

English Vowel Reduction: Why Spelling ≠ Pronunciation

英语的元音弱化（vowel reduction）规律：重音所在的音节，元音发得清楚；非重读音节的元音，几乎都退化成 /ə/（schwa，中央元音）或 /ɪ/。跟字母拼写无关，只跟重音位置有关。

同词根，重音移动，读音跟着变

派生词加后缀时，重音常常跳到别的音节，原来被重读的元音就”塌陷”成 schwa。

基础词音标派生词音标弱化的元音
au·thor/ˈɔːθər/au·thor·i·ty/əˈθɒrɪti/au /ɔː/ → /ə/
pho·to/ˈfoʊtoʊ/pho·tog·ra·phy/fəˈtɒɡrəfi/o /oʊ/ → /ə/
a·tom/ˈeɪtəm/a·tom·ic/əˈtɒmɪk/a /eɪ/ → /ə/
e·qual/ˈiːkwəl/e·qual·i·ty/ɪˈkwɒlɪti/e /iː/ → /ɪ/
te·le·graph/ˈtelɪɡrɑːf/te·leg·ra·phy/tɪˈlɛɡrəfi/e /ɛ/ → /ɪ/
Di·vine/dɪˈvaɪn/di·vin·i·ty/dɪˈvɪnɪti/i /aɪ/ → /ɪ/

介词、冠词、助动词——弱读形式是默认形式

这类词在句子中几乎永远是非重读的，英语母语者日常会自动用弱读形式，重读版反而显得刻意或强调。

词强读（单独说 / 强调时）弱读（句中默认）
and/ænd//ənd/ 或 /ən/
to/tuː//tə/
for/fɔːr//fər/
of/ɒv//əv/ 或 /ə/
can/kæn//kən/
have/hæv//həv/ 或 /əv/
was/wɒz//wəz/
them/ðɛm//ðəm/

“I’ll go to the store and get some for you.” 实际说出来是：
/aɪl ɡoʊ tə ðə stɔːr ən ɡɛt səm fər juː/
其中 to / and / for 全部弱化，这才是自然语速。

-tion / -ment / -ness 后缀——固定弱化模式

这类后缀本身就是非重读音节，o / e 恒弱化为 schwa，不因词根变化。

词音标弱化点
na·tion/ˈneɪʃən/-tion 中 o → /ə/
sta·tion/ˈsteɪʃən/同上
gov·ern·ment/ˈɡʌvərnmənt/-ment 中 e → /ə/
hap·pi·ness/ˈhæpinəs/-ness 中 e → /ə/

为什么会弱化

英语是 stress-timed 语言——重读音节之间的时间间隔大致相等，为了维持节奏，非重读音节必须”压缩”。Schwa /ə/ 是嘴巴最放松时发出的音，发音成本最低，所以成了弱化的终点。

这也是为什么英语听力难：母语者不是”说清楚每个字母”，而是在保持节奏的前提下把非重读部分尽量压缩。

实践口诀

重音在哪，元音在哪；离开重音，退化成 schwa。

背新词时，先标重音位置，再确认非重读音节哪些会弱化——比死记每个字母的读法省力得多。

failure 与 /-jər/ 词尾

failure /ˈfeɪljər/ 的坑：拼写 -ure 容易被中文母语者读成 “your” 的尾音，实际是 /-jər/ —— 一个 /j/（y 滑音）紧跟 /ər/。

把它拆成 fail + y + er，重音在 fail。

字母 U / I 在词尾担任 /j/ 滑音

英语词尾 -ure / -ular / -liar 几乎都是这个套路：U 或 I 不发元音，担任 /j/ 滑音。

词尾发音例
-ure/jər/failure, figure /ˈfɪɡjər/
-ular/jələr/regular /ˈreɡjələr/, popular, particular
-liar/ljər/familiar /fəˈmɪljər/, peculiar /pɪˈkjuːljər/

念法套同一模式：fail-y-er、fig-y-er、reg-y-ler、fa-mil-y-er。

变体：-sure 词尾的 /ʒər/

measure  /ˈmeʒər/
pleasure /ˈpleʒər/
treasure /ˈtreʒər/
leisure  /ˈliːʒər/

U 仍然担任 /j/ 滑音，只是和前面的 s 融合（palatalization），合成一个浊擦音 /ʒ/，表面看不见 U 的痕迹了。

/ʒ/ 汉语里没有，最近似 热 / 日 / 人 的声母 /ʐ/ —— 把翘舌放平就是 /ʒ/。别和拼音的 zh 混：zh 是塞擦音 /ʈʂ/（先停后擦），跟 /ʒ/ 不是一类音。

dogfooding

软件工程黑话之一。直译”吃自己的狗粮”，指：

公司内部员工把自己开发的产品当日常工具用，靠真实使用暴露 QA / metrics 抓不到的体验问题。

更体面的同义说法是 drinking your own champagne。

1988 年微软内部一封题为 “Eating our own Dog Food” 的邮件（经理 Paul Maritz 发出）让这个说法在科技圈固定下来，沿用至今。

为什么 QA / 单元测试替代不了

QA 跟着 checklist 点一遍验功能存在；dogfooding 验长期真实使用下的体感 —— 慢、绕路、别扭、不直觉。这些既不会触发断言，也写不进测试用例。

行为算 dogfooding 吗
团队每天用自己写的内部 CRM✅
自己做金融 SaaS 但从不开账户跑投顾流程❌
QA 跟着用例点一遍❌（缺「真实工作场景 + 长期使用」）

失败模式：internal build ≠ public build

dogfooding 失效的经典姿势是：自己用的版本跟客户拿到的不是同一个。

Anthropic 在 2026-04-23 关于 Claude Code 质量退化的 postmortem 里坦白：员工用的是带内部 flag 的 test build，客户用的是 public build。其中一个缓存清理 bug 在 internal build 里被另一个并行实验掩盖了，员工 dogfooding 完全没暴露问题，客户端持续退化超过两周才定位。

改进措施第一条：让内部员工用 exact public build。

state vs status

两个词中文都译”状态”，工程命名混用会被 review 反复打回。先记一句口诀：

state is what you ARE; status is where you ARE in a process.

判准两步：

  1. 取值是有限几个枚举标签吗？→ status
  2. 是描述对象当下完整情况的一坨数据吗？→ state

status 通常是 state 里的一个字段，不是同级概念。这一层嵌套关系是判断命名时最重要的视角。

最干净的演示：Kubernetes Pod

spec:      # desired state（期望）
  containers: [...]

status:    # 实际情况
  phase: Running           # ← 流程节点标签
  conditions: [...]
  containerStatuses: [...]

spec + status 一起构成 Pod 的完整 state。status.phase 是 state 内部的一个枚举字段。命名 / 查 API 文档时，先判断你在描述哪一层。

反例：GitHub PR 用 state 不用 status

GET /repos/:owner/:repo/pulls/:number
→ {
    "state": "open" | "closed",
    "merged": true | false,
    ...
  }

open / closed 明明是枚举标签，按规则该叫 status，但 GitHub API 字段就叫 state，merged 另起一个 boolean。

为什么？GitHub 把 PR 当成”一个对象当前是什么”在表达，不是把它当流程节点。语义优先于规则，不要死记。

常见场景速查

场景用 status用 state
HTTP200 OK、response.status_codeTCP 连接：ESTABLISHED / TIME_WAIT
数据库字段orders.status = 'paid'—
前端组件antd <Steps status="finish" />React useState({ count, loading })
CI/CDGitHub Actions job status: completedGitHub PR state: open（反例）
即时通讯用户可设 status: 🍵 Tea breakWebSocket readyState: OPEN
游戏—存档（整个世界快照）

易混 trap：OAuth 的 state 参数

/authorize?response_type=code&state=abc123xyz

这里的 state 不是本文讨论的”对象状态”，是 OAuth 2.0 防 CSRF 的随机 nonce（RFC 6749 §4.1.1），授权服务器回调时原样带回，客户端校验是否匹配。

遇到这个词别按”完整数据快照”理解，它跟 state machine 一点关系没有。

一句话决策

  • 数据库字段、订单 / 任务 / 用户的流程节点 → status
  • React 组件内部数据、连接 / 进程的运行时快照 → state
  • 卡在中间 → 看现有 codebase 怎么命名，跟它走，比”正确”重要
