DN42 —— 改 git repo 申请 ASN 的 BGP 沙盒

DN42 是一张「长得跟真实互联网一模一样」的 overlay 网络——同一套 BGP / whois / DNS，但发号不用求 RIR：fork 一个 git 仓库、改几个文件、提 PR，脚本校验通过就合并，你就正式拥有了自己的 ASN 和地址段。核心价值是安全网：你在里面扮演一家 mini-ISP 练 BGP，配错了也不会有大运营商打电话来骂。

最反直觉的一点：它用 WireGuard 这类 VPN 隧道把成员互联，乍看像 Tor 那类 darknet，目标却相反——不做匿名，专做「去 NAT 的真实路由练习」。成员之间要的是能直连、能互相暴露服务，而不是藏起来。

注册是 git PR，不是找管理员

真实互联网上申请 ASN / IP 要走 APNIC 那套官僚流程；DN42 把它压成一次 PR——这是它整个设计的题眼。你要建的对象跟真实 whois 一一对应：

  • mntner（维护者，带签名 auth）、person（nic-hdl 以 -DN42 结尾）
  • aut-num（你的 ASN）、inetnum / inet6num（你的地址段）
  • route / route6——仿照真实互联网的 RPKI/ROA，没有它别人会因 ROA 校验失败而不收你的路由

坑在校验：一个自动脚本会在合并前检查语法，格式错一个字符就打回，拖慢管理员审核。所以是「严格按语法填」而不是「大概填填」。

号段为什么是私有的

资源DN42 用的段备注
ASN4242420000–4242429999（RFC 6996 私有段）新人建议在 4242420000–4242423999 里挑
IPv4172.20.0.0/14默认分 /27（32 个地址）够用
IPv6ULA fd00::/8默认分 /48

全用私有号段是刻意的：这样 DN42 的路由永远不会泄进公网 BGP 表、也不会跟公网 IP 撞车。代价是这些地址空间跟 ChaosVPN（172.31.0.0/16）、NeoNetwork 等兄弟网络重叠——DN42 只敢用 172.20.0.0/14 而非整个 172.16/12，就是给这些互联网络留位置，它们之间是打通的。

AS / ASN / BGP 选路的基础概念这里不重复，见同目录 CN2 那篇（那篇讲的是运营商的真实 ASN 就是线路名，这篇是你自己申请一个私有 ASN 来玩，正好两面）。

加入四步

  1. 注册：按上面建好对象，提 PR，合并即「持号」。
  2. 找 peer：没有中心服务器，必须连另一个成员。新手直接连 Kioubit——它连接数最多、中心性最高，且有全自动 peering 向导（填 ASN + WireGuard 公钥 + 隧道 IP，几分钟出结果），所以大半个网都直连它。也可用官方 Peer Finder 挑低延迟的。
  3. 建隧道：主流是 WireGuard（好配、够安全），也支持 OpenVPN / GRE / IPsec。
  4. 跑 BGP：装 BIRD 或 FRR，跟 peer 起 eBGP 会话，把自己的段 announce 出去。

门槛不高：一台 24/7 在线的 Linux VPS + 会配 WireGuard 和 BGP 就够。

dn42.us 不是官方入口

官方主站是 dn42.eu（wiki + registry 说明都在这）。dn42.us 是成员 xuu 自建的一个节点门户，挂着他自己的 wiki / paste / git 镜像等服务——它本身就是「在 DN42 里对外提供服务」的活样板，别把它当成注册入口。

CN2 GIA / 9929 / CMIN2：三网回国线路分层

回国线路好坏，本质是数据包进国内后走「精品网」还是「普通骨干网」——三网都能按 普通线 / 优质线 / 顶级线 三档理解（VPS、跨境专线圈的俗称）。看线的前提：回程比去程重要——下载占 90%+，回程决定体感。

AS 号就是线路名的真身

AS（Autonomous System，自治系统）= 由同一机构统一管理、对外采用统一路由策略的一组网络。互联网是「网络的网络」，每个大网络就是一个 AS。

  • ASN（AS 号）是网络层的身份证，全球唯一；BGP 选路靠它：IP 像门牌号定位主机，ASN 像区域号——BGP 先决定送到哪个 AS，再在 AS 内细分到具体地址。
  • ASN 原本 16 位（1–65535），号段不够用扩成了 32 位——移动的 AS58453 / AS58807 就是 32 位号，所以位数明显比电信 AS4809 长。

所谓 CN2 GIA、9929、CMIN2，本质就是 AS4809、AS9929、AS58807——运营商不同精品骨干网的「身份证号」。

精品网 vs 普通骨干：差在哪

普通骨干（电信 163、联通 169、移动 CMNET）是共享、超售的大网，国际带宽受限，晚高峰拥堵、抖动大。精品网是独立轻负载链路，绕开大量公共交换点（IX），一跳入境、一跳落地，所以延迟低、丢包少、高峰稳——代价是带宽小、价格贵。这就是 CN2 GIA 被当成回国「黄金标准」的根因。

优质线是半程精品：只在出国段走精品网，国内段仍回普通骨干（电信 CN2 GT 典型——国内段走 163，出境才汇入 CN2），比纯普通线好，但高峰国内段仍会堵；顶级线全程精品、不降级，才是天花板。

三网分层总表

运营商普通线（普通骨干）优质线（半程精品）顶级线（全程精品）
电信163（AS4134）CN2 GT（AS4809，国内仍走 163）CN2 GIA（AS4809）
联通169（AS4837）—（无 GT 式半程档）9929 / CUII（AS9929）
移动CMNET（AS9808）CMI（AS58453）CMIN2（AS58807，CMI 的 N2 精品网）

综合性能大致：CN2 GIA ≈ 9929 ≥ CMIN2 > CN2 GT / CTGNet > 163 / 169 / CMNET。

商家宣传只说「CN2」是泛称，要追问是 GT 还是 GIA——两者差价常有 2–3 倍。

选线一句话：电信用户选 CN2 GIA，联通用户选 9929，移动用户选 CMIN2；三网优化 VPS 就是按运营商把三条线分流，是目前回国线路的天花板。

怎么自测拿到的是哪条线

两个工具都跑在被测的境外 VPS 上：

NodeQuality——一键综合测评，除硬件 / IP 质量外，其 backroute_trace 会自动判断三网回程各走哪条骨干，最省事：

bash <(curl -sL https://run.nodequality.com)

NextTrace（nexttrace）——想手动细看回程路径用它，每跳直接标 AS 号 + 地理位置，比裸 traceroute 清楚。

httpbingo.org —— 按需返回任意 HTTP 响应

写 HTTP 客户端时最难造的是失败：500、慢响应、3xx 链、basic auth 401。靠真实业务 API 不可控也污染数据。httpbingo.org 就是干这个——按 URL 告诉它要什么响应，它就回什么。

curl https://httpbingo.org/status/503         # 强制 503
curl https://httpbingo.org/delay/5            # 5 秒后响应（测超时）
curl https://httpbingo.org/redirect/3         # 3 次 302 链（测 follow 上限）
curl -u user:pass https://httpbingo.org/basic-auth/user/pass
curl https://httpbingo.org/anything           # 把请求 headers / body / query 原样 echo 回来

/anything 是 debug 神器——客户端实际发了什么 header、是不是按预期带了 cookie、proxy 有没有偷改 body，一行 curl 看到。

域名容易和老前辈 httpbin.org 混。httpbingo.org 是 mccutchen/go-httpbin 跑的公网实例，Kenneth Reitz 那个老 httpbin 的 Go 重写。老版自 v0.6.1（2017-08）后没再发版，README 还挂着 travis-ci.org 的 badge（travis 早已对 OSS 停服）。go-httpbin 零依赖、stdlib only、持续发版（v2.22 / 2026），多了 /sse /jsonl /websocket/echo，少了 /brotli（Go stdlib 不带）。

CI 里别直接打公网

免费服务限流 / 下线一次就 red 一片。CI 里起个 sidecar：

docker run -p 8080:8080 ghcr.io/mccutchen/go-httpbin

写 Go 的话还能直接当 library 嵌进单测起 httptest.Server，连 sidecar 都省了。

查 CNAME：dig / host / nslookup 三选一

CNAME（canonical name）是 DNS 把一个域名指向另一个域名的别名记录。常见场景：www.example.com CNAME 到 CDN 边缘域名，或 SaaS 给你一个 <your>.heroku.com 让你 CNAME 过去。

dig example.com CNAME              # 完整记录
dig +short example.com CNAME       # 只要目标
dig +trace example.com             # 完整解析路径（root → TLD → 权威）

host -t CNAME example.com          # 输出最人话："is an alias for ..."

nslookup -type=CNAME example.com   # Windows 时代遗留，少用

CNAME 两条硬规则：

  • 不能和其他记录共存：一个 name 上有 CNAME 就不能再有 A / MX / TXT。所以根域名 example.com 通常不能用 CNAME（MX 在那里），子域才行
  • 链式解析：CNAME 可以指向另一个 CNAME，resolver 会一路 chase，最终必须落到 A/AAAA。链长了拖慢首次解析

挂在云厂商前的根域名查不出 CNAME 多半是因为对方用了 ANAME / ALIAS / flattening——根域 CNAME 限制的绕过手段，权威服务器替你预解析到 A 记录返回，外部看到的就是一条 A 而不是 CNAME。

TCP 三次握手 / 四次挥手

客户端 A、服务器 B，初始序号 seq，确认号 ack。

三次握手（建立连接）

初始：B 监听，A 打开。

  • A → B：seq = x（A 发连接请求，进入 SYN-SENT）
  • B → A：ack = x+1, seq = y（B 确认，进入 SYN-RCVD）
  • A → B：ack = y+1（A 再确认，进入 ESTABLISHED）

B 收到 A 的确认后也进入 ESTABLISHED。

为什么三次？ 防止失效的连接请求突然传到服务器，让服务器误以为要建连接。

四次挥手（释放连接）

  • A → B：seq = u（A 发释放报文，进入 FIN-WAIT-1）
  • B → A：ack = u+1, seq = v（B 确认，半关闭，B 还能发数据，进入 CLOSE-WAIT）
  • B → A：ack = u+1, seq = w（B 重发确认号，进入 LAST-ACK）
  • A → B：ack = w+1, seq = u+1（A 确认，进入 TIME-WAIT）

经过 2MSL 后 A 才进入 CLOSED。

为什么 TIME-WAIT 要等 2MSL？

  • 保证 A 发的最后一个 ACK 能到达 B（否则 B 重传 FIN）
  • 让本次连接的失效报文在网络中消失，不污染下一次连接

抓包验证

sudo tcpdump -i en0 host www.qq.com and tcp -S -c 50

tcpdump 速查

抓 HTTP 流量最常用的一行：

sudo tcpdump host httpforever.com and tcp port 80 -A -nn

完整文档 man tcpdump，下面只列高频项。

常用参数

参数作用
-i <iface>指定接口（不指定取第一个可用；-i any 监听所有）
-nn不解析主机名和端口名
-v / -vv / -vvv详细级别
-c <num>捕获指定数量后退出
-s <len>抓取长度（0 = 全包；新版默认 262144 bytes）
-w <file>写入 pcap 文件
-r <file>读取 pcap 文件离线解析
-A以 ASCII 打印 payload（截 HTTP：tcpdump -nSA port 80）
-X十六进制 + ASCII 打印（不要和 -A 一起用）
-S显示绝对序列号而不是相对编号
-tttt打印完整日期
-e显示以太网头（MAC 地址）

BPF 过滤语法

[proto] [src|dst] [host|net|port] <对象> [逻辑运算]

tcp and dst port 80 and src net 10.0.0.0/8
  • 协议：ip tcp udp icmp arp vlan ip6 …
  • 方向：src dst src or dst src and dst
  • 对象：host 10.0.0.1 / net 192.168.0.0/16 / port 443
  • 逻辑：and (&&) / or (||) / not (!) / ( )
sudo tcpdump -i en0 host 192.168.50.1          # 源或目的
sudo tcpdump -i en0 src host 192.168.50.1      # 源
sudo tcpdump -i en0 dst host 192.168.50.1      # 目的
sudo tcpdump -i en0 port 8080                  # 端口
sudo tcpdump -i en0 net 192.168                # 网段
sudo tcpdump -i en0 tcp                        # 协议

输出格式

14:24:12.196582 IP 192.168.91.29.56930 > 146.190.62.39.http: Flags [SEW], seq 193726297, win 65535, options [...], length 0

时间戳 / 协议层 / 源.端口 > 目标.端口 / TCP 标志位 / seq ack win 等字段 / 报文长度。

TCP Flags

TCP Flag缩写含义
SYNS会话建立请求
ACKA确认发送方的数据
FINF终止标识
RESETR立即中止连接
PUSHP立即推送数据
URGENTU优先于其他数据
NONE.占位符，通常用于 ACK

参考 tcpdump Flags。

接口列表

sudo tcpdump -D
# 1.en0 [Up, Running]
# 2.p2p0 [Up, Running]
# ...

macOS 下监听适配器必须 root。

实例

抓 eth1 上目的地址是 192.168.1.254 或 192.168.1.200、端口 80 的 TCP：

sudo tcpdump -i eth1 '((tcp) and (port 80) and ((dst host 192.168.1.254) or (dst host 192.168.1.200)))'

只抓 SYN：

sudo tcpdump -i eth1 'tcp[tcpflags] = tcp-syn'

抓 SYN + ACK：

sudo tcpdump -i eth1 'tcp[tcpflags] & tcp-syn != 0 and tcp[tcpflags] & tcp-ack != 0'

抓 DNS 请求：

sudo tcpdump -i en0 udp dst port 53

流量大时用 -c 限制数量：

sudo time tcpdump -nn -i en0 'tcp[tcpflags] = tcp-syn' -c 10000 > /dev/null

实时抓 8000 端口的 GET 包写入文件：

sudo tcpdump -i eth0 '((port 8000) and (tcp[(tcp[12]>>2):4]=0x47455420))' -nnAl -w /tmp/GET.log

CORS：带凭证跨域时 Allow-Origin 不能用 *

带 cookie 的跨域请求（fetch(..., { credentials: 'include' })、xhr.withCredentials = true）触发浏览器的「凭证模式」，硬性四条死规矩：

  1. Access-Control-Allow-Origin 必须是具体 origin，禁止 *
  2. 必须返回 Access-Control-Allow-Credentials: true
  3. 必须加 Vary: Origin 防缓存错乱
  4. 预检 OPTIONS 也要重复以上三条

Firefox 报：

Credential is not supported if the CORS header ‘Access-Control-Allow-Origin’ is ’*’

Chrome 报：

The value of the ‘Access-Control-Allow-Origin’ header in the response must not be the wildcard ’*’ when the request’s credentials mode is ‘include’

为什么禁用 * + 凭证

如果允许 * 同时带 cookie：攻击者站点 https://hack.com 直接请求 https://api.example.com，自动带上你已登录的 cookie，服务端无法区分调用方，敏感数据全开。规范因此硬性禁止——这是浏览器侧的强制 CSRF 防线。

Access-Control-Allow-Credentials: true 是服务端的显式授权开关，控制两件事：跨域请求能不能带凭证发出、跨域响应能不能把 Set-Cookie 写入当前域。前端 credentials:include + 服务端 Allow-Credentials: true 是双向确认，少一边浏览器都不携带凭证、也不写入 cookie。

服务端正确返回

读取请求头 Origin，比对白名单，动态回写当前 origin（不是写死一个）：

Access-Control-Allow-Origin: https://app.example.com
Access-Control-Allow-Credentials: true
Vary: Origin

Nginx 关键片段：

if ($http_origin ~* ^https://(app|admin)\.example\.com$) {
    add_header Access-Control-Allow-Origin $http_origin always;
    add_header Access-Control-Allow-Credentials true always;
    add_header Vary Origin always;
}

Vary: Origin 不可省

CDN / 反代按 URL 缓存响应。第一次 https://a.com 请求命中后，缓存里那条响应携带 Access-Control-Allow-Origin: https://a.com。下一次 https://b.com 请求同一 URL，缓存直接返回——浏览器拿到不匹配的 Allow-Origin，跨域失败。

Vary: Origin 告诉缓存层：把 Origin 也算进 cache key，不同 origin 各存各的。预检请求建议扩展为：

Vary: Origin, Access-Control-Request-Method, Access-Control-Request-Headers

常见踩坑

  • 配了具体 Origin，忘了 Allow-Credentials: true → 能过 CORS，但 cookie 发不上、Set-Cookie 也存不下
  • 多域名白名单写死了一个 origin → 其余域名全跨域失败，必须动态回传当前请求的 Origin
  • 同一响应出现多个 Access-Control-Allow-Origin（框架 + Nginx 重复添加）→ 浏览器报错
  • 本地开发 * 一切正常，上线开 credentials 立刻崩
  • 预检 OPTIONS 偷懒返回 * → 主请求过不了

参考：MDN: CORS — Requests with credentials
