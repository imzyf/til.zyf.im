2022-07-28 #network

查 CNAME：dig / host / nslookup 三选一

CNAME（canonical name）是 DNS 把一个域名指向另一个域名的别名记录。常见场景：www.example.com CNAME 到 CDN 边缘域名，或 SaaS 给你一个 <your>.heroku.com 让你 CNAME 过去。

dig example.com CNAME              # 完整记录
dig +short example.com CNAME       # 只要目标
dig +trace example.com             # 完整解析路径（root → TLD → 权威）

host -t CNAME example.com          # 输出最人话："is an alias for ..."

nslookup -type=CNAME example.com   # Windows 时代遗留，少用

CNAME 两条硬规则：

  • 不能和其他记录共存：一个 name 上有 CNAME 就不能再有 A / MX / TXT。所以根域名 example.com 通常不能用 CNAME（MX 在那里），子域才行
  • 链式解析：CNAME 可以指向另一个 CNAME，resolver 会一路 chase，最终必须落到 A/AAAA。链长了拖慢首次解析

挂在云厂商前的根域名查不出 CNAME 多半是因为对方用了 ANAME / ALIAS / flattening——根域 CNAME 限制的绕过手段，权威服务器替你预解析到 A 记录返回，外部看到的就是一条 A 而不是 CNAME。

TCP 三次握手 / 四次挥手

客户端 A、服务器 B，初始序号 seq，确认号 ack。

三次握手（建立连接）

初始：B 监听，A 打开。

  • A → B：seq = x（A 发连接请求，进入 SYN-SENT）
  • B → A：ack = x+1, seq = y（B 确认，进入 SYN-RCVD）
  • A → B：ack = y+1（A 再确认，进入 ESTABLISHED）

B 收到 A 的确认后也进入 ESTABLISHED。

为什么三次？ 防止失效的连接请求突然传到服务器，让服务器误以为要建连接。

四次挥手（释放连接）

  • A → B：seq = u（A 发释放报文，进入 FIN-WAIT-1）
  • B → A：ack = u+1, seq = v（B 确认，半关闭，B 还能发数据，进入 CLOSE-WAIT）
  • B → A：ack = u+1, seq = w（B 重发确认号，进入 LAST-ACK）
  • A → B：ack = w+1, seq = u+1（A 确认，进入 TIME-WAIT）

经过 2MSL 后 A 才进入 CLOSED。

为什么 TIME-WAIT 要等 2MSL？

  • 保证 A 发的最后一个 ACK 能到达 B（否则 B 重传 FIN）
  • 让本次连接的失效报文在网络中消失，不污染下一次连接

抓包验证

sudo tcpdump -i en0 host www.qq.com and tcp -S -c 50

2020-06-29 #network

tcpdump 速查

抓 HTTP 流量最常用的一行：

sudo tcpdump host httpforever.com and tcp port 80 -A -nn

完整文档 man tcpdump，下面只列高频项。

常用参数

参数作用
-i <iface>指定接口（不指定取第一个可用；-i any 监听所有）
-nn不解析主机名和端口名
-v / -vv / -vvv详细级别
-c <num>捕获指定数量后退出
-s <len>抓取长度（0 = 全包；新版默认 262144 bytes）
-w <file>写入 pcap 文件
-r <file>读取 pcap 文件离线解析
-A以 ASCII 打印 payload（截 HTTP：tcpdump -nSA port 80）
-X十六进制 + ASCII 打印（不要和 -A 一起用）
-S显示绝对序列号而不是相对编号
-tttt打印完整日期
-e显示以太网头（MAC 地址）

BPF 过滤语法

[proto] [src|dst] [host|net|port] <对象> [逻辑运算]

tcp and dst port 80 and src net 10.0.0.0/8
  • 协议：ip tcp udp icmp arp vlan ip6 …
  • 方向：src dst src or dst src and dst
  • 对象：host 10.0.0.1 / net 192.168.0.0/16 / port 443
  • 逻辑：and (&&) / or (||) / not (!) / ( )
sudo tcpdump -i en0 host 192.168.50.1          # 源或目的
sudo tcpdump -i en0 src host 192.168.50.1      # 源
sudo tcpdump -i en0 dst host 192.168.50.1      # 目的
sudo tcpdump -i en0 port 8080                  # 端口
sudo tcpdump -i en0 net 192.168                # 网段
sudo tcpdump -i en0 tcp                        # 协议

输出格式

14:24:12.196582 IP 192.168.91.29.56930 > 146.190.62.39.http: Flags [SEW], seq 193726297, win 65535, options [...], length 0

时间戳 / 协议层 / 源.端口 > 目标.端口 / TCP 标志位 / seq ack win 等字段 / 报文长度。

TCP Flags

TCP Flag缩写含义
SYNS会话建立请求
ACKA确认发送方的数据
FINF终止标识
RESETR立即中止连接
PUSHP立即推送数据
URGENTU优先于其他数据
NONE.占位符，通常用于 ACK

参考 tcpdump Flags。

接口列表

sudo tcpdump -D
# 1.en0 [Up, Running]
# 2.p2p0 [Up, Running]
# ...

macOS 下监听适配器必须 root。

实例

抓 eth1 上目的地址是 192.168.1.254 或 192.168.1.200、端口 80 的 TCP：

sudo tcpdump -i eth1 '((tcp) and (port 80) and ((dst host 192.168.1.254) or (dst host 192.168.1.200)))'

只抓 SYN：

sudo tcpdump -i eth1 'tcp[tcpflags] = tcp-syn'

抓 SYN + ACK：

sudo tcpdump -i eth1 'tcp[tcpflags] & tcp-syn != 0 and tcp[tcpflags] & tcp-ack != 0'

抓 DNS 请求：

sudo tcpdump -i en0 udp dst port 53

流量大时用 -c 限制数量：

sudo time tcpdump -nn -i en0 'tcp[tcpflags] = tcp-syn' -c 10000 > /dev/null

实时抓 8000 端口的 GET 包写入文件：

sudo tcpdump -i eth0 '((port 8000) and (tcp[(tcp[12]>>2):4]=0x47455420))' -nnAl -w /tmp/GET.log

CORS：带凭证跨域时 Allow-Origin 不能用 *

带 cookie 的跨域请求（fetch(..., { credentials: 'include' })、xhr.withCredentials = true）触发浏览器的「凭证模式」，硬性四条死规矩：

  1. Access-Control-Allow-Origin 必须是具体 origin，禁止 *
  2. 必须返回 Access-Control-Allow-Credentials: true
  3. 必须加 Vary: Origin 防缓存错乱
  4. 预检 OPTIONS 也要重复以上三条

Firefox 报：

Credential is not supported if the CORS header ‘Access-Control-Allow-Origin’ is ’*’

Chrome 报：

The value of the ‘Access-Control-Allow-Origin’ header in the response must not be the wildcard ’*’ when the request’s credentials mode is ‘include’

为什么禁用 * + 凭证

如果允许 * 同时带 cookie：攻击者站点 https://hack.com 直接请求 https://api.example.com，自动带上你已登录的 cookie，服务端无法区分调用方，敏感数据全开。规范因此硬性禁止——这是浏览器侧的强制 CSRF 防线。

Access-Control-Allow-Credentials: true 是服务端的显式授权开关，控制两件事：跨域请求能不能带凭证发出、跨域响应能不能把 Set-Cookie 写入当前域。前端 credentials:include + 服务端 Allow-Credentials: true 是双向确认，少一边浏览器都不携带凭证、也不写入 cookie。

服务端正确返回

读取请求头 Origin，比对白名单，动态回写当前 origin（不是写死一个）：

Access-Control-Allow-Origin: https://app.example.com
Access-Control-Allow-Credentials: true
Vary: Origin

Nginx 关键片段：

if ($http_origin ~* ^https://(app|admin)\.example\.com$) {
    add_header Access-Control-Allow-Origin $http_origin always;
    add_header Access-Control-Allow-Credentials true always;
    add_header Vary Origin always;
}

Vary: Origin 不可省

CDN / 反代按 URL 缓存响应。第一次 https://a.com 请求命中后，缓存里那条响应携带 Access-Control-Allow-Origin: https://a.com。下一次 https://b.com 请求同一 URL，缓存直接返回——浏览器拿到不匹配的 Allow-Origin，跨域失败。

Vary: Origin 告诉缓存层：把 Origin 也算进 cache key，不同 origin 各存各的。预检请求建议扩展为：

Vary: Origin, Access-Control-Request-Method, Access-Control-Request-Headers

常见踩坑

  • 配了具体 Origin，忘了 Allow-Credentials: true → 能过 CORS，但 cookie 发不上、Set-Cookie 也存不下
  • 多域名白名单写死了一个 origin → 其余域名全跨域失败，必须动态回传当前请求的 Origin
  • 同一响应出现多个 Access-Control-Allow-Origin（框架 + Nginx 重复添加）→ 浏览器报错
  • 本地开发 * 一切正常，上线开 credentials 立刻崩
  • 预检 OPTIONS 偷懒返回 * → 主请求过不了

参考：MDN: CORS — Requests with credentials
